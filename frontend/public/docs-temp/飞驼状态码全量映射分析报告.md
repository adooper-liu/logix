## 五、关键决策点

### 5.1 海关状态码处理方案

**方案A：独立海关状态表**（推荐）
```typescript
// 新建 customs_statuses 表
{
  customs_status_id: PK
  container_number: FK
  customs_code: string  // BLA, BLR, ASA, etc.
  customs_type: enum    // 'export_manifest', 'import_manifest', 'export_clearance', 'import_clearance'
  status: enum          // 'pending', 'released', 'hold', 'exam'
  occurred_at: timestamp
  data_source: string   // 'Feituo'
}
```
**优点**: 
- 不影响现有核心字段
- 支持复杂海关流程
- 便于合规审计

**缺点**: 
- 需要额外表结构
- 需要联表查询

**方案B：复用现有字段**
```typescript
// 继续使用 customs_status 字段
{
  customs_status: 'BLA_RECEIVED'  // 格式: {code}_{status}
  customs_status_detail: '预配舱单已申报'
}
```
**优点**:
- 无需改表结构
- 实现简单

**缺点**:
- 字段含义模糊
- 不利于分析

**建议**: 采用方案A，新建 `process_customs_statuses` 表

### 5.2 预警节点处理方案

**架构设计**:
```typescript
// 预警事件表 alert_events
{
  alert_id: PK
  container_number: FK
  alert_code: string    // WGITM, WDLPT, etc.
  alert_category: enum  // 'DELAY', 'DUMPING', 'DETENTION', 'OVERDUE', 'CHANGE'
  alert_level: enum     // 'warning', 'critical', 'info'
  rule_triggered: string // 规则描述
  detected_at: timestamp
  resolved_at: timestamp
  is_active: boolean
}
```

**处理逻辑**:
1. 飞驼推送预警事件时，写入 alert_events 表
2. 前端查询时，联查 alert_events 获取实时预警
3. 支持预警确认和关闭流程

## 六、风险评估

### 6.1 不补充缺失状态码的风险

| 风险项 | 影响程度 | 发生概率 | 风险等级 | 说明 |
|--------|----------|----------|----------|------|
| 物流状态计算不准确 | 高 | 中 | **高** | 缺失STUF/DUMP等节点导致状态机判断错误 |
| 海关状态无法跟踪 | 高 | 高 | **高** | 无法获取海关放行/查验状态 |
| 预警功能失效 | 中 | 高 | **中** | 预警节点缺失，无法提前发现问题 |
| 客户投诉增加 | 中 | 中 | **中** | 数据不完整导致客户体验下降 |
| 合规风险 | 高 | 低 | **中** | 海关审计时数据不完整 |

### 6.2 实施风险与缓解措施

**风险1**: 状态码冲突
- **现象**: 不同事件类型（transport/equipment/shipment）有相同状态码
- **缓解**: 按事件类型+状态码组合处理，如 `equipment_LOAD` vs `shipment_LOAD`

**风险2**: 数据重复
- **现象**: 飞驼推送与现有数据重复
- **缓解**: 使用 `(container_number, status_code, event_time)` 唯一索引

**风险3**: 优先级判断错误
- **现象**: 多个状态码对应同一核心字段
- **缓解**: 按业务规则定义优先级（如实际时间 > 预计时间）
