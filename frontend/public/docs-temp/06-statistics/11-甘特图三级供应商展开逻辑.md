> 本文档总结甘特图五阶段流转、各阶段完成判断、供应商分组逻辑及实现要点。供开发与维护参考。

## 〇、甘特图第三级数据源综合解读

### 数据结构关系

```
货柜 (biz_containers)
  ├── portOperations[]     ← process_port_operations（目的港 portType=destination）
  ├── truckingTransports[] ← process_trucking_transport
  ├── warehouseOperations[]← process_warehouse_operations  ⚠️ 列表 enrich 当前未返回
  └── emptyReturns[]       ← process_empty_return（无 warehouse_id，需从卸柜推断）
```

### 第三级数据源选择结论

| 候选数据源 | 适用性 | 说明 |
|:---:|:---:|:---|
| **process 表（流程表）** | ✅ **推荐** | 直接反映货柜实际执行时使用的供应商，与完成判断字段同源 |
| dict 表（字典表） | ❌ 不适用 | 仅用于代码→名称解析，不记录货柜执行数据 |
| dict 映射表 | ❌ 不适用 | 描述车队-港口、仓库-车队配对关系，不记录单柜执行 |

**结论**：甘特图第三级（供应商分组）应**从 process 表取**，即货柜 enrich 后的 `portOperations`、`truckingTransports`、`warehouseOperations`。分组键与完成判断字段同源，语义一致，无需额外 JOIN。

### 各节点数据来源明细

| 节点 | 分组字段 | 数据来源（process 表） | 备选字段 |
|:---:|:---:|:---:|:---:|
| 清关 | `customsBrokerCode` | `portOperations[destination].customsBrokerCode` | `customsBroker` |
| 提柜 | `truckingCompanyId` | `truckingTransports[0].truckingCompanyId` | `carrierCompany` |
| 卸柜 | `warehouseId` | `warehouseOperations[0].warehouseId` | `actualWarehouse`、`plannedWarehouse` |
| 还箱 | `warehouseId` | 与卸柜相同（从 `warehouseOperations` 推断） | `process_empty_return` 无 warehouse 字段 |

### 注意事项

1. **warehouseOperations 需在 enrich 中返回**：当前 `enrichContainersList` 未返回 `warehouseOperations`，卸柜/还箱的第三级分组会退化为「未指定仓库」。需在 `container.service.ts` 的 enrich 中补充。
2. **还箱分组**：`process_empty_return` 无 warehouse 字段，还箱与卸柜共用 `warehouseOperations[0].warehouseId` 分组。
3. **名称展示**：若需显示供应商名称而非代码，可额外 JOIN `dict_customs_brokers`、`dict_trucking_companies`、`dict_warehouses` 解析，分组键仍用 process 表代码。

## 一、五阶段流转逻辑

### 1.1 阶段流转示意

```
阶段1: 清关阶段
├── 主任务：清关（实心圆）
├── 虚线任务：提柜 → 卸柜 → 还箱

阶段2: 清关完成 (actualCustomsDate 有值)
├── 清关主任务销毁
├── 提柜变主任务（实心圆）
├── 虚线任务：卸柜 → 还箱

阶段3: 提柜完成 (deliveryDate 有值)
├── 提柜主任务销毁
├── 卸柜变主任务（实心圆）
├── 虚线任务：还箱

阶段4: 卸柜完成 (unloadDate 有值)
├── 卸柜主任务销毁
├── 还箱变主任务（实心圆）

阶段5: 还箱完成 (returnTime 有值)
└── 还箱主任务销毁（流程结束）
```

### 1.2 各阶段完成判断字段

| 阶段 | 完成判断字段 | 数据来源表 | 实体属性 |
|:---:|:---:|:---:|:---:|
| 清关 | `actualCustomsDate` | `process_port_operations` | `PortOperation.actualCustomsDate` |
| 提柜 | `deliveryDate`（送仓日） | `process_trucking_transport` | `TruckingTransport.deliveryDate` |
| 卸柜 | `unloadDate` | `process_warehouse_operations` | `WarehouseOperation.unloadDate` |
| 还箱 | `returnTime` | `process_empty_return` | `EmptyReturn.returnTime` |

## 三、车队-港口-仓库绑定关系（业务约束）

### 3.1 核心业务规则

> **目的港的货柜只能送往对应的仓库，分派任务给对应的车队。**

### 3.2 实际实现：通过车队间接关联

**港口 → 仓库** 没有独立的直接业务表，而是通过 **港口 → 车队** 与 **仓库 → 车队** 间接实现：

```
目的港 (port_code)  ──dict_trucking_port_mapping──→  车队 (trucking_company_id)
                                                              ↑
仓库 (warehouse_code) ──dict_warehouse_trucking_mapping──→  车队 (trucking_company_id)
```

**推导逻辑**：若某车队同时服务目的港 P 和仓库 W，则目的港 P 的货柜可送往仓库 W。即 **港口→仓库 = 通过「同时服务该港口和该仓库的车队」间接推导**。

### 3.3 排产/分派时的约束

| 步骤 | 约束来源 | 校验逻辑 |
|:---:|:---|:---|
| 选车队（提柜） | `dict_trucking_port_mapping` | 目的港 X 的货柜 → 只能在 port_code=X 的映射中选 trucking_company_id |
| 选仓库 | `dict_warehouse_trucking_mapping` | 已选车队 T 的货柜 → 只能在 trucking_company_id=T 的映射中选 warehouse_code |
| 选车队（送仓） | `dict_warehouse_trucking_mapping` | 已选仓库 W 的货柜 → 只能在 warehouse_code=W 的映射中选 trucking_company_id |

### 3.4 相关映射表

| 表名 | 用途 | 关键字段 |
|:---|:---|:---|
| `dict_trucking_port_mapping` | 车队-港口映射 | `truckingCompanyId`、`portCode` |
| `dict_warehouse_trucking_mapping` | 仓库-车队映射 | `warehouseCode`、`truckingCompanyId` |

> 注：业务上 港口→仓库 的关系由上述两表间接推导，字典界面不提供港口-仓库映射维护。

### 3.5 映射表使用场景

- **供应商名称展示**：分组键为代码时，可通过映射表或字典表解析为显示名称
- **路线约束**：智能排产时，按 港口→车队、仓库→车队 校验；港口→仓库 由车队交集推导

### 3.6 智能排柜引擎与流程表的关系（架构澄清）

> **流程表中的车队、仓库是结果，中间需由智能排柜引擎完成分派。**

```
输入：货柜 + ETA + last_free_date + 备货单（港口/目标仓） + 约束
                    ↓
         ┌─────────────────────────────────────────┐
         │  智能排柜引擎（中间环节）                  │
         │  - 映射关系：港口→车队、仓库→车队         │
         │  - 车队能力：日容量、港口-仓库路线        │
         │  - 仓库能力：日产能、卸柜方式             │
         │  - ETA / last_free_date / last_return_date │
         │  - 三种方案决策（直接送仓/堆场/滞港）     │
         └─────────────────────────────────────────┘
                    ↓
输出：分派的车队、仓库、计划时间（planned_*）
                    ↓
         process_trucking_transport（车队、提柜/送仓日）
         process_warehouse_operations（仓库、卸柜日）
         process_empty_return（还箱日）
```

| 层级 | 职责 | 数据 |
|:---|:---|:---|
| **智能排柜引擎** | 根据映射、能力、ETA 等多因素**分派指定**车队与仓库 | 输入：货柜、ETA、约束；输出：truckingCompanyId、warehouseId、planned_* |
| **流程表** | 存储**分派结果**，供甘特图、统计、执行跟踪使用 | `truckingCompanyId`、`warehouseId`、`planned_*`、`actual_*` |

详见：[05-智能排柜与计划调度实现方案](../../11-project/05-智能排柜与计划调度实现方案.md)、[06-智能排柜算法方案](../../11-project/06-智能排柜算法方案.md)。

## 六、开发要点总结

### 6.1 前端

1. **阶段判断**：根据 `actualCustomsDate`、`deliveryDate`、`unloadDate`、`returnTime` 依次判断当前主任务节点
2. **分组键**：按当前阶段使用对应分组字段（`customsBrokerCode` / `truckingCompanyId` / `warehouseId`）
3. **主任务 vs 虚线**：主任务为当前活跃节点（实心圆）；虚线为下游未完成节点
4. **还箱分组**：与卸柜相同，使用 `warehouseId`
5. **名称解析**：使用 `supplierNames` 字段显示供应商名称（字典表解析）

### 6.2 后端

1. **enrich 数据**：`getContainers` 返回的货柜需包含 `portOperations`、`truckingTransports`、`warehouseOperations`、`emptyReturns`，且含 `customsBrokerCode`、`truckingCompanyId`、`warehouseId` 等字段
2. **字段命名**：与数据库 snake_case 一致，实体为 camelCase
3. **supplierNames**：从字典表批量查询并返回供应商名称

### 6.3 扩展建议

- 若需按供应商名称展示，可关联 `dict_customs_brokers`、`dict_trucking_companies`、`dict_warehouses` 解析代码→名称
- 若需按国别/港口再分组，可在现有分组键前追加 `country` 或 `portCode`
