## 1. 评估现有代码

### 1.1 评估结果摘要

| 维度 | 后端 | 前端 |
|------|------|------|
| 日期库 | 原生 Date + 自研 dateTimeUtils | dayjs（组件内）+ 自研 dateTimeUtils |
| 时区处理 | 无统一 UTC | 无统一，部分用 dayjs utc/timezone |
| 格式 | 多种（YYYY-MM-DD、ISO 等） | 多种 |
| 涉及文件数 | ~60+ | ~50+ |

### 1.2 数据库设计

- `TIMESTAMP`：带时间的字段（如 `created_at`、`ata_dest_port`、`gate_in_time`）
- `DATE`：仅日期字段（如 `actual_ship_date`、`last_free_date`）
- PostgreSQL 默认按服务器时区存储，建议统一为 UTC

### 1.3 现有 dateTimeUtils

- **后端/前端**：`parseLocalDate`、`formatDateToLocal`、`parseExcelDate`、`addDays`、`daysBetween`、`getDayStart`、`getDayEnd` 等
- **策略**：以「本地时间」解析，避免时区偏移
- **规格机**：在现有工具基础上新增 `DateTimeUtils` 类，提供 UTC/时区能力

## 4. 使用示例

### 后端

```typescript
import { DateTimeUtils } from '../utils/dateTimeUtils';

// 存储前转为 UTC
const utcDate = DateTimeUtils.toUTC(new Date());

// 按指定时区展示
const localDate = DateTimeUtils.toTimezone(utcDate, 'Asia/Shanghai');

// API 响应格式
const isoStr = DateTimeUtils.toISOString(utcDate);
```

### 前端

```typescript
// 选项 1：inject
const dateTime = inject('dateTime');

// 选项 2：$dateTime（模板内）
// {{ $dateTime.format(row.ataDestPort) }}

// 选项 3：直接导入
import { DateTimeUtils } from '@/utils/dateTimeUtils';
DateTimeUtils.format(apiDate, 'YYYY-MM-DD');
```

## 5. 迁移建议

### 5.1 分阶段

1. **阶段 1**：新功能统一使用 `DateTimeUtils`（当前已完成）
2. **阶段 2**：核心 API 与状态机逐步替换为 `DateTimeUtils`
3. **阶段 3**：Excel 导入、飞驼同步等批量替换

### 5.2 注意事项

- 保留现有 `parseLocalDate`、`formatDateToLocal` 等，保证兼容
- 数据库迁移需评估：现有 `TIMESTAMP` 是否已按 UTC 存储
- 中间件 `dateTimeResponseMiddleware` 会包装 `res.json`，若遇异常可暂时关闭
