# LogiX 业务知识：状态机、滞港费与飞驼 API

本文档解读项目中**物流状态机**、**滞港费（最晚提柜/免费期）**、**飞驼 API 集成**三块业务逻辑，便于开发与排查。

---

## 一、物流状态机

### 1.1 两套状态体系

| 体系 | 用途 | 位置 |
|------|------|------|
| **简化状态（7 层）** | 桑基图、前端展示、统计、DB 字段 `logistics_status` | `backend/src/utils/logisticsStatusMachine.ts`、`frontend/src/utils/logisticsStatusMachine.ts` |
| **详细状态（33 种）** | 外部 API / 物流路径微服务、复杂流转校验 | 同上 + `logistics-path-system` 微服务 |

### 1.2 简化状态枚举（7 层流转）

```
not_shipped → shipped → in_transit → at_port → picked_up → unloaded → returned_empty
  未出运      已装船      在途        已到港     已提柜      已卸柜      已还箱
```

- 状态存于 `biz_containers.logistics_status`。
- 支持跳转：例如在途 → 已提柜（有直接提柜数据时）。

### 1.3 状态如何计算（优先级从高到低）

**核心函数**：`calculateLogisticsStatus()`（`backend/src/utils/logisticsStatusMachine.ts`）

| 优先级 | 条件 | 得到状态 |
|--------|------|----------|
| 1 | `emptyReturn.returnTime` 有值 | `returned_empty` |
| 2 | 仓库 WMS 已确认（`wmsStatus === 'WMS已完成'` 或 `ebsStatus === '已入库'` 或 `wmsConfirmDate` 有值） | `unloaded` |
| 3 | `truckingTransport.pickupDate` 有值 | `picked_up` |
| 4 | 目的港港口操作有 `ataDestPort` | `at_port`（currentPortType = destination） |
| 5 | 中转港港口操作有 `ataDestPort` 或 `gateInTime` | `at_port`（currentPortType = transit） |
| 6 | `seaFreight.shipmentDate` 有值 | `in_transit` |
| 7 | 以上都不满足 | `not_shipped` |

依赖数据来源：`Container`、`PortOperation[]`、`SeaFreight`、`TruckingTransport`、`WarehouseOperation`、`EmptyReturn`。

### 1.4 谁在更新 container 的 logistics_status

- **ContainerStatusService**（`backend/src/services/containerStatus.service.ts`）：
  - `updateStatus(containerNumber)`：单柜根据上述数据调用 `calculateLogisticsStatus`，若结果与当前 `logistics_status` 不同则写回 DB。
  - `batchUpdateStatuses(limit)`：批量对货柜做同样逻辑。
- 入口：接口 `POST /api/v1/containers/:containerNumber/update-status`、`POST /api/v1/containers/update-statuses/batch`。

### 1.5 详细状态 → 简化状态

- `DetailedToSimplifiedMap`：33 种详细状态映射到 7 种简化状态。
- 异常状态（扣货、滞港、延误等）多数映射为 `at_port` 或 `in_transit`；`DETENTION`（滞港费）→ `at_port`。

### 1.6 Excel / 外部 API 状态码 → 简化状态

- **Excel 导入**：`ExcelStatusToSimplifiedMap`（中文如「已到港」「已提柜」→ 简化状态）。
- **外部 API**：`ExternalApiToDetailedMap`（如飞驼 `ARRIVE`、`GATE_OUT`）→ 先到详细状态，再经 `DetailedToSimplifiedMap` 到简化状态。

## 二、滞港费与最晚提柜（免费期）

### 2.1 业务含义

- **last_free_date（最晚免费日）**：目的港「免费堆存/免费用箱」的最后一天；**超过该日后会产生滞港费/滞箱费**。
- 字段位置：**目的港**港口操作表 `process_port_operations.last_free_date`（实体 `PortOperation.lastFreeDate`）。
- 统计与倒计时都围绕「是否在 last_free_date 之前提柜」来区分：已超期、即将超期、预警、时间充裕、无 last_free_date。

### 2.2 数据来源与使用

- **来源**：Excel 导入或后续对接的港口/船公司数据，写入 `process_port_operations` 的**目的港**记录（`port_type = 'destination'`）。
- **取哪一条**：按目的港**最新一条**港口操作（`port_sequence` 最大）的 `last_free_date`；子查询见 `PortOperationQueryBuilder.getLatestDestinationPortWithLastFreeDateAlias()`。

### 2.3 统计分类（最晚提柜维度）

**LastPickupStatisticsService**（`backend/src/services/statistics/LastPickupStatistics.service.ts`）按「出运时间 + 已到目的港且未提柜」的目标集，再按 `last_free_date` 与今天的关系分为 5 类：

| 分类 | 条件 | 说明 |
|------|------|------|
| expired | `last_free_date < 今天` | 已超免费期，产生滞港费风险 |
| urgent | 今天 ≤ `last_free_date` ≤ 3 天内 | 即将超期 |
| warning | 3 天 < `last_free_date` ≤ 7 天 | 预警 |
| normal | `last_free_date` > 7 天 | 时间充裕 |
| noLastFreeDate | 目的港无 `last_free_date`（最晚提柜日为空） | 无免费期数据 |

- 目标集：已到目的港（目的港有 ATA）且（无拖车记录或未实际提柜）。
- SQL 模板在 `LastPickupSubqueryTemplates` 中，与列表筛选共用同一套子集逻辑。

### 2.4 前端与 API

- 统计接口：`GET /api/v1/containers/statistics-detailed` 中包含最晚提柜分布；按条件查列表：`filterCondition` 为 `expired` / `urgent` / `warning` / `normal` / `noLastFreeDate`，走 `GET /api/v1/containers/by-filter`。
- 倒计时卡片等会用到「最晚提柜日」与「最晚免费日」概念，数据均来自上述目的港 `last_free_date`。

## 三、飞驼 API 逻辑

### 3.1 角色与架构

- **飞驼**：外部物流轨迹/状态数据源。
- **适配器**：`FeiTuoAdapter`（`backend/src/adapters/FeiTuoAdapter.ts`）实现 `IExternalDataAdapter`，负责请求飞驼、把响应转成系统标准结构。
- **AdapterManager**：注册/主备切换/健康检查；可接多个数据源（飞驼 Primary、LogisticsPath Secondary 等）。
- **接口**：如 `GET /api/v1/adapters/status`、`GET /api/v1/adapters/container/:containerNumber/status-events`、`POST /api/v1/adapters/container/:containerNumber/sync`。

### 3.2 飞驼数据如何影响状态机

状态机**只认**本库核心时间字段（如 `ata_dest_port`、`gate_in_time`、`pickup_date` 等），**不直接读**飞驼的 statusCode。因此：

- 飞驼同步时需把「飞驼状态事件」**翻译成对核心字段的更新**（写入 `process_port_operations` 等），状态机才会正确推进。
- **FeiTuoStatusMapping**（`backend/src/constants/FeiTuoStatusMapping.ts`）定义：
  - **FEITUO_STATUS_TO_CORE_FIELD_MAP**：飞驼状态码 → 本系统核心字段（如 `ARRIVE`/`ATA` → `ata_dest_port`，`GATE_IN` → `gate_in_time`，`GATE_OUT` → `gate_out_time`，`AVAIL` → `available_time`，`BO`/`DEPARTED` → `shipment_date` 等）。
  - **FEITUO_STATUS_TO_PORT_TYPE_MAP**：飞驼状态码 → 港口类型（origin/transit/destination），用于写入正确的 `port_type`。
- 只有「已发生」的事件（如 hasOccurred / 实际时间）才应回写核心字段；预计时间（如 ETA）通常只更新 ETA 类字段，不覆盖 ATA。

### 3.3 同步与落库

- **ExternalDataService**（`backend/src/services/externalDataService.ts`）：调用飞驼 OpenAPI `POST /application/v1/query`（与 FeiTuoAdapter 同源），拿到 `places` / `containers[].status` 映射后：
  - 根据 `FeiTuoStatusMapping` 决定更新哪张表、哪个字段；
  - 更新 `process_port_operations` 中对应港口类型的记录（或新建港口操作记录），写入 `ata_dest_port`、`gate_in_time`、`gate_out_time` 等；
  - 可写入扩展表如 `ext_container_status_events` 做原始轨迹留存。
- 同步完成后，若需立即反映到前端状态，应触发**状态重算**（调用 `ContainerStatusService.updateStatus(containerNumber)` 或批量接口），把 `logistics_status` 按最新数据算一遍并写回 `biz_containers`。

### 3.4 飞驼状态码示例（与核心字段对应）

| 飞驼状态码 | 含义 | 常用核心字段 | 港口类型 |
|------------|------|--------------|----------|
| BO / DEPARTED | 装船/离港 | shipment_date（海运） | origin |
| ARRIVE / ATA | 到港 | ata_dest_port | destination |
| ETA | 预计到港 | eta_dest_port | destination |
| GATE_IN | 进闸 | gate_in_time | destination |
| GATE_OUT | 出闸 | gate_out_time | destination |
| DISCHARGED | 卸船 | dest_port_unload_date | destination |
| AVAIL | 可提货 | available_time | destination |
| TRANSIT_ARRIVE / TRANSIT_DEPART | 中转港到/离 | ata_dest_port / 离港字段 | transit |

## 四、关联小结

- **状态机**：只依赖本库「核心时间字段」和业务表（柜、海运、港口、拖车、仓库、还箱），按优先级算出 `logistics_status`；Excel 与飞驼都通过**写入这些核心字段**间接驱动状态机。
- **滞港费**：由目的港的 `last_free_date` 驱动；统计与提醒都基于「最晚免费日」与今天的比较；不直接依赖飞驼，但若飞驼将来提供该字段，也可通过适配器写入同一张港口操作表。
- **飞驼**：通过适配器 + FeiTuoStatusMapping 把状态事件转成对港口操作表（及必要时的海运表等）的更新，从而**间接**驱动状态机和前端展示；同步后需调用状态更新接口使 `logistics_status` 与最新数据一致。
