# 飞驼事件触发状态机更新机制检查与审阅

**审阅日期**: 2026-03-18  
**审阅范围**: 飞驼事件 → 核心字段写入 → 状态机重算 → 货柜状态更新

---

## 一、机制总览

```
飞驼数据源
    │
    ├─ API 同步 (FeiTuoAdapter) ──► externalDataController.doSyncContainer
    │
    └─ Excel 导入 ──► feituoImport.service.import
                          │
                          ▼
    ┌─────────────────────────────────────────────────────────────┐
    │ 1. 事件/数据写入                                              │
    │    - ext_container_status_events (状态事件)                    │
    │    - process_port_operations (ata_dest_port, gate_out_time 等)│
    │    - process_sea_freight (shipment_date)                     │
    │    - process_empty_return (return_time)                       │
    └─────────────────────────────────────────────────────────────┘
                          │
                          ▼
    ┌─────────────────────────────────────────────────────────────┐
    │ 2. 状态机重算 calculateLogisticsStatus                       │
    │    - 读取 container + portOperations + trucking + warehouse...│
    │    - 按优先级计算 simplifiedStatus、currentPortType            │
    └─────────────────────────────────────────────────────────────┘
                          │
                          ▼
    ┌─────────────────────────────────────────────────────────────┐
    │ 3. 货柜状态更新                                               │
    │    - biz_containers.logistics_status 持久化                   │
    │    - currentPortType 实时计算（不持久化）                      │
    └─────────────────────────────────────────────────────────────┘
                          │
                          ▼
    ┌─────────────────────────────────────────────────────────────┐
    │ 4. 滞港费重算（可选）                                         │
    │    - ATA 相关字段更新时触发 demurrageService.calculateForContainer │
    └─────────────────────────────────────────────────────────────┘
```

## 二、触发入口与数据流

### 2.1 API 同步（Controller）

| 步骤 | 文件 | 方法 | 说明 |
|------|------|------|------|
| 1 | externalData.controller | doSyncContainer | 入口：POST /api/external/sync/:containerNumber |
| 2 | FeiTuoAdapter | getContainerStatusEvents | 调用飞驼 API，返回 status 节点 |
| 3 | externalData.controller | convertToStatusEvents | 转为 ContainerStatusEvent[] |
| 4 | externalData.controller | saveStatusEvents | 写入 ext_container_status_events |
| 5 | externalData.controller | applyFeiTuoToCoreFields | 按 FeiTuoStatusMapping 写 process_port_operations 等 |
| 6 | externalData.controller | recalculateLogisticsStatus | 调用 calculateLogisticsStatus，更新 container.logistics_status |

**依赖**：FeiTuoStatusMapping.getCoreFieldName、getPortTypeForStatusCode

### 2.2 Excel 导入（feituoImport.service）

| 步骤 | 文件 | 方法 | 说明 |
|------|------|------|------|
| 1 | feituoImport.service | import | 入口：mergeTable1ToCore / mergeTable2ToCore |
| 2 | feituoImport.service | mergeTableXToCore | 写入 process_port_operations、process_sea_freight 等 |
| 3 | feituoImport.service | recalculateStatus | 内部调用 calculateLogisticsStatus，更新 container.logistics_status |

**依赖**：FeituoFieldGroupMapping、FeiTuoStatusMapping（表二状态码）

### 2.3 未使用的 ExternalDataService

⚠️ **发现**：`externalDataService.syncContainerEvents` 未被任何路由调用。

- `externalData.routes.ts` 仅使用 `externalDataController`
- `externalDataService` 含 places 优先、trackingEvents 回退、updatePortOperationCoreFields 等完整逻辑
- **结论**：API 同步走 Controller 路径，未走 ExternalDataService；ExternalDataService 可能为历史遗留或计划中的批量/后台同步入口

## 三、状态机核心逻辑

### 3.1 优先级（logisticsStatusMachine.ts）

| 优先级 | 条件 | 结果 |
|--------|------|------|
| 1 | emptyReturn.returnTime | returned_empty |
| 2 | warehouseOperation.wmsConfirmDate | unloaded |
| 3 | truckingTransport.pickupDate | picked_up |
| 4 | 目的港 ataDestPort | at_port + destination |
| 4a | 目的港 availableTime | at_port + destination |
| 5 | 中转港 ataDestPort / gateInTime / transitArrivalDate | at_port + transit |
| 6 | seaFreight.shipmentDate | in_transit |
| 7 | 默认 | not_shipped |

### 3.2 飞驼事件与核心字段的映射

| 飞驼状态码 | 核心字段 | 影响状态机 |
|------------|----------|------------|
| BDAR, POCA, ARRI | ata_dest_port | 优先级 4 → at_port |
| PCAB, AVLE, AVAIL | available_time | 优先级 4a → at_port |
| DSCH, DISC | dest_port_unload_date | 辅助 |
| STCS, GTOT | gate_out_time | 提柜（需 trucking 表） |
| TSBA | transit_arrival_date | 优先级 5 → at_port + transit |
| RCVE | return_time | 优先级 1 → returned_empty |
| LOBD, BO | shipment_date | 优先级 6 → in_transit |

## 四、审阅发现

### 4.1 正常点 ✅

| 项目 | 说明 |
|------|------|
| 触发顺序正确 | 先写核心字段 → 再重算状态 → 再更新 container |
| 状态机入参完整 | Controller 的 recalculateLogisticsStatus 加载 trucking、warehouse、emptyReturn |
| 滞港费联动 | ATA 相关字段更新后触发 demurrageService.calculateForContainer |
| availableTime 支持 | 状态机优先级 4a 支持「无 ATA 但有可提货」场景 |
| 错误不阻断主流程 | recalculateLogisticsStatus 失败时 catch 不抛出，不影响事件保存 |

### 4.2 潜在问题 ⚠️

| 问题 | 位置 | 说明 |
|------|------|------|
| **ExternalDataService 未接入** | externalData.routes | API 同步走 Controller，未使用 places 优先、 trackingEvents 回退逻辑；若需 places 支持，需将 Controller 改为调用 externalDataService |
| **Controller 无 portOperation 时** | applyFeiTuoToCoreFields | 按 portType 匹配时，若无 portOperation 则跳过更新；需确保货柜已有 portOperation（如从 Excel 导入或海运创建） |
| **ExternalDataService.recalculateLogisticsStatus 入参不全** | externalDataService.ts:752 | 未传 truckingTransport、warehouseOperation、emptyReturn，全部为 undefined；状态机可能无法正确识别 picked_up、unloaded、returned_empty |
| **Controller 与 Service 双路径** | - | Controller 与 ExternalDataService 各自实现一套同步逻辑，存在重复与不一致风险 |

### 4.3 建议改进

| 建议 | 优先级 | 说明 |
|------|--------|------|
| 统一同步入口 | 高 | 让 Controller 调用 externalDataService.syncContainerEvents，复用 places 优先、完整核心字段更新逻辑 |
| 修正 ExternalDataService.recalculateLogisticsStatus | 高 | 加载 trucking、warehouse、emptyReturn 并传入 calculateLogisticsStatus |
| 明确 ExternalDataService 职责 | 中 | 若作为内部服务，需在文档中说明其调用方；若为死代码，应移除或标记 |
| 无 portOperation 时的处理 | 中 | 飞驼同步时若无 portOperation，可考虑自动创建或返回明确错误提示 |

## 五、验证建议

1. **API 同步**：用真实货柜号调用 `POST /api/external/sync/:id`，检查：
   - ext_container_status_events 有新增
   - process_port_operations 有 ata_dest_port / available_time 等
   - container.logistics_status 已更新

2. **Excel 导入**：导入飞驼表二后，检查：
   - process_port_operations 有对应字段
   - container.logistics_status 与预期一致

3. **currentPortType**：列表/详情中「已到中转港」「已到目的港」筛选是否与状态机一致（依赖 container.service 的实时计算）
