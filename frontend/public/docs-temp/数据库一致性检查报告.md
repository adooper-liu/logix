### 2.3 前端 ContainerListItem 接口

ContainerListItem 接口用于列表视图，包含了货柜的核心字段，与后端实体和数据库表保持了基本一致，同时添加了一些前端展示所需的字段。

## 3. 发现的问题

### 3.1 类型不一致

1. **日期时间类型差异**：
   - 数据库中部分日期字段使用 `DATE` 类型，而实体中统一使用 `Date` 类型
   - 虽然 TypeORM 会自动处理类型转换，但建议统一数据库类型以避免潜在问题

2. **字段缺失**：
   - 前端 Container 接口缺少 `scheduleStatus` 字段
   - 前端 PortOperation 接口缺少多个实体字段，如 `revisedEtaDestPort`、`etdTransit`、`atdTransit`、`lastFreeDateMode`、`latitude`、`longitude`、`createdAt`、`updatedAt`

3. **前端额外字段**：
   - 前端 Container 接口包含大量后端实体中没有的字段，如 `orderNumber`、`allOrders`、`summary` 等
   - 这些字段可能是前端特有的计算字段或关联数据，需要确保其来源和更新机制

### 3.2 命名规范

- 数据库表名使用 `snake_case`（如 `biz_containers`）
- 数据库字段使用 `snake_case`（如 `container_number`）
- 实体属性使用 `camelCase`（如 `containerNumber`）
- 前端类型字段使用 `camelCase`（如 `containerNumber`）

命名规范整体一致，符合最佳实践。

## 4. 改进建议

### 4.1 类型统一

1. **日期时间类型统一**：
   - 建议将数据库中所有日期时间字段统一为 `TIMESTAMP` 类型，与实体和前端保持一致
   - 这样可以确保时间精度的一致性，避免日期时间处理中的问题

2. **字段同步**：
   - 前端类型定义应与后端实体保持同步，确保所有必要字段都被包含
   - 对于前端特有的字段，应添加注释说明其用途和来源

### 4.2 数据一致性

1. **状态字段一致性**：
   - 确保 `logistics_status` 等状态字段在数据库、实体和前端的取值范围一致
   - 建议使用枚举类型或常量来统一状态值

2. **关联关系一致性**：
   - 确保数据库外键关系与实体关联关系一致
   - 前端应正确处理关联数据的加载和展示

### 4.3 开发规范

1. **遵循开发流程**：
   - 按照 "SQL → 实体 → API → 前端" 的顺序开发
   - 确保每一步都与数据库表结构保持一致

2. **文档更新**：
   - 及时更新数据库文档，确保表结构、字段含义等信息准确
   - 为前端类型定义添加适当的注释，说明字段来源和用途

## 5. 结论

- 数据库表、实体和前端字段整体保持一致，命名规范符合最佳实践
- 存在一些类型差异和字段缺失，需要进一步调整和同步
- 前端包含一些特有字段，需要确保其数据来源和更新机制
- 建议按照改进建议进行调整，确保系统各组件之间的一致性

## 6. 后续行动

1. **类型统一**：统一日期时间类型，确保数据库、实体和前端的一致性
2. **字段同步**：更新前端类型定义，确保与后端实体保持同步
3. **文档更新**：更新数据库文档和前端类型注释，确保信息准确
4. **测试验证**：进行端到端测试，验证数据流转的一致性
