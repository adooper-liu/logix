# 甘特图圆点销毁逻辑完整梳理

## 业务场景

甘特图中，已完成实际操作的节点应销毁不显示圆点，避免视觉干扰。圆点销毁是甘特图状态管理的核心机制之一。

**核心需求**：

- 有实际日期的节点不显示圆点（已完成，销毁）
- 销毁后的节点在Tooltip中仍显示实际日期（绿色标识）
- 销毁逻辑基于链式依赖：清关 -> 提柜 -> 卸柜 -> 还箱

## 销毁规则总览

### 节点销毁条件

| 节点 | 销毁条件（实际日期字段）            | 数据库表                     | 字段名                      |
| ---- | ----------------------------------- | ---------------------------- | --------------------------- |
| 清关 | `actualCustomsDate` 存在            | process_port_operations      | actual_customs_date         |
| 提柜 | `deliveryDate` 或 `pickupDate` 存在 | process_trucking_transport   | delivery_date / pickup_date |
| 卸柜 | `unloadDate` 存在                   | process_warehouse_operations | unload_date                 |
| 还箱 | `returnTime` 存在                   | process_empty_return         | return_time                 |
| 查验 | 同清关节点                          | process_port_operations      | actual_customs_date         |

**设计原则**：

- 有实际日期 = 节点已完成 = 状态设为 `completed` = 显示类型转为 `null` = 圆点销毁
- 销毁不等于数据丢失，Tooltip中仍可查看实际日期

## 技术实现流程

### 完整判断链路

```
calculateNodeStatus()
    ↓ 计算节点状态 (pending/active/completed)
getNodeDisplayType()
    ↓ 决定显示类型 (main/dashed/null)
getNodeTypeFromLocalStatus()
    ↓ completed → null, active → main, pending → dashed
模板渲染
    ↓ null 不渲染 DOM
圆点销毁
```

#### 5. 反向链式依赖检查（Line 2652-2686）

**业务规则**：后续节点完成意味着前置节点必然已完成，即使前置节点的实际日期字段缺失。

```typescript
// ========== 反向链式依赖检查 ==========
// 业务规则：后续节点完成意味着前置节点必然已完成
// 已提柜 -> 清关完成 | 已卸柜 -> 清关+提柜完成 | 已还箱 -> 清关+提柜+卸柜完成

// 1. 已还箱 -> 反推清关、提柜、卸柜全部完成
if (emptyReturn?.returnTime) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推清关已完成
  }
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate) && pickupSupplier) {
    nodes.提柜.status = 'completed' // 反推提柜已完成
  }
  if (!unloadOp?.unloadDate && unloadSupplier) {
    nodes.卸柜.status = 'completed' // 反推卸柜已完成
  }
}
// 2. 已卸柜 -> 反推清关、提柜完成
else if (unloadOp?.unloadDate) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推清关已完成
  }
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate) && pickupSupplier) {
    nodes.提柜.status = 'completed' // 反推提柜已完成
  }
}
// 3. 已提柜 -> 反推清关完成
else if (pickupTransport?.deliveryDate || pickupTransport?.pickupDate) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推清关已完成
  }
}
```

**设计优势**：

- **容错性强**：即使前置节点日期缺失，也能正确推断状态
- **用户体验好**：避免显示大量虚线圆点（待激活）
- **业务准确性高**：符合物流实际流程（后续完成意味着前置必然完成）

### 第二步：getNodeDisplayType - 显示类型决策

**位置**: `SimpleGanttChartRefactored.vue` Line 1932-1957

```typescript
const getNodeDisplayType = (container: any, nodeName: string): 'main' | 'dashed' | null => {
  const nodeStatus = calculateNodeStatus(container)
  const node = nodeStatus.nodes[nodeName as keyof typeof nodeStatus.nodes]

  // 边界检查：无供应商或供应商未指定
  if (!node?.supplier || node.supplier === '未指定') return null

  // 1. 查验节点特殊处理（后端 gantt-v1 无此节点，仅信本地状态）
  if (nodeName === '查验') {
    return getInspectionNodeType(node)
  }

  // 2. 卸柜/还箱节点：优先使用本地状态（避免后端 ganttDerived 错误标记为 completed）
  if (nodeName === '卸柜' || nodeName === '还箱') {
    return getNodeTypeFromLocalStatus(node)
  }

  // 3. 其他节点使用后端 ganttDerived（如果存在且有效）
  const ganttType = getNodeTypeFromGanttDerived(container, nodeName)
  if (ganttType !== null) {
    return ganttType
  }

  // 4. 回退到本地状态判断
  return getNodeTypeFromLocalStatus(node)
}
```

**关键策略**：

- 卸柜/还箱节点跳过 `ganttDerived`，直接使用本地状态（避免后端错误标记）
- 清关/提柜节点优先使用 `ganttDerived`，失败后回退本地状态

### 第三步：getNodeTypeFromLocalStatus - 状态转换

**位置**: `SimpleGanttChartRefactored.vue` Line 1919-1924

```typescript
const getNodeTypeFromLocalStatus = (node: any): 'main' | 'dashed' | null => {
  if (node.status === 'completed') return null // ← completed 转换为 null，不显示
  if (node.status === 'active') return 'main' // active 转换为 main，实心圆点
  if (node.status === 'pending') return 'dashed' // pending 转换为 dashed，空心圆点
  return null
}
```

**核心转换规则**：

- `completed` → `null`：**销毁圆点，不渲染DOM**
- `active` → `'main'`：显示实心圆点
- `pending` → `'dashed'`：显示空心圆点

```typescript
// 辅助方法：判断是否有实际日期
const hasActualDates = (container: any): boolean => {
  if (!container) return false
  const hasCustoms = container?.portOperations?.[0]?.actualCustomsDate
  const hasPickup =
    container?.truckingTransports?.[0]?.pickupDate ||
    container?.truckingTransports?.[0]?.deliveryDate
  const hasUnload = container?.warehouseOperations?.[0]?.unloadDate
  const hasReturn = container?.emptyReturns?.[0]?.returnTime
  return !!(hasCustoms || hasPickup || hasUnload || hasReturn)
}

// 辅助方法：获取实际提柜日期
const getActualPickupDate = (container: any): string | null => {
  const trucking = container?.truckingTransports?.[0]
  if (trucking?.pickupDate) return trucking.pickupDate
  if (trucking?.deliveryDate) return trucking.deliveryDate
  return null
}

// 辅助方法：获取实际送仓日期
const getActualDeliveryDate = (container: any): string | null => {
  const trucking = container?.truckingTransports?.[0]
  if (trucking?.deliveryDate) return trucking.deliveryDate
  return null
}
```

## 关键技术要点

### 1. 链式依赖检查

每个节点的销毁不仅取决于自身实际日期，还需考虑前置节点状态：

```typescript
// 提柜销毁：只看自身实际日期
if (pickupTransport.deliveryDate || pickupTransport.pickupDate) {
  nodes.提柜.status = 'completed'
}

// 卸柜激活：需检查提柜是否完成
const pickupCompleted = pickupTransport?.deliveryDate || pickupTransport?.pickupDate
if (pickupCompleted) {
  nodes.卸柜.status = 'active'
} else {
  nodes.卸柜.status = 'pending'
}
```

### 2. 后端数据可信度问题

**问题**：后端 `ganttDerived` 可能错误地将后续节点标记为 `completed`

**解决方案**：卸柜/还箱节点优先使用本地状态

```typescript
// 2. 卸柜/还箱节点：优先使用本地状态（避免后端 ganttDerived 错误标记为 completed）
if (nodeName === '卸柜' || nodeName === '还箱') {
  return getNodeTypeFromLocalStatus(node)
}
```

### 3. DOM销毁机制

**原理**：Vue的 `v-if` 指令

```vue
<div v-if="displayType" class="container-dot">
  <!-- displayType 为 null 时，整个 div 不渲染 -->
</div>
```

**效果**：

- `displayType = null`：DOM元素不存在，不占用布局空间
- `displayType = 'main'` 或 `'dashed'`：正常渲染圆点

### 4. 性能优化

**缓存策略**：避免重复计算

```typescript
const nodeStatus = calculateNodeStatus(container) // 只计算一次
if (gd && gd.nodes && gd.nodes.length > 0) {
  return getDisplayItemsFromGanttDerived(container, gd, nodeStatus)
}
```

**优势**：

- 同一货柜的状态只计算一次
- 多个节点共享同一个 `nodeStatus` 对象

2. 辅助函数返回 `null`
3. 模板条件判断失败

**排查步骤**：

2. 链式依赖检查逻辑错误

**排查步骤**：

```typescript
// 检查前置节点完成状态
const customsCompleted = destPortOp?.actualCustomsDate

// 检查节点状态
```
