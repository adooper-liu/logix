# 统计与表格过滤逻辑一致性规范

## 1. 核心原则

### 1.1 唯一真实源原则

**所有统计卡片和表格过滤必须使用相同的数据筛选逻辑**

- ✅ 统计数量 = 表格记录数
- ✅ 统计维度 = 表格过滤条件
- ✅ 国家过滤 = 全局作用域
- ✅ 日期范围 = 统一口径（出运日期）

### 1.2 禁止重复实现

**同一业务逻辑只能有一个实现位置**

- ❌ 禁止在统计方法和表格方法中分别实现相同的过滤逻辑
- ✅ 使用共享工具函数或子查询模板
- ✅ 统计和表格调用同一个底层查询函数

## 2. 标准实现模式

### 2.1 推荐模式：子查询模板模式

```typescript
// ✅ 正确示例：LastPickupStatisticsService

// 步骤 1: 定义 SQL 子查询模板
export class LastPickupSubqueryTemplates {
  static EXPIRED_SUBQUERY = (today: string) => `
    SELECT DISTINCT c.container_number
    FROM biz_containers c
    WHERE c.last_free_date < '${today}'
    -- 其他过滤条件...
  `;
}

// 步骤 2: 统计和表格共用同一套 SQL
private async runSubqueryForCondition(
  innerSql: string,
  startDate?: string,
  endDate?: string
): Promise<string[]> {
  // 统一的日期范围和国家过滤逻辑
  const { sql, params } = this.buildDateRangeQuery(innerSql, startDate, endDate);
  const result = await this.containerRepository.query(sql, params);
  return result.map((r: any) => r.container_number);
}

// 步骤 3: 统计方法
async getExpiredCount(today: string, startDate?: string, endDate?: string): Promise<number> {
  const innerSql = LastPickupSubqueryTemplates.EXPIRED_SUBQUERY(today);
  const containerNumbers = await this.runSubqueryForCondition(innerSql, startDate, endDate);
  return containerNumbers.length; // ✅ 直接返回数组长度
}

// 步骤 4: 表格过滤方法
private async getContainersByExpired(
  today: string,
  startDate?: string,
  endDate?: string
): Promise<Container[]> {
  const innerSql = LastPickupSubqueryTemplates.EXPIRED_SUBQUERY(today);
  const containerNumbers = await this.runSubqueryForCondition(innerSql, startDate, endDate);
  return this.getContainersByNumbers(containerNumbers); // ✅ 复用同一套 SQL
}
```

**优势**：
- ✅ 统计和表格使用完全相同的 SQL
- ✅ 日期范围和国家过滤逻辑统一
- ✅ 无法出现不一致问题

### 2.2 标准模式：统一工具函数模式

```typescript
// ✅ 正确示例：应用 applyDateFilterToQuery

import { applyDateFilterToQuery } from './common/DateRangeSubquery';

// 统计方法
async getReturnUrgentCount(
  today: Date,
  threeDaysLater: Date,
  startDate?: string,
  endDate?: string
): Promise<number> {
  const query = ContainerQueryBuilder.createBaseQuery(this.containerRepository);
  ContainerQueryBuilder.addCountryFilters(query);
  this.filterTargetSet(query);
  this.joinEmptyReturn(query);
  
  // 业务条件
  query.andWhere('er.last_return_date >= :today', { today });
  query.andWhere('er.last_return_date <= :threeDays', { threeDays: threeDaysLater });
  
  // ✅ 统一的日期范围过滤（包含国家过滤的子查询）
  applyDateFilterToQuery(query, this.containerRepository, startDate, endDate, {
    today,
    threeDays: threeDaysLater
  });
  
  const result = await query.select('COUNT(DISTINCT container.containerNumber)', 'count').getRawOne();
  return parseInt(result?.count || '0');
}

// 表格过滤方法
private async getContainersByReturnUrgent(
  today: Date,
  threeDaysLater: Date,
  startDate?: string,
  endDate?: string
): Promise<Container[]> {
  const query = ContainerQueryBuilder.createBaseQuery(this.containerRepository);
  ContainerQueryBuilder.addCountryFilters(query);
  this.filterTargetSet(query);
  this.joinEmptyReturn(query);
  
  // 业务条件（与统计方法完全相同）
  query.andWhere('er.last_return_date >= :today', { today });
  query.andWhere('er.last_return_date <= :threeDays', { threeDays: threeDaysLater });
  
  // ✅ 使用同一个工具函数（确保逻辑一致）
  applyDateFilterToQuery(query, this.containerRepository, startDate, endDate, {
    today,
    threeDays: threeDaysLater
  });
  
  return query.getMany();
}
```

**关键点**：
- ✅ 统计和表格调用同一个工具函数
- ✅ 业务条件完全相同
- ✅ 参数传递一致

## 3. 禁止使用的模式

### 3.1 ❌ 禁止：直接字段比较模式

```typescript
// ❌ 错误示例：统计和表格使用不同的过滤逻辑

// 统计方法（使用子查询）
async getReturnUrgentCount(...): Promise<number> {
  // ... 业务条件
  applyDateFilterToQuery(query, this.containerRepository, startDate, endDate);
  // ✅ 正确：使用了包含国家过滤的子查询
}

// 表格过滤方法（使用直接字段比较）
private async getContainersByReturnUrgent(...): Promise<Container[]> {
  // ... 业务条件
  ContainerQueryBuilder.addDateFilters(query, startDate, endDate);
  // ❌ 错误：只是简单的字段比较，缺少国家过滤
  
  ContainerQueryBuilder.addCountryFilters(query);
  // ❌ 虽然添加了国家过滤，但与统计方法的子查询模式不同
  
  return query.getMany();
}
```

**问题**：
- ❌ 两种不同的实现方式，容易导致逻辑不一致
- ❌ `addDateFilters` 只是直接字段比较
- ❌ 即使调用了 `addCountryFilters`，也可能因为 JOIN 方式不同导致结果差异

### 3.2 ❌ 禁止：复制粘贴模式

```typescript
// ❌ 错误示例：在两个地方复制相同的过滤逻辑

// 文件 A：统计服务
function buildDateFilter(startDate: string, endDate: string) {
  // 50 行复杂的日期过滤逻辑...
  return whereClause;
}

// 文件 B：表格过滤服务
function buildDateFilter(startDate: string, endDate: string) {
  // 复制了 50 行代码，但可能有细微差异...
  return whereClause;
}
```

**问题**：
- ❌ 代码重复，难以维护
- ❌ 一旦业务逻辑变更，需要修改多处
- ❌ 容易出现不一致的 bug

## 4. 日期范围过滤规范

### 4.1 统一使用 applyDateFilterToQuery

```typescript
// ✅ 正确：所有日期范围过滤都使用此函数
import { applyDateFilterToQuery } from './common/DateRangeSubquery';

// 场景 1：有额外参数时
applyDateFilterToQuery(query, this.containerRepository, startDate, endDate, {
  today,
  threeDays: threeDaysLater
});

// 场景 2：无额外参数时
applyDateFilterToQuery(query, this.containerRepository, startDate, endDate);

// 场景 3：只有国家过滤，无日期范围
if (!startDate && !endDate) {
  ContainerQueryBuilder.addDateFilters(query, undefined, undefined);
  DateFilterBuilder.addCountryFilters(query, countryCode);
}
```

### 4.2 日期口径约定

**所有日期范围默认指「出运日期」**

- ✅ 优先使用 `actual_ship_date`（实际出运日期）
- ✅ 备选使用 `expected_ship_date`（预计出运日期）
- ✅ 最后使用 `shipment_date`（海运出运日期）

```typescript
// applyDateFilterToQuery 内部逻辑
const subQuery = createDateRangeSubQuery(containerRepository, startDate, endDate, countryCode);
// 子查询中使用：COALESCE(o.actualShipDate, o.expectedShipDate, sf.shipmentDate)
```

## 5. 国家过滤规范

### 5.1 全局作用域原则

**国家过滤应该作为全局作用域，而非显式参数**

```typescript
// ✅ 正确：从 AsyncLocalStorage 自动获取
const rawCountryCode = getScopedCountryCode();
const countryCode = normalizeCountryCode(rawCountryCode);

// ❌ 错误：手动传递国家代码
function filterByCountry(countryCode: string) {
  // 不应该由调用方决定国家代码
}
```

### 5.2 子查询自动包含

**使用 `applyDateFilterToQuery` 时，国家过滤会自动应用到子查询**

```typescript
// applyDateFilterToQuery 内部逻辑
export function applyDateFilterToQuery(
  query: SelectQueryBuilder<any>,
  containerRepository: Repository<Container>,
  startDate?: string,
  endDate?: string,
  extraParams?: Record<string, unknown>,
  countryCode?: string // 可选，默认从上下文获取
): void {
  if (startDate && endDate) {
    // ✅ 创建子查询时自动包含国家过滤
    const subQuery = createDateRangeSubQuery(
      containerRepository, 
      startDate, 
      endDate, 
      countryCode // 如果未传，内部会调用 getScopedCountryCode()
    );
    query.andWhere(`container.containerNumber IN (${subQuery.getQuery()})`);
  } else {
    // 无日期范围时才手动添加
    ContainerQueryBuilder.addDateFilters(query, startDate, endDate);
    DateFilterBuilder.addCountryFilters(query, countryCode);
  }
}
```

## 8. 架构决策记录

### 8.1 为什么选择子查询模式？

**背景**：早期使用直接字段比较，导致统计和表格逻辑不一致

**决策**：统一使用 SQL 子查询模板模式

**理由**：
1. **单一真实源**：统计和表格共用同一套 SQL，无法出现不一致
2. **易于维护**：业务逻辑变更只需修改一处
3. **性能优化**：PostgreSQL 对 EXISTS 子查询有良好优化

### 8.2 为什么禁止 addDateFilters？

**原因**：
1. `addDateFilters` 只是简单的字段比较：`order.actualShipDate >= :startDate`
2. 不包含国家过滤逻辑
3. 容易与统计方法使用的 `applyDateFilterToQuery` 产生不一致

**替代方案**：
- 有日期范围时使用 `applyDateFilterToQuery`
- 无日期范围时使用 `ContainerQueryBuilder.addDateFilters + DateFilterBuilder.addCountryFilters`

## 9. 常见问题 FAQ

### Q1: 什么时候使用 SQL 模板模式 vs 工具函数模式？

**A**: 
- **SQL 模板模式**：适用于复杂业务逻辑，需要精细控制 SQL 的场景
  - 例如：LastPickupStatistics、PlannedPickupStatistics
- **工具函数模式**：适用于简单业务逻辑，快速开发的场景
  - 例如：LastReturnStatistics、EtaStatistics

### Q2: 如果只需要统计不需要表格，是否可以简化实现？

**A**: **不可以**。必须预留表格过滤的能力，即使当前不需要。

**理由**：
1. 未来需求变更成本高
2. 统计和表格的一致性是关键质量要求
3. 复用已有代码不会增加太多工作量

### Q3: 如何处理分页？

**A**: 统计方法返回总数，表格方法使用分页查询：

```typescript
// 统计方法：返回总数
async getCount(): Promise<number> {
  const containerNumbers = await this.runSubqueryForCondition(sql, start, end);
  return containerNumbers.length;
}

// 表格方法：支持分页
async getContainers(page: number, pageSize: number): Promise<Container[]> {
  const containerNumbers = await this.runSubqueryForCondition(sql, start, end);
  const pagedNumbers = containerNumbers.slice(
    (page - 1) * pageSize, 
    page * pageSize
  );
  return this.getContainersByNumbers(pagedNumbers);
}
```
