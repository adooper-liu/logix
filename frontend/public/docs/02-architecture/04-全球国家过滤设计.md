# 全局国家过滤设计（替代逐接口传 countryCode）

## 一、为何不用「每处传 countryCode」

当前若在**每个**控制器、统计服务、子服务方法里都增加 `countryCode` 参数：

- **缺点**：改动面大、易漏、难维护；且后期国家会与**帐号权限**绑定（帐号仅能看某国/某几国数据），届时仍需从「当前用户」取国家，而不是从每个请求参数里传。
- **目标**：国家作为**全局过滤维度**，首页/布局选定一次，全站生效；后端由**请求上下文**（如 Header / Session）得到「当前生效国家」，在**单一位置**统一施加过滤，而不是在几十个方法签名里传递。

因此推荐采用**请求级全局过滤**，而不是为每个项目添加 `countryCode` 参数。

---

## 二、推荐方案：请求上下文 + 单一应用点

### 2.1 思路

1. **前端**：在顶部/布局放一个国家选择器，选中的国家写入全局状态（如 Pinia）或 Cookie；发请求时通过 **Header** 带给后端（例如 `X-Country-Code: US`）。后续若做登录，可改为「帐号权限范围内的默认国家」或「帐号绑定国家」。
2. **后端**：
   - 用 **中间件** 在请求入口解析 `X-Country-Code`（或从 Session/JWT 中的用户权限得到允许的国家），写入 **请求作用域上下文**（如 `req.scopedCountryCode` 或 AsyncLocalStorage）。
   - 在**唯一一处**施加国家条件：例如在「货柜查询基础构造器」或「日期+国家过滤层」里，从上下文读取 `scopedCountryCode`，若存在则给当前 query 加上国家条件（销住国家 / 客户国家 / 目的港所在国）。  
   - 控制器、统计服务、各子服务**不再**接收或传递 `countryCode` 参数，所有货柜相关查询自动带上国家过滤。

这样：

- 新增/修改过滤逻辑只需改**一处**（例如 `DateFilterBuilder` 或新的 `RequestScopeFilter`）。
- 与帐号权限对接时，只需在中间件里把「当前用户允许的国家」写入同一上下文即可。

### 2.2 后端实现要点

- **中间件**（如 `middleware/scopedCountry.ts`）：  
  `const countryCode = req.headers['x-country-code'] || req.user?.allowedCountryCode;`  
  校验后写入 `req.scopedCountryCode`（或 AsyncLocalStorage 的 `getStore().countryCode`）。
- **单一应用点**：  
  - 在 `DateFilterBuilder.addCountryFilters(query)` 或新建 `RequestScopeFilter.applyCountry(query)` 中：  
    - 从 `req` 或 AsyncLocalStorage 读取 `scopedCountryCode`；  
    - 若存在，则对 query 施加与当前一致的国家条件（销住国家 OR 客户国家 OR 目的港所在国）。  
  - 所有使用 `createBaseQuery` / `createDateRangeSubQuery` / `addDateFilters` 的统计与列表逻辑，**统一**在构建 query 后调用这一层，不再传 `countryCode` 参数。
- **控制器/服务**：  
  移除已添加的 `countryCode` 查询参数和层层传递，列表、统计、by-filter 等接口不再接收 `countryCode`，由中间件+单一应用点完成过滤。

### 2.3 前端配合

- 顶部国家选择器：选中值写入 Pinia（如 `useAppStore().scopedCountryCode`）并持久化到 Cookie/localStorage（可选）。  
- 请求拦截器（axios）：  
  `if (appStore.scopedCountryCode) config.headers['X-Country-Code'] = appStore.scopedCountryCode;`  
  这样所有请求自动带上当前选中国家。  
- 后续与帐号权限绑定：登录后由后端返回「允许的国家列表」或「默认国家」，前端只在该范围内展示/选择，或由后端完全根据 JWT 忽略 Header 中的国家并改用权限国家。

### 2.4 与当前已改代码的关系

- 若已部分实现「逐接口传 countryCode」：可保留后端「国家条件」的 SQL/TypeORM 逻辑（销住国家、客户国家、目的港所在国），只把「参数来源」从「每个方法的 `countryCode` 参数」改为「从请求上下文读取」；然后逐步删除各层方法签名里的 `countryCode`，统一在单一应用点从上下文取数并调用 `addCountryFilters`。
- 这样既避免「为每个项目添加 countryCode」的笨办法，又为后续「国家与帐号权限绑定」留好扩展点。

---

## 三、小结

| 方式 | 做法 | 问题 |
|------|------|------|
| 当前笨办法 | 每个接口、每个服务方法都增加 `countryCode` 参数并传递 | 改动多、易漏、难维护；与未来「帐号国家权限」不一致 |
| **推荐** | 请求级全局过滤：中间件设 `req.scopedCountryCode`，单一应用点（如 DateFilterBuilder/RequestScopeFilter）从上下文取国家并施加过滤 | 改一处即可；易与登录、权限、前端全局选择器对接 |

**结论**：应采用「全局数据过滤」——国家在首页/布局选定，通过 Header 或 Session 传给后端，后端在**唯一应用点**根据请求上下文施加国家条件，而不是在每处添加 `countryCode` 参数。
