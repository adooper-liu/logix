## 四、核心算法

### 4.1 锚点日期计算

```
以 plannedUnloadDate (计划卸柜日) 为锚点:

1. 锚点确定 (findOptimalUnloadDate)
   ├── 从 ETA 后的首个可用日期开始
   ├── 检查仓库日产能 (dailyUnloadCapacity)
   └── 向前查找首个有产能的日期

2. 日期推导
   ├── plannedCustomsDate = plannedPickupDate - 1天
   ├── plannedPickupDate = plannedCustomsDate + 1天
   ├── plannedDeliveryDate = 计算得出
   │     ├── Live load: 提=送=卸
   │     └── Drop off: 送=卸
   ├── plannedReturnDate = plannedUnloadDate + 在仓天数
   │     └── 优先使用 lastReturnDate
   └── plannedUnloadDate = 锚点 (已确定)
```

### 4.2 成本优化逻辑

```
优化目标: 最小化 总成本 (滞港费 + 滞箱费 + 存储费 + 运输费)

优化策略:
1. 生成多个候选日期 (base - 3天, base - 2天, base, base + 1天, ...)
2. 对每个候选日期:
   ├── 调用 DemurrageService.calculateTotalCostWithPlannedDates()
   │     ├── 使用候选日期覆盖 plannedPickupDate
   │     └── 计算各项费用
   └── 汇总总成本
3. 选择总成本最低的方案
4. 如果节省金额 > 0，建议优化
```

## 六、关键配置

### 6.1 排产配置 (dict_scheduling_config)

| 配置键 | 说明 | 默认值 |
|--------|------|--------|
| skip_weekends | 是否跳过周末 | true |
| weekend_days | 周末天数配置 | [0, 6] |
| default_buffer_days | 默认ETA顺延天数 | 2 |
| max_buffer_days | 最大顺延天数 | 7 |

### 6.2 卸柜方案

| 方案 | 说明 | 适用场景 |
|------|------|----------|
| live_load | 提=送=卸（同日） | 急货 |
| drop_off | 先放堆场，送=卸 | 普通货 |
| expedited | 加急模式 | 紧急货物 |

## 七、定时任务

### 7.1 状态同步

```
每5分钟执行: ContainerStatusScheduler
├── 查询需要更新状态的货柜
├── 根据各流程表时间节点计算新状态
└── 更新 logistics_status
```

### 7.2 滞港费写回

```
每5分钟执行: DemurrageWriteBackScheduler
├── 查询待更新货柜
├── 调用 calculateForContainer()
├── 更新 last_free_date / last_return_date
└── 写回数据库
```
