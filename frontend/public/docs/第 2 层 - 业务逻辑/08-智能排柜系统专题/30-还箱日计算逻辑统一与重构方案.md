## 一、背景与问题

### 1.1 现状分析

当前项目中存在**3处独立的还箱日计算逻辑**，分别位于：

| 位置         | 文件                                                      | 行数         | 调用场景                 |
| ------------ | --------------------------------------------------------- | ------------ | ------------------------ |
| 前端甘特图   | `frontend/src/components/common/gantt/useGanttLogic.ts`   | 26行 + 77行  | 拖拽节点时同步还箱日     |
| 后端智能排柜 | `backend/src/services/SchedulingDateCalculator.ts`        | 88行         | 正式排产时计算还箱日     |
| 成本优化服务 | `backend/src/services/schedulingCostOptimizer.service.ts` | 3行 (硬编码) | 批量预估成本时估算还箱日 |

### 1.2 核心问题

#### 问题 1: 规则不一致

```typescript
// 前端: Drop off 模式 max(卸柜日, 原还箱日)
const later = r.isBefore(u) ? u : r

// 后端: Drop off 模式 优先当天 -> 卸+1 -> 顺延 (能力检查)
if (availableOnUnloadDate) return availableOnUnloadDate
const nextDay = new Date(returnDateOnly)
nextDay.setUTCDate(nextDay.getUTCDate() + 1)

// 成本优化: Drop off 模式 硬编码 +3天
plannedReturnDate = dateTimeUtils.addDays(actualPlannedUnloadDate, 3) // ❌ 与实际不符
```

#### 问题 2: 维护成本高

修改还箱日规则需要同时修改 3 处代码，容易遗漏导致行为不一致。

#### 问题 3: 测试分散

每处实现都需要独立测试，难以保证覆盖所有边界场景。

## 三、共享基础函数规范

### 3.1 函数签名

```typescript
/**
 * 共享还箱日计算基础函数
 *
 * @param options 计算参数
 * @param options.unloadDate 卸柜日 (Date 或 YYYY-MM-DD 字符串)
 * @param options.unloadMode 卸柜模式 ('Live load' | 'Drop off')
 * @param options.existingReturnDate 现有还箱日 (可选，用于 Drop off 模式)
 * @param options.oldUnloadDate 原有卸柜日 (可选，用于保持间隔)
 * @returns 计算结果
 */
function calculatePlannedReturnDateBasic(options: {
  unloadDate: Date | string
  unloadMode: 'Live load' | 'Drop off'
  existingReturnDate?: Date | string | null
  oldUnloadDate?: Date | string | null
}): {
  returnDate: Date
  explanation: string
}
```

### 3.2 计算规则详解

#### 规则 1: Live load 模式

```
还箱日 = 卸柜日
explanation = "Live load: 还箱日=卸柜日"
```

#### 规则 2: Drop off 模式 - 有历史数据且保持间隔

```
如果 existingReturnDate 存在 且 oldUnloadDate 存在:
  deltaDays = existingReturnDate - oldUnloadDate  (计算原有间隔)
  calculatedReturn = unloadDate + deltaDays
  returnDate = max(unloadDate, calculatedReturn)  // 不能早于卸柜日
  explanation = "Drop off: 保持原有间隔(N天)"
```

#### 规则 3: Drop off 模式 - 有历史数据但不保持间隔

```
如果 existingReturnDate 存在 但 oldUnloadDate 不存在:
  returnDate = max(unloadDate, existingReturnDate)
  explanation = "Drop off: max(卸柜日, 原还箱日)"
```

#### 规则 4: Drop off 模式 - 无历史数据

```
returnDate = unloadDate + 1天
explanation = "Drop off: 无历史数据，默认卸柜日+1天"
```

### 3.3 时区处理规范

- **输入**: 接受 `Date` 对象或 `YYYY-MM-DD` 字符串
- **内部处理**: 统一转换为 UTC 日期（`setUTCHours(0, 0, 0, 0)`）
- **输出**: 返回 UTC `Date` 对象
- **说明**: 避免跨时区场景下出现日期偏移

```typescript
function computePlannedReturnFromUnload(
  finalUnloadYmd: string,
  unloadMode: string | undefined,
  existingReturnYmd?: string | Date | null
): string {
  const u = dayjs(finalUnloadYmd).startOf('day')
  const mode = String(unloadMode || 'Live load').trim()
  if (mode !== 'Drop off') {
    return u.format('YYYY-MM-DD')
  }
  if (existingReturnYmd != null && existingReturnYmd !== '') {
    const r = dayjs(existingReturnYmd).startOf('day')
    const later = r.isBefore(u) ? u : r
    return later.format('YYYY-MM-DD')
  }
  return u.add(1, 'day').format('YYYY-MM-DD')
}

import { calculatePlannedReturnDateBasic } from '@shared/returnDateCalculator'

function mergeReturnDateIntoUpdateData(
  container: Container,
  updateData: Record<string, string>,
  unloadMode: string | undefined
): void {
  if (!updateData.plannedUnloadDate) return

  const empty = container.emptyReturns?.[0]
  const existingReturn = empty?.plannedReturnDate || null
  const oldUnload = container.warehouseOperations?.[0]?.plannedUnloadDate || null

  const result = calculatePlannedReturnDateBasic({
    unloadDate: updateData.plannedUnloadDate,
    unloadMode: (unloadMode || 'Live load') as 'Live load' | 'Drop off',
    existingReturnDate: existingReturn,
    oldUnloadDate: oldUnload,
  })

  updateData.plannedReturnDate = result.returnDate.toISOString().split('T')[0]
  console.log('[mergeReturnDateIntoUpdateData]', result.explanation)
}
```

**收益**:

- 删除 103 行本地实现
- 规则与后端完全一致
- 自动支持"保持原有间隔"逻辑

```typescript
import { calculatePlannedReturnDateBasic } from '@shared/returnDateCalculator'

async calculatePlannedReturnDate(
  unloadDate: Date,
  unloadMode: string,
  truckingCompanyId: string,
  lastReturnDate?: Date,
): Promise<ReturnDateResult> {
  // Step 1: 获取历史数据
  const emptyReturn = await this.getExistingReturnDate(containerNumber)
  const oldUnload = await this.getOldUnloadDate(containerNumber)

  // Step 2: 调用共享基础函数计算基础还箱日
  const basicResult = calculatePlannedReturnDateBasic({
    unloadDate,
    unloadMode: unloadMode as 'Live load' | 'Drop off',
    existingReturnDate: emptyReturn?.plannedReturnDate,
    oldUnloadDate: oldUnload
  })

  // Step 3: 叠加能力检查逻辑
  const availableDate = await this.findEarliestAvailableReturnDate(
    truckingCompanyId,
    basicResult.returnDate,
    lastReturnDate
  )

  if (!availableDate) {
    return {
      returnDate: basicResult.returnDate,
      explanation: basicResult.explanation + ' (警告: 找不到有能力的日期)'
    }
  }

  const needsAdjustUnload = availableDate.getTime() !== basicResult.returnDate.getTime()

  return {
    returnDate: availableDate,
    adjustedUnloadDate: needsAdjustUnload ? availableDate : undefined,
    explanation: basicResult.explanation + ' (经能力调整后)'
  }
}
```

**收益**:

- 基础规则单一来源
- 能力检查逻辑独立清晰
- 易于单元测试

## 七、风险评估与应对

### 7.1 风险 1: 时区不一致

**风险描述**: 前端使用 dayjs (本地时区)，后端使用 UTC，可能导致跨时区场景下日期差1天。

**应对措施**:

- 共享函数内部统一使用 UTC (`setUTCHours(0, 0, 0, 0)`)
- 前端传入字符串时确保格式为 `YYYY-MM-DD`
- 添加时区相关的单元测试

### 7.2 风险 2: 前端包体积增加

**风险描述**: 引入 shared 模块可能增加前端打包体积。

**应对措施**:

- shared 模块仅包含纯函数，无外部依赖
- Vite Tree Shaking 会自动剔除未使用代码
- 监控打包前后体积变化

### 7.3 风险 3: 后端能力检查逻辑丢失

**风险描述**: 重构时可能误删能力检查逻辑。

**应对措施**:

- 明确分层：基础函数只负责计算，不负责校验
- 后端完整方法必须调用 `findEarliestAvailableReturnDate`
- 添加回归测试验证能力检查仍生效
