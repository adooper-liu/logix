## 🔧 关键技术决策

### 1. 分层架构设计

```
┌─────────────────────────────────────┐
│  shared/returnDateCalculator.ts     │  ← 纯计算、同步、无依赖
└──────────────┬──────────────────────┘
               │
    ┌──────────┼──────────┐
    ▼          ▼          ▼
  前端      后端排产    成本预估
  甘特图   (能力检查)  (简化估算)
```

### 2. 暂缓重构 SchedulingDateCalculator

**原因**:
- 当前实现已正确（Live load/Drop off + 能力检查 + 最晚还箱日约束）
- 职责不同：共享函数用于预览/估算，SchedulingDateCalculator 用于正式排产
- 强行统一会增加复杂度，收益有限
- 遵循 SKILL 原则："保持现状"

### 3. 成本预估场景简化

在 `evaluateTotalCost` 中，暂时不使用 `existingReturnDate` 参数：
- 成本预估是批量场景，不需要精确的历史数据
- 使用默认规则（Drop off: 卸+1天）足够准确
- 避免引入复杂的 container 查询逻辑

## ✨ 经验总结

### 成功经验

1. **先文档后代码**: 先生成完整技术方案，再实施重构，避免方向偏差
2. **分层架构清晰**: 基础规则与能力检查分离，职责明确
3. **单元测试先行**: 覆盖所有规则分支，保证重构安全性
4. **渐进式重构**: 从简单到复杂逐步推进
5. **遵循 SKILL 原则**: 简洁、真实、业务导向
6. **务实决策**: 暂缓不必要的重构，保持现状

### 遇到的问题

1. **路径别名配置**: tsx 和 Jest 都需要正确配置
2. **TypeScript 类型错误**: 原有代码存在类型问题（与本次重构无关）
3. **作用域问题**: `evaluateTotalCost` 中无法访问 `container` 变量

### 解决方案

1. **路径别名**: 
   - tsconfig.json: `"@shared/*": ["../shared/src/*"]`
   - vite.config.ts: `'@shared': path.resolve(__dirname, '../shared/src')`
   - jest.config.js: `'^@shared/(.*)$': '<rootDir>/../shared/src/$1'`

2. **作用域问题**: 
   - 成本预估场景不需要历史数据
   - 暂时移除 `existingReturnDate` 参数
   - 使用默认规则（Drop off: 卸+1天）
