## 三、手动验证场景

### 场景 1: 前端甘特图拖拽还箱日同步

#### 测试步骤

1. 打开浏览器访问 `http://localhost:5173`
2. 进入甘特图页面
3. 选择一个 **Drop off 模式** 的货柜
4. 记录当前的：
   - 计划卸柜日（例如：2026-04-10）
   - 计划还箱日（例如：2026-04-15）
   - 原有间隔 = 5天
5. 拖拽卸柜节点到新日期（例如：2026-04-12）
6. 观察还箱日是否自动调整为 2026-04-17（保持 5天间隔）
7. 打开浏览器控制台，查看日志：
   ```
   [mergeReturnDateIntoUpdateData] Drop off: 保持原有间隔(5天) -> 2026-04-17
   ```

#### 预期结果

| 测试场景          | 输入                                 | 预期输出          | 说明            |
| ----------------- | ------------------------------------ | ----------------- | --------------- |
| Live load 拖拽    | 卸柜日=2026-04-10                    | 还箱日=2026-04-10 | 还=卸           |
| Drop off 无历史   | 卸柜日=2026-04-10                    | 还箱日=2026-04-11 | 卸+1天          |
| Drop off 有历史   | 卸柜日=2026-04-10, 原还箱=2026-04-15 | 还箱日=2026-04-15 | max(卸, 原还箱) |
| Drop off 保持间隔 | 卸柜日=2026-04-10, 原间隔=5天        | 还箱日=2026-04-15 | 新卸+5天        |

### 场景 2: 后端成本预估准确性

#### 测试步骤

1. 调用成本优化 API（使用 Postman 或 curl）：

   ```bash
   curl -X POST http://localhost:3001/api/v1/scheduling/optimize-container \
     -H "Content-Type: application/json" \
     -d '{
       "containerNumber": "ECMU5399797",
       "strategy": "Drop off"
     }'
   ```

2. 检查响应中的 `plannedReturnDate` 字段

3. 对比重构前后的差异：
   - **重构前**: Drop off 模式还箱日 = 卸柜日 + 3天（硬编码错误）
   - **重构后**: Drop off 模式还箱日 = max(卸柜日, 原还箱日) 或 卸+1天

#### 预期结果

假设货柜数据：

- 卸柜日: 2026-04-10
- 原还箱日: 2026-04-15

**重构前**（错误）:

```json
{
  "plannedUnloadDate": "2026-04-10",
  "plannedReturnDate": "2026-04-13", // ❌ 卸+3天（硬编码）
  "totalCost": 1250.0
}
```

**重构后**（正确）:

```json
{
  "plannedUnloadDate": "2026-04-10",
  "plannedReturnDate": "2026-04-15", // ✅ max(卸, 原还箱)
  "totalCost": 1180.0 // 滞箱费减少
}
```

## 五、常见问题排查

### 问题 1: 前端报错 `Cannot find module '@shared/returnDateCalculator'`

**原因**: Vite 路径别名配置未生效

**解决**:

1. 检查 `frontend/vite.config.ts` 中是否有：
   ```typescript
   resolve: {
     alias: {
       '@shared': path.resolve(__dirname, '../shared/src'),
     },
   }
   ```
2. 重启前端开发服务器
3. 清除浏览器缓存

### 问题 2: 后端报错 `MODULE_NOT_FOUND: @shared/returnDateCalculator`

**原因**: tsconfig.json 路径映射不正确

**解决**:

1. 检查 `backend/tsconfig.json` 中是否有：
   ```json
   "paths": {
     "@shared/*": ["../shared/src/*"]
   }
   ```
2. 重启后端服务（tsx watch 会自动重载）

### 问题 3: 还箱日计算结果不符合预期

**排查步骤**:

1. 查看控制台日志中的 `explanation` 字段
2. 确认输入的 `unloadMode` 是否正确（'Live load' 或 'Drop off'）
3. 确认 `existingReturnDate` 和 `oldUnloadDate` 是否有值
4. 对照共享函数的计算规则验证
