## 一、项目背景

### 1.1 问题描述

项目中存在 **3处独立的还箱日计算逻辑**，导致：

1. **规则不一致**:
   - 前端: max(卸柜日, 原还箱日)
   - 后端智能排柜: 优先当天 → 卸+1 → 顺延
   - 成本优化服务: 硬编码 +3天（❌ 错误）

2. **维护成本高**: 修改规则需同时改 3 处代码

3. **Bug 风险**: 成本优化服务的硬编码 +3天 与实际业务不符

### 1.2 解决目标

1. 提取共享基础函数，统一还箱日计算规则
2. 修复成本优化服务的硬编码错误
3. 减少代码重复，降低维护成本
4. 建立可复用的日期计算模块

## 二、实施方案

### 2.1 架构设计

采用**分层架构**：

```
┌─────────────────────────────────────┐
│  shared/returnDateCalculator.ts     │  ← 共享基础函数（纯计算）
│  - calculatePlannedReturnDateBasic  │
└──────────────┬──────────────────────┘
               │
    ┌──────────┼──────────┐
    ▼          ▼          ▼
  前端      后端排产    成本预估
  甘特图   (能力检查)  (简化估算)
```

**核心原则**:

- 共享函数：纯计算、同步、无外部依赖
- 后端排产：在共享函数基础上叠加能力检查
- 前端/成本预估：直接调用共享函数

### 2.2 计算规则

| 模式              | 规则                           | 说明         |
| ----------------- | ------------------------------ | ------------ |
| Live load         | 还箱日 = 卸柜日                | 同日还箱     |
| Drop off 无历史   | 还箱日 = 卸柜日 + 1天          | 默认堆存1天  |
| Drop off 有历史   | 还箱日 = max(卸柜日, 原还箱日) | 不早于卸柜日 |
| Drop off 保持间隔 | 还箱日 = 新卸柜日 + 原有间隔   | 用户习惯保持 |

**修复前**（成本优化服务）:

```typescript

if (option.strategy === 'Drop off') {
  plannedReturnDate = dateTimeUtils.addDays(actualPlannedUnloadDate, 3)

```

**修复后**:

```typescript
// ✅ 使用共享基础函数，规则与实际业务一致
const returnResult = calculatePlannedReturnDateBasic({
  unloadDate: actualPlannedUnloadDate,
  unloadMode: option.strategy === 'Drop off' ? 'Drop off' : 'Live load',
  existingReturnDate: container.emptyReturns?.[0]?.plannedReturnDate,
})
const plannedReturnDate = returnResult.returnDate
```

## 六、决策记录

### 6.1 为什么暂缓重构 SchedulingDateCalculator？

根据 **SKILL 原则 - 简洁即美** 和 **代码重构决策经验**：

1. **当前实现已经正确**：
   - Live load / Drop off 规则清晰
   - 车队能力检查完整
   - 最晚还箱日约束生效

2. **职责不同**：
   - 共享函数：纯计算，适用于预览/估算
   - SchedulingDateCalculator：完整排产，包含能力检查

3. **强行统一会增加复杂度**：
   - 能力检查无法共享（需要 TypeORM）
   - 拆分两层增加调用链复杂度
   - 收益不足以覆盖风险

4. **遵循"保持现状"原则**：
   - 代码清晰、可维护、已测试通过
   - 无明显 Bug 或性能问题

**结论**: 暂不重构 `SchedulingDateCalculator`，保持当前实现。
