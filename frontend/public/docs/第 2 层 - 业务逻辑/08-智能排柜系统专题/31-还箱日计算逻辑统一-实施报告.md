## 二、核心成果

### 2.1 共享模块功能

```typescript
// shared/src/returnDateCalculator.ts

export function calculatePlannedReturnDateBasic(options: {
  unloadDate: Date | string
  unloadMode: 'Live load' | 'Drop off'
  existingReturnDate?: Date | string | null
  oldUnloadDate?: Date | string | null
}): {
  returnDate: Date
  explanation: string
}
```

**支持的规则**:

1. Live load: 还箱日 = 卸柜日
2. Drop off 无历史: 卸柜日 + 1天
3. Drop off 有历史: max(卸柜日, 原还箱日)
4. Drop off 保持间隔: 新卸柜日 + 原有间隔（不低于卸柜日）

## 三、关键技术决策

### 3.1 为什么选择同步纯函数？

**决策**: 共享基础函数设计为同步纯函数，不包含异步操作。

**理由**:

1. 前端需要快速响应 (<10ms)，不能等待数据库查询
2. 成本预估场景不需要能力检查，只需基础规则
3. 后端可以在调用基础函数后再叠加能力检查
4. 纯函数易于单元测试和调试

### 3.2 时区处理策略

**决策**: 内部统一使用 UTC，输入输出支持字符串和 Date 对象。

**实现**:

```typescript
// 标准化为 UTC Date
const result = new Date(Date.UTC(year, month - 1, day))
result.setUTCHours(0, 0, 0, 0)
```

**好处**:

- 避免跨时区场景下日期偏移
- 前后端行为一致
- 与数据库存储格式对齐

### 3.3 保持原有间隔的实现

**决策**: 通过 `oldUnloadDate` 参数可选支持"保持间隔"逻辑。

**算法**:

```typescript
deltaDays = existingReturnDate - oldUnloadDate // 计算原有间隔
calculatedReturn = unloadDate + deltaDays // 应用间隔
returnDate = max(unloadDate, calculatedReturn) // 不低于卸柜日
```

**适用场景**:

- 前端拖拽时保持用户原有的调度习惯
- 后端正式排产时可选择不启用（由业务决定）

#### 为什么暂缓重构 SchedulingDateCalculator？

根据 **SKILL 原则 - 简洁即美** 和 **代码重构决策经验**：

1. **当前实现已经正确**：`SchedulingDateCalculator.calculatePlannedReturnDate` 方法实现了完整的还箱日计算逻辑，包括：
   - Live load 模式：还 = 卸（同日）
   - Drop off 模式：优先当天 → 卸+1 → 顺延查找
   - 车队能力检查：查询 `ExtTruckingReturnSlotOccupancy` 表
   - 最晚还箱日约束：不超过 `lastReturnDate`

2. **与共享基础函数的职责不同**：
   - 共享函数 `calculatePlannedReturnDateBasic`：纯计算逻辑，无外部依赖，适用于预览/估算场景
   - `SchedulingDateCalculator`：完整排产逻辑，包含能力检查和数据库查询，适用于正式排产

3. **强行统一会导致复杂度增加**：
   - 若将能力检查逻辑纳入共享模块，会引入 TypeORM 依赖，前端无法使用
   - 若拆分为"基础规则 + 能力检查"两层，会增加调用链复杂度，但收益有限

4. **遵循"保持现状"原则**：
   - 当前代码清晰、可维护、已测试通过
   - 没有明显的 Bug 或性能问题
   - 重构带来的收益不足以覆盖风险

**结论**：暂不重构 `SchedulingDateCalculator`，保持当前实现。如果未来需要统一规则，可以考虑在注释中引用共享函数作为参考。
