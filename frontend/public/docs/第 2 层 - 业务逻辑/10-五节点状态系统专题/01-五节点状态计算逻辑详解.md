## 节点计算逻辑

### 1. 清关节点计算

#### 算法伪代码

```
FUNCTION calculateCustomsStatus(customsStatus):
  IF customsStatus IS NULL OR EMPTY:
    RETURN { kind: 'bad', text: '未清关' }
  
  SWITCH customsStatus:
    CASE 'COMPLETED':
      RETURN { kind: 'ok', text: '已完成' }
    CASE 'FAILED':
      RETURN { kind: 'bad', text: '失败' }
    CASE 'IN_PROGRESS':
      RETURN { kind: 'warn', text: '清关中' }
    CASE 'PENDING':
      RETURN { kind: 'warn', text: '待清关' }
    DEFAULT:
      RETURN { kind: 'warn', text: '未知状态' }
```

#### 测试场景

| 场景 | customsStatus | 预期 kind | 预期 text |
|------|---------------|-----------|-----------|
| 无数据 | null | bad | 未清关 |
| 无数据 | undefined | bad | 未清关 |
| 无数据 | '' | bad | 未清关 |
| 已完成 | 'COMPLETED' | ok | 已完成 |
| 清关中 | 'IN_PROGRESS' | warn | 清关中 |
| 失败 | 'FAILED' | bad | 失败 |
| 待清关 | 'PENDING' | warn | 待清关 |
| 未知状态 | 'UNKNOWN' | warn | 未知状态 |

### 2. 提柜节点计算

#### 算法伪代码

```
FUNCTION calculatePickupStatus(pickupDate, plannedPickupDate):
  // 优先级: 实际提柜日 > 计划提柜日 > 无数据
  
  IF pickupDate IS NOT NULL AND NOT EMPTY:
    RETURN { kind: 'ok', text: '已提柜' }
  
  IF plannedPickupDate IS NOT NULL AND NOT EMPTY:
    RETURN { kind: 'warn', text: '已计划提柜' }
  
  RETURN { kind: 'bad', text: '未计划提柜' }
```

#### 关键设计决策

**为什么有计划但未提显示为 "warn" 而不是 "ok"?**

- **业务含义**: "ok" 表示任务已完成,"warn" 表示进行中
- **用户期望**: 看到"已计划提柜"应该知道还需要跟进
- **风险管控**: 有计划但未执行可能存在延误风险

**错误示例**:
```typescript
// ❌ 错误: 有计划就显示完成
const pickup = plannedPickupDate || pickupDate ? 'ok' : 'bad'

// ✅ 正确: 区分三种状态
const pickup = pickupDate ? 'ok' : (plannedPickupDate ? 'warn' : 'bad')
```

#### 测试场景

| 场景 | pickupDate | plannedPickupDate | 预期 kind | 预期 text |
|------|------------|-------------------|-----------|-----------|
| 已提柜 | '2024-03-16' | '2024-03-15' | ok | 已提柜 |
| 已提柜(无计划) | '2024-03-16' | null | ok | 已提柜 |
| 已计划未提 | null | '2024-03-15' | warn | 已计划提柜 |
| 无数据 | null | null | bad | 未计划提柜 |
| 空字符串 | '' | '' | bad | 未计划提柜 |

### 3. 卸柜节点计算

#### 算法伪代码

```
FUNCTION calculateUnloadStatus(logisticsStatus):
  status = LOWERCASE(logisticsStatus OR '')
  
  // 已完成: 已卸柜或已还箱(还箱必然已卸柜)
  IF status IN ['unloaded', 'returned_empty']:
    RETURN { kind: 'ok', text: '已卸柜' }
  
  // 进行中: 已到港或已提柜
  IF status IN ['at_port', 'picked_up']:
    RETURN { kind: 'warn', text: '未卸柜' }
  
  // 未完成: 其他所有状态
  RETURN { kind: 'bad', text: '未卸柜' }
```

#### 状态映射关系

| logisticsStatus | 卸柜状态 | 说明 |
|-----------------|---------|------|
| not_shipped | bad | 未出运 |
| shipped | bad | 已出运 |
| in_transit | bad | 在途 |
| arrived_at_transit | bad | 已到中转港 |
| at_port / arrived_at_destination | warn | 已到目的港 |
| picked_up | warn | 已提柜 |
| unloaded | ok | 已卸柜 |
| returned_empty | ok | 已还箱(隐含已卸柜) |

|------|-----------------|-----------|-----------|
| 未出运 | 'not_shipped' | bad | 未卸柜 |
| 海运中 | 'shipped' | bad | 未卸柜 |
| 在途 | 'in_transit' | bad | 未卸柜 |
| 已到港 | 'at_port' | warn | 未卸柜 |

| 空字符串 | '' | bad | 未卸柜 |

### 4. 还箱节点计算

#### 算法伪代码

```
FUNCTION calculateEmptyReturnStatus(returnTime, logisticsStatus):
  status = LOWERCASE(logisticsStatus OR '')
  
  // 已完成: 有实际还箱时间
  IF returnTime IS NOT NULL AND NOT EMPTY:
    RETURN { kind: 'ok', text: '已还箱' }
  
  // 进行中: 已卸柜或已提柜,但还未还箱
  IF status IN ['unloaded', 'picked_up']:
    RETURN { kind: 'warn', text: '未还箱' }
  
  // 未完成: 其他所有状态
  RETURN { kind: 'bad', text: '未还箱' }
```

#### 业务规则

**为什么已卸柜/已提柜时显示 "warn" 而不是 "bad"?**

- **业务含义**: 货柜已经在仓库或运输途中,还箱是下一步操作
- **用户期望**: 看到"未还箱"但状态为黄色,知道这是正常流程中的等待状态
- **风险区分**: 
  - `warn` + "未还箱" = 正常等待还箱
  - `bad` + "未还箱" = 还未到还箱环节(如还在海上)

#### 测试场景

| 场景 | returnTime | logisticsStatus | 预期 kind | 预期 text |
|------|------------|-----------------|-----------|-----------|
| 已还箱 | '2024-03-20' | 'returned_empty' | ok | 已还箱 |
| 已还箱(状态滞后) | '2024-03-20' | 'unloaded' | ok | 已还箱 |
| 已卸柜未还箱 | null | 'unloaded' | warn | 未还箱 |
| 已提柜未还箱 | null | 'picked_up' | warn | 未还箱 |
| 未到港 | null | 'at_port' | bad | 未还箱 |
| 海运中 | null | 'shipped' | bad | 未还箱 |
| 无数据 | null | null | bad | 未还箱 |

### 5. 查验节点计算

#### 算法伪代码

```
FUNCTION calculateInspectionStatus(inspectionRequired):
  IF inspectionRequired IS TRUE:
    RETURN { kind: 'warn', text: '需查验', type: 'warning' }
  
  RETURN { kind: 'ok', text: '免查验', type: 'info' }
```

#### 特殊说明

**为什么查验只有两种状态?**

- 查验是布尔属性,不是流程节点
- 不需要"已完成查验"的状态,因为查验完成后会更新其他字段
- `inspectionRequired = true` 表示需要关注,所以用 `warn`

#### 测试场景

| 场景 | inspectionRequired | 预期 kind | 预期 text | 预期 type |
|------|--------------------|-----------|-----------|-----------|
| 需查验 | true | warn | 需查验 | warning |
| 免查验 | false | ok | 免查验 | info |
| 无数据 | null | ok | 免查验 | info |
| 无数据 | undefined | ok | 免查验 | info |

## 性能优化建议

### 1. 避免重复计算

```typescript
// ❌ 错误: 多次调用 toLowerCase()
const isUnloaded = row.logisticsStatus.toLowerCase() === 'unloaded'
const isReturned = row.logisticsStatus.toLowerCase() === 'returned_empty'

// ✅ 正确: 只调用一次
const s = String(row.logisticsStatus || '').toLowerCase()
const isUnloaded = s === 'unloaded'
const isReturned = s === 'returned_empty'
```

### 2. 提前返回

```typescript
// ✅ 优先判断最常见的情况
if (row.returnTime) return 'ok'  // 大部分已还箱的货柜直接返回
```

### 3. 缓存结果

如果同一行数据在页面中多次使用五节点状态,可以考虑缓存:

```typescript
const fiveNodeCache = new Map<string, FiveNodeResult>()

function getFiveNodeRowsCached(row: any, key: string) {
  if (!fiveNodeCache.has(key)) {
    fiveNodeCache.set(key, getFiveNodeRows(row))
  }
  return fiveNodeCache.get(key)!
}
```

## 常见错误与修复

### 错误 1: 提柜状态判断错误

**问题**: 有计划提柜日就显示"已完成"

**原因**:
```typescript
// ❌ 错误代码
const pickup = plannedPickupDate || pickupDate ? 'ok' : 'bad'
```

**修复**:
```typescript
// ✅ 正确代码
const pickup = pickupDate ? 'ok' : (plannedPickupDate ? 'warn' : 'bad')
```

---

### 错误 2: 卸柜状态未考虑已还箱的情况

**问题**: 已还箱的货柜卸柜状态显示"未卸柜"

**原因**:
```typescript
// ❌ 错误代码
if (logisticsStatus === 'unloaded') unload = 'ok'
```

**修复**:
```typescript
// ✅ 正确代码
if (['unloaded', 'returned_empty'].includes(s)) unload = 'ok'
```

---

### 错误 3: 未处理空字符串

**问题**: 空字符串导致判断异常

**原因**:
```typescript
// ❌ 错误代码
const s = row.logisticsStatus.toLowerCase()  // 如果为 null 会报错
```

**修复**:
```typescript
// ✅ 正确代码
const s = String(row.logisticsStatus || '').toLowerCase()
```
