## 📋 概述

飞驼（Freightower）API 是 LogiX 系统获取实时物流状态数据的核心渠道。本指南深度解析飞驼事件的产生、映射、同步全流程。

### 核心价值

1. **实时性**: API 自动同步，无需人工录入
2. **准确性**: 直接对接船公司和码头数据源
3. **完整性**: 覆盖从装船到还箱的全流程节点
4. **智能化**: 区分预计/实际事件，智能验证数据

**Token 获取流程**:
```typescript
// 使用 OAuth 时自动获取 Token
async function getToken(): Promise<string> {
  // 1. 检查缓存
  if (this.openApiTokenCache && !isExpired()) {
    return this.openApiTokenCache.token;
  }
  
  // 2. 调用认证接口
  const response = await axios.post('/auth/api/token', {
    clientId: CLIENT_ID,
    secret: CLIENT_SECRET
  });
  
  // 3. 缓存 Token
  this.openApiTokenCache = {
    token: response.data.access_token,
    expiresAt: Date.now() + response.data.expires_in * 1000
  };
  
  return response.data.access_token;
}
```

### 1.2 数据同步流程

#### 完整同步流程

```
1. 触发同步
   ↓
2. 构建查询请求
   - containerNo (必填)
   - billNo (从 process_sea_freight 补全)
   - carrierCode (从 shipping_company 补全)
   ↓
3. 调用飞驼 API
   ↓
4. 解析响应数据
   - 提取 status[] 数组
   - 映射为标准事件格式
   ↓
5. 写入原始数据表
   - ext_feituo_status_events
   - 保留所有原始字段
   ↓
6. 处理实际事件
   - 过滤 is_estimated=false 的事件
   - 更新 process_port_operations 核心字段
   ↓
7. 触发状态机重算
   - calculateLogisticsStatus()
   - 更新 biz_containers.logistics_status
   ↓
8. 触发后续动作
   - 滞港费重算
   - 预警检查
   - 前端通知
```

## 二、飞驼状态码映射体系

### 2.1 映射层次结构

```
飞驼状态码 (eventCode)
    ↓ FEITUO_STATUS_TO_CORE_FIELD_MAP
核心时间字段 (ata, eta, gate_in_time 等)
    ↓ 写入
process_port_operations 表字段
    ↓ 读取
calculateLogisticsStatus() 计算
    ↓ 返回
logistics_status (not_shipped, at_port 等)
```

#### 字段更新策略

```typescript
/**
 * 判断是否应该更新核心字段
 * @param statusCode 状态码
 * @param hasOccurred 是否已发生 (true=实际，false=预计)
 */
function shouldUpdateCoreField(
  statusCode: string,
  hasOccurred: boolean
): boolean {
  // 原则 1: 只有实际发生的事件才能更新核心字段
  if (!hasOccurred) {
    return false;
  }
  
  // 原则 2: 只有映射表中存在的状态码才能更新
  const coreFieldName = FEITUO_STATUS_TO_CORE_FIELD_MAP[statusCode];
  if (!coreFieldName) {
    return false;
  }
  
  // 原则 3: 特殊状态码额外检查
  if (statusCode === 'ETA') {
    // ETA 只在没有 ATA 时才更新（作为预测参考）
    return !hasActualArrival();
  }
  
  return true;
}
```

### 3.2 智能验证机制

#### ATA 更新前验证

```typescript
/**
 * ATA 更新前智能验证
 * 防止错误数据污染数据库
 */
interface ATAValidationContext {
  ata: Date;                    // 新的 ATA 时间
  eta?: Date | null;            // 当前 ETA
  shipDate?: Date | null;       // 开船日
  logisticsStatus?: string;     // 当前物流状态
  previousPort?: {              // 上一港信息
    portSequence: number;
    atd?: Date | null;          // 上一港离港时间
  };
  truckingTransport?: {         // 拖车信息
    pickupDate?: Date | null;
  };
}

function validateATA(context: ATAValidationContext): ValidationResult {
  const { ata, eta, shipDate, logisticsStatus, previousPort } = context;
  
  // 验证 1: ATA 不能早于开船日
  if (shipDate && ata < shipDate) {
    return {
      valid: false,
      reason: 'ATA 不能早于开船日',
      suggestion: '检查 ATA 时间或开船日是否正确'
    };
  }
  
  // 验证 2: ATA 不能早于上一港 ATD（如果有）
  if (previousPort?.atd && ata < previousPort.atd) {
    return {
      valid: false,
      reason: 'ATA 不能早于上一港离港时间',
      suggestion: '检查港口顺序或时间是否正确'
    };
  }
  
  // 验证 3: ATA 不能晚于提柜日（如果已有）
  if (context.truckingTransport?.pickupDate && ata > context.truckingTransport.pickupDate) {
    return {
      valid: false,
      reason: 'ATA 不能晚于提柜日',
      suggestion: '检查时间逻辑是否正确'
    };
  }
  
  // 验证 4: 如果已经是 in_transit 或 at_port，需要特别小心
  if (['in_transit', 'at_port'].includes(logisticsStatus || '')) {
    // 检查时间差异是否过大
    if (eta && Math.abs(ata.getTime() - eta.getTime()) > 7 * 24 * 60 * 60 * 1000) {
      return {
        valid: false,
        reason: 'ATA 与 ETA 差异超过 7 天',
        suggestion: '请人工核实数据准确性'
      };
    }
  }
  
  return { valid: true };
}
```

### 3.3 字段更新流程

#### 单个事件更新流程

```typescript
async function updateCoreFieldFromEvent(
  containerNumber: string,
  event: FeituoEvent
): Promise<void> {
  // Step 1: 检查是否应该更新
  if (!shouldUpdateCoreField(event.statusCode, event.hasOccurred)) {
    logger.debug(`跳过事件 ${event.statusCode} (预计事件或无映射)`);
    return;
  }
  
  // Step 2: 获取核心字段名
  const coreFieldName = FEITUO_STATUS_TO_CORE_FIELD_MAP[event.statusCode];
  if (!coreFieldName) {
    logger.warn(`未找到状态码 ${event.statusCode} 的映射`);
    return;
  }
  
  // Step 3: 智能验证（针对特殊字段）
  if (coreFieldName === 'ata') {
    const validation = await validateATA({
      ata: new Date(event.eventTime),
      // ... 其他验证参数
    });
    
    if (!validation.valid) {
      logger.warn(`ATA 验证失败：${validation.reason}`);
      // 记录警告，但不阻止更新（信任数据源）
    }
  }
  
  // Step 4: 确定目标表和字段
  const { tableName, poFieldName } = resolveTableAndField(coreFieldName);
  
  // Step 5: 更新数据库
  await updateDatabase(tableName, poFieldName, event);
  
  // Step 6: 自动同步到关联表（如果需要）
  if (coreFieldName === 'gate_out_time') {
    await syncToTruckingPickupDate(containerNumber, event.eventTime);
  }
  
  logger.info(`成功更新 ${coreFieldName} = ${event.eventTime}`);
}
```

## 四、is_estimated 核心原则

### 4.1 基本原则

#### 数据准确性优先

```
实际发生数据 (is_estimated=false) > 预计数据 (is_estimated=true)
```

**禁止反向操作**: 不允许用预计数据覆盖实际数据

---

#### 单一数据源

```
process_port_operations (权威源)
    ↓ 自动同步
process_trucking_transport.pickup_date
```

**禁止**: 预计事件不能写入权威源

---

#### 等待实际事件

如果当前只有预计事件:
- ✅ 保持核心字段为 NULL
- ✅ 等待 `is_estimated=false` 的实际事件
- ❌ 不用预计时间填充
