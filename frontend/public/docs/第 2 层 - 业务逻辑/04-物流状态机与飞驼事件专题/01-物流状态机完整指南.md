## 📋 概述

物流状态机是 LogiX 系统的核心引擎，负责根据货柜的实际业务节点自动计算和更新物流状态。

### 核心价值

1. **自动化**: 无需手动维护状态，系统根据业务数据自动计算
2. **准确性**: 基于真实业务事件（ATA、提柜日、还箱日等）而非人工判断
3. **一致性**: 前后端使用同一套状态机逻辑，确保数据展示一致
4. **可追溯**: 每个状态变更都有明确的触发字段和原因说明

## 一、7 层简化状态

### 1.1 状态定义与流转

```
not_shipped → shipped → in_transit → at_port → picked_up → unloaded → returned_empty
未出运        已出运      在途         已到港     已提柜     已卸柜     已还箱
```

| 状态代码         | 中文名称 | 英文说明       | 业务含义                         |
| ---------------- | -------- | -------------- | -------------------------------- |
| `not_shipped`    | 未出运   | Not Shipped    | 货物尚未装船，仍在仓库或工厂     |
| `shipped`        | 已出运   | Shipped        | 货物已装船，但尚未开航           |
| `in_transit`     | 在途     | In Transit     | 船舶正在航行中                   |
| `at_port`        | 已到港   | At Port        | 船舶已到达目的港（含可提货状态） |
| `picked_up`      | 已提柜   | Picked Up      | 货柜已从码头提出，正在运输途中   |
| `unloaded`       | 已卸柜   | Unloaded       | 货柜已到达仓库并卸货             |
| `returned_empty` | 已还箱   | Returned Empty | 空箱已归还堆场                   |

---

### 1.2 状态类型分类

| 分类           | 状态                                                        | 说明         | Element Plus 类型            |
| -------------- | ----------------------------------------------------------- | ------------ | ---------------------------- |
| **初始状态**   | `not_shipped`                                               | 尚未开始运输 | `info` (灰色)                |
| **进行中状态** | `shipped`, `in_transit`, `at_port`, `picked_up`, `unloaded` | 运输流程中   | `primary`/`warning`/`danger` |
| **完成状态**   | `returned_empty`                                            | 整个流程结束 | `success` (绿色)             |

## 二、状态计算逻辑

### 2.1 核心规则（优先级从高到低）

状态计算采用**优先级匹配**机制，一旦满足某个条件即返回对应状态，不再继续向下匹配。

```typescript
export const calculateLogisticsStatus = (
  container: Container,
  portOperations: PortOperation[],
  seaFreight?: SeaFreight,
  truckingTransport?: TruckingTransport,
  warehouseOperation?: WarehouseOperation,
  emptyReturn?: EmptyReturn
): LogisticsStatusResult => {
  // 优先级 1: 还空箱日期（最高优先级）
  if (emptyReturn?.returnTime) {
    return {
      status: 'returned_empty',
      triggerFields: { returnTime: emptyReturn.returnTime },
      reason: '已还空箱',
    }
  }

  // 优先级 2: 仓库卸柜（WMS 已确认）
  if (isWmsConfirmed(warehouseOperation)) {
    return {
      status: 'unloaded',
      triggerFields: {
        wmsStatus: warehouseOperation?.wmsStatus,
        wmsConfirmDate: warehouseOperation?.wmsConfirmDate,
      },
      reason: '仓库已卸柜（WMS 确认）',
    }
  }

  // 优先级 3: 拖车提柜日期
  if (truckingTransport?.pickupDate) {
    return {
      status: 'picked_up',
      triggerFields: { pickupDate: truckingTransport.pickupDate },
      reason: '已提柜',
    }
  }

  // 优先级 4: 目的港有实际到港时间
  const destWithArrival = portOperations.find(po => po.portType === 'destination' && po.ata)
  if (destWithArrival) {
    return {
      status: 'at_port',
      currentPortType: 'destination',
      latestPortOperation: destWithArrival,
      triggerFields: { ata: destWithArrival.ata },
      reason: '目的港已到港（ATA）',
    }
  }

  // 优先级 4a: 目的港有可提货时间（飞驼 PCAB/AVLE/AVAIL 触发）
  const destWithAvailable = portOperations.find(
    po => po.portType === 'destination' && po.availableTime
  )
  if (destWithAvailable) {
    return {
      status: 'at_port',
      currentPortType: 'destination',
      latestPortOperation: destWithAvailable,
      triggerFields: { availableTime: destWithAvailable.availableTime },
      reason: '目的港可提货（PCAB/AVLE/AVAIL）',
    }
  }

  // 优先级 5: 中转港 ATA + 海运出运
  const transitWithArrival = portOperations.find(po => po.portType === 'transit' && po.ata)
  if (transitWithArrival && seaFreight?.shipmentDate) {
    return {
      status: 'in_transit',
      currentPortType: 'transit',
      latestPortOperation: transitWithArrival,
      triggerFields: {
        ata: transitWithArrival.ata,
        shipmentDate: seaFreight.shipmentDate,
      },
      reason: '中转港已到港 + 已出运',
    }
  }

  // 优先级 6: 有海运记录（已实际出运）
  if (seaFreight?.shipmentDate) {
    return {
      status: 'shipped',
      triggerFields: { shipmentDate: seaFreight.shipmentDate },
      reason: '已出运',
    }
  }

  // 默认状态（无出运记录）
  return {
    status: 'not_shipped',
    triggerFields: {},
    reason: '无出运记录',
  }
}
```

### 2.2 优先级详解

#### 优先级 1: 还空箱（最高优先级）

**判断条件**:

```typescript
emptyReturn?.returnTime !== null && emptyReturn?.returnTime !== undefined
```

**数据来源**:

- 表：`process_empty_return`
- 字段：`return_time`
- 来源：飞驼 RCVE 状态码（实际事件）

**业务含义**:

- 一旦空箱归还，整个运输流程结束
- 这是最终状态，优先级最高

---

#### 优先级 2: 仓库卸柜

**判断条件**:

```typescript
isWmsConfirmed(warehouseOperation)
// warehouseOperation.wmsStatus === 'WMS 已完成'
// OR warehouseOperation.ebsStatus === '已入库'
// OR warehouseOperation.wmsConfirmDate !== null
```

**数据来源**:

- 表：`process_warehouse_operations`
- 字段：`wms_status`, `ebs_status`, `wms_confirm_date`
- 来源：WMS 系统同步

**业务含义**:

- 货柜已到达仓库并完成卸货
- WMS 确认具有权威性

---

#### 优先级 3: 拖车提柜

**判断条件**:

```typescript
truckingTransport?.pickupDate !== null && truckingTransport?.pickupDate !== undefined
```

**数据来源**:

- 表：`process_trucking_transport`
- 字段：`pickup_date`
- 来源：飞驼 GATE_OUT/STCS 状态码（实际事件）

**业务含义**:

- 货柜已从码头提出
- 正在运往仓库的途中

---

#### 优先级 4: 目的港实际到港 (ATA)

**判断条件**:

```typescript
portOperations.some(po => po.portType === 'destination' && po.ata)
```

**数据来源**:

- 表：`process_port_operations`
- 字段：`ata` (Actual Time of Arrival)
- 来源：飞驼 ATA/ARRIVE/BDAR 等状态码（实际事件）

**业务含义**:

- 船舶已到达目的港
- 可能还在码头等待提货

---

#### 优先级 4a: 目的港可提货（特殊场景）

**判断条件**:

```typescript
portOperations.some(po => po.portType === 'destination' && po.availableTime)
```

**数据来源**:

- 表：`process_port_operations`
- 字段：`available_time`
- 来源：飞驼 PCAB/AVLE/AVAIL 状态码

**业务含义**:

- 即使没有 ATA，只要有可提货时间也视为已到港
- 这是飞驼 API 特有的状态码触发

---

#### 优先级 5: 中转港 + 已出运

**判断条件**:

```typescript
portOperations.some(po => po.portType === 'transit' && po.ata) && seaFreight?.shipmentDate
```

**数据来源**:

- 表：`process_port_operations` (中转港 ATA)
- 表：`process_sea_freight` (出运日期)

**业务含义**:

- 船舶已到达中转港并且已经出运
- 正在前往目的港的途中

---

#### 优先级 6: 已出运

**判断条件**:

```typescript
seaFreight?.shipmentDate !== null && seaFreight?.shipmentDate !== undefined
```

**数据来源**:

- 表：`process_sea_freight`
- 字段：`shipment_date`
- 来源：海运提单数据

**业务含义**:

- 货物已装船出运
- 但尚未到达任何港口

---

#### 优先级 7: 未出运（默认状态）

**判断条件**:

- 以上所有条件都不满足

**业务含义**:

- 货物尚未开始运输
- 可能在仓库备货阶段

## 三、状态计算实战场景

### 3.1 场景 1: 正常流程

**场景描述**: 货柜按正常流程从起运港到目的港再到仓库

**数据演变**:

```
1. 初始状态: not_shipped
   - 无任何业务数据

2. 装船出运: shipped
   - process_sea_freight.shipment_date = 2026-03-01

3. 航行中: in_transit
   - shipment_date = 2026-03-01
   - process_port_operations (transit).ata = 2026-03-10

4. 到达目的港: at_port
   - process_port_operations (destination).ata = 2026-03-15

5. 提柜: picked_up
   - process_trucking_transport.pickup_date = 2026-03-18

6. 卸柜: unloaded
   - process_warehouse_operations.wms_confirm_date = 2026-03-19

7. 还箱: returned_empty
   - process_empty_return.return_time = 2026-03-20
```

---

### 3.2 场景 2: 中转港经停

**场景描述**: 船舶先到达中转港，再到达目的港

**关键点**:

- 中转港 ATA 不会直接触发 `at_port` 状态
- 必须结合 `shipment_date` 才能触发 `in_transit`
- 只有目的港 ATA 才能触发 `at_port`

**数据示例**:

```typescript
// 中转港记录
{
  portType: 'transit',
  ata: '2026-03-10',
  portName: 'Singapore'
}

// 目的港记录
{
  portType: 'destination',
  ata: '2026-03-15',
  portName: 'Los Angeles'
}

// 状态计算过程:
// 1. 有 transit.ata + shipment_date → in_transit
// 2. 有 destination.ata → at_port (覆盖 in_transit)
```

---

### 3.3 场景 3: PCAB 可提货状态（无 ATA）

**场景描述**: 码头系统显示可提货，但没有 ATA 记录

**业务背景**:

- 某些港口不记录 ATA，只记录可提货状态
- 飞驼提供 PCAB/AVLE/AVAIL 状态码

**数据处理**:

```typescript
// 飞驼事件
{
  statusCode: 'PCAB',
  eventTime: '2026-03-15 10:00:00',
  isEstimated: false
}

// 映射逻辑
FEITUO_STATUS_TO_CORE_FIELD_MAP['PCAB'] = 'availableTime'

// 写入数据库
process_port_operations.availableTime = '2026-03-15 10:00:00'

// 状态计算
// 虽然没有 ATA，但有 availableTime → at_port
```

---

### 3.4 场景 4: is_estimated 预计事件处理

**场景描述**: 飞驼返回预计事件和实际事件混合

**错误案例** (历史教训):

```typescript
// ❌ 错误逻辑（已修复）
const statuses = [
  { statusCode: 'ATA', eventTime: '2026-03-20', isEstimated: false }, // 实际到港
  { statusCode: 'ATA', eventTime: '2026-03-22', isEstimated: true }, // 预计到港（未来）
]

// 错误行为：预计事件覆盖了实际数据
// 结果：ata 从 2026-03-20 变成 2026-03-22（错误！）
```

**正确逻辑** (当前实现):

```typescript
// ✅ 正确逻辑
for (const event of feituoEvents) {
  // 只处理实际事件
  if (event.isEstimated === true || event.hasOccurred === false) {
    continue // 跳过预计事件
  }

  // 更新核心字段
  await updateCoreField(event)
}

// 结果：预计事件被忽略，实际数据保持不变
```

## 四、状态机触发更新机制

### 4.1 触发时机

以下场景会触发状态机重新计算：

1. **飞驼 API 同步后**
   - 外部数据服务更新核心字段后
   - 调用 `recalculateLogisticsStatus(containerNumber)`

2. **Excel 导入后**
   - 飞驼导入服务处理完状态数组后
   - 调用 `updateContainerStatus(containerNumber)`

3. **手动编辑后**
   - 用户在前端修改了关键日期字段
   - 后端保存时触发

4. **定时调度器**
   - 每 5 分钟批量检查需要更新的货柜
   - 调用 `containerStatusScheduler`

---

### 4.2 更新流程

```
1. 接收触发事件（如飞驼同步完成）
   ↓
2. 获取货柜及相关数据
   - Container (基本信息)
   - PortOperation[] (港口操作记录)
   - SeaFreight (海运信息)
   - TruckingTransport (拖车信息)
   - WarehouseOperation (仓库信息)
   - EmptyReturn (还箱信息)
   ↓
3. 调用 calculateLogisticsStatus()
   ↓
4. 比较新旧状态
   - 如果状态改变 → 记录审计日志
   - 如果 ganttDerived 改变 → 更新甘特图
   ↓
5. 写回数据库
   - biz_containers.logistics_status
   - biz_containers.gantt_derived
   ↓
6. 触发后续动作
   - 通知前端更新（WebSocket）
   - 触发预警检查
   - 触发滞港费重算
```

## 五、状态机与其他模块的协同

### 5.1 与飞驼 API 的协同

```
飞驼 API → status[] → is_estimated 检查
                      ↓
                更新核心字段
                      ↓
                触发状态机重算
                      ↓
                更新 logistics_status
```

**关键原则**:

- ✅ 只有实际事件 (`is_estimated=false`) 才能更新核心字段
- ❌ 预计事件不能触发状态变更
- ⚠️ 状态机只认核心字段，不直接读取飞驼事件

---

### 5.2 与滞港费计算的协同

```
状态机计算 → at_port
              ↓
        触发滞港费计算
              ↓
        查询 ext_demurrage_standards
              ↓
        计算免费期和计费期
              ↓
        生成滞港费记录
```

**依赖关系**:

- 滞港费计算前必须先判定是否到达目的港
- 使用 `calculateLogisticsStatus()` 的快照做预判
- 只有 `at_port` 状态才启动滞港费计算

---

### 5.3 与甘特图的协同

```
状态机计算 → 同时生成 ganttDerived
              ↓
        包含:
        - plannedPickupDate
        - plannedReturnDate
        - lastFreeDate
        - 各节点实际时间
              ↓
        前端渲染甘特图
```

**ganttDerived 结构**:

```typescript
interface GanttDerived {
  plannedPickupDate?: string // 计划提柜日
  plannedReturnDate?: string // 计划还箱日
  lastFreeDate?: string // 最晚免费日
  actualPickupDate?: string // 实际提柜日
  actualReturnDate?: string // 实际还箱日
  nodes: GanttNode[] // 节点详情
}
```

## 六、常见问题排查

### 6.1 问题 1: 状态一直是 not_shipped

**现象**: 货柜已装船，但状态一直显示 `not_shipped`

**排查步骤**:

```sql
-- 1. 检查是否有 shipment_date
SELECT container_number, shipment_date
FROM process_sea_freight
WHERE container_number = 'xxxxx';

-- 2. 如果为 NULL，说明缺少出运数据
-- 解决方案：补充海运数据或等待飞驼同步
```

**常见原因**:

- ❌ 缺少 `shipment_date`
- ❌ 飞驼 API 未同步
- ❌ Excel 导入遗漏

---

### 6.2 问题 2: 已到港但显示 in_transit

**现象**: 船舶已到达目的港，但状态仍是 `in_transit`

**排查步骤**:

```sql
-- 1. 检查目的港 ATA
SELECT container_number, port_type, ata
FROM process_port_operations
WHERE container_number = 'xxxxx'
  AND port_type = 'destination';

-- 2. 如果 ata 为 NULL
-- 检查飞驼事件
SELECT event_code, event_time, is_estimated
FROM ext_feituo_status_events
WHERE container_number = 'xxxxx'
  AND event_code IN ('ATA', 'ARRIVE', 'BDAR');
```

**常见原因**:

- ❌ `ata` 字段为空
- ❌ 飞驼事件标记为预计 (`is_estimated=true`)
- ❌ 港口类型错误（transit vs destination）

---

### 6.3 问题 3: 已提柜但显示 at_port

**现象**: 货柜已从码头提出，但状态仍是 `at_port`

**排查步骤**:

```sql
-- 1. 检查提柜日期
SELECT container_number, pickup_date
FROM process_trucking_transport
WHERE container_number = 'xxxxx';

-- 2. 如果 pickup_date 为 NULL
-- 检查飞驼 GATE_OUT 事件
SELECT event_code, event_time, is_estimated
FROM ext_feituo_status_events
WHERE container_number = 'xxxxx'
  AND event_code IN ('GATE_OUT', 'STCS', 'GTOT');
```

**常见原因**:

- ❌ `pickup_date` 字段为空
- ❌ GATE_OUT 事件未同步
- ❌ 事件标记为预计

---

### 6.4 问题 4: 预计事件覆盖了实际数据

**现象**: 实际发生的日期被未来的预计时间覆盖

**排查步骤**:

```sql
-- 1. 查询原始事件
SELECT
  container_number,
  event_code,
  event_time,
  is_estimated,
  data_source
FROM ext_feituo_status_events
WHERE container_number = 'xxxxx'
ORDER BY created_at DESC;

-- 2. 查询核心字段
SELECT
  container_number,
  ata,
  gate_out_time,
  available_time
FROM process_port_operations
WHERE container_number = 'xxxxx';
```

**常见原因**:

- ❌ is_estimated 检查逻辑失效
- ❌ 错误的"最终状态例外"逻辑（历史问题，已修复）

**解决方案**:

- ✅ 修复 is_estimated 检查逻辑
- ✅ 删除历史错误数据
- ✅ 重新导入正确数据

## 八、最佳实践

### 8.1 开发建议

1. **始终使用状态机计算状态**
   - ✅ 调用 `calculateLogisticsStatus()`
   - ❌ 不要手动判断和赋值

2. **优先检查实际事件**
   - ✅ `if (!status.isEstimated)`
   - ❌ `if (status.isEstimated || isFinalStatus)` (错误逻辑)

3. **保持数据一致性**
   - ✅ 前后端使用相同的计算逻辑
   - ❌ 前后端分别实现不同的判断逻辑

4. **记录状态变更原因**
   - ✅ 返回 `triggerFields` 和 `reason`
   - ❌ 只返回状态代码
