# 批量免费日更新 - 自动分批处理优化记录

## 概述

**优化日期**: 2026-04-08  
**优化内容**: 为手动批量免费日更新功能添加自动分批处理机制，避免 HTTP 超时  
**影响范围**: 后端服务、前端调用、API 接口

---

## 一、问题背景

### 1.1 原有问题

当需要处理的货柜数量超过 500 个时：

- 单次批量更新耗时过长（可能超过 5-10 分钟）
- 容易导致 HTTP 请求超时（默认 30s-60s）
- 用户无法看到进度，只能等待完成或失败
- 数据库连接长时间占用，影响其他操作

### 1.2 业务场景

典型场景：

- 导入 2000+ 货柜后，需要批量计算免费日
- 滞港费标准表更新后，需要重新计算所有受影响货柜
- 系统迁移或数据修复后，需要补全历史数据

---

## 二、解决方案

### 2.1 核心思路

实现**类懒加载的自动分批处理机制**：

- 每次调用只处理一小批货柜（默认 100 个）
- 通过多次调用逐步处理完所有货柜
- 用户可控，可随时停止
- 避免 HTTP 超时，提升用户体验

### 2.2 设计原则

#### **原则一：智能判断是否需要分批**

```typescript
if (!autoBatch || totalCount <= limitLastFree) {
  // 不需要分批：直接处理前 limitLastFree 个
  processCount = Math.min(totalCount, limitLastFree)
} else {
  // 需要分批：只处理 batchSize 个
  processCount = batchSize
}
```

#### **原则二：每次只处理一小批**

- `limitLastFree`: 500（单次调用允许的最大值）
- `batchSize`: 100（实际每批处理的数量）
- 执行流程：第 1 次处理 0-100，第 2 次处理 101-200，依此类推

#### **原则三：透明化处理，用户无感知**

- 前端：用户点击按钮，系统自动判断是否分批
- 后端：记录日志，显示已处理数量和剩余数量
- 用户：可多次点击，逐步处理完所有货柜

## 三、技术实现

### 3.1 后端改动

**文件**: `backend/src/services/demurrage.service.ts`

#### **改动 1: 方法签名增加参数**

```typescript
async batchWriteBackComputedDates(options?: {
  limitLastFree?: number;
  limitLastReturn?: number;
  autoBatch?: boolean;   // ✅ 新增：是否启用自动分批，默认 true
  batchSize?: number;    // ✅ 新增：每批大小，默认 100
})
```

#### **改动 2: 查询时不加 LIMIT**

```typescript
// ❌ 旧逻辑：直接在 SQL 中加 LIMIT
const lastFreeRows = await this.containerRepo.query(
  `(SELECT ...) UNION (...) LIMIT ${Math.max(1, limitLastFree)}`
)

// ✅ 新逻辑：先查全部，再在内存中切片
const allLastFreeRows = await this.containerRepo.query(
  `(SELECT ...) UNION (...)` // 不加 LIMIT
)
const allLastFreeNumbers = allLastFreeRows.map(r => r.container_number)

// 根据配置决定处理方式
if (!autoBatch || allLastFreeNumbers.length <= limitLastFree) {
  lastFreeNumbers = allLastFreeNumbers.slice(0, limitLastFree)
} else {
  lastFreeNumbers = allLastFreeNumbers.slice(0, batchSize)
}
```

#### **改动 3: 记录处理进度**

```typescript
let lastFreeProcessedTotal = 0

for (const cn of lastFreeNumbers) {
  try {
    const { result } = await this.calculateForContainer(cn)
    if (result?.writeBack?.lastFreeDateWritten) lastFreeWritten++
    lastFreeProcessedTotal++ // ✅ 累加计数
  } catch (e) {
    logger.warn(`Failed for ${cn}:`, e)
  }
}

// 记录剩余数量
if (autoBatch && allLastFreeNumbers.length > limitLastFree) {
  const remaining = allLastFreeNumbers.length - lastFreeProcessedTotal
  logger.info(
    `[Demurrage] LFD batch complete: processed=${lastFreeProcessedTotal}, remaining=${remaining}. ` +
      `Call API again to process next batch.`
  )
}
```

#### **改动 4: 返回真实的处理数量**

```typescript
return {
  lastFreeWritten,
  lastReturnWritten,
  lastFreeProcessed: lastFreeProcessedTotal, // ✅ 实际处理数量
  lastReturnProcessed: lastReturnProcessedTotal, // ✅ 实际处理数量
}
```

## 四、使用示例

### 4.1 场景：处理 1250 个货柜

**步骤**:

1. **第 1 次点击**

   ```
   后端日志:
   [Demurrage] Found 1250 containers needing LFD update
   [Demurrage] Auto-batching LFD: total=1250, batchSize=100
   [Demurrage] Processing first LFD batch: 100 containers
   [Demurrage] LFD batch complete: processed=100, remaining=1150

   前端提示:
   "计算完成：最晚提柜日 85/100，最晚还箱日 62/100"
   ```

2. **第 2 次点击**

   ```
   后端日志:
   [Demurrage] Found 1150 containers needing LFD update
   [Demurrage] Auto-batching LFD: total=1150, batchSize=100
   [Demurrage] Processing first LFD batch: 100 containers
   [Demurrage] LFD batch complete: processed=100, remaining=1050

   前端提示:
   "计算完成：最晚提柜日 82/100，最晚还箱日 58/100"
   ```

3. **重复点击，直到剩余数量为 0**
   ```
   预计次数: 1250 / 100 = 13 次（最后一次处理 50 个）
   预计时间: 13 * 20秒 = 约 4 分钟
   ```

---

### 4.2 场景：禁用自动分批（兼容旧逻辑）

如果希望保持原有行为（一次性处理最多 500 个）：

```typescript
const result = await containerService.batchWriteBackDemurrageDates({
  limitLastFree: 500,
  limitLastReturn: 500,
  autoBatch: false, // ❌ 禁用自动分批
})
```

**效果**:

- 如果总数 <= 500，处理全部
- 如果总数 > 500，只处理前 500 个
- 不会在日志中提示剩余数量

## 五、优势与局限

### 5.1 优势

✅ **避免 HTTP 超时**

- 每次请求响应时间控制在 10-30 秒
- 不会因为处理大量货柜导致超时

✅ **用户可见进度**

- 通过日志和返回结果，用户可看到已处理数量和剩余数量
- 可预估还需要多少次操作

✅ **可随时中断**

- 用户可在任何时候停止点击
- 已处理的货柜不会回滚

✅ **向后兼容**

- 默认启用自动分批（`autoBatch: true`）
- 可通过参数禁用，保持原有行为

✅ **保护机制完善**

- 不覆盖手工维护的 LFD（`lastFreeDateSource = 'manual'`）
- 单个货柜失败不影响整体流程

---

### 5.2 局限性

⚠️ **不是真正的"全自动"**

- 需要用户多次点击才能处理完所有货柜
- 不是后台自动循环处理所有批次

**原因**:

- 避免长时间占用 HTTP 连接
- 让用户有控制权，可随时停止
- 便于监控每一批的执行情况

**未来优化方向**:

- 实现 WebSocket 推送进度
- 提供"后台任务"接口，异步处理所有批次
- 前端显示进度条和预计完成时间

---

⚠️ **分批边界可能变化**

- 每次调用时，都会重新查询需要处理的货柜
- 如果在两次调用之间，某些货柜的状态发生了变化，可能会导致：
  - 某些货柜被跳过
  - 某些货柜被重复处理（但有保护机制）

**建议**:

- 尽量在短时间内连续点击
- 或者在业务低峰期执行，避免并发修改

---

⚠️ **日志中的"剩余数量"仅供参考**

- 基于本次调用时的查询结果
- 下次调用时可能有新的货柜符合条件，或某些货柜已被定时任务处理

## 六、测试建议

### 6.1 单元测试

测试以下场景：

1. `autoBatch = false` 时，按原有逻辑处理
2. `autoBatch = true` 且总数 <= limit 时，处理全部
3. `autoBatch = true` 且总数 > limit 时，只处理 batchSize 个
4. 返回的 `lastFreeProcessed` 等于实际处理数量

### 6.2 集成测试

测试以下场景：

1. 模拟 1000 个货柜，分 10 次调用，验证每次处理 100 个
2. 验证日志中正确显示剩余数量
3. 验证中途停止后，再次调用能继续处理
4. 验证手工维护的 LFD 不会被覆盖

### 6.3 性能测试

测试以下指标：

1. 处理 100 个货柜的平均耗时
2. 处理 1000 个货柜的总耗时（分 10 次）
3. 数据库连接池的使用情况
4. HTTP 请求的响应时间分布
