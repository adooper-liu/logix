# 免费日更新三种方式逻辑梳理与对比矩阵

## 概述

LogiX 系统提供**三种免费日更新方式**，用于计算和写回货柜的最晚提柜日（`last_free_date`）和最晚还箱日（`last_return_date`）。这三种方式分别适用于不同的业务场景，具有不同的触发条件、写回规则和使用限制。

---

## 一、三种更新方式总览

| 更新方式         | 英文名称                | 触发方式              | 适用场景               | API 端点                                             |
| ---------------- | ----------------------- | --------------------- | ---------------------- | ---------------------------------------------------- |
| **定时自动更新** | Scheduled Update        | 定时任务（每 6 小时） | 全量货柜批量更新       | `DemurrageWriteBackScheduler`                        |
| **手动批量更新** | Manual Batch Update     | 前端按钮触发          | 用户主动触发批量更新   | `POST /api/v1/demurrage/batch-write-back`            |
| **单条货柜更新** | Single Container Update | 前端表格操作列按钮    | 针对单个货柜的精准更新 | `POST /api/v1/demurrage/write-back/:containerNumber` |

---

## 二、核心概念说明

### 2.1 最晚提柜日 (last_free_date)

**定义**: 滞港费免费期的截止日期，超过此日期将开始收取滞港费。

**计算公式**:

```
last_free_date = 基准日 + (免费天数 - 1)天
```

**基准日优先级**（从高到低）:

1. 修正 ETA (`revised_eta_dest_port`)
2. 原始 ETA (`eta_dest_port`)
3. 实际到港 ATA (`ata_dest_port`)
4. 实际卸船日 (`dest_port_unload_date`)

**免费天数来源**: `ext_demurrage_standards` 滞港费标准表中的 `free_days` 字段

**示例**:

- ETA = 2026-02-16
- 免费天数 = 7 天（来自滞港费标准表）
- last_free_date = 2026-02-22（2026-02-16 + 6 天）

**存储位置**: `process_port_operations.last_free_date`

---

### 2.2 最晚还箱日 (last_return_date)

**定义**: 滞箱费免费期的截止日期，超过此日期将开始收取滞箱费。

**计算公式**:

```
last_return_date = 起算日 + 免费用箱天数
```

**起算日优先级**:

1. 实际提柜日 (`process_trucking_transport.pickup_date`)
2. 最晚提柜日 (`process_port_operations.last_free_date`)

**免费用箱天数来源**: `ext_demurrage_standards` 滞箱费标准表中 Detention 类型的 `free_days` 字段

**示例**:

- last_free_date = 2026-02-22
- 免费用箱天数 = 7 天
- last_return_date = 2026-03-01（2026-02-22 + 7 天）

**存储位置**: `process_empty_return.last_return_date`

---

### 2.3 计算模式 (calculation_mode)

系统根据货柜的物流状态自动判断使用哪种计算模式：

| 模式                    | 触发条件                   | 说明                        |
| ----------------------- | -------------------------- | --------------------------- |
| **forecast** (预测模式) | 未到港、无 ATA、无实际卸船 | 基于 ETA 预测，每日滚动更新 |
| **actual** (实际模式)   | 已到港（有 ATA）或已卸船   | 基于实际时间，更准确        |

**判断依据**: 调用 [`calculateLogisticsStatus`](./05-state-machine/04-calculateLogisticsStatus详解.md) 函数，如果状态为 `AT_PORT` 或之后环节则为 actual，否则为 forecast。

### 3.2 写回条件对比（核心差异）

#### **定时自动更新 & 手动批量更新**（严格限制）

这两种方式使用相同的写回逻辑 `writeBackComputedDatesIfNeeded()`，具有严格的写回条件：

**最晚提柜日 (LFD) 写回条件**:

```typescript
// 条件 1: 无实际提柜日时才写回
if (!pickupDateActual && computedLastFreeDate) {
  // 条件 2: 只覆盖 computed 来源或空值，保留手工维护的值
  const canOverwrite = !destPort.lastFreeDateSource || destPort.lastFreeDateSource === 'computed'
  if (canOverwrite) {
    // 执行写回
  }
}
```

**关键限制**:

1. ✅ **无实际提柜日**时才写回 LFD
   - 原因: 有实际提柜日后，滞港费已封顶，LFD 不再变化
2. ✅ **只覆盖 computed 来源**或空值
   - 原因: 保留用户手工维护的 LFD，避免被自动计算覆盖
3. ✅ **同时更新 trucking 表**
   - 同步写入 `process_trucking_transport.last_pickup_date`

**最晚还箱日 (LRD) 写回条件**:

```typescript
// 条件 1: 仅 actual 模式且有实际提柜日时才计算
if (calculationMode === 'actual' && pickupDateActual && computedLastReturnDate) {
  // 条件 2: 无实际还箱日时才写回
  if (!returnTime) {
    // 执行写回
  }
}
```

**关键限制**:

1. ✅ **仅 actual 模式**才写回 LRD
   - 原因: forecast 模式下无法准确预测还箱时间
2. ✅ **有实际提柜日**时才计算
   - 原因: LRD 从提柜日起算，无提柜日无法计算
3. ✅ **无实际还箱日**时才写回
   - 原因: 已还箱后 LRD 不再有意义

---

#### **单条货柜更新**（宽松条件）

单条更新使用独立的写回逻辑 `writeBackSingleContainerFreeDates()`，条件更为宽松：

**最晚提柜日 (LFD) 写回条件**:

```typescript
// 只要计算出了 LFD，就尝试写回（不检查是否有实际提柜日）
if (computedLastFreeDate && destPort) {
  // 仍然保留对 manual 来源的保护
  const canOverwrite = !destPort.lastFreeDateSource || destPort.lastFreeDateSource === 'computed'
  if (canOverwrite) {
    // 执行写回
  }
}
```

**关键特点**:

1. ✅ **不限制实际提柜日**
   - 即使已有实际提柜日，也会重新计算并写回 LFD
   - 用途: 修正历史数据或重新计算
2. ✅ **仍保护 manual 来源**
   - 如果用户手工维护了 LFD（`lastFreeDateSource = 'manual'`），不会覆盖
3. ✅ **同步 trucking 表**
   - 同样会更新 `process_trucking_transport.last_pickup_date`

**最晚还箱日 (LRD) 写回条件**:

```typescript
// 只要计算出了 LRD，就尝试写回（不限制 actual 模式）
if (computedLastReturnDate) {
  // 允许覆盖旧值，确保人工触发可对齐到最新计算结果
  let emptyReturn = await this.emptyReturnRepo.findOne({ where: { containerNumber } })
  if (!emptyReturn) {
    // 新建记录
  } else {
    // 直接覆盖（不检查是否有实际还箱日）
    await this.emptyReturnRepo.update(
      { containerNumber },
      { lastReturnDate: computedLastReturnDate }
    )
  }
}
```

**关键特点**:

1. ✅ **不限制 actual 模式**
   - forecast 模式下也会计算并写回 LRD
2. ✅ **允许覆盖旧值**
   - 即使已有 LRD，也会用最新计算结果覆盖
   - 用途: 确保人工触发时能对齐到最新计算结果
3. ✅ **支持新建记录**
   - 如果 `process_empty_return` 中无记录，会自动创建

### 3.3 代码实现对比

#### **定时/批量更新的写回逻辑**

**文件**: `backend/src/services/demurrage.service.ts`  
**方法**: `writeBackComputedDatesIfNeeded()` (第 3419-3510 行)

```typescript
private async writeBackComputedDatesIfNeeded(
  containerNumber: string,
  pickupDateActual: Date | null,
  returnTime: Date | null,
  computedLastFreeDate: Date | null,
  computedLastReturnDate: Date | null,
  calculationMode: 'actual' | 'forecast'
): Promise<{ lastFreeDateWritten: boolean; lastReturnDateWritten: boolean }> {

  // LFD 写回：无实际提柜日 + 非 manual 来源

    if (canOverwrite) {
      await this.portOpRepo.update(

        {
          lastFreeDate: computedLastFreeDate,

        { lastPickupDate: computedLastFreeDate }
      );
    }
  }

  if (calculationMode === 'actual' && pickupDateActual && computedLastReturnDate && !returnTime) {
    // 写回逻辑...
  }

  return { lastFreeDateWritten, lastReturnDateWritten };
}
```

#### **单条更新的写回逻辑**

**文件**: `backend/src/services/demurrage.service.ts`  
**方法**: `writeBackSingleContainerFreeDates()` (第 3316-3395 行)

```typescript
private async writeBackSingleContainerFreeDates(
  containerNumber: string,
  computedLastFreeDate: Date | null,
  computedLastReturnDate: Date | null,
  lastFreeDateMode: 'actual' | 'forecast'
): Promise<{ lastFreeDateWritten: boolean; lastReturnDateWritten: boolean }> {

  // LFD 写回：不检查实际提柜日，只保护 manual 来源
  if (computedLastFreeDate && destPort) {
    const canOverwrite = !destPort.lastFreeDateSource || destPort.lastFreeDateSource === 'computed';
    if (canOverwrite) {
      await this.portOpRepo.update(
        { id: destPort.id },
        {
          lastFreeDate: computedLastFreeDate,
          lastFreeDateMode,
          lastFreeDateSource: 'computed'
        }
      );
      // 同步 trucking 表
      await this.truckingRepo.update(
        { containerNumber },
        { lastPickupDate: computedLastFreeDate }
      );
    }
  }

  // LRD 写回：不限制 actual 模式，允许覆盖旧值
  if (computedLastReturnDate) {
    let emptyReturn = await this.emptyReturnRepo.findOne({ where: { containerNumber } });
    if (!emptyReturn) {
      // 新建记录
      emptyReturn = this.emptyReturnRepo.create({
        containerNumber,
        lastReturnDate: computedLastReturnDate
      });
      await this.emptyReturnRepo.save(emptyReturn);
    } else {
      // 直接覆盖（不检查是否有实际还箱日）
      await this.emptyReturnRepo.update(
        { containerNumber },
        { lastReturnDate: computedLastReturnDate }
      );
    }
  }

  return { lastFreeDateWritten, lastReturnDateWritten };
}
```

## 四、使用场景与建议

### 4.1 定时自动更新

**适用场景**:

- ✅ 日常数据维护，保持全量货柜的免费日数据最新
- ✅ ETA 变更后自动更新 forecast 模式的 LFD
- ✅ ATA 到港后自动从 forecast 切换到 actual 模式并重算

**优点**:

- 自动化，无需人工干预
- 保证数据一致性
- 及时反映 ETA 变化

**缺点**:

- 写回条件严格，某些特殊情况无法更新
- 无法针对性处理问题货柜

**配置**:

- 定时任务类: `DemurrageWriteBackScheduler`
- 默认频率: 每 6 小时执行一次
- 可通过环境变量调整: `DEMURRAGE_WRITE_BACK_INTERVAL_HOURS`

---

### 4.2 手动批量更新

**适用场景**:

- ✅ 批量修复数据问题（如导入错误后重新计算）
- ✅ 紧急更新大量货柜的免费日
- ✅ 滞港费标准表更新后，重新计算受影响货柜

**优点**:

- 用户可控，按需触发
- **✅ 支持自动分批处理**：当货柜数量 > limit 时，自动分批次执行，避免超时
- 可限制处理数量，避免长时间运行
- 与定时任务使用相同逻辑，安全可靠

**缺点**:

- 仍需等待批量处理完成
- 写回条件严格，可能遗漏特殊情况

**前端入口**:

- 位置: 货柜列表页面顶部工具栏
- 按钮: "免费日更新"
- 确认对话框: 显示计算逻辑说明
- **✅ 自动分批配置**:
  - `limitLastFree`: 500（LFD 最大处理数量）
  - `limitLastReturn`: 500（LRD 最大处理数量）
  - `autoBatch`: true（启用自动分批）
  - `batchSize`: 100（每批处理 100 个货柜）
- **工作原理**: 当需要处理的货柜数 > 500 时，自动只处理前 100 个，并在日志中提示剩余数量。用户可多次点击按钮，逐步处理完所有货柜。

**API 调用**:

```typescript

  limitLastFree: 500,      // LFD 单次批次上限
  limitLastReturn: 500,    // LRD 单次批次上限
  autoBatch: true,         // ✅ 启用自动分批
  batchSize: 100           // 每批处理 100 个
}
```

**返回结果示例**:

## 五、自动分批处理机制（新增）

### 5.1 问题背景

**原有问题**:

- 当需要处理的货柜数量 > 500 时，单次批量更新可能耗时过长
- 可能导致 HTTP 请求超时（默认 30s-60s）
- 用户无法看到进度，只能等待完成或失败

**解决方案**: 实现类懒加载的自动分批处理机制

---

### 5.2 核心设计原则

#### **原则一：智能判断是否需要分批**

```typescript
// 伪代码逻辑
if (!autoBatch || totalCount <= limitLastFree) {
  // 不需要分批：直接处理前 limitLastFree 个
  processCount = Math.min(totalCount, limitLastFree)
} else {
  // 需要分批：只处理 batchSize 个
  processCount = batchSize
}
```

**判断条件**:

1. `autoBatch = false`: 禁用自动分批，按原有逻辑处理
2. `totalCount <= limitLastFree`: 总数未超限，无需分批
3. 其他情况: 启用自动分批，每批处理 `batchSize` 个

---

#### **原则二：每次只处理一小批**

**配置参数**:

- `limitLastFree`: 500（单次调用允许的最大值）
- `batchSize`: 100（实际每批处理的数量）

**执行流程**:

```
第 1 次调用: 处理 0-100 号货柜
第 2 次调用: 处理 101-200 号货柜
第 3 次调用: 处理 201-300 号货柜
...
直到所有货柜处理完毕
```

**优点**:

- ✅ 每次请求响应时间可控（约 10-30 秒）
- ✅ 避免 HTTP 超时
- ✅ 用户可看到进度（通过日志或返回结果）
- ✅ 可随时中断，已处理的不会回滚

---

#### **原则三：透明化处理，用户无感知**

**前端行为**:

- 用户点击"免费日更新"按钮
- 系统自动判断是否需要分批
- 如果需要分批，只处理第一批
- 返回结果中显示已处理数量和剩余数量

**后端日志**:

```
[Demurrage] Found 1250 containers needing LFD update
[Demurrage] Auto-batching LFD: total=1250, batchSize=100
[Demurrage] Processing first LFD batch: 100 containers
[Demurrage] LFD batch complete: processed=100, remaining=1150. Call API again to process next batch.
```

**用户操作**:

- 第一次点击: 处理 100 个，提示"还有 1150 个待处理"
- 第二次点击: 再处理 100 个，提示"还有 1050 个待处理"
- ...
- 最后一次点击: 处理剩余的 50 个，提示"全部处理完成"

---

### 5.3 技术实现细节

#### **后端实现** (`demurrage.service.ts`)

**关键改动**:

1. **查询时不加 LIMIT**

   ```typescript
   // ❌ 旧逻辑：直接在 SQL 中加 LIMIT
   const rows = await query(`SELECT ... LIMIT ${limitLastFree}`)

   // ✅ 新逻辑：先查全部，再在内存中切片
   const allRows = await query(`SELECT ...`) // 不加 LIMIT
   const allNumbers = allRows.map(r => r.container_number)

   if (autoBatch && allNumbers.length > limitLastFree) {
     // 只取前 batchSize 个
     numbersToProcess = allNumbers.slice(0, batchSize)
   } else {
     // 取前 limitLastFree 个
     numbersToProcess = allNumbers.slice(0, limitLastFree)
   }
   ```

2. **记录处理进度**

   ```typescript
   let lastFreeProcessedTotal = 0

   for (const cn of lastFreeNumbers) {
     try {
       const { result } = await this.calculateForContainer(cn)
       if (result?.writeBack?.lastFreeDateWritten) lastFreeWritten++
       lastFreeProcessedTotal++ // ✅ 累加计数
     } catch (e) {
       logger.warn(`Failed for ${cn}:`, e)
     }
   }

   // 记录剩余数量
   if (autoBatch && allLastFreeNumbers.length > limitLastFree) {
     const remaining = allLastFreeNumbers.length - lastFreeProcessedTotal
     logger.info(`LFD batch complete: processed=${lastFreeProcessedTotal}, remaining=${remaining}`)
   }
   ```

3. **返回真实的处理数量**
   ```typescript
   return {
     lastFreeWritten,
     lastReturnWritten,
     lastFreeProcessed: lastFreeProcessedTotal, // ✅ 实际处理数量
     lastReturnProcessed: lastReturnProcessedTotal, // ✅ 实际处理数量
   }
   ```

---

```typescript
  const payload = result.data
  ElMessage.success(
    `计算完成：最晚提柜日 ${payload.lastFreeWritten}/${payload.lastFreeProcessed}，` +
      `最晚还箱日 ${payload.lastReturnWritten}/${payload.lastReturnProcessed}`
  )
}
```

**用户体验**:

- 点击按钮后，loading 状态持续约 10-30 秒
- 完成后显示"已处理 100 个，成功写回 85 个"
- 如果还有剩余，用户可再次点击继续处理

### 5.4 与定时任务的对比

| 维度             | 定时任务          | 手动批量更新（自动分批） |
| ---------------- | ----------------- | ------------------------ |
| **触发方式**     | 自动（每 6 小时） | 手动（用户点击）         |
| **处理范围**     | 全量货柜          | 分批处理，每次 100 个    |
| **是否需要干预** | ❌ 否             | ✅ 是（需多次点击）      |
| **适用场景**     | 日常维护          | 紧急修复、数据补全       |
| **超时风险**     | 低（后台运行）    | 低（分批处理）           |
| **可见性**       | 日志              | 前端提示 + 日志          |

**建议**:

- 日常使用定时任务即可，无需手动干预
- 只有在特殊情况下（如导入大量数据后）才使用手动批量更新
- 手动批量更新适合需要立即生效的场景

### 5.5 最佳实践

#### **场景一：导入 2000 个货柜后需要更新免费日**

**步骤**:

#### **场景二：定时任务已经运行，但想立即看到结果**

**步骤**:

2. 如果急需结果，可手动点击"免费日更新"

4. 由于有保护机制（不覆盖 manual 来源），不会造成数据冲突

### 5.6 注意事项

#### **⚠️ 警告 1: 不是真正的"全自动"分批**

**说明**:

- 当前实现是"半自动"：每次调用只处理一批
- 需要用户多次点击才能处理完所有货柜
- 不是后台自动循环处理所有批次

**原因**:

- 避免长时间占用 HTTP 连接
- 让用户有控制权，可随时停止
- 便于监控每一批的执行情况

**未来优化**:

- 可考虑实现 WebSocket 推送进度
- 或者提供"后台任务"接口，异步处理所有批次

---

#### **⚠️ 警告 2: 分批边界可能变化**

**说明**:

- 每次调用时，都会重新查询需要处理的货柜
- 如果在两次调用之间，某些货柜的状态发生了变化（如被手工修改），可能会导致：
  - 某些货柜被跳过
  - 某些货柜被重复处理（但有保护机制，不会覆盖 manual 来源）

**建议**:

- 尽量在短时间内连续点击，减少状态变化的可能性
- 或者先在业务低峰期执行，避免并发修改

---

#### **⚠️ 警告 3: 日志中的"剩余数量"仅供参考**

**说明**:

- 日志中的 `remaining` 是基于本次调用时的查询结果
- 如果下次调用时有新的货柜符合条件，实际剩余数量可能更多
- 反之，如果有货柜被处理或状态变化，实际剩余数量可能更少

**示例**:

```
第 1 次调用: Found 1250, processed 100, remaining 1150
第 2 次调用: Found 1200, processed 100, remaining 1100  // 可能有 50 个已被定时任务处理
```

## 六、特殊场景处理

### 5.1 手工维护的免费日保护

**机制**: 通过 `last_free_date_source` 字段区分来源

| 来源类型   | 说明         | 是否会被自动覆盖 |
| ---------- | ------------ | ---------------- |
| `computed` | 系统自动计算 | ✅ 是            |
| `manual`   | 用户手工维护 | ❌ 否（受保护）  |
| `null`     | 初始状态     | ✅ 是            |

**手工维护入口**:

- 前端表格操作列 → "LFD 手工维护"
- 弹窗输入新的 LFD 日期
- 后端设置 `lastFreeDateSource = 'manual'`

**保护逻辑**:

```typescript
const canOverwrite = !destPort.lastFreeDateSource || destPort.lastFreeDateSource === 'computed'
if (!canOverwrite) {
  logger.info(`[Demurrage] Skipped write back (manual source preserved)`)
}
```

---

### 5.2 forecast 与 actual 模式切换

**切换时机**:

- 当货柜从"未出运"变为"已到港"时
- 由 `calculateLogisticsStatus` 函数判断

**切换影响**:

- **forecast → actual**:
  - 基准日从 ETA 变为 ATA
  - LFD 可能发生变化（如果 ATA ≠ ETA）
  - 开始可以计算 LRD（如果有实际提柜日）
- **actual → forecast**:
  - 理论上不会发生（状态不可逆）
  - 如果 ATA 被误删除，会回退到 forecast

**模式标记**:

- 字段: `process_port_operations.last_free_date_mode`
- 值: `'forecast'` 或 `'actual'`
- 用途: 追溯 LFD 的计算来源

---

### 5.3 实际提柜日对写回的影响

**关键规则**:

```
有实际提柜日 → LFD 不再写回（滞港费已封顶）
无实际提柜日 → LFD 可以写回（仍在计费期）
```

**业务逻辑**:

- 实际提柜后，滞港费截止日固定为提柜日
- LFD 仅用于判断是否产生滞港费，不再变化
- 因此无需频繁更新，节省数据库写入

**例外情况**:

- 单条货柜更新不受此限制
- 可用于修正错误的实际提柜日

---

### 5.4 同步更新 Trucking 表

**同步字段**:

- `process_trucking_transport.last_pickup_date` = `process_port_operations.last_free_date`

**同步时机**:

- 所有三种更新方式都会同步
- 确保排产系统能看到最新的 LFD

**同步逻辑**:

```typescript
const tt = await this.truckingRepo.findOne({ where: { containerNumber } })
if (tt) {
  await this.truckingRepo.update({ containerNumber }, { lastPickupDate: computedLastFreeDate })
}
```

**注意事项**:

- 只在 `trucking_transport` 记录存在时同步
- 如果不存在，不会自动创建（需先有排产记录）

## 七、常见问题与排查

### 6.1 为什么我的 LFD 没有被更新？

**可能原因**:

1. **已有实际提柜日**
   - 检查: `SELECT pickup_date FROM process_trucking_transport WHERE container_number = 'xxx'`
   - 解决: 这是正常行为，LFD 在有提柜日后不再更新
   - 如需强制更新: 使用单条货柜更新

2. **手工维护的 LFD**
   - 检查: `SELECT last_free_date_source FROM process_port_operations WHERE container_number = 'xxx'`
   - 如果值为 `'manual'`: 系统保护手工值，不会覆盖
   - 解决: 如需重新计算，先清除手工标记或改用单条更新

3. **未匹配到滞港费标准**
   - 检查: 目的港、船公司、货代是否在标准表中配置
   - 解决: 补充标准表数据

4. **缺少基准日（ETA/ATA/卸船日）**
   - 检查: `SELECT eta, ata, dest_port_unload_date FROM process_port_operations WHERE container_number = 'xxx'`
   - 解决: 补充缺失的时间字段

---

### 6.2 为什么 LRD 一直没有值？

**可能原因**:

1. **仍处于 forecast 模式**
   - LRD 仅在 actual 模式下计算
   - 检查: `SELECT logistics_status FROM biz_containers WHERE container_number = 'xxx'`
   - 如果状态不是 `at_port` 或之后，说明未到港

2. **无实际提柜日**
   - LRD 从提柜日起算
   - 检查: `SELECT pickup_date FROM process_trucking_transport WHERE container_number = 'xxx'`
   - 解决: 录入实际提柜日

3. **已有实际还箱日**
   - 已还箱后不再更新 LRD
   - 检查: `SELECT return_time FROM process_empty_return WHERE container_number = 'xxx'`
   - 这是正常行为

---

### 6.3 如何验证计算是否正确？

**步骤**:

1. **查看计算结果**

   ```typescript
   // 调用单条更新 API，查看返回的 calculationDates
   POST /api/v1/demurrage/calculate/:containerNumber
   ```

2. **检查关键字段**

   ```sql
   -- 查看 LFD 相关信息
   SELECT
     container_number,
     last_free_date,
     last_free_date_mode,
     last_free_date_source,
     eta,
     revised_eta,
     ata,
     dest_port_unload_date
   FROM process_port_operations
   WHERE container_number = 'xxx';

   -- 查看 LRD 相关信息
   SELECT
     container_number,
     last_return_date,
     return_time
   FROM process_empty_return
   WHERE container_number = 'xxx';

   -- 查看提柜信息
   SELECT
     container_number,
     pickup_date,
     planned_pickup_date,
     last_pickup_date
   FROM process_trucking_transport
   WHERE container_number = 'xxx';
   ```

3. **手动验算**
   - 确认基准日（ETA/ATA/卸船日）
   - 查询匹配的滞港费标准（免费天数）
   - 计算: `基准日 + (免费天数 - 1)`
   - 对比数据库中的 `last_free_date`

## 十、总结

### 9.1 三种方式的选择指南

| 场景             | 推荐方式     | 理由                 |
| ---------------- | ------------ | -------------------- |
| 日常数据维护     | 定时自动更新 | 自动化，无需人工干预 |
| ETA 频繁变更     | 定时自动更新 | 及时反映变化         |
| 批量数据修复     | 手动批量更新 | 用户可控，可限制数量 |
| 标准表更新后重算 | 手动批量更新 | 按需触发，避免等待   |
| 问题排查         | 单条货柜更新 | 即时反馈，便于调试   |
| 个别货柜修正     | 单条货柜更新 | 精准处理，不影响其他 |
| 开发测试         | 单条货柜更新 | 快速验证，灵活可控   |

---

### 9.2 核心要点回顾

1. **写回条件的差异是关键**
   - 定时/批量: 严格限制，保护数据稳定性
   - 单条: 宽松条件，适合问题修复

2. **手工维护值的保护机制**
   - `last_free_date_source = 'manual'` 不会被自动覆盖
   - 三种方式都遵守此规则

3. **forecast 与 actual 模式的自动切换**
   - 由 `calculateLogisticsStatus` 判断
   - 影响 LFD 的基准日和 LRD 的可计算性

4. **同步更新 trucking 表**
   - 确保排产系统能看到最新的 LFD
   - 三种方式都会同步

5. **性能与可靠性的平衡**
   - 批量处理采用串行方式，避免数据库压力
   - 单个失败不影响整体流程
   - 完善的日志记录便于排查
