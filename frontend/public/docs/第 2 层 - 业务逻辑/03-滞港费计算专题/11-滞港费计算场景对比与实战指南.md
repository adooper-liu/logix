## 一、Storage Charge 深度解析

### 1.1 Storage Charge 是什么？

**堆存费 (Storage Charge)** 是集装箱在港区或堆场存放期间产生的费用，与 Demurrage（滞港费）可能同时发生，但计费逻辑不同。

**关键特征**:
- **计费对象**: 集装箱在堆场的存放
- **起算日**: ATA/卸船日/ETA（根据标准）
- **截止日**: 提柜日（实际或计划）
- **免费期**: 通常较短（3-7 天）
- **费率**: 可能高于滞港费

### 1.2 Storage vs Demurrage 对比

| 维度 | Storage (堆存费) | Demurrage (滞港费) |
|------|------------------|-------------------|
| **收费主体** | 码头/堆场经营方 | 船公司/港口当局 |
| **计费原因** | 集装箱占用堆场资源 | 货物占用港区资源 |
| **起算日** | ATA/卸船日/ETA | ATA/ETA/卸船日 |
| **截止日** | 提柜日 | 提柜日 |
| **免费期** | 较短（3-7 天） | 较长（5-10 天） |
| **费率水平** | 较高（USD 100-200/天） | 中等（USD 80-150/天） |
| **是否依赖提柜日** | ✅ **是** | ❌ 否（可用 Today） |

### 1.3 Storage Charge 的两种计算模式

Storage Charge 最复杂的特点是它有**两种子模式**：

#### 模式 A: actual 区间（已有 ATA/卸船日）

```typescript
// 触发条件
storageUsesActualInterval = 
  calculationMode === 'actual' || 
  (calculationMode === 'forecast' && hasStorageActualStart);

// hasStorageActualStart 判断
const hasStorageActualStart = useDischargeStorage
  ? !!params.calculationDates.dischargeDate  // 按卸船：只需卸船日
  : !!(params.calculationDates.ataDestPort ?? params.calculationDates.dischargeDate);
```

**示例**:
```
场景：ATA 已到，但未提柜
ATA: 2024-04-01
计划提柜日：2024-04-08
Today: 2024-04-06
免费天数：5 天

起算日 = 2024-04-01 (ATA)
最晚提柜日 = 2024-04-05
截止日 = max(2024-04-06, 2024-04-08) = 2024-04-08
计费天数 = 2024-04-08 - 2024-04-05 = 3 天
```

#### 模式 B: forecast 区间（无 ATA/卸船日）

```typescript
// 触发条件
else {
  // 纯预测模式：无 ATA/卸船日
  if (!plannedPickupDate) {
    skippedItems.push({
      reasonCode: 'missing_planned_pickup_date',
      reason: '堆存 (预测) 需计划提柜日（planned_pickup_date）'
    });
    continue;
  }
}
```

**示例**:
```
场景：未到港，只有 ETA
修正 ETA: 2024-04-01
计划提柜日：2024-04-08
Today: 2024-04-06
免费天数：5 天

起算日 = 2024-04-01 (修正 ETA)
最晚提柜日 = 2024-04-05
截止日 = max(2024-04-06, 2024-04-08) = 2024-04-08
计费天数 = 2024-04-08 - 2024-04-05 = 3 天
```

### 1.4 没有 planned_pickup_date 会不会计算 Storage Charge？

**答案：分情况**

#### 情况 1: forecast 模式，无 ATA/卸船日，无 planned_pickup_date

```
❌ **不会计算**

原因：缺少计划提柜日，无法确定截止日
错误码：missing_planned_pickup_date
错误信息：'堆存 (预测) 需计划提柜日（planned_pickup_date）'
```

**代码逻辑**:
```typescript
if (isStorage && !storageUsesActualInterval) {
  // forecast 区间模式
  if (!plannedPickupDate) {
    skippedItems.push({
      reasonCode: 'missing_planned_pickup_date',
      reason: '堆存 (预测) 需计划提柜日（planned_pickup_date）'
    });
    continue; // 跳过，不计算
  }
}
```

---

#### 情况 2: forecast 模式，已有 ATA/卸船日，无 planned_pickup_date

```
✅ **会计算**（使用 actual 区间模式）

原因：已有 ATA/卸船日，切换到实际区间模式
起算日 = ATA/卸船日
截止日 = Today（因为无计划提柜日）
```

**示例**:
```
ATA: 2024-04-01
计划提柜日：null
Today: 2024-04-06
免费天数：5 天

// ✅ 自动切换到 actual 区间模式
起算日 = 2024-04-01 (ATA)
最晚提柜日 = 2024-04-05
截止日 = 2024-04-06 (Today)
计费天数 = 2024-04-06 - 2024-04-05 = 1 天
费用 = 1 × USD 120 = USD 120
```

**代码逻辑**:
```typescript
if (isStorage) {
  const hasStorageActualStart = useDischargeStorage
    ? !!params.calculationDates.dischargeDate
    : !!(params.calculationDates.ataDestPort ?? params.calculationDates.dischargeDate);
  
  storageUsesActualInterval =
    calculationMode === 'actual' || 
    (calculationMode === 'forecast' && hasStorageActualStart);
  
  if (storageUsesActualInterval && hasStorageActualStart) {
    // ✅ 使用 actual 区间，不需要 planned_pickup_date
    startDate = hasStorageActualStart;
    endDate = pickupDateActual ?? today; // 用 Today 替代
    // 正常计算...
  }
}
```

---

#### 情况 3: actual 模式，有实际提柜日

```
✅ **会计算**

原因：actual 模式下，提柜日已确定
起算日 = ATA/卸船日
截止日 = 实际提柜日（金额封顶）
```

**示例**:
```
ATA: 2024-04-01
实际提柜日：2024-04-10
Today: 2024-04-15
免费天数：5 天

起算日 = 2024-04-01 (ATA)
最晚提柜日 = 2024-04-05
截止日 = 2024-04-10 (实际提柜日，金额封顶)
计费天数 = 2024-04-10 - 2024-04-05 = 5 天
费用 = 5 × USD 120 = USD 600（固定值，不再增长）
```

### 1.5 Storage Charge 计算流程图

```
开始计算 Storage Charge
         ↓
检查 calculation_basis
    (按到港 or 按卸船)
         ↓
是否有起算日？
├─ 按到港：检查 ATA
├─ 按卸船：检查卸船日
         ↓
    有起算日？
    ├─ Yes → storageUsesActualInterval = true
    │         ↓
    │       使用 actual 区间
    │         ↓
    │       起算日 = ATA/卸船日
    │       截止日 = 实际/计划提柜日 或 Today
    │         ↓
    │       正常计算
    │
    └─ No → storageUsesActualInterval = false
              ↓
            forecast 区间模式
              ↓
            有计划提柜日？
            ├─ Yes → 起算日=修正 ETA/ETA
            │        截止日=max(Today, 计划提柜日)
            │        正常计算
            │
            └─ No → 跳过计算
                      ↓
                    reason='missing_planned_pickup_date'
```

## 二、四种费用类型场景对比

### 2.1 完整对比矩阵

| 费用类型 | forecast 无计划提柜日 | forecast 有计划提柜日 | actual 无实际提柜日 | actual 有实际提柜日 |
|----------|---------------------|---------------------|-------------------|-------------------|
| **Demurrage** | ✅ 计算<br>(Today 截止) | ✅ 计算<br>(计划提柜日截止) | ✅ 计算<br>(Today 截止) | ✅ 计算<br>(实际提柜日封顶) |
| **Detention** | ❌ **跳过** | ✅ 计算 | ❌ **跳过** | ✅ 计算 |
| **Storage** | ❌ **跳过**<br>(无 ATA) | ✅ 计算 | ✅ 计算<br>(无 ATA) | ✅ 计算 |
| | ✅ 计算<br>(有 ATA) | ✅ 计算<br>(有 ATA) | ✅ 计算<br>(有 ATA) | ✅ 计算 |
| **D&D Combined** | ❌ **跳过**<br>(无 ETA) | ✅ 计算 | ❌ **跳过**<br>(无 ATA) | ✅ 计算 |
| | ✅ 计算<br>(有 ETA) | ✅ 计算<br>(有 ETA) | ✅ 计算<br>(有 ATA) | ✅ 计算 |

### 2.2 Demurrage 场景详解

#### 场景 D1: forecast，无计划提柜日

```
条件：
- 修正 ETA: 2024-04-01
- 计划提柜日：null
- Today: 2024-04-06

计算：
起算日 = 2024-04-01 (修正 ETA)
截止日 = 2024-04-06 (Today)
✅ 可以计算，不依赖 planned_pickup_date

结果：
计费天数 = Today - 最晚提柜日
```

**关键点**: Demurrage 在 forecast 模式下**不一定需要**计划提柜日，可以用 ETA 作为起算日，Today 作为截止日。

---

#### 场景 D2: forecast，有计划提柜日

```
条件：
- 修正 ETA: 2024-04-01
- 计划提柜日：2024-04-08
- Today: 2024-04-06

计算：
起算日 = 2024-04-01 (修正 ETA)
截止日 = max(2024-04-06, 2024-04-08) = 2024-04-08
✅ 优先使用计划提柜日

结果：
计费天数 = 计划提柜日 - 最晚提柜日
```

---

#### 场景 D3: actual，有实际提柜日

```
条件：
- ATA: 2024-04-01
- 实际提柜日：2024-04-10
- Today: 2024-04-15

计算：
起算日 = 2024-04-01 (ATA)
截止日 = 2024-04-10 (实际提柜日)
✅ 金额封顶，不再增长

结果：
计费天数 = 实际提柜日 - 最晚提柜日
费用 = 固定值
```

---

### 2.3 Detention 场景详解

#### 场景 T1: forecast，无计划提柜日

```
条件：
- 计划提柜日：null
- 计划还箱日：2024-04-15

结果：
❌ **跳过计算**
reason = 'missing_planned_pickup_date'
原因 = '缺少计划提柜日（planned_pickup_date），该费用项暂不计算'
```

**关键点**: Detention **必须依赖**提柜日（起算日），没有提柜日无法计算。

---

#### 场景 T2: forecast，有计划提柜日

```
条件：
- 计划提柜日：2024-04-08
- 计划还箱日：2024-04-15
- Today: 2024-04-06

计算：
起算日 = 2024-04-08 (计划提柜日)
最晚还箱日 = 2024-04-14 (起算日 + 7 天)
截止日 = max(2024-04-06, 2024-04-15) = 2024-04-15

结果：
计费天数 = 2024-04-15 - 2024-04-14 = 1 天
```

---

#### 场景 T3: actual，有实际提柜日和还箱日

```
条件：
- 实际提柜日：2024-04-08
- 实际还箱日：2024-04-18
- Today: 2024-04-20

计算：
起算日 = 2024-04-08 (实际提柜日)
最晚还箱日 = 2024-04-14 (起算日 + 7 天)
截止日 = 2024-04-18 (实际还箱日，金额封顶)

结果：
计费天数 = 2024-04-18 - 2024-04-14 = 4 天
费用 = 4 × USD 80 = USD 320（固定值）
```

## 三、planned_pickup_date 的影响

### 3.1 各费用类型对 planned_pickup_date 的依赖程度

| 费用类型 | 无 planned_pickup_date (forecast) | 无 planned_pickup_date (actual) | 说明 |
|----------|----------------------------------|--------------------------------|------|
| **Demurrage** | ❌ 不依赖<br>(可用 ETA+Today) | ❌ 不依赖<br>(可用 ATA+Today) | 最不敏感 |
| **Detention** | ✅ **强依赖**<br>(无则跳过) | ❌ 不依赖<br>(需实际提柜日) | forecast 模式必需 |
| **Storage** | ⚠️ **条件依赖**<br>(无 ATA 时需) | ❌ 不依赖<br>(有 ATA 即可) | 有 ATA 时不依赖 |
| **D&D Combined** | ❌ 不依赖<br>(需 ETA) | ❌ 不依赖<br>(需 ATA) | 依赖到港日 |

### 3.2 planned_pickup_date 的来源

#### 来源 1: 智能排柜自动生成


#### 来源 3: Excel 导入

通过 Excel 导入备货单时，可附带计划日期：
```

映射字段：process_trucking_transport.planned_pickup_date
```

### 3.3 没有 planned_pickup_date 的影响

#### 影响范围评估

| 场景 | Demurrage | Detention | Storage | D&D | 总影响 |
|------|-----------|-----------|---------|-----|--------|
| **forecast，无 ATA** | ✅ 可计算 | ❌ 跳过 | ❌ 跳过 | ⚠️ 需 ETA | 部分影响 |
| **forecast，有 ATA** | ✅ 可计算 | ❌ 跳过 | ✅ 可计算 | ⚠️ 需 ETA | 轻微影响 |
| **actual** | ✅ 可计算 | ⚠️ 需实际提柜日 | ✅ 可计算 | ⚠️ 需实际还箱日 | 无影响 |

#### 业务影响

1. **成本估算不准确**
   - 智能排柜无法计算总成本
   - 无法进行成本优化比较

2. **预警功能失效**
   - 滞港费预警需要完整的费用计算
   - 最晚提柜日统计不完整

3. **决策支持缺失**
   - 无法提供最优提柜日期建议
   - 无法评估延期成本

## 四、智能排柜成本计算实战

### 4.1 成本计算流程

```
智能排柜引擎
    ↓
生成候选方案
(多个日期组合)
    ↓
for each 候选方案:
    ├── DemurrageService.calculateTotalCost()
    │     ├── calculateForContainer()
    │     │     ├── getContainerMatchParams()
    │     │     ├── matchStandards()
    │     │     ├── calculateSingleDemurrage() × 4 种费用
    │     │     └── 汇总各项费用
    │     │
    │     └── calculateTransportationCost()
    │           ├── 车队运输费
    │           ├── 仓库操作费
    │           └── Drop off 堆场费
    │
    └── 记录总成本
          ↓
选择最低成本方案
    ↓
输出最优日期 + 成本明细
```

### 4.2 成本计算代码示例

```typescript
// intelligentScheduling.service.ts
async calculateEstimatedCosts(
  containerNumber: string,
  plannedPickupDate: Date,
  plannedUnloadDate: Date,
  plannedReturnDate: Date,
  unloadMode: string,
  warehouse: Warehouse,
  truckingCompany: TruckingCompany
): Promise<{
  demurrageCost?: number;
  detentionCost?: number;
  storageCost?: number;
  ddCombinedCost?: number;
  transportationCost?: number;
  yardStorageCost?: number;
  handlingCost?: number;
  totalCost?: number;
  currency?: string;
}> {
  // 使用统一的 calculateTotalCost 方法
  const totalCostResult = await this.demurrageService.calculateTotalCost(
    containerNumber,
    {
      mode: 'forecast',
      plannedDates: {
        plannedPickupDate,
        plannedUnloadDate,
        plannedReturnDate
      },
      includeTransport: true,
      warehouse,
      truckingCompany,
      unloadMode
    }
  );
  
  // 计算外部堆场堆存费（Drop off 模式专属）
  let yardStorageCost = 0;
  if (unloadMode === 'Drop off' && truckingCompany.hasYard) {
    const plannedDeliveryDate = plannedUnloadDate; // Drop off: 送 = 卸
    const usesYard = plannedPickupDate < plannedDeliveryDate;
    
    if (usesYard) {
      const yardDays = daysBetween(plannedPickupDate, plannedDeliveryDate);
      yardStorageCost = yardDays * (truckingCompany.yardDailyRate || 50);
    }
  }
  
  return {
    demurrageCost: totalCostResult.demurrageCost,
    detentionCost: totalCostResult.detentionCost,
    storageCost: totalCostResult.storageCost,
    ddCombinedCost: totalCostResult.ddCombinedCost,
    transportationCost: totalCostResult.transportationCost,
    yardStorageCost,
    handlingCost: 0, // Live load 模式无操作费
    totalCost: totalCostResult.totalCost + yardStorageCost,
    currency: totalCostResult.currency
  };
}
```

## 五、特殊场景处理

### 5.1 场景 1: 中转港 ATA，目的港 ETA

```
条件:
- 中转港 ATA: 2024-04-01 (新加坡)
- 目的港 ETA: 2024-04-10 (温哥华)
- 目的港 ATA: null
- 计划提柜日：null

处理:
1. 判断计算模式：forecast (目的港未到达)
2. Demurrage 起算日：使用目的港 ETA (2024-04-10)
3. Detention: 跳过 (无计划提柜日)
4. Storage: 跳过 (无 ATA，无计划提柜日)
5. D&D: 使用目的港 ETA 计算

结果:
- Demurrage: ✅ 可计算
- Detention: ❌ 跳过
- Storage: ❌ 跳过
- D&D: ✅ 可计算
```

### 5.2 场景 2: 卸船日早于 ATA

```
条件:
- ATA: 2024-04-05
- 卸船日：2024-04-03 (提前卸船)
- 计划提柜日：2024-04-08

处理:
1. 判断 calculation_basis
   - 按到港：使用 ATA (2024-04-05)
   - 按卸船：使用卸船日 (2024-04-03)
   
2. 如果标准允许选择：
   - 优先使用较早日期（对货主有利）
   - 或按标准明确规定的口径

结果:
根据标准的 calculation_basis 决定起算日
```

### 5.3 场景 3: 实际提柜日早于计划提柜日

```
条件:
- 计划提柜日：2024-04-10
- 实际提柜日：2024-04-08 (提前)
- Today: 2024-04-09

处理:
1. 判断计算模式：actual (已提柜)
2. Demurrage 截止日：使用实际提柜日 (2024-04-08，金额封顶)
3. Detention 起算日：使用实际提柜日 (2024-04-08)
4. Storage 截止日：使用实际提柜日 (2024-04-08)

结果:
所有费用都按实际提柜日计算，金额封顶
```

### 6.3 常见错误码速查

| 错误码 | 含义 | 解决方案 |
|--------|------|----------|
| `no_arrival_at_dest` | 未到达目的港 | 等待到港或维护 ETA |
| `missing_arrival_dates` | 缺少到港/卸船日起算日 | 维护 ATA/ETA/卸船日 |
| `no_matching_standards` | 无匹配标准 | 检查四字段配置 |
| `missing_dates` | forecast 模式缺计划提柜日 | 维护计划提柜日 |
| `missing_pickup_date_actual` | actual 模式缺实际提柜日 | 更新实际提柜日 |
| `missing_planned_pickup_date` | forecast 模式缺计划提柜日 | 维护计划提柜日 |
| `missing_arrival_storage_actual` | 堆存费 (实际) 缺 ATA/卸船日 | 维护 ATA/卸船日 |
| `missing_eta_storage_forecast` | 堆存费 (预测) 缺 ETA | 维护 ETA |
| `missing_eta_combined_forecast` | D&D 合并缺 ETA | 维护 ETA |
| `missing_arrival_for_combined_actual` | D&D 合并 (实际) 缺 ATA | 维护 ATA |
