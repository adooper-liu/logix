# 货柜计划编辑功能详解

## 功能概述

**编辑计划**功能用于在货柜详情页手工调整货柜的物流计划安排，包括清关、拖卡、仓库和还箱四个环节的日期与资源配置。

**适用场景**:
- 智能排柜后微调特定日期
- 特殊情况下的手工排产
- 紧急计划变更
- 资源重新分配（更换车队、仓库、清关行）

---

## 功能入口

**路径**: 货柜详情页 → 页面右上角  
**按钮**: "编辑计划"（主按钮，蓝色）

**访问权限**: 所有已登录用户

---

## 编辑内容

计划编辑对话框分为四个部分，涵盖货柜物流的完整流程。

### 一、清关计划

| 字段 | 类型 | 说明 | 数据源 |
|------|------|------|--------|
| 计划清关日 | 日期选择器 | 货柜在目的港的计划清关日期 | `process_port_operations.planned_customs_date` |
| 清关公司 | 下拉选择 | 负责清关的清关行 | `dict_customs_brokers` |

**清关公司下拉**:
- 支持搜索（模糊匹配名称和代码）
- 支持清空
- 根据目的国过滤（显示该国清关行）
- 显示中文名和英文名

---

### 二、拖卡计划

| 字段 | 类型 | 说明 | 数据源 |
|------|------|------|--------|
| 计划提柜日 | 日期选择器 | 计划从港口提柜的日期 | `process_trucking_transport.planned_pickup_date` |
| 计划送仓日 | 日期选择器 | 计划将货柜送到仓库的日期 | `process_trucking_transport.planned_delivery_date` |
| 车队 | 下拉选择 | 负责拖卡运输的车队 | `dict_trucking_companies` |
| 卸柜方式 | 下拉选择 | 卸柜模式 | 固定选项 |

**卸柜方式选项**:
- `Drop off`（直接卸柜）：货柜送到仓库后直接卸货
- `Live load`（车板等待卸柜）：车队在仓库等待装货

**车队下拉**:
- 支持搜索
- 支持清空
- 根据目的国过滤
- 显示中文名和英文名

---

### 三、仓库计划

| 字段 | 类型 | 说明 | 数据源 |
|------|------|------|--------|
| 计划卸柜日 | 日期选择器 | 计划在仓库卸货的日期 | `process_warehouse_operations.planned_unload_date` |
| 仓库 | 下拉选择 | 进行卸货操作的仓库 | `dict_warehouses` |

**仓库下拉**:
- 支持搜索
- 支持清空
- 根据目的国过滤
- 显示中文名和英文名

---

### 四、还箱计划

| 字段 | 类型 | 说明 | 数据源 |
|------|------|------|--------|
| 计划还箱日 | 日期选择器 | 计划将空箱返还到堆场的日期 | `process_empty_return.planned_return_date` |

## 业务规则校验

保存前会进行严格校验，失败时直接回滚所有操作。

### 一、映射关系校验

#### 1. 港口 → 车队映射

**校验规则**: 车队必须支持目的港

**查询逻辑**:
```typescript
const portMapping = await truckingPortMappingRepository.findOne({
  where: {
    truckingCompanyId,
    portCode: seaFreight.portOfDischarge,  // 目的港代码
    isActive: true
  }
});

if (!portMapping) {
  validationErrors.push(`车队 ${truckingCompanyId} 不支持港口 ${seaFreight.portOfDischarge}`);
}
```

**依赖表**: `dict_trucking_port_mapping`

---

#### 2. 仓库 → 车队映射

**校验规则**: 仓库必须支持该车队

**查询逻辑**:
```typescript
const warehouseMapping = await warehouseTruckingMappingRepository.findOne({
  where: {
    warehouseCode: warehouseId,
    truckingCompanyId,
    isActive: true
  }
});

if (!warehouseMapping) {
  validationErrors.push(`仓库 ${warehouseId} 不支持车队 ${truckingCompanyId}`);
}
```

**依赖表**: `dict_warehouse_trucking_mapping`

---

### 二、时间顺序约束

**强制规则**: 所有计划日期必须按先后顺序排列

```
计划清关日 ≤ 计划提柜日 ≤ 计划送仓日 ≤ 计划卸柜日 ≤ 计划还箱日
```

**校验逻辑**:
```typescript
// 1. 计划清关日不能晚于计划提柜日
if (plannedCustoms && plannedPickup && plannedCustoms > plannedPickup) {
  validationErrors.push('计划清关日不能晚于计划提柜日');
}

// 2. 计划提柜日不能晚于计划送仓日
if (plannedPickup && plannedDelivery && plannedPickup > plannedDelivery) {
  validationErrors.push('计划提柜日不能晚于计划送仓日');
}

// 3. 计划送仓日不能晚于计划卸柜日
if (plannedDelivery && plannedUnload && plannedDelivery > plannedUnload) {
  validationErrors.push('计划送仓日不能晚于计划卸柜日');
}

// 4. 计划卸柜日不能晚于计划还箱日
if (plannedUnload && plannedReturn && plannedUnload > plannedReturn) {
  validationErrors.push('计划卸柜日不能晚于计划还箱日');
}
```

**校验失败处理**:
```typescript
if (validationErrors.length > 0) {
  await queryRunner.rollbackTransaction();
  res.status(400).json({ 
    success: false, 
    message: '校验失败', 
    errors: validationErrors 
  });
  return;
}
```

## 事务处理

### 事务保证

使用 TypeORM 事务确保所有表的更新要么全部成功，要么全部回滚。

**事务流程**:
```typescript
const queryRunner = AppDataSource.createQueryRunner();
await queryRunner.connect();
await queryRunner.startTransaction();

try {
  // 1. 业务规则校验
  // 2. 更新多个表
  // 3. 提交事务
  await queryRunner.commitTransaction();
  
  // 4. 自动更新物流状态
  await containerStatusService.updateStatus(id);
  
  res.json({ success: true, message: '计划更新成功' });
} catch (error) {
  // 5. 失败回滚
  await queryRunner.rollbackTransaction();
  res.status(500).json({ success: false, message: '更新计划失败' });
} finally {
  await queryRunner.release();
}
```

**保证事项**:
- ✅ 所有表同时更新成功
- ✅ 任一表更新失败则全部回滚
- ✅ 数据始终保持一致性
- ✅ 校验失败自动回滚

## 注意事项

### ⚠️ 时间顺序约束

所有计划日期必须严格遵守以下顺序：

```
计划清关日 → 计划提柜日 → 计划送仓日 → 计划卸柜日 → 计划还箱日
```

**违反后果**: 保存失败，提示校验错误

---

### ⚠️ 映射关系约束

- 车队必须支持目的港（通过 `dict_trucking_port_mapping` 校验）
- 仓库必须支持车队（通过 `dict_warehouse_trucking_mapping` 校验）

**违反后果**: 保存失败，提示"车队不支持该港口"或"仓库不支持该车队"

---

### ⚠️ 手工调整标记

所有手工编辑的计划都会标记为 `schedule_status = 'adjusted'`

**作用**:
- 区分智能排柜和手工调整
- 便于后续统计和分析
- 不影响智能排柜的重新执行

---

### ⚠️ 状态自动更新

保存成功后会自动触发物流状态机重算

**影响**:
- 货柜物流状态可能发生变化
- 预警信息可能更新
- 统计报表数据更新

## 常见问题

### Q1: 为什么保存时提示"校验失败"？

**可能原因**:
1. 时间顺序不正确（如提柜日晚于送仓日）
2. 车队不支持目的港
3. 仓库不支持车队

**解决方法**:
- 检查所有日期的先后顺序
- 确认车队与港口的映射关系
- 确认仓库与车队的映射关系

---

### Q2: 手工调整后会覆盖智能排柜结果吗？

**回答**: 不会覆盖，但会标记为 `adjusted`

**说明**:
- 手工调整和智能排柜是独立的
- 手工调整会标记 `schedule_status = 'adjusted'`
- 可以重新执行智能排柜，会覆盖手工调整的结果

---

### Q3: 保存后为什么物流状态变了？

**回答**: 系统自动触发了物流状态机重算

**说明**:
- 计划更新后，系统会自动重新计算物流状态
- 这是正常行为，确保状态与计划保持一致
- 状态更新是异步的，可能有短暂延迟

---

### Q4: 可以只修改部分字段吗？

**回答**: 可以，只提交有值的字段

**说明**:
- 前端会自动过滤空值字段
- 后端只更新传入的字段
- 未修改的字段保持不变
