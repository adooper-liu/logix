#### 代码示例

**批量查询示例**:

```typescript
// ✅ 好的做法：批量查询
async function getStatistics() {
  const sql = `
    SELECT
      COUNT(DISTINCT FILTER (WHERE po.ata_dest_port::date = CURRENT_DATE)) as arrivalToday,
      COUNT(DISTINCT FILTER (WHERE po.ata_dest_port::date < CURRENT_DATE AND tt.pickup_date IS NULL)) as arrivedBeforeTodayNotPickedUp,
      -- ... 其他维度
    FROM biz_containers c
    INNER JOIN process_port_operations po ON c.container_number = po.container_number
    LEFT JOIN process_trucking_transport tt ON c.container_number = tt.container_number
    WHERE po.port_type = 'destination'
      AND po.port_sequence = (SELECT MAX(port_sequence) ...)
  `

  const result = await this.containerRepository.query(sql)
  return result[0]
}
```

#### 方案对比

**方案 A：每次统计时逐柜调用 demurrage 计算** ❌ 不推荐

| 维度     | 说明                                                                   |
| -------- | ---------------------------------------------------------------------- |
| **做法** | LastPickupStatisticsService 对每个货柜逐个调用 calculateForContainer() |
| **性能** | 目标集可能有几十～几百个货柜，每个需匹配标准、多表查询、计算           |
| **结果** | 统计接口会从几百 ms 变为数秒甚至更久                                   |
| **结论** | 会明显拖慢 Shipments 页加载，不推荐                                    |

---

**方案 B：写回库（推荐）** ✅

| 维度         | 说明                                                                                           |
| ------------ | ---------------------------------------------------------------------------------------------- |
| **做法**     | 在滞港费计算或同步时，若计算出 lastPickupDateComputed 且 DB 为空，则写回                       |
| **触发时机** | 1. 用户打开滞港费详情时（异步写回）<br>2. 定时任务：每 6 小时批量计算并写回                    |
| **优点**     | ✅ 统计、列表、甘特图均沿用现有 SQL/逻辑<br>✅ 性能无影响（写回是异步的）<br>✅ 数据一致性保证 |
| **缺点**     | ⚠️ 需明确写回策略（仅系统计算、不覆盖人工维护等）                                              |

---

**方案 C：批量预计算 + 统计时合并** ⚠️ 次选

| 维度     | 说明                                                                                        |
| -------- | ------------------------------------------------------------------------------------------- |
| **做法** | 新增 batchComputeLastFreeDates()，统计时对空值子集批量计算                                  |
| **优点** | ✅ 可批量查标准、减少重复查询，比逐柜计算快                                                 |
| **缺点** | ❌ 每次统计仍会触发批量计算，目标集大时仍有明显延迟<br>❌ 需在统计服务中引入 demurrage 依赖 |

---

**方案 D：SQL 层内联计算** ❌ 不推荐

| 维度     | 说明                                                                                                                               |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| **做法** | 在 LastPickupSubqueryTemplates 中用 COALESCE(po.last_free_date, 计算表达式)                                                        |
| **缺点** | ❌ 标准匹配、免费天数基准（Natural/Working）在 SQL 中难以完整表达<br>❌ Working Days 公式复杂，维护成本高<br>❌ 一对多匹配逻辑复杂 |
