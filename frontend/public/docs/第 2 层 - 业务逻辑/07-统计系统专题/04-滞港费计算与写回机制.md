# 滞港费计算与写回机制

**版本**: v1.0

## 📋 概述

本文档详细介绍 last_free_date（最晚提柜日）和 last_return_date（最晚还箱日）的自动计算与写回机制，确保统计卡片、列表、甘特图的数据一致性。

### 核心价值

1. **自动计算**: 基于滞港费标准自动计算免费期截止日
2. **智能写回**: 计算完成后自动写回数据库
3. **数据一致**: 统一数据来源，避免统计/列表/详情不一致
4. **性能优化**: 定时任务批量计算，避免实时计算影响性能

---

## 一、问题背景

### 1.1 现状分析

| 数据来源 | 最晚提柜日 | 最晚还箱日 |
|----------|------------|------------|
| **统计卡片** | `process_port_operations.last_free_date` | `process_empty_return.last_return_date` |
| **列表行** | `latestPortOperation.lastFreeDate` | `emptyReturn.lastReturnDate` |
| **滞港费详情** | 自动计算：起算日 + 免费天数 | 自动计算：用箱起算日 + 免费用箱天数 |

**问题**: 当 DB 无 `last_free_date` / `last_return_date` 时：
- 统计卡片会归入「最晚提柜日为空」「最晚还箱日为空」
- 滞港费详情可基于标准自动计算并展示
- **数据不一致**：统计显示为空，滞港费详情有值

---

### 1.2 解决方案对比

#### 方案 A：每次统计时逐柜调用 demurrage 计算 ❌ 不推荐

**做法**: LastPickupStatisticsService 对目标集中 `last_free_date` 为空的货柜，逐个调用 `demurrageService.calculateForContainer()` 取计算值，再按日期分类统计。

**性能**: 目标集可能有几十～几百个货柜，每个货柜需：
- 匹配标准
- 多表查询
- 费用计算

**结果**: 统计接口会从几百 ms 变为数秒甚至更久。

**结论**: 会明显拖慢 Shipments 页加载，不推荐。

---

#### 方案 B：写回库（推荐） ✅

**做法**: 在滞港费计算或同步时，若计算出 `lastPickupDateComputed` 且 DB 的 `last_free_date` 为空，则**写回** `process_port_operations.last_free_date`；最晚还箱日同理写回 `process_empty_return.last_return_date`。

**触发时机**:
1. 用户打开滞港费详情时（`calculateForContainer` 返回后，异步写回）
2. 定时任务：对「last_free_date 为空且已到目的港」的货柜批量计算并写回

**优点**:
- ✅ 统计、列表、甘特图均沿用现有 SQL/逻辑，无需改动
- ✅ 性能无影响（写回是异步的）
- ✅ 数据一致性保证

**缺点**:
- ⚠️ 需明确写回策略（仅系统计算、不覆盖人工维护等）

---

#### 方案 C：批量预计算 + 统计时合并

**做法**: 新增 `DemurrageService.batchComputeLastFreeDates(containerNumbers: string[])`，对一批货柜批量匹配标准并计算；LastPickupStatisticsService 在统计时，对 `last_free_date` 为空的子集调用该批量接口，在内存中合并后再分类统计。

**优点**: 可批量查标准、减少重复查询，比逐柜计算快。

**缺点**:
- ❌ 每次统计仍会触发批量计算，目标集大时仍有明显延迟
- ❌ 需在统计服务中引入 demurrage 依赖

---

#### 方案 D：SQL 层内联计算

**做法**: 在 LastPickupSubqueryTemplates 中，用 `COALESCE(po.last_free_date, 计算表达式)`，计算表达式需 JOIN `ext_demurrage_standards` 并按目的港、船公司、货代等匹配。

**缺点**:
- ❌ 标准匹配、免费天数基准（Natural/Working）在 SQL 中难以完整表达
- ❌ Working Days 公式复杂，维护成本高
- ❌ 一对多匹配逻辑复杂

---

## 二、推荐实现：方案 B（写回库）

### 2.1 写回逻辑

#### 最晚提柜日

**条件**:
- ✅ `process_port_operations.last_free_date IS NULL`
- ✅ demurrage 计算出 `lastPickupDateComputed`
- ✅ 无实际提柜日（`pickup_date IS NULL`）

**写回**: 更新对应目的港 `process_port_operations.last_free_date`

**注意**: 取目的港最新一条港口操作（port_sequence 最大）

---

#### 最晚还箱日

**条件**:
- ✅ `process_empty_return.last_return_date IS NULL`
- ✅ demurrage 计算出 `lastReturnDateComputed`
- ✅ 无实际还箱日（`return_time IS NULL`）

**写回**: 插入或更新 `process_empty_return` 的 `last_return_date`

**注意**: 部分货柜可能尚无 `process_empty_return` 记录，需按业务决定是否新建

---

### 2.2 写回终止条件

**当有了实际提柜日、实际还箱日，就不再计算与回写。**

| 条件 | 不再写回 |
|------|----------|
| `process_trucking_transport.pickup_date` 有值 | 不再计算/写回最晚提柜日 |
| `process_empty_return.return_time` 有值 | 不再计算/写回最晚还箱日 |

**理由**: 已提柜/已还箱后，最晚提柜日/最晚还箱日对业务无意义，且统计目标集已排除这些货柜。

---

### 2.3 写回策略建议

- ✅ **不覆盖人工维护**: 仅当 DB 值为空时写回；若 DB 已有值，以 DB 为准。
- ✅ **幂等**: 同一货柜多次计算，结果一致时可重复写回。
- ✅ **可配置**: 通过配置开关控制是否启用自动写回，便于灰度与回滚。

## 三、触发点

### 3.1 滞港费计算接口

**文件**: `backend/src/services/demurrage.service.ts`

**方法**: `calculateForContainer()`

**流程**:

```typescript
async calculateForContainer(containerNumber: string) {
  // 1. 获取货柜信息
  const container = await this.getContainer(containerNumber);
  
  // 2. 匹配滞港费标准
  const standard = await this.matchDemurrageStandard(container);
  
  // 3. 计算费用
  const result = this.calculateDemurrage(container, standard);
  
  // 4. 异步写回（不阻塞响应）
  if (result.lastPickupDateComputed || result.lastReturnDateComputed) {
    setImmediate(async () => {
      await this.writeBackComputedDatesIfNeeded(containerNumber, result);
    });
  }
  
  return result;
}
```

## 四、定时任务详解

### 4.1 DemurrageWriteBackScheduler

**文件**: `backend/src/schedulers/demurrage-write-back.scheduler.ts`

**核心方法**:

```typescript
class DemurrageWriteBackScheduler {
  constructor(
    private containerRepository: Repository<Container>,
    private demurrageService: DemurrageService
  ) {}
  
  /**
   * 批量写回计算的最晚提柜日/最晚还箱日
   */
  async batchWriteBackComputedDates() {
    console.log('[DemurrageWriteBack] Starting batch write-back...');
    
    // 1. 查找需要计算的货柜
    const containers = await this.findContainersNeedingCalculation();
    
    console.log(`[DemurrageWriteBack] Found ${containers.length} containers`);
    
    // 2. 批量计算（每批 50 个）
    const batchSize = 50;
    for (let i = 0; i < containers.length; i += batchSize) {
      const batch = containers.slice(i, i + batchSize);
      
      await Promise.all(
        batch.map(c => this.calculateAndWriteBack(c.container_number))
      );
      
      console.log(`[DemurrageWriteBack] Processed ${Math.min(i + batchSize, containers.length)}/${containers.length}`);
    }
    
    console.log('[DemurrageWriteBack] Batch write-back completed');
  }
  
  /**
   * 查找需要计算的货柜
   */
  private async findContainersNeedingCalculation(): Promise<Container[]> {
    return await this.containerRepository
      .createQueryBuilder('c')
      .innerJoin('c.portOperations', 'po')
      .leftJoin('c.truckingTransport', 'tt')
      .leftJoin('c.emptyReturn', 'er')
      .where('po.port_type = :portType', { portType: 'destination' })
      .andWhere('po.last_free_date IS NULL')
      .andWhere('(tt.pickup_date IS NULL OR tt.pickup_date::date > CURRENT_DATE)')
      .orderBy('po.ata_dest_port', 'ASC')
      .getMany();
  }
  
  /**
   * 计算并写回单个货柜
   */
  private async calculateAndWriteBack(containerNumber: string) {
    try {
      const result = await this.demurrageService.calculateForContainer(containerNumber);
      
      if (result.lastPickupDateComputed) {
        console.log(`[DemurrageWriteBack] Written back last_pickup_date=${result.lastPickupDateComputed} for ${containerNumber}`);
      }
    } catch (error) {
      console.error(`[DemurrageWriteBack] Error calculating ${containerNumber}:`, error);
    }
  }
}
```

## 五、数据一致性

### 5.1 当前数据来源差异

| 场景 | 最晚提柜日来源 | 最晚还箱日来源 |
|------|----------------|----------------|
| **统计/列表** | `process_port_operations.last_free_date` | `process_empty_return.last_return_date` |
| **滞港费计算**（getContainerMatchParams） | `process_trucking_transport.last_pickup_date` | 无（仅计算 lastReturnDateComputed） |

**问题**: 滞港费计算**不读** `process_port_operations.last_free_date`，只读 `process_trucking_transport.last_pickup_date`。若写回只写 `process_port_operations`，则统计/列表与滞港费详情可能**来源不同、值不一致**。

---

### 5.2 解决方案：统一口径

**修改 `getContainerMatchParams`**，使 `calculationDates.lastPickupDate` 的取值优先级为：

```typescript
calculationDates.lastPickupDate = 
  destPort.lastFreeDate                    // 优先级 1: process_port_operations.last_free_date
  ?? truckingTransport.lastPickupDate      // 优先级 2: process_trucking_transport.last_pickup_date
  ?? computedLastPickupDate;               // 优先级 3: 计算值
```

**代码实现**:

```typescript
private getContainerMatchParams(container: Container) {
  const destPort = container.portOperations?.find(po => po.port_type === 'destination');
  const truckingTransport = container.truckingTransport;
  
  return {
    calculationDates: {
      lastPickupDate: 
        destPort?.lastFreeDate 
        ?? truckingTransport?.lastPickupDate 
        ?? undefined,
      lastReturnDate: 
        container.emptyReturn?.lastReturnDate 
        ?? undefined
    },
    // ... 其他参数
  };
}
```

---

### 5.3 效果对比

#### 修改前

```
统计/列表：process_port_operations.last_free_date (空)
滞港费：process_trucking_transport.last_pickup_date (有值)
结果：不一致 ❌
```

#### 修改后

```
写回前:
统计/列表：process_port_operations.last_free_date (空)
滞港费：process_trucking_transport.last_pickup_date (有值)
结果：不一致（但行为与现有一致）✅

写回后:
统计/列表：process_port_operations.last_free_date (有值)
滞港费：process_port_operations.last_free_date (有值，优先级最高)
结果：一致 ✅
```

---

### 5.4 最晚还箱日

**当前**:
- 统计/列表：`process_empty_return.last_return_date`
- 滞港费：目前只计算 `lastReturnDateComputed`，不读 DB

**写回后**:
- `process_empty_return.last_return_date` 有值
- 滞港费计算可增加：`emptyReturn.lastReturnDate ?? computedLastReturnDate`
- 与写回值对齐

## 六、实施坑点

### 6.1 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| **写回失败** | DB 字段约束冲突 | 检查字段类型和长度 |
| **写回后统计仍为空** | 缓存未刷新 | 清除 Redis 缓存或等待过期 |
| **统计卡片值不一致** | 滞港费计算未统一口径 | 修改 getContainerMatchParams |
| **定时任务不执行** | Scheduler 未注册 | 检查模块导入和 providers |

---

### 6.2 排查步骤

1. **检查写回日志**

CREATE INDEX idx_port_ops_last_free_null

CREATE INDEX idx_port_ops_type_null_free
