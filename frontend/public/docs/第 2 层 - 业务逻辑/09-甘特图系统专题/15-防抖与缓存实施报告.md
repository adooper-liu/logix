## 二、技术实现细节

### 2.1 防抖逻辑

**实现方式**: 原生 `setTimeout`（避免引入 lodash-es 依赖）

**核心代码**:
```typescript
// 防抖定时器
let optimizationDebounceTimer: ReturnType<typeof setTimeout> | null = null

/**
 * 🎯 防抖版本的优化触发器（500ms）
 */
const debouncedTriggerCostOptimization = (
  container: Container,
  updateData: Record<string, string>
) => {
  // 清除之前的定时器
  if (optimizationDebounceTimer) {
    clearTimeout(optimizationDebounceTimer)
    console.log('[CostOptimization] 清除之前的防抖定时器')
  }

  // 设置新的定时器
  optimizationDebounceTimer = setTimeout(() => {
    console.log('[CostOptimization] 防抖延迟结束，执行优化')
    triggerCostOptimization(container, updateData)
    optimizationDebounceTimer = null
  }, 500) // 500ms 防抖

  console.log('[CostOptimization] 设置 500ms 防抖定时器')
}
```

**工作流程**:
```
用户拖拽 → 确认对话框 → executeUpdateWithData 
  ↓
调用 debouncedTriggerCostOptimization
  ↓
清除之前的定时器（如果有）
  ↓
设置新的 500ms 定时器
  ↓
[500ms 后] 执行 triggerCostOptimization
  ↓
检查缓存 → 命中则返回，未命中则调用 API
```

**优势**:
- ✅ 快速连续拖拽只触发一次 API 调用
- ✅ 减少服务器负载
- ✅ 提升用户体验（避免频繁弹窗）
- ✅ 无需外部依赖

### 2.2 结果缓存

**缓存策略**: LRU-like（基于时间过期）

**核心代码**:
```typescript
// 缓存接口
interface OptimizationCache {
  data: any
  timestamp: number
}

// 缓存配置
const CACHE_TTL = 5 * 60 * 1000 // 5分钟
const optimizationCache = new Map<string, OptimizationCache>()

/**
 * 生成缓存键
 */
const getCacheKey = (
  containerNumber: string,
  warehouseCode: string,
  truckingCompanyId: string,
  basePickupDate: string
): string => {
  return `${containerNumber}_${warehouseCode}_${truckingCompanyId}_${basePickupDate}`
}

/**
 * 从缓存获取结果
 */
const getCachedResult = (key: string): any | null => {
  const cached = optimizationCache.get(key)
  if (cached && Date.now() - cached.timestamp < CACHE_TTL) {
    console.log('[CostOptimization] 使用缓存结果')
    return cached.data
  }
  // 缓存过期，删除
  if (cached) {
    optimizationCache.delete(key)
  }
  return null
}

/**
 * 设置缓存结果
 */
const setCachedResult = (key: string, data: any): void => {
  optimizationCache.set(key, {
    data,
    timestamp: Date.now(),
  })
  console.log('[CostOptimization] 缓存结果已设置')
}
```

**缓存键设计**:
```
key = containerNumber_warehouseCode_truckingCompanyId_basePickupDate

示例:
ECMU5399797_WH001_TRUCK001_2026-04-15
```

**缓存生命周期**:
```
API 调用成功 → 设置缓存（timestamp = now）
  ↓
后续相同请求 → 检查缓存
  ↓
timestamp < 5分钟 → 使用缓存
  ↓
timestamp ≥ 5分钟 → 删除缓存，重新调用 API
```

**优势**:
- ✅ 相同参数的重复请求直接返回缓存
- ✅ 减少 API 调用次数
- ✅ 提升响应速度（毫秒级 vs 秒级）
- ✅ 自动过期，避免内存泄漏

## 三、性能对比

### 3.1 防抖效果

**场景**: 用户快速连续拖拽 5 次（每次间隔 200ms）

**无防抖**:
```
拖拽1 → API 调用 1 (耗时 2s)
拖拽2 → API 调用 2 (耗时 2s)
拖拽3 → API 调用 3 (耗时 2s)
拖拽4 → API 调用 4 (耗时 2s)
拖拽5 → API 调用 5 (耗时 2s)

总计: 5 次 API 调用，总耗时 ~10s
```

**有防抖（500ms）**:
```
拖拽1 → 设置定时器
拖拽2 → 清除定时器1，设置定时器2
拖拽3 → 清除定时器2，设置定时器3
拖拽4 → 清除定时器3，设置定时器4
拖拽5 → 清除定时器4，设置定时器5
[500ms 后] → API 调用 1 (耗时 2s)

总计: 1 次 API 调用，总耗时 ~2.5s
```

**性能提升**: 
- API 调用次数: **↓ 80%** (5次 → 1次)
- 总耗时: **↓ 75%** (10s → 2.5s)

### 3.2 缓存效果

**场景**: 同一货柜多次查看优化建议

**无缓存**:
```
第1次: API 调用 (2s)
第2次: API 调用 (2s)
第3次: API 调用 (2s)

总计: 3 次 API 调用，总耗时 6s
```

**有缓存（5分钟TTL）**:
```
第1次: API 调用 (2s) → 设置缓存
第2次: 命中缓存 (<1ms)
第3次: 命中缓存 (<1ms)

总计: 1 次 API 调用，总耗时 ~2s
```

**性能提升**:
- API 调用次数: **↓ 67%** (3次 → 1次)
- 响应速度: **↑ 2000倍** (2s → <1ms)

## 五、边界情况处理

### 5.1 缓存过期

```typescript
// 缓存设置时间: 2026-04-06 10:00:00
// 当前时间: 2026-04-06 10:06:00 (6分钟后)

const cached = optimizationCache.get(key)
if (cached && Date.now() - cached.timestamp < CACHE_TTL) {
  // 6分钟 > 5分钟，条件不满足
  return cached.data // ❌ 不会执行
}

// 缓存过期，删除
if (cached) {
  optimizationCache.delete(key) // ✅ 执行
}
return null // ✅ 返回 null，重新调用 API
```

### 5.2 组件卸载

```typescript
// 用户正在查看优化面板，然后切换到其他页面
onUnmounted(() => {
  // 清除 pending 的防抖定时器
  if (optimizationDebounceTimer) {
    clearTimeout(optimizationDebounceTimer)
    optimizationDebounceTimer = null
  }
  
  // 清空所有缓存
  optimizationCache.clear()
})

// 结果: 
// - 不会有 pending 的 API 调用
// - 不会占用内存
// - 下次进入页面会重新计算
```

### 5.3 参数变化

```typescript
// 第1次: 拖拽到仓库 WH001

// 第2次: 拖拽到仓库 WH002（不同仓库）
```

## 七、潜在问题与解决方案

### 7.1 内存泄漏风险

**问题**: 长时间使用后，缓存 Map 可能积累大量数据

**解决方案**:
1. ✅ 已实现：5分钟 TTL 自动过期
2. ✅ 已实现：组件卸载时清空缓存
3. 🔧 建议：添加最大缓存数量限制（如 100 条）

**改进代码**:
```typescript
const MAX_CACHE_SIZE = 100

const setCachedResult = (key: string, data: any): void => {
  // 如果超过最大数量，删除最旧的缓存
  if (optimizationCache.size >= MAX_CACHE_SIZE) {
    const oldestKey = optimizationCache.keys().next().value
    optimizationCache.delete(oldestKey)
  }
  
  optimizationCache.set(key, {
    data,
    timestamp: Date.now(),
  })
}
```

### 7.2 并发请求问题

**问题**: 多个组件实例同时调用可能导致竞态条件

**解决方案**:
1. ✅ 当前实现：每个组件实例有独立的定时器和缓存
2. 🔧 建议：如需全局共享，可使用 Pinia store

### 7.3 缓存一致性

**问题**: 后端数据更新后，前端缓存可能过时

**解决方案**:
1. ✅ 当前实现：5分钟 TTL 保证最终一致性
2. 🔧 建议：提供手动刷新按钮
3. 🔧 建议：监听后端 WebSocket 通知，主动失效缓存
