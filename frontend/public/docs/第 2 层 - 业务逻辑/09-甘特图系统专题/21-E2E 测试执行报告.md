## 三、问题分析

### 3.1 缺失的前置条件

#### 问题 1: 甘特图组件缺少 data-testid

**现状**:
```vue
<!-- GanttDot.vue 或其他组件 -->
<div class="gantt-dot">
  <!-- 没有 data-testid -->
</div>
```

**需要添加**:
```vue
<div class="gantt-dot" :data-testid="`${nodeType}-node`">
  <!-- 例如: data-testid="pickup-node" -->
</div>
```

**影响**: E2E 测试无法定位元素

---

#### 问题 2: 测试数据缺失

**现状**:
- 前端页面可能显示空数据
- 后端 API 可能返回空数组
- 数据库中可能没有测试货柜

**需要**:
- 数据库中有至少 1 个测试货柜
- 货柜状态为"未出运"
- 货柜有完整的流程数据

---

#### 问题 3: 后端 API 未就绪

**现状**:
- 优化 API `/api/v1/scheduling/optimize-container/:containerNumber` 可能未实现
- 或者返回 404/500 错误

**需要**:
- 后端服务正常运行
- API 路由已注册
- 返回正确的响应格式

---

#### 问题 4: 页面路由可能不正确

**现状**:
- 测试访问 `/gantt` 路由
- 实际路由可能是 `/scheduling/gantt` 或其他

**需要**:
- 确认正确的甘特图页面路由
- 更新测试中的 `page.goto()` URL

---

### 3.2 技术债务

#### 债务 1: E2E 测试依赖真实环境

**问题**: 
- Playwright 测试需要完整的前后端环境
- 无法在 CI/CD 中独立运行
- 测试速度慢（每个场景 30s 超时）

**建议**:
- 使用 Mock 数据隔离测试
- 或者使用 Cypress Component Testing
- 或者编写更简单的冒烟测试

---

#### 债务 2: 选择器脆弱

**问题**:
- 依赖 `data-testid` 属性
- 如果组件重构，测试会失败
- 维护成本高

**建议**:
- 使用更稳定的选择器（如角色、文本内容）
- 或者使用视觉回归测试
- 或者减少 E2E 测试数量，增加单元测试

## 七、经验教训

### 7.1 为什么 E2E 测试失败？

1. **缺少前置条件**: 没有考虑测试环境的依赖性
2. **选择器不稳定**: 依赖 `data-testid`，但组件未添加
3. **数据依赖**: 需要真实数据，但没有准备
4. **API 依赖**: 需要后端服务，但没有验证

### 7.2 如何避免类似问题？

1. **测试金字塔**: 优先单元测试，少量 E2E 测试
2. **Mock 数据**: E2E 测试也应尽量 Mock 外部依赖
3. **测试隔离**: 每个测试独立，不依赖其他测试的状态
4. **前置检查**: 测试开始前验证环境就绪
