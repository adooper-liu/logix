## 一、系统架构与原理

### 1.1 核心设计原则

**单一真相源（Single Source of Truth）**：

- **后端**：`ganttDerivedBuilder.ts` 负责计算 `gantt_derived` JSONB 字段，存储到 `biz_containers` 表
- **前端**：`calculateNodeStatus()` 基于 API 返回的流程表数据实时计算节点状态
- **优先级**：前端本地计算 > 后端 ganttDerived（避免后端数据过期）

**前后端职责分离**：

- **后端**：提供原始数据（流程表快照）、预计算 ganttDerived（用于报表/缓存）
- **前端**：实时计算节点状态、应用反向链式依赖、渲染圆点

**数据流向**：

```
数据库流程表 → 后端 API (enrichContainersList) → 前端 calculateNodeStatus() → getNodeDisplayType() → 模板渲染
```

### 1.2 技术栈

## 二、三级分组架构

### 2.1 分组层级

甘特图采用**三级树形结构**组织数据：

```
一级：目的港（Port）
  └─ 二级：五节点（Node: 清关/提柜/卸柜/还箱/查验）
      └─ 三级：供应商（Supplier: 清关行/车队/仓库）
```

**示例**：

```
IT-NAP（那不勒斯港）
  ├─ 清关
  │   └─ XX清关行
  ├─ 提柜
  │   └─ RT LOGISTICA Srl
  ├─ 卸柜
  │   └─ IT-S002（仓库）
  └─ 还箱
      └─ IT-S002（仓库）
```

---

### 2.2 分组实现

**前端数据结构**：


**供应商来源与默认值**：

- **清关**：`process_port_operations.customs_broker_code` 或 `customs_broker`，**缺省时默认 "未指定清关公司"**
- **提柜**：`process_trucking_transport.trucking_company_id` 或 `carrier_company`
- **卸柜**：`process_warehouse_operations.warehouse_id`
- **还箱**：`process_empty_return.return_terminal_code` 或卸柜仓库

## 三、五节点定义

### 3.1 节点映射表

| 中文名称 | 英文 Key   | 对应流程表                   | 关键字段                                           |
| -------- | ---------- | ---------------------------- | -------------------------------------------------- |
| **清关** | customs    | process_port_operations      | actual_customs_date, planned_customs_date          |
| **提柜** | pickup     | process_trucking_transport   | pickup_date, delivery_date, planned_pickup_date    |
| **卸柜** | unload     | process_warehouse_operations | unload_date, planned_unload_date                   |
| **还箱** | return     | process_empty_return         | return_time, last_return_date, planned_return_date |
| **查验** | inspection | process_port_operations      | （复用清关日期）                                   |

---

### 3.2 节点状态机

每个节点有 **3 种状态**：

| 状态       | 英文标识  | 显示类型           | 含义                                |
| ---------- | --------- | ------------------ | ----------------------------------- |
| **已完成** | completed | null（销毁）       | 有实际日期，圆点不显示              |
| **活跃**   | active    | main（实心圆点）   | 前置已完成 + 有计划日期，当前应执行 |
| **待激活** | pending   | dashed（虚线圆点） | 有计划日期但前置未完成，等待激活    |

**状态转换规则**：

```
pending --(前置完成)--> active --(有实际日期)--> completed（销毁）
```

**清关节点特殊规则**：

- **清关节点无论 pending 还是 active，均显示为实心圆点（main）**
- 目的：让用户清楚看到所有待清关的货柜，即使尚未分配清关行
- 只有在清关实际完成（`actual_customs_date` 有值）时才销毁圆点

## 五、正向与反向链式依赖

### 5.1 正向链式依赖（Forward Dependency）

**定义**：后续节点的激活依赖于前置节点的完成。

**规则**：

```
清关完成 → 提柜激活
提柜完成 → 卸柜激活
卸柜完成 → 还箱激活
```

**实现逻辑**（`calculateNodeStatus` Line 2636-2643）：

```typescript
// 提柜节点状态判断
if (pickupTransport.deliveryDate || pickupTransport.pickupDate) {
  nodes.提柜.status = 'completed' // 有实际日期 = 完成
} else if (nodes.提柜.plannedDate) {
  const customsCompleted = destPortOp?.actualCustomsDate
  if (customsCompleted) {
    nodes.提柜.status = 'active' // 前置已完成 + 有计划 = 活跃
  } else {
    nodes.提柜.status = 'pending' // 前置未完成 + 有计划 = 待激活
  }
}
```

**业务意义**：

- 确保操作流程的顺序性
- 避免跳过必要步骤（如未清关就提柜）
- 直观展示当前应执行的节点

---

### 5.2 反向链式依赖（Backward Dependency）

**定义**：后续节点的完成意味着前置节点必然已完成，应销毁前置节点的圆点。

**规则**：

```
已提柜 → 反推清关完成（销毁清关节点）
已卸柜 → 反推清关+提柜完成（销毁清关+提柜节点）
已还箱 → 反推清关+提柜+卸柜完成（销毁清关+提柜+卸柜节点）
```

**实现逻辑**（`calculateNodeStatus` Line 2733-2778）：

```typescript
// 1. 已还箱 -> 反推清关、提柜、卸柜全部完成
if (emptyReturn?.returnTime) {
  if (!destPortOp?.actualCustomsDate) {
    nodes.清关.status = 'completed' // 反推清关已完成
  }
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate)) {
    nodes.提柜.status = 'completed' // 反推提柜已完成
  }
  if (!unloadOp?.unloadDate) {
    nodes.卸柜.status = 'completed' // 反推卸柜已完成
  }
}
// 2. 已卸柜 -> 反推清关、提柜完成
else if (unloadOp?.unloadDate) {
  if (!destPortOp?.actualCustomsDate) {
    nodes.清关.status = 'completed'
  }
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate)) {
    nodes.提柜.status = 'completed'
  }
}
// 3. 已提柜 -> 反推清关完成
else if (pickupTransport?.deliveryDate || pickupTransport?.pickupDate) {
  if (!destPortOp?.actualCustomsDate) {
    nodes.清关.status = 'completed'
  }
}
```

**业务意义**：

- **数据缺失补偿**：即使缺少清关实际日期，只要已提柜就能推断清关已完成
- **视觉简化**：销毁已完成节点的圆点，避免界面混乱
- **逻辑一致性**：符合业务流程的因果关系

### 6.2 getNodeDisplayType 函数

**位置**：`SimpleGanttChartRefactored.vue` Line 1973-2006

**逻辑**：

```typescript
const getNodeDisplayType = (container: any, nodeName: string): 'main' | 'dashed' | null => {
  const nodeStatus = calculateNodeStatus(container)
  const node = nodeStatus.nodes[nodeName]

  // 1. 边界检查：无供应商或供应商未指定
  if (!node?.supplier || node.supplier === '未指定') return null

  // 2. 查验节点特殊处理
  if (nodeName === '查验') {
    return getInspectionNodeType(node)
  }

  // 3. 清关/卸柜/还箱/提柜：优先使用本地状态
  if (nodeName === '清关' || nodeName === '卸柜' || nodeName === '还箱' || nodeName === '提柜') {
    return getNodeTypeFromLocalStatus(node)
  }

  // 4. 回退到后端 ganttDerived（备用）
  const ganttType = getNodeTypeFromGanttDerived(container, nodeName)
  if (ganttType !== null) {
    return ganttType
  }

  // 5. 最终回退到本地状态
  return getNodeTypeFromLocalStatus(node)
}
```

**关键决策**：

- **优先本地计算**：避免后端 ganttDerived 数据过期
- **供应商检查**：无供应商的节点不显示
- **查验特殊处理**：后端 gantt-v1 无此节点，仅用本地状态

### 7.3 后端计算逻辑

**文件**：`backend/src/utils/ganttDerivedBuilder.ts`

**核心函数**：

```typescript
export function buildGanttDerived(
  portOperations: PortOperation[],
  trucking: TruckingTransport | null,
  warehouse: WarehouseOperation | null,
  emptyReturn: EmptyReturn | null
): GanttDerived
```

**计算步骤**：

1. 确定当前阶段（1-5）
2. 遍历 5 个节点，根据阶段和反向链式依赖确定 taskRole
3. 提取各节点的 plannedDate 和 actualDate
4. 判断 completed 状态

**反向链式依赖实现**（Line 214-229）：

```typescript
// 已还箱 -> 所有前置节点销毁
if (hasReturnTime) {
  taskRole = 'none'
}
// 已卸柜 -> 清关、提柜销毁
else if (hasUnloadDate && (key === 'customs' || key === 'pickup')) {
  taskRole = 'none'
}
// 已提柜 -> 清关、提柜销毁
else if ((hasDeliveryDate || hasPickupDate) && (key === 'customs' || key === 'pickup')) {
  taskRole = 'none'
}
// 其他情况：显示为虚线
else {
  taskRole = 'dashed'
}
```

## 十、常见问题与陷阱

### 10.1 多套渲染逻辑陷阱

**问题**：组件中存在多处渲染 `container-dot` 的模板，容易遗漏某处的修复。

**案例**：

- Line 309 曾使用旧的 `isMainTask/isDashedTask` 逻辑
- 其他 4 处使用新的 `getNodeDisplayType` 逻辑
- 导致反向链式依赖在 Line 309 不生效

**解决方案**：

1. 全局搜索所有渲染点：`grep_code(regex="container-dot")`
2. 确保所有模板使用统一的 `getNodeDisplayType` 函数
3. 定期审查 CSS 样式，避免 `!important` 覆盖内联样式

---

### 10.2 CSS 优先级陷阱

**问题**：CSS 中的 `!important` 会覆盖 Vue 内联样式。

**案例**：

```css
/* 错误写法 */
.container-dot.dashed-task {
  background: transparent !important; /* ← 覆盖了内联的 backgroundColor */
}
```

**解决方案**：

- 移除 `!important`，使用更具体的选择器
- 添加 `.has-warning` 修饰符支持预警状态

---

### 10.3 后端 ganttDerived 不可信

**问题**：后端的 `ganttDerived` 可能因为数据迁移、手动修改等原因与实际不符。

**解决方案**：

- 前端优先使用本地计算（`calculateNodeStatus`）
- 后端 ganttDerived 仅作为备用（报表、缓存）
- 定期运行迁移脚本同步数据

## 十二、总结

### 12.1 核心要点

1. **三级分组**：目的港 → 五节点 → 供应商
2. **五节点状态机**：pending → active → completed（销毁）
3. **正向链式依赖**：前置完成才能激活后续
4. **反向链式依赖**：后续完成反推前置销毁
5. **圆点类型**：实心（活跃）、虚线（待激活）、销毁（完成）
6. **颜色系统**：背景色=物流状态，边框色=预警等级（#8c939d深灰色）
7. **前端优先**：本地计算 > 后端 ganttDerived
8. **清关节点特殊规则**：默认显示实心主任务，即使无清关行也显示在“未指定清关公司”泳道
9. **图例组件**：提供颜色和节点类型的视觉参考

---

### 12.2 架构优势

- **数据一致性**：前后端都实现反向链式依赖，互为备份
- **视觉清晰**：通过颜色、形状、动画区分不同状态
- **性能优化**：销毁已完成节点，减少 DOM 数量
- **易于维护**：统一的 `getNodeDisplayType` 函数，避免逻辑分散
- **用户友好**：清关节点始终可见，便于跟踪所有待清关货柜
