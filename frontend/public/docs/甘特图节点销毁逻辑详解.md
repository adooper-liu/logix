**版本**: v1.0  
**创建时间**: 2026-04-04

## 核心概念：什么是"销毁"？

**销毁 = 不显示圆点**

当一个节点的状态变为 `completed`（已完成）时，它会转换为显示类型 `null`，Vue 的 `v-if` 指令不会渲染该 DOM 元素，圆点从页面上消失。

```typescript
// 状态转换规则
completed → null  (销毁，不显示)
active    → main  (实心圆点)
pending   → dashed (空心圆点)
```

---

## 销毁的两种触发方式

### 方式1：正向销毁（有实际日期）

**规则**：节点有自己的实际日期字段 → 直接销毁

| 节点 | 实际日期字段                   | 销毁条件           |
| ---- | ------------------------------ | ------------------ |
| 清关 | `actualCustomsDate`            | 字段存在即销毁     |
| 提柜 | `deliveryDate` 或 `pickupDate` | 任一字段存在即销毁 |
| 卸柜 | `unloadDate`                   | 字段存在即销毁     |
| 还箱 | `returnTime`                   | 字段存在即销毁     |

**示例**：

```typescript
// 清关节点
if (destPortOp.actualCustomsDate) {
  nodes.清关.status = 'completed' // 有实际清关日期 -> 销毁
}

// 提柜节点
if (pickupTransport.deliveryDate || pickupTransport.pickupDate) {
  nodes.提柜.status = 'completed' // 有实际提柜/送仓日期 -> 销毁
}
```

### 方式2：反向推导销毁（后续节点已完成）

**规则**：后续节点完成了，反推前置节点也必然完成 → 间接销毁

| 触发条件   | 反推结果                                     | 说明                           |
| ---------- | -------------------------------------------- | ------------------------------ |
| **已提柜** | 清关 -> 销毁                                 | 能提柜说明清关已完成           |
| **已卸柜** | 清关 -> 销毁<br>提柜 -> 销毁                 | 能卸柜说明清关和提柜都已完成   |
| **已还箱** | 清关 -> 销毁<br>提柜 -> 销毁<br>卸柜 -> 销毁 | 能还箱说明前面所有步骤都已完成 |

**示例**：

```typescript
// 场景：只记录了还箱日期，其他日期都缺失
actualCustomsDate: null // 清关日期缺失
deliveryDate: null // 提柜日期缺失
unloadDate: null // 卸柜日期缺失
returnTime: '2026-04-15' // 但已有还箱日期

// 反向推导
if (emptyReturn?.returnTime) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推清关已完成 -> 销毁
  }
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate) && pickupSupplier) {
    nodes.提柜.status = 'completed' // 反推提柜已完成 -> 销毁
  }
  if (!unloadOp?.unloadDate && unloadSupplier) {
    nodes.卸柜.status = 'completed' // 反推卸柜已完成 -> 销毁
  }
}
```

## 各节点销毁逻辑详解

### 1. 清关节点销毁

#### 触发条件（满足任一即销毁）

```
条件A: actualCustomsDate 存在（正向）
   OR
条件B: 提柜已完成（deliveryDate/pickupDate 存在）且 customsSupplier 存在（反向）
   OR
条件C: 卸柜已完成（unloadDate 存在）且 customsSupplier 存在（反向）
   OR
条件D: 还箱已完成（returnTime 存在）且 customsSupplier 存在（反向）
```

#### 代码实现

```typescript
// 第一步：正向检查
if (destPortOp.actualCustomsDate) {
  nodes.清关.status = 'completed' // 有实际日期 -> 销毁
}

// 第二步：反向推导（在 calculateNodeStatus 末尾）
if (emptyReturn?.returnTime) {
  // 已还箱 -> 反推清关完成
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推销毁
  }
} else if (unloadOp?.unloadDate) {
  // 已卸柜 -> 反推清关完成
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推销毁
  }
} else if (pickupTransport?.deliveryDate || pickupTransport?.pickupDate) {
  // 已提柜 -> 反推清关完成
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推销毁
  }
}
```

#### 场景演示

**场景1：有实际清关日期**

```
数据: actualCustomsDate = "2026-04-08"
结果: 清关 -> completed -> null (销毁)
显示: ❌ 不显示圆点
```

**场景2：无实际日期，但已提柜**

```
数据:
  actualCustomsDate = null
  deliveryDate = "2026-04-11"
  customsSupplier = "XX清关行"

推导: 已提柜 -> 反推清关完成
结果: 清关 -> completed -> null (销毁)
显示: ❌ 不显示圆点
```

**场景3：无任何日期，无供应商**

```
数据:
  actualCustomsDate = null
  customsSupplier = null 或 "未指定"

推导: 无供应商，不应用反向推导
结果: 清关 -> pending (如果有计划日期)
显示: ⚪ 空心圆点（待激活）
```

### 2. 提柜节点销毁

#### 触发条件（满足任一即销毁）

```
条件A: deliveryDate 或 pickupDate 存在（正向）
   OR
条件B: 卸柜已完成（unloadDate 存在）且 pickupSupplier 存在（反向）
   OR
条件C: 还箱已完成（returnTime 存在）且 pickupSupplier 存在（反向）
```

#### 代码实现

```typescript
// 第一步：正向检查
if (pickupTransport.deliveryDate || pickupTransport.pickupDate) {
  nodes.提柜.status = 'completed' // 有实际日期 -> 销毁
}

// 第二步：反向推导
if (emptyReturn?.returnTime) {
  // 已还箱 -> 反推提柜完成
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate) && pickupSupplier) {
    nodes.提柜.status = 'completed' // 反推销毁
  }
} else if (unloadOp?.unloadDate) {
  // 已卸柜 -> 反推提柜完成
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate) && pickupSupplier) {
    nodes.提柜.status = 'completed' // 反推销毁
  }
}
```

#### 场景演示

**场景1：有实际提柜日期**

```
数据: deliveryDate = "2026-04-11"
结果: 提柜 -> completed -> null (销毁)
显示: ❌ 不显示圆点
```

**场景2：无实际日期，但已卸柜**

```
数据:
  deliveryDate = null
  pickupDate = null
  unloadDate = "2026-04-13"
  pickupSupplier = "XX车队"

推导: 已卸柜 -> 反推提柜完成
结果: 提柜 -> completed -> null (销毁)
显示: ❌ 不显示圆点
```

---

### 3. 卸柜节点销毁

#### 触发条件（满足任一即销毁）

```
条件A: unloadDate 存在（正向）
   OR
条件B: 还箱已完成（returnTime 存在）且 unloadSupplier 存在（反向）
```

**注意**：卸柜没有更后续的节点可以反推它，只有还箱可以反推卸柜

#### 代码实现

```typescript
// 第一步：正向检查
if (unloadOp.unloadDate) {
  nodes.卸柜.status = 'completed' // 有实际日期 -> 销毁
}

// 第二步：反向推导
if (emptyReturn?.returnTime) {
  // 已还箱 -> 反推卸柜完成
  if (!unloadOp?.unloadDate && unloadSupplier) {
    nodes.卸柜.status = 'completed' // 反推销毁
  }
}
```

#### 场景演示

**场景1：有实际卸柜日期**

```
数据: unloadDate = "2026-04-13"
结果: 卸柜 -> completed -> null (销毁)
显示: ❌ 不显示圆点
```

**场景2：无实际日期，但已还箱**

```
数据:
  unloadDate = null
  returnTime = "2026-04-15"
  unloadSupplier = "XX仓库"

推导: 已还箱 -> 反推卸柜完成
结果: 卸柜 -> completed -> null (销毁)
显示: ❌ 不显示圆点
```

---

### 4. 还箱节点销毁

#### 触发条件（仅正向）

```
条件A: returnTime 存在（正向）
```

**注意**：还箱是最后一个节点，没有后续节点可以反推它

#### 代码实现

```typescript
// 只有正向检查，无反向推导
if (emptyReturn?.returnTime) {
  nodes.还箱.status = 'completed' // 有实际日期 -> 销毁
}
```

#### 场景演示

**场景1：有实际还箱日期**

```
数据: returnTime = "2026-04-15"
结果: 还箱 -> completed -> null (销毁)
显示: ❌ 不显示圆点
```

**场景2：无实际还箱日期**

```
数据: returnTime = null, lastReturnDate = "2026-04-16"
结果: 还箱 -> active 或 pending (取决于卸柜是否完成)
显示: 🔵 实心圆点 或 ⚪ 空心圆点
```

## 关键要点总结

### 1. 销毁的本质

```
销毁 ≠ 数据删除
销毁 = 不显示圆点（DOM 不渲染）

数据仍然存在，可以在 Tooltip 中查看实际日期
```

### 2. 销毁的两个阶段

```
阶段1: 正向检查（优先）
  - 检查节点自身是否有实际日期
  - 有 -> 直接标记为 completed

阶段2: 反向推导（补充）
  - 检查后续节点是否已完成
  - 是 -> 反推当前节点也为 completed
  - 前提: 供应商信息存在
```

### 3. 为什么需要反向推导？

**问题场景**：

```
业务上货柜已经还箱了，但操作人员只录入了还箱日期，
前面的清关、提柜、卸柜日期都没有录入。

如果只有正向检查:
  清关: pending (空心圆点) <- 错误！实际上已完成
  提柜: pending (空心圆点) <- 错误！实际上已完成
  卸柜: pending (空心圆点) <- 错误！实际上已完成
  还箱: completed (销毁)   <- 正确

用户体验: 看到3个空心圆点，以为还有3个节点待执行 ❌

使用反向推导:
  清关: completed (销毁) <- 正确！反推已完成
  提柜: completed (销毁) <- 正确！反推已完成
  卸柜: completed (销毁) <- 正确！反推已完成
  还箱: completed (销毁) <- 正确

用户体验: 看到0个圆点，知道所有节点都已完成 ✅
```

### 4. 反向推导的前提条件

```typescript
// 必须同时满足两个条件才进行反向推导
if (!actualDate && supplierExists) {
  // 1. 实际日期不存在（否则不需要反推）
  // 2. 供应商存在（确保这个节点应该显示）
  status = 'completed'
}
```

**为什么检查供应商？**

- 如果没有供应商，说明这个节点本来就不应该显示
- 避免为不存在的节点设置状态

### 5. 销毁顺序

```
还箱完成 -> 反推卸柜、提柜、清关全部销毁
卸柜完成 -> 反推提柜、清关销毁
提柜完成 -> 反推清关销毁
清关完成 -> 只能销毁自己（没有前置节点）
```

## 常见疑问解答

### Q1: 为什么已完成的节点要销毁，而不是显示为灰色圆点？

**A**: 设计原则是"只显示待执行和可执行的节点"

- 已完成 = 不需要关注 = 不显示
- Tooltip 中仍可查看实际日期（绿色标识）
- 减少视觉干扰，聚焦当前任务

### Q2: 反向推导会不会导致错误的状态？

**A**: 不会，因为：

1. 反向推导只在实际日期缺失时才生效
2. 业务逻辑保证：后续完成意味着前置必然完成
3. 如果能提柜，说明清关一定已完成（海关放行才能提货）
4. 如果能卸柜，说明提柜一定已完成（提到仓库才能卸）
5. 如果能还箱，说明卸柜一定已完成（卸完才能还空箱）

### Q3: 如果数据录入错误怎么办？

**A**:

- 反向推导基于业务逻辑，不是基于数据准确性
- 如果确实录错了，应该修正数据库中的实际日期
- 一旦有了正确的实际日期，正向检查会优先生效

### Q4: 查验节点会销毁吗？

**A**: 会的，查验节点的销毁逻辑与清关类似：

```typescript
if (destPortOp?.actualCustomsDate && needsInspection) {
  nodes.查验.status = 'completed' // 有实际清关日期且有查验需求 -> 销毁
}
```

### Q5: 销毁后的节点还能查看吗？

**A**: 可以！通过 Tooltip 查看：

```vue
<!-- 实际日期区 -->
<div class="tooltip-section" v-if="hasActualDates(tooltipContainer)">
  <div class="tooltip-row" v-if="actualCustomsDate">
    <span class="label">实际清关：</span>
    <span class="value is-arrived">{{ formatDate(actualCustomsDate) }}</span>
  </div>
  <!-- ... 其他节点的实际日期 -->
</div>
```

## 快速判断流程图

```
节点要销毁吗？
  ↓
有实际日期？
  ├─ 是 → ✅ 销毁（正向）
  └─ 否 ↓
       后续节点已完成？
         ├─ 是 → 有供应商？
         │        ├─ 是 → ✅ 销毁（反向推导）
         │        └─ 否 → ❌ 不销毁（节点本就不该显示）
         └─ 否 → ❌ 不销毁（显示 pending 或 active）
```
