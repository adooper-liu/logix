# 五大统计卡片 - 统计口径说明文档

## 📊 概述

本文档详细说明货柜列表页面五个统计卡片的统计口径、业务含义和使用场景。

---

## 1️⃣ 按状态分布 (Status Distribution)

### 🎯 统计范围
**全部货柜**（无状态限制）

### 📋 分类依据
- **字段**: `logistics_status`（物流状态机）
- **分类数**: 8 个状态
- **互斥性**: ✅ 严格互斥

### 🏷️ 状态明细

| 状态 | 字段名 | 颜色 | 业务含义 |
|-----|--------|------|---------|
| 未出运 | not_shipped | 🟤 #909399 | 备货装柜阶段 |
| 已出运 | shipped | 🔵 #409eff | 已装船出运 |
| 在途 | in_transit | 🟠 #e6a23c | 海运运输中 |
| 已到中转港 | arrived_at_transit | 🟤 #909399 | 经停中转港（特殊子集） |
| 已到目的港 | at_port | 🟢 #67c23a | 到达目的港待提柜 |
| 已提柜 | picked_up | 🟡 #f39c12 | 已从港口提出 |
| 已卸柜 | unloaded | 🔵 #3498db | 货物已卸载 |
| 已还箱 | returned_empty | ⚪ #95a5a6 | 空箱已返还 |

### 💼 业务用途
监控货柜在全流程中的实时分布状态，识别瓶颈环节。

### ⚠️ 特殊说明
- `arrived_at_transit` 是特殊子集标记，不计入总数
- 基于子查询判断：有 transit 记录且无 destination 记录

---

## 2️⃣ 按到港分布 (Arrival Distribution)

### 🎯 统计范围
**shipped + in_transit + at_port + picked_up + unloaded + returned_empty**（扩展后）

> **重要更新**: 2026-03-04 起，扩展目标集包含 `picked_up`、`unloaded`、`returned_empty` 状态。已还箱货柜纳入「今日之前到港已提柜」，方便追踪历史到港数据，确保按到港合计与按状态一致。

### 📋 分类依据
- **核心字段**: 
  - `ata_dest_port`（实际到港日期，ATA）
  - `eta_dest_port`（预计到港日期，ETA）
- **分类数**: 7 个
- **互斥性**: ✅ 严格互斥

### 🔄 优先级规则
```
IF ATA IS NOT NULL:
  → 按 ATA 分类（今日到港 / 今日之前到港）
ELSE IF ETA IS NOT NULL:
  → 按 ETA 分类（逾期 / 3 天内 / 7 天内 / >7 天）
ELSE:
  → other
```

### 🏷️ 分类明细

| 分类 | 字段名 | 颜色 | 核心条件 | 业务含义 |
|-----|--------|------|---------|---------|
| 今日到港 | today | 🟢 #67c23a | ATA = 今日 | 今天刚到港 |
| 今日之前到港 | arrivedBeforeToday | 🟤 #909399 | ATA < 今日 | 以前到港的（含已提柜） |
| 已逾期未到港 | overdue | 🔴 #f56c6c | ETA < 今日 且 ATA is null | 应该到但还没到 ⚠️ |
| 3 天内预计到港 | within3Days | 🟠 #e6a23c | 今日 ≤ ETA ≤ 3 天后 | 近期预计到港 |
| 7 天内预计到港 | within7Days | 🔵 #409eff | 3 天后 < ETA ≤ 7 天后 | 中期预计到港 |
| >7 天预计到港 | over7Days | 🟢 #67c23a | ETA > 7 天后 | 远期预计到港 |
| 其他情况 | other | ⚪ #c0c4cc | ETA is null | 没有预计到港日期 |

### 💼 业务用途
监控海运段的到港进度，提前预警可能逾期的货柜。

### 🔍 关键逻辑
- 支持多港经停（使用 `port_type = 'destination'` + `port_sequence`）
- 逾期判断同时检查 `eta_dest_port` 和 `eta_correction`（修正后的 ETA）
- 如果一个货柜有多条目的港记录，取最新的 ATA
- **包含已提柜状态**：方便追踪历史到港数据，确保数据前后一致

### 📊 数据流转示例

```
集装箱 A 的生命周期：

Day 1: shipped (已出运)
  → ETA = Day 10
  → 计入"7 天内预计到港"

Day 5: in_transit (在途)
  → ETA = Day 10
  → 计入"3 天内预计到港"

Day 8: at_port (已到目的港)
  → ATA = Day 8
  → 计入"今日之前到港"（如果是 Day 8 当天则是"今日到港"）

Day 10: picked_up (已提柜)
  → ATA = Day 8（保持不变）
  → ✅ 仍计入"今日之前到港"（扩展后的统计）

Day 12: unloaded (已卸柜)
  → ATA = Day 8（保持不变）
  → ✅ 仍计入"今日之前到港"（扩展后的统计）

Day 15: returned_empty (已还箱)
  → ATA = Day 8（保持不变）
  → ✅ 仍计入"今日之前到港已提柜"（扩展后的统计）
```

### ⚠️ 特殊说明

#### 包含已提柜状态的意义

**之前的问题**:
```
目标集 = shipped + in_transit + at_port

当货柜流转到 picked_up/unloaded 后：
❌ 从"按到港"统计中消失
❌ 导致历史数据无法追踪
❌ 前后数据不一致
```

**现在的解决方案**:
```
目标集 = shipped + in_transit + at_port + picked_up + unloaded + returned_empty

优势：
✅ 可以追踪任意时间点到港的货柜最终去向
✅ 数据前后一致，便于分析
✅ 完整记录货柜生命周期
```

**示例场景**:
```
统计 3 月 1 日 到港的货柜：

Day 1 (3 月 1 日): 
  - 10 个货柜今日到港
  - 按到港总数 = 10

Day 5 (3 月 5 日):
  - 这 10 个货柜中有 6 个已提柜
  - 按到港总数 = 10（仍包含这 6 个）
  - "今日之前到港" = 10（全部）

如果没有包含已提柜：
  - 按到港总数 = 4（只有未提柜的）
  - ❌ 数据不一致，无法解释另外 6 个去哪了
```

## 3️⃣ 按提柜分布 (Pickup Distribution)

### 🎯 统计范围
**at_port 状态的货柜**（且有目的港操作记录）

### 📋 分类依据
- **核心字段**:
  - `planned_pickup_date`（计划提柜日期）
  - `pickup_date`（实际提柜日期）
- **分类数**: 6 个
- **互斥性**: ✅ 严格互斥

### 🏷️ 分类明细

| 分类 | 字段名 | 颜色 | 核心条件 | 业务含义 |
|-----|--------|------|---------|---------|
| 计划提柜逾期 | overdue | 🔴 #f56c6c | planned < 今日 且 pickup is null | 计划已过期但未执行 ⚠️ |
| 今日计划提柜 | todayPlanned | 🟠 #e6a23c | planned = 今日 且 pickup is null | 今天计划要提的 |
| 今日实际提柜 | todayActual | 🟢 #67c23a | pickup = 今日 | 今天已经提走的 ✅ |
| 待安排提柜 | pending | 🟤 #909399 | 无拖卡运输记录 | 完全没安排的 |
| 3 天内预计提柜 | within3Days | 🔵 #409eff | 今日 < planned ≤ 3 天后 | 近期计划提柜 |
| 7 天内预计提柜 | within7Days | 🟢 #67c23a | 3 天后 < planned ≤ 7 天后 | 中期计划提柜 |

### 💼 业务用途
监控已到港货柜的提柜整体进度，包括已安排和未安排的。

### 🔍 关键逻辑
- "今日实际提柜"只统计当前状态仍为 `at_port` 的货柜
- 所有分类都要求有目的港操作记录（`port_type = 'destination'`）
- 已安排未执行的货柜会被多次统计（overdue/todayPlanned/within3Days/within7Days）

## 4️⃣ 最晚提柜分布 (Last Pickup Distribution)

### 🎯 统计范围
**at_port + 未实际提柜**（包括已安排未执行）

### 📋 分类依据
- **核心字段**: `last_free_date`（最后免费提柜日期）
- **过滤条件**: `pickup_date IS NULL`（未实际提柜）
- **分类数**: 5 个
- **互斥性**: ✅ 严格互斥

### 🏷️ 分类明细

| 分类 | 字段名 | 颜色 | 核心条件 | 业务含义 |
|-----|--------|------|---------|---------|
| 已超时 | expired | 🔴 #f56c6c | lastFreeDate < 今日 | 免租期已过，可能产生滞港费 ⚠️⚠️ |
| 即将超时 (1-3 天) | urgent | 🟠 #e6a23c | 今日 ≤ lastFreeDate ≤ 3 天后 | 紧急，需要立即处理 ⚠️ |
| 预警 (4-7 天) | warning | 🔵 #409eff | 3 天后 < lastFreeDate ≤ 7 天后 | 需要关注，提前准备 |
| 时间充裕 (7 天以上) | normal | 🟢 #67c23a | lastFreeDate > 7 天后 | 时间充足，按计划执行 |
| 最晚提柜日为空 | noLastFreeDate | 🟤 #909399 | lastFreeDate is null | 数据缺失，需补充 |

### 💼 业务用途
**聚焦免租期倒计时风险**，预警可能产生滞港费的货柜。

### 🔍 与"按提柜"的关键区别

| 维度 | 按提柜 | 最晚提柜 |
|-----|----------|---------|
| **业务定位** | 整体进度监控 | **风险预警** |
| **包含已安排未执行** | ✅ 是 | ✅ **是（修复后）** |
| **核心指标** | planned/pickup | **last_free_date** |
| **颜色语义** | 进度状态 | **风险等级** |

### ⚠️ 重要说明
- 2026-03-03 修复：从"无拖卡记录"改为"未实际提柜"
- 现在包含**已安排但未执行**的货柜（更全面的风险分析）
- 与"按提柜"部分重叠是设计使然（各有业务价值）

---

## 5️⃣ 最晚还箱分布 (Return Distribution)

### 🎯 统计范围
**picked_up + unloaded**（且未填写还箱时间）

### 📋 分类依据
- **核心字段**: `last_return_date`（最后还箱日期）
- **过滤条件**: 
  - `logistics_status IN ('picked_up', 'unloaded')`
  - `return_time IS NULL`（未还箱）
- **分类数**: 5 个
- **互斥性**: ✅ 严格互斥

### 🏷️ 分类明细

| 分类 | 字段名 | 颜色 | 核心条件 | 业务含义 |
|-----|--------|------|---------|---------|
| 已超时 | expired | 🔴 #f56c6c | lastReturnDate < 今日 | 已超过还箱期限 ⚠️⚠️ |
| 即将超时 (1-3 天) | urgent | 🟠 #e6a23c | 今日 ≤ lastReturnDate ≤ 3 天后 | 紧急，需尽快还箱 ⚠️ |
| 预警 (4-7 天) | warning | 🔵 #409eff | 3 天后 < lastReturnDate ≤ 7 天后 | 需要提前准备 |
| 时间充裕 (7 天以上) | normal | 🟢 #67c23a | lastReturnDate > 7 天后 | 时间充足 |
| 最后还箱日为空 | noLastReturnDate | 🟤 #909399 | lastReturnDate is null | 数据缺失，需补充 |

### 💼 业务用途
监控空箱返还期限，避免产生滞箱费（detention charge）。

### 🔍 关键逻辑
- 只统计已提柜（picked_up/unloaded）但未还箱的货柜
- 依赖空箱返还记录（empty_return 表）
- 与最晚提柜采用相同的日期范围划分标准

## 📊 跨维度关系图

### 目标集覆盖关系

```
┌─────────────────────────────────────┐
│       全部货柜 (按状态)              │
│                                     │
│  ┌───────────────────────────────┐  │
│  │ shipped + in_transit + at_port│  │ ← 按到港
│  │                               │  │
│  │  ┌─────────────────────────┐  │  │
│  │  │    at_port (按提柜)     │  │  │
│  │  │                         │  │  │
│  │  │  ┌───────────────────┐  │  │  │
│  │  │  │ at_port + 未提柜  │  │  │  │ ← 最晚提柜
│  │  │  └───────────────────┘  │  │  │
│  │  └─────────────────────────┘  │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘

          ┌──────────────────────┐
          │ picked_up + unloaded │  ← 最晚还箱
          └──────────────────────┘
```

### 数据流转与触发

```
not_shipped → [按状态]
    ↓
shipped → [按状态] + [按到港]
    ↓
in_transit → [按状态] + [按到港]
    ↓
at_port → [按状态] + [按到港] + [按提柜] + [最晚提柜]
    ↓
picked_up → [按状态] + [最晚还箱]
    ↓
unloaded → [按状态] + [最晚还箱]
    ↓
returned_empty → [按状态]（完成）
```

## ✅ 数据一致性验证

### 互斥性保证
每个维度内部的分类都**严格互斥**，确保货柜不会被重复计数。

### 完整性验证
- **按状态**: 总和 = 全部货柜
- **按到港**: 总和 = shipped + in_transit + at_port
- **按提柜**: 总和 = at_port（有目的港记录）
- **最晚提柜**: 总和 = at_port + 未实际提柜
- **最晚还箱**: 总和 = picked_up + unloaded（未还箱）

### 跨维度检查
```sql
-- 验证按到港 vs 按状态
按到港总数 ≤ (按状态：shipped + in_transit + at_port)

-- 验证按提柜 vs 按状态
按提柜总数 ≤ (按状态：at_port)

-- 验证最晚提柜 vs 按提柜
最晚提柜总数 ≤ 按提柜总数
差异 = todayActual（已实际提柜的部分）
```
