# 甘特图日期取值逻辑（5级优先级）

本文档说明甘特图货柜圆点显示时日期字段的取值逻辑，以及后端如何确保数据一致性。

---

## 一、问题背景

在甘特图开发过程中，发现部分货柜无法显示在甘特图上，控制台日志显示这些货柜"has no date"。经排查，这些货柜具有以下特征：

1. **没有港口操作记录**：`latestPortOperation` 为 `null`
2. **有海运记录**：`seaFreight` 对象存在且包含 `eta` 字段
3. **根级别日期字段为空**：`ataDestPort`、`etaDestPort`、`etaCorrection` 均为 `null`

这导致前端 `getContainerDate` 函数无法获取到有效日期，货柜无法在甘特图上显示。

---

## 二、日期取值5级优先级

### 2.1 前端取值逻辑（SimpleGanttChart.vue）

```typescript
const getContainerDate = (container: Container): Date | null => {
  // 优先级1: 实际到港日
  if (container.ataDestPort) return new Date(container.ataDestPort)
  
  // 优先级2: 修正ETA
  if (container.etaCorrection) return new Date(container.etaCorrection)
  
  // 优先级3: 预计到港日
  if (container.etaDestPort) return new Date(container.etaDestPort)
  
  // 优先级4: 港口操作记录
  if (container.portOperations && container.portOperations.length > 0) {
    const destPortOp = container.portOperations.find((op: any) => op.portType === 'destination')
    if (destPortOp?.ataDestPort) return new Date(destPortOp.ataDestPort)
    if (destPortOp?.etaCorrection) return new Date(destPortOp.etaCorrection)
    if (destPortOp?.etaDestPort) return new Date(destPortOp.etaDestPort)
  }
  
  // 优先级5: 海运表ETA（最后fallback）
  if (container.seaFreight?.eta) return new Date(container.seaFreight.eta)
  
  return null
}
```

### 2.2 后端数据回写（container.service.ts）

后端 `enrichContainersList` 方法在第106行实现日期回写逻辑：

```typescript
etaDestPort: latestPortOperation?.etaDestPort || container.seaFreight?.eta || null,
```

**关键点**：
- 当 `latestPortOperation` 为 `null` 时（即没有港口操作记录）
- 使用 `container.seaFreight?.eta` 作为 `etaDestPort` 的 fallback
- 确保前端直接从根级别字段读取，无需遍历嵌套对象

## 三、完整取值流程

```
货柜数据（后端 enrichContainersList）
    ↓
检查 latestPortOperation
    ↓
┌─────────────┬──────────────┐
│ 有港口操作  │ 无港口操作   │
└─────────────┴──────────────┘
    ↓              ↓
使用 latestPortOperation  使用 seaFreight.eta
    ↓              ↓
写回 etaDestPort   写回 etaDestPort
    ↓              ↓
前端读取 container.etaDestPort
```

## 四、为什么需要这个逻辑

### 4.1 业务场景

- **已出运但未到港**：货柜已装船出运（有海运记录），但尚未到达目的港（无港口操作记录）
- **数据同步延迟**：飞驼API数据同步存在延迟，港口操作记录可能晚于海运记录更新
- **新货柜**：刚创建的货柜只有海运记录，港口操作记录尚未生成

### 4.2 用户体验

- **完整性**：确保所有已出运的货柜都能在甘特图上显示
- **一致性**：前后端使用相同的取值逻辑，避免数据不一致
- **性能**：后端回写根级别字段，前端无需多次嵌套查询

## 六、验证方法

### 6.1 前端验证

打开浏览器控制台，查看日志：
```
[SimpleGantt] Container XXXXX has no date (port=XXX)
```

如果看到这个日志，说明该货柜的所有5个日期字段均为 `null`。

### 6.2 后端验证

查询 `process_sea_freight` 表，确认 `eta` 字段有值：
```sql
SELECT bill_of_lading_number, eta, port_of_discharge 
FROM process_sea_freight 
WHERE bill_of_lading_number = 'XXX';
```

查询 `process_port_operations` 表，确认无目的港记录：
```sql
SELECT container_number, port_type, eta_dest_port 
FROM process_port_operations 
WHERE container_number = 'XXX' AND port_type = 'destination';
```

---

## 七、注意事项

1. **seaFreight.eta 只能作为 fallback**：因为海运记录中的ETA可能不准确，应该优先使用港口操作记录中的ATA或ETA
2. **日期格式统一**：所有日期字段使用 `Date` 类型或 ISO 8601 字符串
3. **时区处理**：使用 dayjs 处理时区转换，确保日期比较准确性
4. **空值处理**：所有优先级都为空时，返回 `null`，前端跳过该货柜
