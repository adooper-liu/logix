## 一、概述

### 1.1 背景

随着项目规模扩大,前端 HTTP 请求面临以下挑战:

1. **重复请求** - 用户快速点击导致相同请求多次发出
2. **网络波动** - 临时性网络错误导致请求失败
3. **超时配置单一** - 所有接口统一 10s 超时,不合理
4. **并发失控** - 大量请求同时发出,可能压垮后端
5. **性能盲区** - 无法追踪慢请求和错误请求

### 1.2 目标

建立统一的前端请求治理体系,实现:

- ✅ 自动去重 - 防止重复请求浪费资源
- ✅ 智能重试 - GET 请求失败自动重试
- ✅ 分级超时 - 根据接口类型设置不同超时
- ✅ 并发控制 - 限制最大并发请求数
- ✅ 性能监控 - 统计耗时、告警慢请求

### 1.3 适用范围

**生效范围**: 仅对使用 `@/services/api` 的请求生效

**不生效范围**:

- ❌ `@/api/httpClient` (如 fiveNode.ts)
- ❌ 直接使用 axios 的代码
- ❌ 第三方库内置的请求

**建议**: 逐步迁移所有 HTTP 客户端到统一的 `@/services/api`

```
frontend/src/utils/
└── logger.ts                 # 日志工具
```

### 2.2 请求流程

```
发起请求
    ↓
[0] 性能监控 - 记录开始
    ↓
[1] Token + 国家代码注入
    ↓
[2] 动态超时设置(根据 URL)
    ↓
[3] 取消重复请求(相同参数)
    ↓
[4] 并发控制(超出则排队)
    ↓
发送请求
    ↓
成功 → [5a] 清除去重记录 → [6a] 记录结束 → 返回 data
    ↓
失败 → [5b] 清除去重记录 → [6b] 记录失败 → [7] 重试拦截器
                                              ↓
                                         可重试? → 是 → 重新走流程
                                              ↓
                                             否 → 显示错误
```

## 三、功能详解

### 3.1 请求去重

**模块**: `requestDeduplication.ts`

**原理**:

- 生成唯一键: `method:url:params:data`
- 新请求发出时,检查是否有相同 key 的请求在 pending
- 如有,取消旧请求,执行新请求

**示例**:

```typescript
// 连续两次相同请求
api.get('/containers?page=1')
api.get('/containers?page=1')

// 结果: 第一个被取消,只执行第二个
// 控制台: [RequestDedup] Canceled duplicate request: GET:/containers:...
```

**注意事项**:

- 仅对完全相同的请求生效(method + url + params + data 都相同)
- POST/PUT 也会参与去重,但因 data 不同,实际很少触发
- 取消的请求不会触发错误提示

---

### 3.2 自动重试

**模块**: `retryInterceptor.ts`

**策略**:

- **仅 GET 请求**可重试
- **最多重试 2 次**(共 3 次尝试)
- **指数退避延迟**: 1s, 2s
- **仅网络错误重试**: 408, 429, 500, 502, 503, 504

**配置**:

```typescript
// api.get 默认带重试配置
api.get('/containers', {
  retryConfig: {
    maxRetries: 2,
    retryDelay: 1000,
    retryableMethods: ['GET'],
    retryableStatuses: [408, 429, 500, 502, 503, 504],
  },
})
```

**示例**:

```typescript
// 模拟 500 错误
api.get('/test-error')

// 控制台:
// [Retry] Retrying request (1/2) after 1000ms
// [Retry] Retrying request (2/2) after 2000ms
```

**注意事项**:

- 重试会重新走拦截链(Token、去重、超时等)
- 重试成功后返回形态与首次成功一致(response.data)
- POST/PUT/DELETE 不重试,避免副作用

---

### 3.3 分级超时

**模块**: `timeoutConfig.ts`

**配置**:

```typescript
export const TIMEOUT_CONFIG = {
  default: 10000, // 10秒 - 默认
  fast: 5000, // 5秒  - 字典/配置查询
  normal: 15000, // 15秒 - 列表查询
  slow: 30000, // 30秒 - 统计/导出/同步
  upload: 60000, // 60秒 - 文件上传
}
```

**URL 匹配规则**:

```typescript
if (url.includes('/statistics')) return 30000 // 统计
if (url.includes('/import') || url.includes('/export')) return 60000 // 导入/导出
if (url.includes('/dict')) return 5000 // 字典
if (url.includes('/external') || url.includes('/sync')) return 30000 // 外部同步
return 15000 // 默认
```

**手动覆盖**:

```typescript
// 显式设置 timeout,优先级高于自动推断
api.get('/containers', { timeout: 20000 })
```

**注意事项**:

- 仅在未显式设置 timeout 时生效
- URL 匹配为子串匹配,需注意路径命名
- 真实路径需对照路由配置

---

### 3.4 并发控制

**模块**: `concurrencyControl.ts`

**策略**:

- **全局并发上限**: 10 (可配置)
- **单域名并发上限**: max(5, floor(maxConcurrent / 2))
- **超出限制的请求进入队列等待**
- **FIFO 队列**,先入先出

**示例**:

```typescript
// 同时发起 15 个请求
const promises = []
for (let i = 0; i < 15; i++) {
  promises.push(api.get(`/containers?page=${i}`))
}
await Promise.all(promises)

// 结果:
// - 前 10 个立即执行
// - 后 5 个进入队列
// - 每完成一个,队列中的下一个开始执行
```

**监控接口**:

```typescript
// 查看当前状态
concurrencyControl.getStatus()
// 返回: {
//   activeRequests: 8,
//   queuedRequests: 2,
//   maxConcurrent: 10,
//   domainStats: Map { 'localhost' => 8 }
// }

// 动态调整并发上限
concurrencyControl.setMaxConcurrent(20)

// 清空队列(紧急情况下)
concurrencyControl.clearQueue()
```

**注意事项**:

- 队列无上限,极端情况可能导致内存泄漏
- 建议添加队列长度限制(如最多 100 个)
- 队列中的请求会被取消如果调用 clearQueue()

---

### 3.5 性能监控

**模块**: `performanceMonitor.ts`

**功能**:

- 自动统计每个请求的耗时
- 记录慢请求(超过阈值,默认 5s)
- 提供性能统计数据
- 支持性能告警

**监控接口**:

```typescript
// 获取性能统计
performanceMonitor.getStats()
// 返回: {
//   totalRequests: 150,
//   avgDuration: 234,
//   maxDuration: 8765,
//   minDuration: 45,
//   slowRequests: 3,
//   errorRequests: 2
// }

// 获取最近的慢请求
performanceMonitor.getSlowRequests(10)

// 获取最近的错误请求
performanceMonitor.getErrorRequests(10)

// 清空统计数据
performanceMonitor.clear()
```

**告警规则**:

- 慢请求(duration > 5000ms) → logger.warn
- 错误请求(status >= 400) → logger.error

**注意事项**:

- 数据保存在内存中,刷新页面会丢失
- 限制记录数量为 1000 条,避免内存泄漏
- 预留日志上报接口,待实现

## 五、最佳实践

### 5.1 必须遵守

1. **统一使用 `@/services/api`**

   ```typescript
   // ✅ 正确
   import { api } from '@/services/api'
   const data = await api.get('/containers')

   // ❌ 错误
   import axios from 'axios'
   const data = await axios.get('/containers')
   ```

2. **禁止使用 console**

   ```typescript
   // ✅ 正确
   import { logger } from '@/utils/logger'
   logger.info('Message')

   // ❌ 错误
   console.log('Message')
   ```

3. **GET 请求确保幂等**
   - 重试机制依赖 GET 请求幂等性
   - 避免在 GET 中修改数据

4. **敏感操作禁用重试**
   ```typescript
   // 支付、下单等操作,手动禁用重试
   api.post('/payment', data, {
     retryConfig: { maxRetries: 0 },
   })
   ```

---

### 5.2 推荐做法

1. **关键接口手动设置超时**

   ```typescript
   // 重要统计接口,设置更长超时
   api.get('/containers/statistics', { timeout: 60000 })
   ```

2. **定期检查性能统计**

   ```typescript
   // 在开发环境定期查看
   if (process.env.NODE_ENV === 'development') {
     setInterval(() => {
       console.table(performanceMonitor.getStats())
     }, 30000)
   }
   ```

3. **慢请求优化**
   ```typescript
   // 发现慢请求后,优化后端或增加缓存
   const slowRequests = performanceMonitor.getSlowRequests(10)
   slowRequests.forEach(req => {
     logger.warn(`Slow API: ${req.method} ${req.url} took ${req.duration}ms`)
   })
   ```

## 六、常见问题

### Q1: 为什么我的请求没有被去重?

**A**: 检查以下几点:

1. 是否使用 `@/services/api`?
2. 两个请求的参数是否完全相同(method + url + params + data)?
3. 第一个请求是否已经完成?(已完成的不算 pending)

---

### Q2: 重试后返回的数据结构不对?

**A**: 确认使用的是修正后的版本:

- `retryInterceptor` 应使用 `apiClient.request(config)`
- 不应使用全局 `axios(config)`
- 返回形态应为 `response.data`,不是整包 response

---

### Q3: 如何禁用某个接口的重试?

**A**: 手动设置 `maxRetries: 0`:

```typescript
api.post('/critical-operation', data, {
  retryConfig: { maxRetries: 0 },
})
```

---

### Q4: 并发控制导致请求变慢?

**A**: 检查并发配置:

```typescript
// 查看当前状态
concurrencyControl.getStatus()

// 如果队列过长,可适当增加并发上限
concurrencyControl.setMaxConcurrent(20)
```

---

### Q5: 性能监控数据在哪里查看?

**A**: 在浏览器控制台执行:

```typescript
import { performanceMonitor } from '@/services/api'
performanceMonitor.getStats()
```
