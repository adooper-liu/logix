## 何时创建 SKILL

### 判断流程

```
需要记录的内容
   ↓
是业务知识/规则吗？
   ↓ 否 -> 不创建 SKILL（可能是代码注释或 README）
   ↓ 是
对 6 个月后的开发者有价值吗？
   ↓ 否 -> 临时笔记即可
   ↓ 是
需要跨项目/团队共享吗？
   ↓ 否 -> 放在代码注释中
   ↓ 是
创建 SKILL 文档
```

### 适合创建 SKILL 的场景

1. **核心业务逻辑**
   - 成本计算规则
   - 状态机流转
   - 数据映射关系

2. **复杂算法**
   - 智能排柜算法
   - 日期计算逻辑
   - 费用分摊规则

3. **踩坑经验**
   - TypeScript 类型错误修复
   - 数据库性能优化
   - 第三方 API 集成

4. **开发规范**
   - 命名约定
   - 代码风格
   - 最佳实践

### 不适合创建 SKILL 的场景

1. **简单的 CRUD 操作**
2. **框架使用说明**
3. **临时问题解决方案**
4. **个人学习笔记**

### 文件命名规范

**强制要求**：

1. **文件名使用中文**
   - 格式：`SKILL-{编号}-{中文主题}.md`
   - 示例：`SKILL-001-时间线标签显示规则.md`
   - 禁止：`SKILL-001-timeline-label-rules.md`（英文）

2. **主题简洁明了**
   - 控制在 10 个字以内
   - 准确反映文档内容
   - 使用专业术语

3. **编号规则**
   - SKILL-001 ~ SKILL-999
   - 按创建顺序编号
   - 编号唯一且连续

## 编写要点

### 1. 标题

**要求**:

- 简洁明了
- 体现核心价值
- 包含 SKILL 编号
- **使用中文主题**

**示例**:

```markdown
正确：SKILL-001-时间线标签显示规则
错误：关于时间线的一些显示规则说明
错误：SKILL-001-timeline-label-rules（英文）
```

### 2. 业务场景

**要求**:

- 描述真实业务问题
- 说明为什么需要这个功能
- 包含具体场景示例

**示例**:

```markdown
## 业务场景

在集装箱管理列表中，运营人员需要快速了解每个集装箱的物流状态和时间信息：

1. 已经航行了多少天？（历时）
2. 距离预计到港还有多少天？（倒计时）
3. 是否已经超过免费期？（超期）

这些时间信息对于安排提柜、还箱等后续操作至关重要。
```

### 3. 核心规则

**要求**:

- 使用表格归纳
- 条理清晰
- 优先级明确

**示例**:

```markdown
## 核心规则

| 节点类型 | 显示内容   | 计算逻辑               | 优先级 |
| -------- | ---------- | ---------------------- | ------ |
| 实际节点 | 历时天数   | today - actual_time    | 1      |
| 预测节点 | 倒计时天数 | estimated_time - today | 2      |
| 超期节点 | 超期天数   | today - last_free_date | 3      |
```

### 4. 代码实现

**要求**:

- 完整可运行
- 包含必要注释
- 有错误处理

**示例**:

```typescript
/**
 * 计算时间线节点的显示信息
 *
 * @param node - 时间线节点
 * @param referenceDate - 参考日期（通常为今天）
 * @returns 显示信息对象
 */
function calculateNodeDisplay(node: TimelineNode, referenceDate: Date): NodeDisplayInfo {
  // 1. 优先显示实际发生时间
  if (node.actualTime) {
    return {
      type: 'actual',
      days: calculateDays(node.actualTime, referenceDate),
      label: `${calculateDays(node.actualTime, referenceDate)}天`,
    }
  }

  // 2. 其次显示预计时间
  if (node.estimatedTime) {
    return {
      type: 'estimated',
      days: calculateDays(referenceDate, node.estimatedTime),
      label: `预计${formatDate(node.estimatedTime)}`,
    }
  }

  // 3. 无时间信息
  return {
    type: 'unknown',
    days: null,
    label: '待定',
  }
}
```

### 5. 显示效果

**要求**:

- 文字描述清晰
- 可用截图辅助
- 说明交互逻辑

**示例**:

```markdown
## 显示效果

### 列表视图

每个集装箱卡片右上角显示时间标签：

- 蓝色标签：历时 X 天（已发生）
- 绿色标签：倒计时 X 天（未发生）
- 红色标签：超期 X 天（已超期）

### 甘特图视图

时间轴上用不同颜色标识：

- 实线：实际发生的节点
- 虚线：预计的节点
- 红色区域：超期区间
```

### 6. 常见错误

**要求**:

- 列举真实错误
- 说明错误原因
- 给出正确做法

**示例**:

````markdown
## 常见错误

### 错误 1: 同时显示历时和倒计时

```typescript
// 错误：逻辑混乱
if (node.actualTime && node.estimatedTime) {
  display = `${actualDays}天 (历时) / ${countdownDays}天 (倒计时)`
}
```
````

**问题**: 用户无法理解一个节点为什么有两个时间。

**正确**: 根据优先级只显示一种。

### 错误 2: 忽略时区问题

```typescript
// 错误：直接使用 Date
const days = Math.floor((now - date) / (1000 * 60 * 60 * 24))
```

**问题**: 跨时区时计算错误。

**正确**: 统一使用 UTC 或指定时区。

````

## 文档格式规范

### 排版要求

1. **禁止 emoji**
   - 不使用任何表情符号
   - 保持专业风格

2. **使用 ASCII 箭头**
   ```markdown
   正确：步骤 1 -> 步骤 2 -> 步骤 3
   错误：步骤 1 → 步骤 2 → 步骤 3
````

3. **状态标记用文字**
   ```markdown
   正确：OK 通过：6 FAIL 失败：0
   错误：✅ 通过：6 ❌ 失败：0
   ```

### 代码块

1. **必须标注语言**

   ````markdown
   ```typescript
   // 代码内容
   ```
   ````

   ```

   ```

2. **添加必要注释**
   ```typescript
   /**
    * 函数说明
    * @param param - 参数说明
    */
   ```

### 表格

1. **表头清晰**
2. **对齐统一**
3. **内容简洁**

```markdown
| 字段 | 类型   | 必填 | 说明     |
| ---- | ------ | ---- | -------- |
| id   | string | 是   | 唯一标识 |
| name | string | 是   | 名称     |
```
