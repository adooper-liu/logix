## 一、甘特图调度显示

### 1.1 显示维度与泳道

| 维度 | 现有 | 建议扩展 | 与资源/约束的对应 |
|------|------|----------|-------------------|
| **泳道类型** | 按到港、按提柜计划、按最晚提柜、按最晚还箱 | 按计划清关、按计划卸柜；或**资源泳道**（车队、仓库卸柜、堆场） | 每泳道对应一类资源或一类时间分组 |
| **行（时间分组）** | 今日到港、已逾期、3 日内到港等 | 清关：待/中/已/超期；卸柜：待卸/已卸/超期 | 与 last_free_date、last_return_date 分段一致 |
| **格子内** | 货柜圆点 + tooltip | 计划 vs 实际（形状/颜色）；预警（红框/三角）；资源占用数 | 每格可展示「当日该资源已占用/总容量」 |
| **时间轴** | 日期 | 7/15/30 天可切换；与 planned_*、last_* 对齐 | 便于发现冲突与顺延 |

### 1.2 货柜在甘特图中的位置

货柜在甘特图中的**位置**由以下决定：

| 泳道类型 | 货柜落格规则 | 数据来源 |
|----------|--------------|----------|
| **按到港** | 按 ETA/修正 ETA/ATA 的日期落格 | process_port_operations |
| **按提柜计划** | 按 planned_pickup_date 落格 | process_trucking_transport |
| **按最晚提柜** | 按 last_free_date 落格；无则「最晚提柜日为空」行 | process_port_operations |
| **按最晚还箱** | 按 last_return_date 落格；无则「最晚还箱日为空」行 | process_empty_return |
| **按计划清关** | 按 planned_customs_date 落格 | process_port_operations |
| **按计划卸柜** | 按 planned_unload_date 落格 | process_warehouse_operations |
| **资源泳道**（车队/仓库/堆场） | 按**占用该资源的日期**落格：拖车=送仓日或提柜日；仓库=卸柜日；堆场=堆场停留日 | process_trucking_transport、process_warehouse_operations、堆场占用 |

### 1.3 三池视图（可选，参考 container-system）

```
港口池（清关前）  →  资源池（拖卡+卸柜）  →  空箱池（还箱后）
     │                    │                        │
  按清关状态分组      按车队/仓库/堆场泳道        按还箱状态
  可提柜后可拖拽      可拖拽调整日期/泳道         只读或筛选
```

### 1.4 三池评估：价值、劣势与行业实践

#### 1.4.1 价值与意义

| 维度 | 说明 |
|------|------|
| **流程可视化** | 港口→资源→空箱 三阶段与业务链（到港→清关→拖卡→卸柜→还箱）一一对应，货柜流转路径一目了然 |
| **调度操作直观** | 港口池「待排」、资源池「已排/可调」、空箱池「已完成」分区明确；拖拽从港口池到资源池即完成排产，符合「从待办到执行」的心智 |
| **状态分组清晰** | 每池内按状态分组（在途/将到港/已清关、车队/仓库/堆场、已还箱），便于按阶段筛选与统计 |
| **资源占用可见** | 资源池按车队/仓库/堆场泳道展示，同一格子内可显示占用数，便于发现冲突与顺延 |

#### 1.4.2 劣势与风险

| 劣势 | 说明 |
|------|------|
| **空间割裂** | 同一货柜在三个池中「各占一位」，难以一眼看到单柜的**全流程时间线**；需点击或悬停才能串联 |
| **时间轴弱化** | 池内多为「状态分组」而非「时间轴」；资源池虽有日期格子，但港口池、空箱池常按状态聚合，**时间维度**不如纯甘特突出 |
| **实现复杂度高** | 三池布局、拖拽逻辑、状态同步、实体/虚拟任务联动，前后端与状态机均需配套，开发与维护成本高 |
| **与现有 LogiX 差异大** | LogiX 当前为「按时间分组泳道」（按到港/提柜/还箱等），三池为「按阶段分区」，需较大改版 |
| **多国别/多仓库时扩展难** | 三池若按国别/仓库再拆子池，界面会急剧膨胀；筛选与折叠逻辑复杂 |

#### 1.4.3 行业实践对比

| 实践 | 适用场景 | 与三池的关系 |
|------|----------|--------------|
| **纯甘特（时间轴 + 资源泳道）** | 以「何时占用何资源」为核心，强调排产与档期 | 三池的资源池本质是甘特；港口池、空箱池为**前置/后置看板**，甘特是主视图 |
| **看板（Kanban）** | 按阶段列（待清关/待拖卡/待卸柜/已还箱），卡片流转，强调 WIP 与流动 | 三池与看板类似，但看板通常**无时间轴**；三池在资源池引入时间格子，是看板+甘特的混合 |
| **单一货柜时间线** | 以柜为主线，横向展示五节点计划/实际，强调单柜全流程 | 与三池互补：三池看「全局分布」，时间线看「单柜详情」；可结合 |
| **码头/堆场调度** | 场桥、外卡、内卡协同；预约制、动态分配 | 行业多用**甘特+热力图**展示资源负荷，较少用「池」；三池更偏**货代/仓配**视角 |
| **集成调度（Integrated Scheduling）** | 车队、仓库、堆场同时优化，多目标 | 学术与高端系统采用；三池为**可视化载体**，优化算法可后台运行，结果通过三池展示 |

#### 1.4.4 建议：分阶段选型

| 阶段 | 建议 | 理由 |
|------|------|------|
| **当前/阶段一** | **保留现有泳道甘特**，不引入三池 | 实现成本低；按到港/提柜/还箱等时间分组已满足大部分监控与统计；先补齐五节点泳道、预警、资源线 |
| **阶段二** | 若需**拖拽排产**，可引入**资源池**（单池） | 仅做「资源池」：车队+仓库+堆场泳道 + 时间格子 + 拖拽；港口池、空箱池用**列表/筛选**替代，降低复杂度 |
| **阶段三** | 若业务强需求「港口待排 → 拖拽到资源池」的完整流程，再引入**完整三池** | 与 container-system 对齐；需投入三池布局、状态机、实体/虚拟任务联动 |

**小结**：三池在**流程可视化**与**调度操作直观性**上有价值，但存在**空间割裂、时间轴弱化、实现复杂**等劣势。行业更常见的是**甘特+资源泳道**或**看板+甘特混合**。建议优先做强现有泳道甘特与资源线，再视业务需求决定是否引入三池或简化版（单资源池）。

## 二、货柜在资源间的关联与依赖、约束

### 2.1 货柜 ↔ 资源关联矩阵

| 资源 | 货柜占用方式 | 关联字段 | 约束 |
|------|--------------|----------|------|
| **港口** | 到港、清关、待提 | port_code、ata_dest_port、customs_status | 清关放行前不可排拖卡 |
| **清关能力**（可选） | 计划清关日 | planned_customs_date | 清关公司/口岸日产能 |
| **拖车/车队** | 提柜日、送仓日、还箱日（Drop 时） | trucking_company、planned_pickup_date、planned_delivery_date | 拖车日容量、**港口-仓库路线** |
| **仓库** | 卸柜日 | warehouse_code、planned_unload_date | 仓库日产能、卸柜方式（Drop/Live） |
| **第三堆场** | 堆场停留日（方案二时） | yard_code、停留天数 | 堆场日容量、费率 |
| **还箱堆场**（可选） | 还箱日 | return_terminal_*、planned_return_date | 还箱点开放/预约 |

### 2.2 货柜 ↔ 资源依赖链（顺序约束）

```
清关放行 ──→ 可排拖卡（提柜）
    │
    └── 提柜 ──→ 送仓 ──→ 卸柜 ──→ 还箱
         │         │        │        │
         │         │        │        └── 依赖：卸柜完成（空箱）
         │         │        └── 依赖：送仓日；约束：仓库日产能、卸柜方式
         │         └── 依赖：提柜日；约束：拖车档期、港口-仓库路线
         └── 约束：last_free_date、清关放行
```

- **拖卡** 依赖清关放行；受 last_free_date、拖车档期、仓库预约、卸柜方式、第三堆场方案约束。
- **卸柜** 依赖送仓日；受仓库日产能、已占用、卸柜方式约束。
- **还箱** 依赖卸柜完成；受 last_return_date、Live loading 时=卸柜日约束。

### 2.3 货柜 ↔ 资源约束校验规则

| 约束 | 校验条件 | 违反时 |
|------|----------|--------|
| **last_free_date** | planned_pickup_date ≤ last_free_date | 滞港风险；甘特标红/预警 |
| **last_return_date** | planned_return_date ≤ last_return_date | 滞箱风险；甘特标红/预警 |
| **仓库日产能** | 当日该仓 planned_count < capacity | 不可排入；或顺延至有剩余能力日 |
| **拖车日容量** | 当日该路线 planned_trips < capacity | 不可排入；或顺延 |
| **港口-仓库路线** | 拖车可服务该 港口↔仓库 | 不可选该拖车 |
| **卸柜方式** | Live：delivery=unload=return；Drop：可错开 | 计划须符合卸柜方式 |
| **schedule_status** | 仅 initial 可被智能排产覆盖 | issued/adjusted 不可覆盖 |

## 三、智能排柜机制

### 3.1 触发方式（三种可选）

| 方式 | 说明 | 适用场景 |
|------|------|----------|
| **手动一键排产** | 用户在甘特/列表页点击「一键排产」，选择日期范围、国家等 | 日常批量排产、补排 |
| **导入后自动排产** | Excel 导入货柜后，对 schedule_status=initial 的柜自动排产 | 新数据接入 |
| **定时任务** | 定时（如每日）对未排产或需重排的柜执行排产 | 周期性刷新 |

### 3.2 执行流程

```
1. 过滤：排除 schedule_status ≠ initial
2. 排序：按清关可放行日升序，同日内按 last_free_date 升序
3. 规则引擎：依次为每柜分配五节点计划时间（见 05、06）
   - 按备货单港口-仓库关系选仓
   - findEarliestAvailableDay（仓库）
   - findEarliestAvailableDeliveryDay（拖车档期）
   - findEarliestAvailableFleetReturnDay（车队送还能力，Drop 时）
4. 三种方案决策：仓库满负荷时比较总费用，选最小
5. 写回 planned_*、更新资源占用、设置 schedule_status=initial
6. 甘特图刷新展示
```

### 3.3 与甘特图的衔接

- 智能排产完成后，**甘特图**读取最新的 planned_*、资源占用，刷新泳道与格子内货柜位置。
- 可选：排产前展示「模拟结果」，用户确认后再写库。

## 四、手工拖拽调整并更新计划机制

### 4.1 可拖拽对象与目标

| 拖拽对象 | 来源位置 | 可拖拽至 | 含义 |
|----------|----------|----------|------|
| **货柜卡片/圆点** | 港口池（已清关） | 车队泳道 × 日期格子 | 排定提柜/送仓日、车队 |
| **货柜卡片/圆点** | 车队泳道 | 同泳道其他日期、或其他车队泳道 | 调整提柜/送仓日、换车队 |
| **货柜卡片/圆点** | 仓库卸柜泳道 | 同泳道其他日期、或其他仓库泳道 | 调整卸柜日、换仓 |
| **货柜卡片/圆点** | 堆场泳道 | 同泳道其他日期 | 调整堆场停留 |

### 4.2 拖拽落点校验（前端 + 后端）

| 校验项 | 前端 | 后端 |
|--------|------|------|
| **schedule_status** | 已下发等不可拖拽或仅只读 | 拒绝更新 |
| **last_free_date** | 拖拽至的提柜日若 > last_free_date，提示预警 | 可允许但标预警；或拒绝 |
| **last_return_date** | 还箱日若 > last_return_date，提示预警 | 同上 |
| **仓库日产能** | 落格前请求「当日剩余能力」 | 无剩余能力则拒绝或顺延 |
| **拖车档期** | 落格前请求「当日剩余档期」 | 无档期则拒绝或顺延 |
| **顺序约束** | 送仓日 ≥ 提柜日；卸柜日 ≥ 送仓日；还箱日 ≥ 卸柜日 | 违反则拒绝 |
| **卸柜方式** | Live 时 delivery=unload=return 须同 day | 违反则拒绝 |

### 4.3 拖拽完成后的更新逻辑

```
用户拖拽货柜 A 至「车队2 × 2025-03-15」
  ↓
1. 前端：校验 last_free_date、顺序、capacity（可先请求接口）
2. 前端：调用 PATCH /containers/:id/schedule 或 PUT
   Body: {
     planned_pickup_date: "2025-03-15",
     planned_delivery_date: "2025-03-16",  // 或由后端按卸柜日推算
     trucking_company_id: "车队2",
     schedule_status: "adjusted"
   }
  ↓
3. 后端：
   a. 校验 schedule_status（若 issued 可配置为允许手工覆盖）
   b. 校验 last_free_date、last_return_date、顺序
   c. 校验仓库/拖车/堆场剩余能力（释放旧占用 + 占用新格子）
   d. 写回 process_trucking_transport、process_warehouse_operations 等
   e. 更新 ext_warehouse_daily_occupancy、ext_trucking_slot_occupancy 等
   f. 若计划时间变更，触发 last_return_date 重算（若实现自动计算）
   g. 设置 schedule_status = "adjusted"
  ↓
4. 前端：刷新甘特图、列表、详情
```

### 4.4 覆盖规则与 schedule_status

| schedule_status | 智能排产 | 手工拖拽 | Excel 导入 |
|----------------|----------|----------|------------|
| **initial** | ✅ 可覆盖 | ✅ 可调整；调整后→adjusted | ✅ 可覆盖 |
| **issued** | ❌ 不可覆盖 | 可配置：❌ 仅手工或 ✅ 允许 | 可配置 |
| **adjusted** | ❌ 不可覆盖 | ✅ 可继续调整 | 可配置 |

## 五、甘特图与资源的联动展示

### 5.1 资源线（可选）

在泳道下方或第二层展示：

| 资源线 | 内容 | 用途 |
|--------|------|------|
| **仓库日产能** | 仓库 × 日期 → 已占用/总容量 | 发现冲突、顺延 |
| **拖车档期** | 车队 × 日期 × 路线 → 已占用/总容量 | 同上 |
| **堆场占用** | 堆场 × 日期 → 已占用/总容量 | 方案二时 |

### 5.2 约束冲突提示

| 场景 | 展示方式 |
|------|----------|
| 计划提柜日 > last_free_date | 货柜边框红色、tooltip「滞港风险」 |
| 计划还箱日 > last_return_date | 货柜边框红色、tooltip「滞箱风险」 |
| 当日仓库已满仍排入 | 落格前拒绝或标黄「超能力」 |
| 顺序约束违反 | 落格前拒绝、toast 提示 |

### 5.3 依赖关系线（可选）

- 同一货柜的 清关→提柜→送仓→卸柜→还箱 可画**依赖线**，便于理解顺序。
- 查验插入时，清关→查验→放行 可单独标注。

## 六、实体任务与虚拟占位任务的变动、互动机制

> 参考 container-system：主任务（实线色块）与虚线关联任务（预测性占位）的区分；主任务可拖拽、虚线任务由主任务派生并随主任务联动。

### 6.1 定义与区分

| 类型 | 说明 | 数据来源 | 展示 | 可操作性 |
|------|------|----------|------|----------|
| **实体任务** | 有实际数据支撑的调度任务，对应货柜在某一资源上的**计划占用** | process_trucking_transport、process_warehouse_operations、process_empty_return 等 | 实线色块、圆点 | 可拖拽、可编辑、可删除 |
| **虚拟占位任务** | 由实体任务**派生**的预测性占位，用于展示**下游节点预期**，无独立持久化 | 由主任务时间推算（如 送仓日→卸柜日、卸柜日→还箱日） | 虚线、浅色点 | 不可单独拖拽；随主任务联动 |

### 6.2 典型对应关系（五节点视角）

| 实体任务 | 虚拟占位任务 | 派生规则 |
|----------|--------------|----------|
| **车队主任务**（提柜/送仓） | **预计到库**（仓库泳道虚线） | 送仓日 = 预计到库日；虚线落在仓库泳道 × 送仓日 |
| **车队主任务**（送仓后） | **已到库未还箱**（仓库泳道虚线） | 送仓日～卸柜日之间；卸柜完成后虚线转为实体 |
| **仓库卸柜任务** | **预计还箱**（还箱泳道虚线） | 卸柜日 + 1～2 天（Drop）或当日（Live）；还箱完成后进入空箱池 |
| **堆场停留任务**（方案二） | **预计送仓**（仓库泳道虚线） | 堆场释放日 = 预计送仓日 |

### 6.3 变动机制

| 触发 | 实体任务 | 虚拟占位任务 |
|------|----------|--------------|
| **智能排产** | 新建/覆盖 planned_*，写入主表 | 由新计划时间推算，前端渲染时生成 |
| **手工拖拽主任务** | 更新 planned_pickup_date、planned_delivery_date 等；释放旧占用、占用新格子 | **联动更新**：送仓日变→预计到库日变→虚线位置变 |
| **确认到仓** | 车队主任务可标记完成/隐藏；仓库任务由「虚线」转为「实体」 | 虚线锁定为实体，可独立拖拽 |
| **卸柜完成** | 仓库任务完成；还箱任务可新建或已存在 | 虚线「预计还箱」转为实体或进入空箱池 |
| **退回港口池** | 主任务删除；释放车队/仓库占用 | 关联虚线一并移除 |

### 6.4 互动机制（联动规则）

```
主任务（实体）变更
    │
    ├── 提柜日变更 ──→ 送仓日可能变 ──→ 预计到库（虚线）位置变
    │                      │
    │                      └── 卸柜日可能变 ──→ 预计还箱（虚线）位置变
    │
    ├── 送仓日变更 ──→ 预计到库（虚线）位置变
    │                      │
    │                      └── 卸柜日可能变 ──→ 预计还箱（虚线）位置变
    │
    ├── 卸柜日变更 ──→ 预计还箱（虚线）位置变
    │
    └── 换仓/换车队 ──→ 虚线落格泳道变（仓库泳道/车队泳道）
```

- **单向派生**：虚拟占位任务由实体任务推算，不反向写库。
- **级联更新**：主任务时间变更时，所有关联虚线在前端或后端统一重算并刷新。
- **状态提升**：确认到仓、卸柜完成时，虚线可「锁定」为实体，获得独立 ID 与可编辑性。

### 6.5 存储与渲染建议

| 层级 | 实体任务 | 虚拟占位任务 |
|------|----------|--------------|
| **存储** | 写入 process_trucking_transport、process_warehouse_operations、process_empty_return 等 | 不单独建表；或扩展 is_virtual、related_task_id 字段做关联 |
| **渲染** | 从主表读取，按 planned_* 落格 | 前端根据主任务 planned_* 推算；或后端接口返回「派生任务」列表 |
| **拖拽** | 可拖拽；落点校验后写库 | 不可单独拖拽；拖拽主任务时虚线联动移动 |

### 6.6 与 container-system 的对应

| container-system | LogiX 方案 |
|------------------|------------|
| 车队主任务（实线 #5470c6） | 实体任务：process_trucking_transport |
| 预计到库（虚线 #91cc75） | 虚拟占位：由 planned_delivery_date 推算 |
| 已到库未还箱（#ee6666） | 虚拟占位：送仓后～卸柜前 |
| 已到库已还箱（#91cc75 实线） | 实体任务：process_warehouse_operations + process_empty_return |
| 「确认到仓」→ 主任务销毁、虚线锁定 | 状态提升：虚线→实体，可独立管理 |
| is_virtual、related_task_id | 可选：扩展表字段或仅前端逻辑 |

## 八、小结

| 主题 | 要点 |
|------|------|
| **甘特图显示** | 泳道=时间分组或资源类型；货柜按 planned_*/last_* 落格；可扩展资源泳道、资源线、依赖线 |
| **三池评估** | 价值：流程可视化、调度直观；劣势：空间割裂、时间轴弱化、实现复杂；建议分阶段，优先做强泳道甘特，再视需求引入三池或单资源池 |
| **货柜-资源关联** | 港口、拖车、仓库、堆场、还箱点；关联字段与约束校验规则 |
| **依赖与约束** | 顺序：清关→提柜→送仓→卸柜→还箱；last_free_date、last_return_date、日产能、路线 |
| **智能排产** | 三种触发；规则引擎+三种方案决策；仅 initial 可覆盖；排产后刷新甘特 |
| **手工拖拽** | 货柜在泳道间/日期间拖拽；落点校验（能力、约束）；更新 planned_*、资源占用、schedule_status=adjusted |
| **实体 vs 虚拟任务** | 实体任务有持久化、可拖拽；虚拟占位由主任务派生、联动更新；确认到仓/卸柜完成时虚线可锁定为实体 |
