# 排柜数据补全与缺省方案

> 解决 customer_code 为空、last_free_date/last_return_date 为空等问题，保障智能排柜所需参数完整。

---

## 一、customer_code 为空

### 1.1 问题

- `biz_replenishment_orders.customer_code` 为空时，无法通过 customer 关联取 `country`
- 当前回退：`biz_customers WHERE customer_name = sell_to_country` 取 country，依赖 customer_name 与 sell_to_country 完全一致
- 风险：customer_name 变更、重名、拼写差异会导致取不到 country

### 1.2 解决方案

#### 方案 A：数据回填（推荐，一次性修复历史数据）

```sql
-- 根据 sell_to_country = customer_name 回填 customer_code
UPDATE biz_replenishment_orders o
SET customer_code = c.customer_code
FROM biz_customers c
WHERE c.customer_name = o.sell_to_country
  AND o.customer_code IS NULL
  AND o.sell_to_country IS NOT NULL;
```

**迁移脚本**：`migrations/backfill_customer_code_from_sell_to_country.sql`

**导入增强**：`backend/src/controllers/import.controller.ts` 中 `fillCustomerCodeFromSellToCountry()` 已在 Excel 导入时自动补全。

#### 方案 B：Excel 导入时自动补全

- 导入备货单时，若「销往国家」/「客户名称」有值且 customer_code 为空，根据 `biz_customers.customer_name` 或 `sell_to_country` 查找 `customer_code` 并写入
- 实现位置：`backend/src/controllers/import.controller.ts` 备货单写入逻辑
- 逻辑：`customer_name` 或 `sell_to_country` 列 → 查 `biz_customers WHERE customer_name = 值` → 取 `customer_code` 写入 `customer_code` 字段

#### 方案 C：保存时校验

- 备货单新增/编辑时，若 `sell_to_country` 有值且 `customer_code` 为空，提示或自动补全
- 实现位置：备货单 API 或前端表单提交前

**建议**：先执行方案 A 回填历史数据，再在导入（方案 B）中增加补全逻辑，防止新数据再次出现空值。

## 二、last_free_date 为空

### 2.1 计算口径

**注意**：以下迁移脚本已废弃，请勿执行：

- `migrations/backfill_last_free_date.sql`（原简单 ETA+7 天逻辑已注释）
- `migrations/backfill_last_return_date.sql`（原 last_free_date+7 天逻辑已注释）

正确做法：使用滞港费服务批量写回。

### 2.2 计算规则

**last_free_date（最后免费日）应按以下规则计算**，而非简单 ETA+7 天：

**基准日（免费期起算日）**：按优先级取一

| 计算方式 | actual 模式 | forecast 模式 |
|----------|-------------|---------------|
| 按到港 | ATA → 实际卸船日 | ATA → 实际卸船日 → **修正 ETA** → **ETA** |
| 按卸船 | 实际卸船日（无则退回按到港） | 实际卸船日 → 修正 ETA → ETA |

**免费期结束日**：`last_free_date = 基准日 + (免费天数 − 1)`，按 **免费天数基准**（Natural Days / Working Days / Natural+Working）折算。

**免费天数来源**：`ext_demurrage_standards` 滞港费标准表，四字段匹配（境外公司、目的港、船公司、货代）取 `free_days`、`free_days_basis`、`calculation_basis`。

详见：`frontend/public/docs/demurrage/02-CONTAINER_SCHEDULING_AND_COST_OPTIMIZATION_PLAN.md` §3.1、`08-DEMURRAGE_CALCULATION_MODES.md`。

### 2.3 解决方案：使用滞港费服务批量写回

**项目已实现**：`backend/src/services/demurrage.service.ts` 的 `batchWriteBackComputedDates()` 会：

1. 查找 last_free_date 为空的货柜（**目标集扩展**：已到目的港 或 未到目的港但有 ETA）
2. 匹配滞港费标准（四字段 + 有效期）
3. 按 修正 ETA → ETA 取基准日，按标准 free_days 计算 last_free_date
4. 写回 `process_port_operations.last_free_date`（forecast 模式也允许写回，支持智能排柜）

**触发方式**：

- **定时任务**：`DemurrageWriteBackScheduler` 默认每 6 小时执行（`DEMURRAGE_WRITEBACK_SCHEDULER_INTERVAL`）
- **手动触发**：`POST /api/v1/demurrage/batch-write-back`
- **智能排产前**：`batchSchedule` 入口自动调用批量写回

**废弃迁移脚本**：
- `migrations/backfill_last_free_date.sql` - 原简单 ETA+7 天逻辑已注释
- `migrations/backfill_last_return_date.sql` - 原 last_free_date+7 天逻辑已注释

## 三、last_return_date 为空

### 3.1 计算口径

**last_return_date（最晚还箱日）**：滞箱费标准中的免费用箱天数，起算日 = 实际提柜日（有则用）或 last_free_date。

**项目已实现**：`demurrage.service` 的 `batchWriteBackComputedDates()` 会一并计算并写回 last_return_date（需有实际提柜日或计划提柜日，且匹配滞箱费标准）。

**触发**：与 last_free_date 相同，调用 `POST /api/v1/demurrage/batch-write-back` 或等待定时任务。

---

## 四、执行顺序

1. **customer_code 回填**：`migrations/backfill_customer_code_from_sell_to_country.sql`
2. **last_free_date / last_return_date**：调用 `POST /api/v1/demurrage/batch-write-back` 或等待定时任务（按修正 ETA、ETA、滞港费标准计算）
3. **导入增强**：Excel 导入时 `fillCustomerCodeFromSellToCountry` 自动补全 customer_code
