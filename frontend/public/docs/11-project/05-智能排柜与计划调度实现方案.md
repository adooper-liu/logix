# 智能排柜与计划调度实现方案

> 基于 [04-五节点调度与可视化方案.md](./04-五节点调度与可视化方案.md) 中的业务规则、资源约束与决策目标，本文档描述如何实现**智能排柜**与**计划调度**，生成五节点的计划时间与资源占用。

---

## 一、目标与产出

### 1.1 目标

在给定约束下，为每个货柜**自动或半自动**生成五节点计划时间，并记录资源占用，支撑甘特图展示与排产决策。

### 1.2 产出（per 柜）

| 节点 | 计划时间字段 | 资源占用 |
|------|--------------|----------|
| **清关** | planned_customs_date | 清关公司/口岸（可选） |
| **拖卡** | planned_pickup_date, planned_delivery_date | 拖车、日期档期（拖车日容量 + 港口-仓库路线） |
| **卸柜** | planned_unload_date | 仓库、卸柜口/日产能（按备货单的港口-仓库关系选仓） |
| **还箱** | planned_return_date | 还箱堆场（可选） |
| **查验** | inspection_planned_date（若需） | 查验档期（可选） |
| **计划状态** | schedule_status | initial（初始计划）/ issued（已下发）/ adjusted（手工调整）等；仅 initial 可被智能排产覆盖 |

---

## 二、输入与约束

### 2.1 输入数据

| 来源 | 字段/表 | 用途 |
|------|---------|------|
| 货柜 | container_number、目的港、船公司、货代、柜型 | 匹配滞港/滞箱标准 |
| 备货单 | 关联的港口、目标仓库 | **港口-仓库对应关系**，多仓库时在该关系内选仓 |
| 到港 | ATA、修正 ETA、ETA、dest_port_unload_date | 基准日、last_free_date 计算 |
| 清关 | customs_status、document_status、inspection_required | 可提柜判断、查验排期 |
| 计算字段 | last_free_date、last_return_date | 拖卡、还箱硬约束 |
| 计划状态 | schedule_status | 仅 initial 时可覆盖；issued/adjusted 不可覆盖 |
| 配置 | 滞港费标准、滞箱费标准、仓库日产能、卸柜方式、**第三堆场容量与费用**、**拖车日容量与港口-仓库路线** | 费用计算、能力校验 |

### 2.2 约束（来自 04 方案 §2.3、§2.3.1、§2.3.2）

| 类型 | 约束内容 |
|------|----------|
| **时间约束** | planned_pickup_date ≤ last_free_date；planned_return_date ≤ last_return_date |
| **顺序约束** | 清关放行 → 提柜 → 送仓 → 卸柜 → 还箱 |
| **资源约束** | 仓库日产能、卸柜方式（Drop off / Live loading）、**拖车日容量与港口-仓库路线**、**第三堆场容量与费用** |
| **决策目标** | 总费用最少：在直接送仓 / 先提送堆场再送仓 / 滞港等仓再送仓 三者中择一 |

## 三、实现思路（分阶段）

### 3.1 阶段一：规则引擎（先到先得 + 约束校验）

**思路**：按「到港/清关可放行日」排序，依次为每柜分配计划时间与资源，满足硬约束。

**流程**：

```
1. 取目标货柜集（按出运日期、国家等筛选）；**排除 schedule_status ≠ initial 的柜**（已下发等不可覆盖）
2. 按 清关可放行日（ATA 或 修正 ETA） 升序排序
3. 对每个货柜：**按备货单的港口-仓库对应关系确定候选仓库**，在候选仓库内选仓
   a. 计划清关日 = 清关可放行日（或 + 1 天缓冲）
   b. 计划提柜日 = 计划清关日 + N 天（N 可配置），且 ≤ last_free_date
   c. 查仓库剩余能力：从 计划提柜日 起，找第一个「当日剩余能力 > 0」的日期 → 计划卸柜日
   d. 计划送仓日 = 计划卸柜日（Live loading）或 计划卸柜日前一天（Drop off，柜可落地）
   e. 计划还箱日 = 计划卸柜日（Live loading）或 计划卸柜日 + 1～2 天（Drop off），且 ≤ last_return_date
   f. 写回 planned_* 字段；扣减仓库日产能
4. 输出：每柜五节点计划时间 + 仓库日产能占用表
```

**资源占用**：维护「仓库 × 日期 → 已占用柜数」，分配时查询并更新。

### 3.2 阶段二：三种方案决策（总费用最少）

**思路**：当仓库满负荷时，对三种方案分别估算总费用，选最小方案并生成计划。

**流程**：

```
1. 估算仓库释放能力天数 N（排队柜数 ÷ 日产能，或配置「预计 N 天后可接新柜」）
2. 方案一（直接送仓）：若当前有剩余能力 → 排期；否则顺延，产生滞港费
3. 方案二（先提送堆场再送仓）：港→堆场拖车费 + N 天堆场费 + 堆场→仓拖车费 + 多环节风险成本
4. 方案三（滞港等仓再送仓）：N 天滞港费 + 之后一次性提柜送仓
5. 比较总费用，选最小方案
6. 按选定方案生成 planned_pickup_date、planned_delivery_date、planned_unload_date、planned_return_date
```

**资源占用**：方案二需占用第三堆场 N 天；方案一、三占用仓库日产能。

### 3.3 阶段三：智能优化（可选）

**思路**：在满足约束的前提下，做多目标优化（滞港费、滞箱费、拖车费、仓库均衡等）。

**可选手段**：

| 手段 | 说明 |
|------|------|
| **启发式** | 按 last_free_date 紧迫度、优先级排序，优先排「即将超期」柜 |
| **整数规划** | 以总费用最小为目标，决策变量为每柜的提柜日、送仓日、卸柜日、还箱日 |
| **遗传/模拟退火** | 在可行解空间内搜索更优排产 |

**参考**：container-system 的「AI 智能排柜」（GANTT-CHART-COMPLETE-DESIGN），用历史数据 + 实时数据做多目标优化。

## 四、资源占用数据结构

### 4.1 建议表

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| **ext_warehouse_daily_occupancy** | 仓库日产能占用 | warehouse_code, date, planned_count, capacity, remaining |
| **ext_trucking_slot_occupancy** | 拖车日容量占用 | trucking_company_id, date, planned_trips, capacity；需配置**港口-仓库路线**（哪些拖车可跑哪些港口↔仓库） |
| **ext_yard_daily_occupancy** | 第三堆场日占用 | yard_code, date, planned_count, capacity |
| **dict_yards**（或 ext_yard_capacity_fee） | **第三堆场容量与费用参数表** | yard_code, yard_name, daily_capacity, fee_per_day, fee_per_container, port_code, warehouse_code（与港口/仓库对应关系） |
| **港口-仓库映射** | 备货单选仓依据 | 可在 dict_ports、dict_warehouses 扩展，或独立 port_warehouse_mapping(port_code, warehouse_code) |

### 4.2 更新时机

- **计划生成时**：写入或更新占用；每次批量排产重新计算并写回
- **实际发生时**：可选「计划占用 + 实际扣减」双轨，或仅以计划占用为准

### 4.3 与 ext_warehouse_unload_capacity 的关系

- `ext_warehouse_unload_capacity`：仓库日产能配置（每日可卸柜数、已预约等）
- `ext_warehouse_daily_occupancy`：按日记录计划占用，可从 capacity 表读配置，写回占用

## 六、实现路径建议

| 步骤 | 内容 | 优先级 |
|------|------|--------|
| 1 | 建仓库日产能与占用表、**第三堆场容量与费用参数表**、**港口-仓库映射**、**拖车日容量与港口-仓库路线** | 高 |
| 2 | 计划表增加 **schedule_status**（initial/issued/adjusted）；覆盖规则：仅 initial 可被智能排产覆盖 | 高 |
| 3 | 实现规则引擎：按清关可放行日排序 → 依次分配五节点计划时间；**按备货单港口-仓库关系选仓**；校验 last_free_date、last_return_date、仓库剩余能力、拖车日容量 | 高 |
| 4 | 资源占用：分配时更新仓库日占用、拖车档期、第三堆场占用；支持 Drop off / Live loading 区分 | 高 |
| 5 | 三种方案决策：仓库满负荷时，比较直接送仓 / 堆场 / 滞港的总费用（堆场费来自第三堆场费用参数表），选最小并生成计划 | 中 |
| 6 | 触发方式：支持**手动一键排产**、**导入后自动排产**、**定时任务**三种，可配置切换 | 中 |
| 7 | API：`POST /containers/schedule` 或 `POST /containers/batch-schedule`，入参为柜号列表 + 日期范围，返回计划时间 + 资源占用 | 中 |
| 8 | 前端：甘特图展示计划时间，可选「一键排产」触发批量调度 | 中 |
| 9 | 智能优化：在规则引擎稳定后，再引入多目标优化算法 | 低 |

## 七、API 设计建议

### 7.1 批量排产

```
POST /api/v1/containers/batch-schedule

Request:
{
  "containerNumbers"?: string[],   // 可选，不传则按筛选条件
  "startDate": "2025-03-01",
  "endDate": "2025-03-31",
  "countryCode"?: string,
  "options"?: {
    "overwriteExisting"?: boolean,  // 是否覆盖已有计划；仅当 schedule_status=initial 时可覆盖，issued/adjusted 不可覆盖
    "scheduleMode"?: "direct" | "yard" | "demurrage" | "auto"  // auto 表示自动选三种方案
  }
}

Response:
{
  "scheduled": number,
  "skipped": number,
  "errors"?: Array<{ containerNumber: string, reason: string }>,
  "occupancy"?: { warehouse: [...], yard: [...] }
}
```

### 7.2 单柜模拟（不写库）

```
POST /api/v1/containers/:containerNumber/schedule-preview

Response:
{
  "recommendedPlan": "direct" | "yard" | "demurrage",
  "plannedDates": { customs, pickup, delivery, unload, return },
  "costComparison": { direct, yard, demurrage },
  "constraints": { lastFreeDate, lastReturnDate, warehouseCapacity }
}
```

## 八、已确认规则

| 事项 | 已确认规则 |
|------|------------|
| **触发方式** | **三种方式都要有，可选**：① 手动「一键排产」；② 导入后自动排产；③ 定时任务。系统支持配置或切换触发方式。 |
| **覆盖规则** | 已有计划时：若计划状态为**已下发**等（非初始计划），**不能覆盖**，只能手工调整；**仅当计划状态为「初始计划」时可覆盖**。需在计划表增加 `schedule_status` 字段（如 `initial` / `issued` / `adjusted` 等）。 |
| **多仓库** | **按备货单的在港口对应仓库关系间选择**。需维护「港口 → 仓库」映射或备货单关联的候选仓库，排产时在该关系内选仓。 |
| **拖车档期** | **只建模拖车日容量**，及**港口、仓库间的对应关系**。即：拖车公司/车队有日容量；拖车可服务的「港口 ↔ 仓库」路线需配置，排产时校验路线可用性与日容量。 |
| **第三堆场** | **要建立容量与费用参数表**。独立主数据表：堆场编码、名称、**日容量**、**费率**（按天或按柜）、与港口/仓库的对应关系；用于三种方案决策中的堆场费计算与容量校验。 |
