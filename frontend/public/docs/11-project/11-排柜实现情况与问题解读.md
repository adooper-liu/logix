## 二、手工排柜实现情况

### 2.1 已实现

| 功能 | 实现位置 | 说明 |
|------|----------|------|
| **计划编辑表单** | `ScheduleEditDialog.vue` | 清关日、提柜日、送仓日、卸柜日、还箱日、车队、仓库、清关公司、卸柜方式 |
| **PATCH schedule API** | `container.controller.ts` → `updateSchedule` | 更新 process_port_operations、process_trucking_transport、process_warehouse_operations、process_empty_return |
| **映射校验** | `updateSchedule` 内 | 港口→车队（TruckingPortMapping）、仓库→车队（WarehouseTruckingMapping） |
| **顺序约束** | `updateSchedule` 内 | 清关≤提柜≤送仓≤卸柜≤还箱 |
| **schedule_status=adjusted** | `updateSchedule` 内 | 手工调整后写回 TruckingTransport |
| **甘特图拖拽** | `useGanttLogic.ts` → `handleDrop` | 调用 `containerService.updateSchedule`，支持 plannedPickupDate、plannedReturnDate |

### 2.2 手工排柜缺口

| 缺口 | 说明 |
|------|------|
| **last_free_date 校验** | 计划提柜日 > last_free_date 时未在前端/后端强制校验或预警 |
| **last_return_date 校验** | 计划还箱日 > last_return_date 时未强制校验或预警 |
| **甘特图拖拽字段不全** | 仅支持 plannedPickupDate、plannedReturnDate，未支持 plannedDeliveryDate、plannedUnloadDate、车队、仓库 |
| **资源泳道拖拽** | 无车队/仓库泳道，无法拖拽至「车队×日期」「仓库×日期」 |

## 三、智能排柜实现情况

### 3.1 已实现

| 功能 | 实现位置 | 说明 |
|------|----------|------|
| **批量排产 API** | `POST /scheduling/batch-schedule` | 入参 country、startDate、endDate、forceSchedule |
| **规则引擎（阶段一）** | `intelligentScheduling.service.ts` | 先到先得、按清关可放行日排序 |
| **过滤条件** | `getContainersToSchedule` | schedule_status=initial，且有 ATA 或 ETA |
| **计划时间计算** | `scheduleSingleContainer` | 清关日、提柜日、送仓日、卸柜日、还箱日 |
| **候选仓库** | `getCandidateWarehouses` | 按 sell_to_country 过滤 |
| **车队选择** | `selectTruckingCompany` | 按仓库→车队映射 + 拖车档期 |
| **资源占用** | `decrementWarehouseOccupancy`、`decrementTruckingOccupancy` | 扣减仓库日产能、拖车档期 |
| **一键排产入口** | `Shipments.vue` → `handleBatchSchedule` | 使用顶部日期、国家筛选 |

### 3.2 智能排柜问题（需修复）

| 问题 | 位置 | 说明 | 状态 |
|------|------|------|------|
| **1. destPo.destinationPort 不存在** | `intelligentScheduling.service.ts:237, 268` | PortOperation 实体无 `destinationPort`，应改为 `destPo.portCode` | ✅ 已修复 |
| **2. mapping.truckingCompanyCode 错误** | `intelligentScheduling.service.ts:432, 441` | WarehouseTruckingMapping 字段为 `truckingCompanyId`，非 `truckingCompanyCode` | ✅ 已修复 |
| **3. destPo.lastReturnDate 不存在** | `intelligentScheduling.service.ts:231` | PortOperation 无 lastReturnDate；last_return_date 在 process_empty_return，需从 EmptyReturn 或计算得出 | ✅ 已修复 |
| **4. schedule_status 双存与来源** | Container / TruckingTransport | Container 实体有 scheduleStatus，但 `biz_containers` 表在 03_create_tables 中无 schedule_status 列；实际在 process_trucking_transport。getContainersToSchedule 用 `c.scheduleStatus` 过滤，若 biz_containers 无此列会报错。需统一：以 TruckingTransport.scheduleStatus 为准，或给 biz_containers 增加 schedule_status 并同步 | ✅ 已修复 |
| **5. updateContainerSchedule 不创建 TruckingTransport** | `intelligentScheduling.service.ts:336-348` | 仅 `update` 已存在的 truckingTransport；若货柜无 process_trucking_transport 记录，不会创建，导致计划未写入 | ✅ 已修复 |
| **6. 候选仓库未按港口-仓库映射** | `getCandidateWarehouses` | 仅按国家过滤，未按 dict_trucking_port_mapping、dict_warehouse_trucking_mapping 推导「目的港→可送仓库」 | ✅ 已修复 |
| **7. 卸柜方式写死** | `scheduleSingleContainer:219` | `unloadMode = 'Drop off'` 写死，应从备货单或 process_warehouse_operations 读取 | ⚠️ 部分修复（仍为TODO，需业务数据支持） |
| **8. findEarliestAvailableDay 日期类型** | `intelligentScheduling.service.ts:356` | `where: { warehouseCode, date }` 中 date 为 Date 对象，可能与数据库 date 类型比较有精度问题 | ✅ 已修复 |
| **9. selectTruckingCompany 用 today** | `intelligentScheduling.service.ts:328` | 占用校验用 `today`，应使用 plannedPickupDate 对应日期 | ✅ 已修复 |
| **10. schedule-preview 未实现** | `scheduling.controller.ts:47` | 返回「预览功能开发中」，未实现 | ✅ 已修复 |

## 四、数据流与 API 汇总

### 4.1 手工排柜

```
ScheduleEditDialog / 甘特图 handleDrop
    → containerService.updateSchedule(containerNumber, scheduleData)
    → PATCH /api/v1/containers/:id/schedule
    → container.controller.updateSchedule
    → 校验映射、顺序 → 写 process_port_operations、process_trucking_transport、
      process_warehouse_operations、process_empty_return
    → schedule_status = 'adjusted'
```

### 4.2 智能排柜

```
Shipments 一键排产
    → containerService.batchSchedule({ country, startDate, endDate })
    → POST /api/v1/scheduling/batch-schedule
    → intelligentSchedulingService.batchSchedule
    → getContainersToSchedule (schedule_status=initial)
    → sortByClearanceDate
    → scheduleSingleContainer (每柜)
    → updateContainerSchedule、decrementWarehouseOccupancy、decrementTruckingOccupancy
    → schedule_status = 'issued'
```

## 五、修复建议优先级

| 优先级 | 问题 | 修复方向 |
|--------|------|----------|
| **P0** | destPo.destinationPort → portCode | 改 intelligentScheduling.service.ts |
| **P0** | mapping.truckingCompanyCode → truckingCompanyId | 改 intelligentScheduling.service.ts |
| **P0** | lastReturnDate 来源 | 从 EmptyReturn 或 last_free_date+免费用箱天数计算 |
| **P0** | updateContainerSchedule 不创建记录 | 若 truckingTransport/warehouseOp/emptyReturn 不存在则 create |
| **P1** | schedule_status 来源统一 | 以 TruckingTransport 为准，或给 biz_containers 加 schedule_status 并同步 |
| **P1** | selectTruckingCompany 用 plannedPickupDate | 占用校验日期改为实际排产日 |
| **P1** | 手工排柜 last_free_date、last_return_date 校验 | 在 updateSchedule 中增加校验 |
| **P2** | 候选仓库按港口-仓库映射 | 用 dict_trucking_port_mapping、dict_warehouse_trucking_mapping 推导 |
| **P2** | 卸柜方式从数据读取 | 不再写死 Drop off |
| **P2** | schedule-preview 实现 | 单柜模拟不写库 |

## 七、单柜排柜模拟

可用脚本模拟单柜排柜全流程：

```bash
cd backend

# Dry-run（仅展示流程，不写库）
npx tsx scripts/simulate-schedule-one.ts
npx tsx scripts/simulate-schedule-one.ts TEMU1234567

# 实际执行（写库）
npx tsx scripts/simulate-schedule-one.ts TEMU1234567 --execute
```

或使用 npm 脚本：`npm run simulate:schedule [柜号] [--execute]`

**前置条件**：柜子需满足 `schedule_status=initial` 且有目的港 ATA 或 ETA。
