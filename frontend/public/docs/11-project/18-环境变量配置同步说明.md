## 配置优先级

TypeORM 和 Node.js 的环境变量加载顺序：

```
命令行参数 > .env.dev > .env > .env.example > 系统环境变量
```

**实际使用**:

- 开发环境: `.env.dev` 优先
- 生产环境: `.env` 优先
- 新成员加入: 复制 `.env.example` 为 `.env`

## 如何同步配置

### 场景 1: 添加新的配置项

**步骤**:

1. **更新 `.env.example`**（模板）

   ```bash
   # 添加新配置项
   NEW_CONFIG=value
   ```

2. **更新 `.env`**（生产配置）

   ```bash
   # 填入实际值
   NEW_CONFIG=actual_value
   ```

3. **更新 `.env.dev`**（开发配置）

   ```bash
   # 开发环境可能需要不同的值
   NEW_CONFIG=dev_value
   ```

4. **提交 `.env.example` 到 Git**
   ```bash
   git add backend/.env.example
   git commit -m "docs: 添加新配置项 NEW_CONFIG"
   ```

---

### 场景 2: 修改现有配置

**示例**: 修改 AI 提供商配置

1. **检查三个文件是否一致**

   ```bash
   # 对比差异
   diff backend/.env backend/.env.dev
   diff backend/.env backend/.env.example
   ```

2. **统一修改**
   - 在三个文件中同时更新相关配置
   - 确保 `.env.example` 使用占位符或示例值

3. **验证配置**
   ```bash
   # 重启服务使配置生效
   npm run dev
   ```

---

## 最佳实践

### ✅ 推荐做法

1. **保持同步**
   - 三个文件的配置项应该保持一致
   - 只有具体值可以不同（如 API Key、数据库密码等）

2. **使用占位符**
   - `.env.example` 中使用 `xxx` 或空值
   - 清晰的注释说明如何获取真实值

3. **分类组织**
   - 按功能分组（数据库、Redis、AI 等）
   - 添加清晰的分隔注释

4. **文档化**
   - 每个配置项都有注释说明
   - 复杂配置提供示例值

---

### ❌ 避免的做法

1. **不要提交敏感信息**

   ```bash
   # ❌ 错误：提交真实的 API Key
   git add backend/.env

   # ✅ 正确：只提交模板
   git add backend/.env.example
   ```

2. **不要遗漏配置项**
   - 新增配置时，确保三个文件都更新
   - 定期检查配置同步状态

3. **不要硬编码配置**

   ```typescript
   // ❌ 错误：硬编码
   const apiKey = 'sk-xxxxx'

   // ✅ 正确：从环境变量读取
   const apiKey = process.env.SILICON_FLOW_API_KEY
   ```

## 故障排查

### 问题 1: 配置不生效

**原因**: 服务未重启

**解决**:

```bash
# 停止服务
# Ctrl+C

# 重新启动
npm run dev
```

---

### 问题 2: 配置文件冲突

**症状**: 某些配置项在不同文件中不一致

**解决**:

```bash
# 1. 检查差异
diff backend/.env backend/.env.dev

# 2. 手动同步
# 确保三个文件的配置项一致

# 3. 重启服务
npm run dev
```

---

### 问题 3: 敏感信息泄露

**预防**:

```bash
# 1. 检查 .gitignore
cat backend/.gitignore

# 应该包含：
# .env
# .env.dev
# .env.local

# 2. 如果已提交，立即撤销
git rm --cached backend/.env
git commit -m "chore: 移除敏感配置文件"
```
