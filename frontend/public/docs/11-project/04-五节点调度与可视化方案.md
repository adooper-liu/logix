# 以货柜为主的五节点：清关、拖卡、卸柜、还箱、查验 — 业务与可视化方案

> 基于现有 Shipments（`/#/shipments`）与甘特图实现，不急于改代码，先出方案。  
> 目标：按货柜为对象，对**清关、拖卡、卸柜、还箱、查验**五个节点做**计划调度、实际执行、异常预警、异常处理、费用计算、风险管理**的梳理与可视化设计，并结合行业最佳实践。

---

## 一、范围与目标

### 1.1 五节点定义

| 节点 | 英文 | 含义 | 现有主数据来源 |
|------|------|------|----------------|
| **清关** | Customs | 目的港报关/放行，决定可否提柜 | process_port_operations（目的港记录） |
| **拖卡** | Trucking / Pickup | 从港口提柜并送仓（或经堆场） | process_trucking_transport |
| **卸柜** | Unload / Warehouse | 在仓库完成卸货、WMS 确认 | process_warehouse_operations |
| **还箱** | Empty Return | 空箱还回船东/码头 | process_empty_return |
| **查验** | Inspection | 海关/检验检疫查验（可能造成扣货、延迟） | biz_containers / 备货单 inspection_required；可扩展为查验记录 |

### 1.2 六类能力（每个节点均需考虑）

| 能力 | 说明 | 行业常见做法 |
|------|------|--------------|
| **计划调度** | 计划日期/时间、责任方、资源（车、仓、人） | 计划 vs 实际对比；依赖上一节点完成再排下一节点 |
| **实际执行** | 实际完成时间、执行人、执行结果 | 状态流转、时间戳记录、与计划对比 |
| **异常预警** | 超期、延误、即将超免费期、单据缺失等 | 基于规则（日期、阈值）提前 N 天/小时提醒 |
| **异常处理** | 记录异常原因、处理动作、责任人 | 异常单/工单、状态从「异常」到「已处理」 |
| **费用计算** | 滞港、滞箱、堆存、拖车、查验等 | 按标准与免费期/超期天数计算；支持预估与实绩 |
| **风险管理** | 高查验率、高滞港风险、客户/港口风险标签 | 风险等级、看板/列表突出显示、报表聚合 |

### 1.3 与现有实现的关系

- **Shipments 列表**：已有柜号、物流状态、到港/提柜/还箱等关键日期、清关状态、查验（是/否）；可扩展为「五节点」列/标签、预警标识、费用摘要。
- **甘特图**：当前四泳道为「按到港 / 按提柜计划 / 按最晚提柜 / 按最晚还箱」；可增加「按清关计划」「按卸柜计划」等泳道，或在同一泳道内用不同形状/颜色区分五节点关键日。
- **货柜详情**：已有港口操作、拖卡、仓库、还箱等区块；可统一为「五节点时间线 + 计划/实际 + 异常/费用」一张视图。

## 二、业务与作业流程（行业最佳实践）

### 2.1 各节点目标集与倒计时口径（必守规则）

| 节点 | 目标货柜范围 | 倒计时依据 | 倒计时含义 |
|------|--------------|------------|------------|
| **清关** | **所有已出运的货柜**（logistics_status ≥ shipped，即已装船/在途/已到港等） | **修正 ETA**（eta_correction）有则用，否则用 **ETA**（eta_dest_port）；目的港港口操作 | 「距离预计到港还有 X 天」；到港后转为清关状态/计划清关日倒计时 |
| **拖卡** | **所有已到港、未提柜的货柜**（at_port 且无 pickup_date） | **最晚提柜日** last_free_date（目的港） | 「距离最晚免费提柜日还有 X 天」；超期即滞港风险 |
| **卸柜** | **所有已提柜、未还空箱的货柜**（picked_up ～ returned_empty 之间） | 与还箱共用同一批货柜；卸柜本身可用 planned_unload_date 做计划倒计时 | 计划卸柜日/实际卸柜日 |
| **还箱** | **所有已提柜、未还空箱的货柜**（同上） | **最晚还箱日**（last_return_date 或免费用箱截止日） | 「距离最晚还箱日还有 X 天」；超期即滞箱费风险 |
| **查验** | 标记为需查验的货柜（inspection_required=true）；可能处于在途或已到港 | 查验全流程：通知日→查验日→结果→放行日；见下文 2.1.1 | 全过程记录与跟踪、费用管理，不做单一「倒计时」而做多阶段进度 |

- **清关倒计时**：取目的港 port_operations 的 **eta_correction**（有值）或 **eta_dest_port**，与「今天」比较得到「距预计到港 X 天」；已到港（有 ATA）后，倒计时可切换为「距计划清关日」或清关状态展示。
- **拖卡倒计时**：与现有「最晚提柜」统计一致，目标集 = 已到目的港（有 ATA）+ 未提柜；取目的港 **last_free_date** 做倒计时。
- **卸柜/还箱**：目标集 = 已提柜且未还箱（logistics_status ∈ picked_up, unloaded）；还箱倒计时取 **last_return_date** 或（提柜日 + 免费用箱天数）；卸柜可用 planned_unload_date 做计划倒计时。

**说明：最晚提柜日与最晚还箱日为计算得出**（项目内口径见 [demurrage/01-DEMURRAGE_LOGIC_FROM_CONTAINER_SYSTEM.md](../demurrage/01-DEMURRAGE_LOGIC_FROM_CONTAINER_SYSTEM.md)、[demurrage/02-CONTAINER_SCHEDULING_AND_COST_OPTIMIZATION_PLAN.md](../demurrage/02-CONTAINER_SCHEDULING_AND_COST_OPTIMIZATION_PLAN.md)）：

| 日期 | 存储位置 | 计算口径（项目约定） |
|------|----------|----------------------|
| **最晚提柜日 last_free_date** | `process_port_operations.last_free_date`（目的港） | **免费期结束日**：基准日 + 免费天数（按免费天数基准折算）。**基准日**：按到港时优先 ATA（`ata_dest_port`）/ 卸船日（`dest_port_unload_date`），无则修正 ETA（`eta_correction`）、再否则 ETA（`eta_dest_port`）；按卸船时仅卸船日。免费天数与基准（Natural/Working/Natural+Working）来自滞港费标准或配置；计算后可写回目的港记录。当前实现中多为 Excel 导入/人工维护，文档建议在导入或飞驼同步后按上述口径自动计算并写回。 |
| **最晚还箱日 last_return_date** | `process_empty_return.last_return_date` | **免费用箱截止日**：用箱起算日 + 免费用箱天数。**用箱起算日**：有实际提柜日（`process_trucking_transport.pickup_date`）时用提柜日，否则用目的港 **last_free_date**。免费用箱天数来自滞箱费标准或配置。即 最晚还箱日 =（提柜日 ?? last_free_date）+ 免费用箱天数；计算后可写回还空箱表。 |

#### 2.1.1 查验：全过程记录与跟踪、费用管理

- **范围**：对「需查验」货柜做**全流程**管理，而非仅布尔标记。
- **过程记录**：建议覆盖 — 查验通知日、预约/实际查验日、查验类型（海关/商检等）、查验结果（放行/扣货/整改）、放行/放行日期、异常说明；可与清关状态联动（如 customs_status = inspection_hold → 查验中）。
- **跟踪**：状态流转 待查验 → 查验中 → 已放行/已扣货；每步可带时间戳与操作人。
- **费用管理**：查验费（政府/码头）、代理费、滞留费等；支持按柜登记、汇总与报表；与 ext_container_charges 或专项「查验费用」类型对接。
- **数据**：扩展 process_port_operations 或独立 ext_inspection_records，含 inspection_date、inspection_result、inspection_release_date、inspection_fee 等；详见第四节扩展建议。

### 2.2 节点顺序与依赖（典型目的港到还箱链）

```
到港(ATA) → 清关(放行) → 拖卡(提柜→送仓) → 卸柜(仓库卸货) → 还箱(空箱还回)
                ↑
            查验(可能插入在清关前后，导致扣货/延迟)
```

- **清关**：依赖到港（ATA/卸船）；放行后才有「可提柜」状态；若**查验**发生，常表现为清关状态 = 扣货/查验中，完成查验后才放行。
- **拖卡**：依赖清关放行（部分港口可「先提后补」以政策为准）；计划提柜日通常参考 last_free_date 倒推，避免滞港。
- **卸柜**：依赖拖卡送仓；计划卸柜日受**仓库产能**（每日可卸门数/柜数）约束。
- **还箱**：依赖卸柜完成（空箱才能还）；计划还箱日受**免费用箱期**约束，超期产生滞箱费。
- **查验**：可与清关并行或嵌入清关流程；需**全过程记录与跟踪 + 费用管理**，见 2.1.1。

### 2.3 计划调度（S&OP 思路）

| 节点 | 计划内容 | 数据来源/约束 | 最佳实践 |
|------|----------|----------------|----------|
| 清关 | 计划清关日、清关公司、单据齐套日 | planned_customs_date、document_status、last_free_date 前完成 | 单据先行；与船期、到港日衔接 |
| 拖卡 | 计划提柜日、计划送仓日、拖车公司 | planned_pickup_date、planned_delivery_date、last_free_date | 以 last_free_date 为硬约束；考虑仓库预约与卸柜方式 |
| 卸柜 | 计划卸柜日、仓库、卸货方式 | planned_unload_date、**仓库卸柜能力参数表**、**卸柜方式**（见 2.3.1） | 按仓库能力与卸柜方式排产；与拖卡送仓日衔接 |
| 还箱 | 计划还箱日、还箱堆场 | planned_return_date、免费用箱期 | 提柜日 + 免费用箱天数内还箱；Drop off 时还箱可滞后 |
| 查验 | 预计查验日、预计放行日（若需） | inspection_required；可扩展 inspection_planned_date | 高查验率客户/品名提前排期 |

#### 2.3.1 卸柜计划调度：仓库能力、卸柜方式与第三堆场决策

**决策目标**：计划调度的结果以**总费用最少**为决策目标，在以下三种方案中择一执行：
- **直接送仓**：到港/清关后按仓库能力排期，直接提柜送仓卸柜还箱；
- **先提送堆场再送仓**：先提柜至第三堆场暂存，等仓库能力释放后再从堆场送仓卸柜还箱；
- **滞港等仓再直接送仓**：在港口等待 N 天（产生滞港费），待仓库有能力时再提柜直接送仓卸柜还箱。

比较三种方案的总费用（含滞港费、堆场费、拖车费、多环节带来的风险与操作成本等），选择**总费用最少**的方案作为计划调度结果。下文（1）（2）（3）为计算与比较所需的能力、卸柜方式与费用要素。

计划调度需**综合**以下要素，才能合理排定「何时提柜、何时送仓、何时卸柜、是否经第三堆场」。

**（1）仓库卸柜能力参数表**

- 建议维护**仓库维度的卸柜能力参数**（可在 dict_warehouses 扩展或独立表，如 `ext_warehouse_unload_capacity`）：
  - **日卸柜能力**：每日可卸柜数（或每日可卸门数/卸货口数）；
  - **已占用/已预约**：按日或按时间段的已排产柜数，用于计算「剩余能力」；
  - 可选：按仓库、按国家/港口、按淡旺季的可配置能力值。
- 排产时：计划卸柜日必须落在「该仓库当日剩余能力 > 0」的日期，否则顺延或改仓/改第三堆场。

**（2）仓库卸柜方式：Drop off 与 Live loading**

| 卸柜方式 | 英文 | 含义 | 对排产与资源的影响 |
|----------|------|------|---------------------|
| **Drop off** | Drop off | 货柜可**落地**：拖车把柜放到仓库后即可离开，仓库卸完后**再由拖车下次来还箱** | 可利用**仓库自身堆场**暂存柜子；卸柜节奏按仓库产能排即可，不占用拖车「等待时间」；还箱可与卸柜日错开，计划还箱日更灵活 |
| **Live loading** | Live loading | 货柜**不可落地**：拖车在现场等，**当天卸当天还箱** | 只能利用仓库**实际卸柜能力**（同一时间段内能卸完的柜数），且占用拖车与司机等待时间；还箱日 = 卸柜日，计划还箱受卸柜完成时间约束 |

- **数据**：process_warehouse_operations 已有 `unload_mode_plan` / `unload_mode_actual`，可约定取值如 `DROP_OFF`、`LIVE_LOADING`；排产与能力计算时按该字段区分逻辑。
- **计划调度**：同一仓库下，Drop off 按「日卸柜能力 + 堆场缓冲」排产；Live loading 按「当日实际可卸柜数」与拖车档期排产，且送仓日 = 卸柜日 = 还箱日（同一天）。

**（3）高峰期：三种方案的总费用比较与决策**

- **场景**：仓库满负荷或接近满负荷时，需在**直接送仓**（若能力允许）、**先提送堆场再送仓**、**滞港等仓再直接送仓**三者中选一。
- **总费用比较所需输入**：
  - **仓库预计释放能力的天数 N**：当前排队柜数 ÷ 日卸柜能力，或维护「预计 N 天后可接新柜」；
  - **方案一（直接送仓）**：拖车费 + 仓内卸柜/还箱相关成本；若当前无能力则顺延，产生等待期滞港费；
  - **方案二（先提送堆场再送仓）**：港→堆场拖车费 + N 天堆场费 + 堆场→仓拖车费 + 卸柜还箱；另计入多一道环节的**风险与操作成本**（装卸/交接、异常与时效风险）；
  - **方案三（滞港等仓再直接送仓）**：N 天滞港费 + 之后一次性提柜送仓卸柜还箱（拖车 + 卸柜还箱）。
- **决策规则**：对三种方案分别估算**总费用**，选择**总费用最少**的方案作为计划调度结果。因此：
  - 若**滞港费明显高于**堆场费且多一程拖车与风险成本仍更省，则倾向「先提送堆场再送仓」；
  - 若**滞港费与堆场费相差不大**，则**滞港等仓再直接送仓**往往更优：多一道环节会带来额外成本与风险，总费用易更高。
- **数据与产品**：需维护第三堆场主数据（名称、费率、容量可选）、仓库「预计释放能力天数」或排队模型；排产/模拟时对三种方案分别计算总费用，输出**总费用最少**的推荐方案（或展示三项总费用由人工决策）。

#### 2.3.2 计划调度间的资源约束关系（细分与可视化）

计划调度受多类**资源**约束，且节点间存在**约束传递**：上一节点释放的时间/能力会成为下一节点的输入。下面按「资源清单 → 资源与节点约束矩阵 → 约束传递链」细分，并给出可视化表与示意图。

**（1）资源清单（参与计划调度的约束资源）**

| 资源类型 | 说明 | 约束对象 | 数据/配置来源 |
|----------|------|----------|----------------|
| **港口免费期** | 最晚提柜日 last_free_date 前须提柜，否则滞港费 | 拖卡（计划提柜日） | 计算得出（基准日+免费天数）；见 2.1 |
| **免费用箱期** | 提柜后须在 N 天内还箱，否则滞箱费 | 还箱（计划还箱日） | 滞箱费标准；last_return_date 计算得出；见 2.1 |
| **仓库卸柜能力** | 每日可卸柜数/门数，已预约占用剩余能力 | 卸柜（计划卸柜日）、间接约束送仓日 | 仓库卸柜能力参数表；见 2.3.1（1） |
| **卸柜方式** | Drop off（可落地）/ Live loading（当天卸当天还） | 卸柜节奏、还箱是否与卸柜同天、拖车占用 | process_warehouse_operations.unload_mode_*；见 2.3.1（2） |
| **第三堆场** | 容量、费率、可用性 | 是否采用「先提送堆场再送仓」方案 | 第三堆场主数据、费率；见 2.3.1（3） |
| **拖车/车板** | 可用车辆数、档期、单日可跑趟数 | 提柜日、送仓日、还箱日（Live loading 时与卸柜同程） | 拖车公司、排车计划（可扩展主数据与档期表） |
| **清关能力** | 清关公司/口岸每日可处理票数（可选） | 清关（计划清关日） | 可扩展清关产能或约定为「单据齐套即排期」 |
| **查验能力** | 查验预约档期、放行节奏（可选） | 查验（预计查验日/放行日）、进而延迟可提柜日 | 查验预约/码头或海关档期（可扩展） |
| **还箱堆场** | 还箱点开放时间/预约（可选） | 还箱（计划还箱日） | 还箱堆场主数据、预约规则（可扩展） |

**（2）资源 ↔ 节点约束矩阵（谁约束谁）**

| 节点 | 受哪些资源约束（计划时必考虑） | 产出/释放给下游 |
|------|--------------------------------|------------------|
| **清关** | 到港日(ATA)、单据齐套、清关能力（可选）、查验（若发生则延迟放行） | **可提柜日**：清关放行后拖卡才能排提柜 |
| **拖卡** | **港口免费期** last_free_date、**清关放行**、仓库预约与**卸柜方式**、**拖车档期**、是否经**第三堆场** | **送仓日**、**提柜日**（用箱起算，影响 last_return_date） |
| **卸柜** | **仓库卸柜能力**（日产能、已占用）、**卸柜方式**、拖卡**送仓日** | **卸柜完成日**：还箱可排在此之后（Drop off 可错开，Live loading 即日还） |
| **还箱** | **免费用箱期**（last_return_date）、**卸柜完成**（空箱才能还）、**还箱堆场**（可选）、Live loading 时=卸柜日 | 用箱结束，滞箱费截止 |
| **查验** | 查验预约/能力（可选） | 若发生：延迟**清关放行** → 连锁延迟拖卡/卸柜/还箱 |

**（3）约束传递链（时序与依赖可视化）**

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  资源约束传递链（从左到右为时间顺序；↑ 表示该资源约束该节点）                        │
└─────────────────────────────────────────────────────────────────────────────────┘

  到港(ATA)    清关            拖卡                    卸柜              还箱
     │          │               │                      │                 │
     │    ┌─────┴─────┐   ┌─────┴─────┐          ┌─────┴─────┐     ┌─────┴─────┐
     │    │ 单据齐套   │   │last_free │          │ 仓库日产能 │     │last_return│
     │    │ 清关能力?  │   │ 拖车档期 │          │ 卸柜方式   │     │ 还箱堆场? │
     │    │ 查验? ─────┼──→ 放行 ────┼──→ 提柜 ─┼→ 送仓 ───┼──→ 卸柜 ─┼→ 还箱   │
     │    └───────────┘   │ 仓库预约 │          │ 已占用    │     │           │
     │                    │ 第三堆场?│          └───────────┘     └───────────┘
     │                    └──────────┘
     │
     └──→ 基准日 → last_free_date 计算
                    提柜日 / last_free_date → last_return_date 计算
```

- **清关**：依赖到港(ATA)；输出「可提柜日」；若查验插入，放行日推迟 → 拖卡计划整体右移。
- **拖卡**：依赖清关放行；受港口免费期、拖车档期、仓库预约与卸柜方式、第三堆场方案约束；输出送仓日、提柜日（后者参与 last_return_date 计算）。
- **卸柜**：依赖送仓日；受仓库日产能与已占用、卸柜方式约束；输出卸柜完成日。
- **还箱**：依赖卸柜完成（空箱）；受免费用箱期(last_return_date)、Live loading 时还箱=卸柜日约束。

### 2.6 异常处理与费用、风险

- **异常处理**：建议每节点支持「异常原因」「处理措施」「责任人」「关闭时间」；可落在扩展表（如 ext_container_*_exceptions）或现有 remarks + 状态。
- **费用计算**：与现有滞港/滞箱设计对齐（见 demurrage 文档）；五节点视角下：清关→影响可提柜日→滞港；拖卡/卸柜→影响用箱天数→滞箱；查验→可能产生查验费、延迟费。
- **风险管理**：可按客户/港口/品名维护「查验率」「历史滞港率」等，在列表/甘特上打风险标签（高/中/低），驱动优先排程或预警级别。

### 2.7 节点时间与三种数据来源的适应性

所有节点涉及的时间字段，需**同时考虑**三种更新方式：**飞驼 API 动态更新**、**手工维护**、**Excel 导入**；设计时约定优先级与覆盖规则，避免冲突或覆盖丢失。

#### 2.7.1 三种来源说明

| 来源 | 说明 | 典型更新内容 |
|------|------|--------------|
| **飞驼 API** | 外部轨迹同步，通过 FeiTuoAdapter 写入 process_port_operations 等；同步后触发状态重算 | ETA、ATA、gate_in_time、gate_out_time、dest_port_unload_date、available_time、shipment_date 等（见 FeiTuoStatusMapping） |
| **手工维护** | 用户在系统内编辑货柜/港口操作/拖卡/仓库/还箱等 | 任意可编辑字段：计划日、实际日、状态、备注、修正 ETA、last_free_date、清关/查验相关 |
| **Excel 导入** | 批量导入更新或补录 | 与当前 Excel 映射一致：到港/清关/拖卡/卸柜/还箱/查验等；可扩展列以支持更多节点字段 |

#### 2.7.2 各节点关键时间字段与来源适应性

| 节点 | 关键时间/状态字段 | 飞驼 API | 手工维护 | Excel 导入 | 建议策略 |
|------|-------------------|----------|----------|------------|----------|
| **清关** | eta_dest_port, eta_correction, ata_dest_port, planned_customs_date, actual_customs_date, customs_status | 可更新 ETA、ATA（目的港） | 可更新全部；**修正 ETA** 多为手工或 Excel | 支持 ETA、计划/实际清关日、清关状态等 | **ETA/ATA**：飞驼与导入可覆盖；**修正 ETA**：仅手工/导入，不被飞驼覆盖；**清关日期/状态**：手工与导入为主，飞驼若有清关状态可映射写入 |
| **拖卡** | last_free_date（目的港）, planned_pickup_date, pickup_date, planned_delivery_date, delivery_date | 可更新 ATA/卸船等**计算基准**，触发 last_free_date 重算（若实现自动计算）；目前不直接提供 last_free_date、提柜日；若后续提供可映射 | 可更新全部；可**覆盖**自动计算得到的 last_free_date | 支持最晚提柜日、计划/实际提柜/送仓；导入后可为 last_free_date 提供值或触发重算 | **last_free_date**：**计算得出**（基准日 + 免费天数，见 2.1），也可手工或 Excel 录入；若实现自动计算，建议在导入/飞驼同步后按口径写回（可仅当为空时写回或支持强制重算）；**提柜/送仓日**：手工或 Excel；飞驼若将来提供 GATE_OUT 等可映射 |
| **卸柜** | planned_unload_date, unload_date, unload_mode_plan/actual, wms_confirm_date, wms_status | 一般不来自飞驼 | 可更新全部 | 支持计划/实际卸柜、卸柜方式、WMS 状态等 | 以手工与 Excel 为主；WMS 若对接可走接口写入；仓库能力与卸柜方式见 2.3.1 |
| **还箱** | last_return_date, planned_return_date, return_time | 一般不来自飞驼；若实现自动计算，提柜日等由飞驼写入后可**触发 last_return_date 重算** | 可更新全部；可**覆盖**自动计算得到的 last_return_date | 支持最晚还箱日、计划/实际还箱日；导入后可为 last_return_date 提供值或触发重算 | **last_return_date**：**计算得出**（提柜日或 last_free_date + 免费用箱天数，见 2.1），也可手工或 Excel 录入；若实现自动计算，可在提柜或 last_free_date 更新后写回；**计划/实际还箱**：以手工与 Excel 为主 |
| **查验** | inspection_required, inspection_date, inspection_result, inspection_release_date, 查验费用 | 一般不来自飞驼 | 可更新全部 | 支持是否查验及扩展的查验日期/结果/费用 | 以手工与 Excel 为主；全过程与费用见 2.1.1 |

#### 2.7.3 统一规则建议

- **写入口**：所有三种来源最终都写入**同一套表与字段**（如 process_port_operations、process_trucking_transport 等），不按来源分表；这样倒计时、状态机、列表/甘特只读一套数据。
- **计算字段**：**last_free_date**、**last_return_date** 为计算得出（见 2.1）。若实现自动计算：last_free_date 在目的港数据写入或更新后（导入/飞驼同步后）按基准日 + 免费天数写回；last_return_date 在提柜日或 last_free_date 更新后按用箱起算日 + 免费用箱天数写回。手工与 Excel 可**覆盖**计算结果（或策略上仅当该字段为空时自动写回）。
- **优先级/覆盖**：  
  - **实际时间**（如 ATA、pickup_date）：飞驼若推送「已发生」事件，可覆盖旧值；手工与 Excel 可覆盖飞驼（以「最后更新时间」或「数据来源」标记决定，可选）。  
  - **计划时间**（如 planned_customs_date、planned_pickup_date）：通常不由飞驼更新，仅手工与 Excel。  
  - **修正 ETA**：仅手工或 Excel，飞驼只更新 eta_dest_port 时不应清空 eta_correction。
- **审计与追溯**：可选在扩展表记录「最后更新来源」（feituo / manual / excel）与时间，便于排查；核心业务表可保留 updated_at 即可。
- **Excel 模板**：新增或扩展列时，与 2.6.2 对齐，确保计划/实际/状态均可导入；飞驼已覆盖的字段也可在 Excel 中提供列，用于「补录或覆盖」。

| **时间维度** | 未来 15 天按天展示，每格一天 | 与 last_free_date、last_return_date、planned_* 对齐；支持 7/15/30 天切换（与 GANTT-ENHANCED 时间范围一致） |
| **颜色与优先级** | 高 #f5222d、中 #faad14、低 #1890ff；任务类型区分（实线/虚线） | 五节点预警：超期红、即将超期橙、正常蓝；计划 vs 实际用形状或边框区分 |
| **拖拽与调度** | 港口池 → 车队泳道；资源池内任务可拖拽到其他泳道/日期 | 可选：已清关柜子拖拽到拖车/仓库泳道排期；需校验 last_free_date、仓库剩余能力（2.3.2 约束） |
| **自动/智能调度** | 先到先得、按天排满；智能优化优先高优先级 | 可选：按 last_free_date 紧迫度自动建议排期；与「总费用最少」决策（2.3.1）结合做排产建议 |
| **状态流转** | fleet_allocated → fleet_picking → warehouse_arrived → warehouse_completed | 五节点：清关放行 → 拖卡排期 → 提柜 → 送仓 → 卸柜完成 → 还箱；可与 logistics_status 及排柜状态扩展字段联动 |
| **多维度筛选** | 国别、仓库、时间范围、资源类型、车队堆场开关 | 与 LogiX 现有日期+国家过滤一致；增加仓库、清关状态、预警类型筛选 |
| **资源热力图** | 按小时展示资源负荷 | 可选：仓库日产能占用、拖车档期占用，与 2.3.2（4）「甘特图+资源线」对应 |
| **依赖关系线** | 任务间依赖线 | 五节点顺序依赖（清关→拖卡→卸柜→还箱）可画依赖线；查验插入时标注 |
| **Tooltip 与详情** | 柜号、任务类型、操作说明；点击打开详情抽屉 | 五节点：每格 tooltip 显示计划/实际日、状态、预警；点击打开货柜详情（3.4 五节点时间线） |

**（4）与 LogiX 现有实现的衔接**

- LogiX 甘特图已有：按到港/按提柜计划/按最晚提柜/按最晚还箱四泳道，与 containerDimensions 口径一致。
- 扩展时可**分阶段**：  
  - **阶段一**：新增泳道（按计划清关、按计划卸柜）、同一泳道内五节点颜色/形状区分、预警标红，不引入三池与拖拽。  
  - **阶段二**：若需调度能力，可参考 container-system 引入三池布局或资源泳道 + 拖拽/自动调度，并与 process_* 表及 2.3.2 资源约束对接。

### 3.4 货柜详情（弹窗/抽屉或详情页）

## 四、数据与模型支撑（现状与扩展建议）

### 4.1 现有表与五节点映射

| 节点 | 主表 | 计划/实际/状态字段（现有） | 缺口（扩展建议） |
|------|------|----------------------------|------------------|
| 清关 | process_port_operations | planned_customs_date, actual_customs_date, customs_status, customs_broker_code, document_status, customs_remarks | 异常原因/处理单独字段或扩展表；与查验关联 |
| 拖卡 | process_trucking_transport | last_pickup_date, planned_pickup_date, pickup_date, planned_delivery_date, delivery_date, cost | 异常/预警、与 last_free_date 的关联展示在应用层 |
| 卸柜 | process_warehouse_operations | planned_unload_date, unload_date, unload_mode_plan, unload_mode_actual, wms_confirm_date, wms_status | **仓库卸柜能力参数表**（日产能、已占用）；**卸柜方式** DROP_OFF / LIVE_LOADING；**第三堆场**主数据与费率、与滞港费的决策支撑 |
| 还箱 | process_empty_return | last_return_date, planned_return_date, return_time, return_terminal_* | 异常/柜况已有 return_remarks；滞箱费计算见 demurrage |
| 查验 | biz_containers, biz_replenishment_orders | inspection_required | **全过程**：inspection_date, inspection_result, inspection_status, inspection_release_date, 查验费；可放在 process_port_operations 或 **ext_inspection_records**（推荐独立表便于跟踪与费用）；费用与 ext_container_charges 或 type=inspection 对接 |

### 4.2 扩展建议（仅方案，不落代码）

- **查验**：推荐独立表 **ext_inspection_records**，支持全过程与费用：container_number, inspection_notice_date, inspection_date, inspection_type, inspection_result, inspection_release_date, inspection_status, fee_amount, fee_currency, remarks；与清关状态联动。若暂不独立表，可在 process_port_operations 上增加上述部分字段。
- **仓库卸柜能力与卸柜方式**：dict_warehouses 扩展或独立 **ext_warehouse_unload_capacity**（warehouse_code, effective_date, daily_unload_capacity, 已预约/占用量等）；卸柜方式已有 unload_mode_plan / unload_mode_actual，约定枚举 DROP_OFF、LIVE_LOADING。**第三堆场**：独立主数据表（堆场编码、名称、费率、容量可选）+ 与「滞港费 vs 堆场费」决策逻辑对接；仓库「预计释放能力天数」可配置或由排队模型计算。
- **异常**：统一 ext_container_node_exceptions(container_number, node_type, occurred_at, reason, action, owner, closed_at) 或各 process_* 表增加 exception_reason、exception_closed_at 等。
- **费用**：滞港/滞箱已有 demurrage 方案；拖车费、查验费、**第三堆场费**可在现有 cost 或 ext_container_charges 中区分 type=demurrage/detention/trucking/inspection/yard_storage；查验费支持按柜登记与汇总。

## 五、实施顺序建议（方案层）

1. **共识与定稿**：与业务确认五节点顺序、计划/实际口径、预警规则、费用口径（与 demurrage 文档一致）。
2. **数据层**：补全查验相关字段或表；如需统一异常表则设计 ext 表。
3. **后端**：五节点聚合接口（按柜返回五节点计划/实际/状态/预警/费用摘要）；预警规则服务；费用计算服务（与现有滞港滞箱对接）。
4. **前端**：  
   - Shipments 列表：五节点状态列、预警图标、可选费用列；  
   - 甘特：新增泳道或同一泳道多类型关键日；可参考 **D:\Gihub\container-system** 的三池、资源泳道、拖拽与状态流转（见 3.3.1）；  
   - 货柜详情：五节点时间线 + 每节点卡片（计划/实际/异常/费用）+ 费用汇总 + 异常列表。
5. **看板/预警中心**：在列表与详情稳定后再做聚合看板与预警中心。

## 六、小结

| 维度 | 要点 |
|------|------|
| **目标集与倒计时** | **清关**：已出运货柜，按修正 ETA / ETA 倒计时（距预计到港）；**拖卡**：已到港未提柜，按最晚提柜日倒计时；**卸柜与还箱**：已提柜未还箱，按最晚还箱日倒计时；**查验**：全过程记录与跟踪、费用管理，不做单一倒计时。 |
| **业务** | 五节点顺序：到港→清关（含查验）→拖卡→卸柜→还箱；计划调度以**总费用最少**为决策目标，在**直接送仓、先提送堆场再送仓、滞港等仓再直接送仓**三者中择一，依赖 last_free_date、last_return_date（计算得出）、仓库卸柜能力与卸柜方式；**计划调度间资源约束**见 2.3.2（资源清单、资源↔节点矩阵、约束传递链及产品可视化建议）。 |
| **三种数据来源** | 所有节点时间需兼顾**飞驼 API 动态更新**、**手工维护**、**Excel 导入**；同一套表与字段，约定优先级（如实际时间飞驼可写、计划时间与修正 ETA 仅手工/Excel）；修正 ETA 不被飞驼覆盖。 |
| **计划/实际** | 每节点明确计划日与实际日、状态；列表与详情并排或对比展示，甘特可双维度（计划/实际）或双形状。 |
| **预警** | 滞港/滞箱/清关延误/单据缺失/查验扣货/卸柜积压等规则；列表与甘特外露图标或颜色，详情内可展开。 |
| **费用** | 滞港、滞箱、堆存、拖车、**查验费**；与 demurrage 文档一致；列表可摘要，详情可明细；查验支持按柜登记与汇总。 |
| **可视化** | Shipments 列与筛选扩展；甘特新泳道或同泳道多类型；详情五节点时间线 + 节点卡片 + 费用汇总 + 异常列表。**甘特图参考**：D:\Gihub\container-system 三池结构、资源泳道、拖拽调度、状态流转（见 3.3.1 解读与五节点映射）。 |
| **数据** | 清关/拖卡/卸柜/还箱已有主表；查验推荐 ext_inspection_records 全过程+费用；异常与费用可复用或扩展现有表与 demurrage 设计。 |

本文档仅做方案梳理与设计，具体表结构变更、接口与前端实现以后续开发任务为准。
