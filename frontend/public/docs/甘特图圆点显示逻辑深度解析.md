# 甘特图圆点显示逻辑

## 业务场景

甘特图通过圆点展示货柜在各物流节点的计划与实际进度。圆点类型反映节点的执行状态和依赖关系。

**核心需求**：

- 已提柜的货柜应显示卸柜和还箱节点的计划圆点
- 前置节点未完成时，后续节点显示虚线圆点（待激活）
- 前置节点完成后，后续节点显示实线圆点（可执行）
- 有实际日期的节点不显示圆点（已完成，销毁）

## 圆点样式定义

### 圆点类型与视觉表现

| 类型   | 状态      | 含义   | 背景色                   | 边框样式   | 边框颜色                 |
| ------ | --------- | ------ | ------------------------ | ---------- | ------------------------ |
| main   | active    | 可执行 | 有颜色（见下方颜色定义） | 2px solid  | 根据预警等级（红/橙/灰） |
| dashed | pending   | 待激活 | 白色填充（transparent）  | 2px dashed | 根据预警等级（红/橙/灰） |
| null   | completed | 已销毁 | 不显示                   | 不显示     | 不显示                   |

**设计原则**：

- **实心圆点**：前置节点已完成，当前节点可以执行，显示货柜状态颜色 + 颜色边框
- **空心圆点**：前置节点未完成，当前节点待激活，白色填充 + 颜色虚线边框清晰区分
- **所有圆点都有边框颜色**：不显示透明，统一使用红/橙/灰三色

## 圆点显示规则

### 显示类型定义

| 类型   | 状态      | 含义   | 视觉表现                            |
| ------ | --------- | ------ | ----------------------------------- |
| main   | active    | 可执行 | 实心圆点（有背景色 + 颜色边框）     |
| dashed | pending   | 待激活 | 空心圆点（白色填充 + 颜色虚线边框） |
| null   | completed | 已销毁 | 不显示                              |

**视觉设计原则**：

- **实心圆点**：前置节点已完成，当前节点可以执行，显示货柜状态颜色
- **空心圆点**：前置节点未完成，当前节点待激活，白色填充 + 颜色虚线边框清晰区分

### 链式依赖规则

**完整依赖链条**：清关 -> 提柜 -> 卸柜 -> 还箱

节点状态由三个因素决定：**自身日期** + **前置节点完成状态** + **反向链式推导**

#### 正向依赖（前置检查）

| 节点 | 前置依赖     | 判断逻辑                                                                               |
| ---- | ------------ | -------------------------------------------------------------------------------------- |
| 清关 | 无           | 有计划日期 -> active；有实际日期 -> completed                                          |
| 提柜 | **清关完成** | 清关未完成+有计划 -> pending<br>清关已完成+有计划 -> active<br>有实际日期 -> completed |
| 卸柜 | **提柜完成** | 提柜未完成+有计划 -> pending<br>提柜已完成+有计划 -> active<br>有实际日期 -> completed |
| 还箱 | **卸柜完成** | 卸柜未完成+有计划 -> pending<br>卸柜已完成+有计划 -> active<br>有实际日期 -> completed |
| 查验 | 无           | 同清关/提柜逻辑                                                                        |

**前置节点完成判断**：

- 清关完成：`destPortOp?.actualCustomsDate` 存在
- 提柜完成：`pickupTransport?.deliveryDate || pickupTransport?.pickupDate` 存在
- 卸柜完成：`unloadOp?.unloadDate` 存在
- 还箱完成：`emptyReturn?.returnTime` 存在

#### 反向依赖（后续推导）

**业务规则**：后续节点完成意味着前置节点必然已完成，即使前置节点的实际日期字段缺失。

| 触发条件                                         | 反推结果                                                    | 适用场景                               |
| ------------------------------------------------ | ----------------------------------------------------------- | -------------------------------------- |
| **已提柜** (`deliveryDate` 或 `pickupDate` 存在) | 清关 -> completed                                           | 数据录入不完整，但业务上清关必然已完成 |
| **已卸柜** (`unloadDate` 存在)                   | 清关 -> completed<br>提柜 -> completed                      | 只记录了卸柜日期，前置日期缺失         |
| **已还箱** (`returnTime` 存在)                   | 清关 -> completed<br>提柜 -> completed<br>卸柜 -> completed | 只记录了还箱日期，所有前置日期缺失     |

**设计原则**：

- 优先使用实际日期字段判断状态
- 如果实际日期缺失，但后续节点已完成，则反推前置节点为 completed
- 反推的前提是供应商信息存在（确保节点应该显示）
- 避免显示大量虚线圆点（待激活），提升用户体验

**代码实现**（`SimpleGanttChartRefactored.vue` Line 2652-2686）：

```typescript
// ========== 反向链式依赖检查 ==========
// 业务规则：后续节点完成意味着前置节点必然已完成
// 已提柜 -> 清关完成 | 已卸柜 -> 清关+提柜完成 | 已还箱 -> 清关+提柜+卸柜完成

// 1. 已还箱 -> 反推清关、提柜、卸柜全部完成
if (emptyReturn?.returnTime) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推清关已完成
  }
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate) && pickupSupplier) {
    nodes.提柜.status = 'completed' // 反推提柜已完成
  }
  if (!unloadOp?.unloadDate && unloadSupplier) {
    nodes.卸柜.status = 'completed' // 反推卸柜已完成
  }
}
// 2. 已卸柜 -> 反推清关、提柜完成
else if (unloadOp?.unloadDate) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推清关已完成
  }
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate) && pickupSupplier) {
    nodes.提柜.status = 'completed' // 反推提柜已完成
  }
}
// 3. 已提柜 -> 反推清关完成
else if (pickupTransport?.deliveryDate || pickupTransport?.pickupDate) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推清关已完成
  }
}
```

**代码实现**（`SimpleGanttChartRefactored.vue` Line 2525-2535）：

| 货柜类型   | 清关 | 提柜         | 卸柜       | 还箱       | 清关状态            | 提柜状态            | 卸柜状态            | 还箱状态  | 圆点总数 |

**设计优势**：

- **容错性强**：即使前置节点日期缺失，也能正确推断状态
- **用户体验好**：避免显示大量虚线圆点（待激活）
- **业务准确性高**：符合物流实际流程（后续完成意味着前置必然完成）

## 技术实现

### 判断流程

```
calculateNodeStatus() → 计算节点状态 (pending/active/completed)
    ↓
getNodeDisplayType() → 决定显示类型 (null/main/dashed)
    ↓
模板渲染 → 显示圆点
```

### 关键函数

#### 1. calculateNodeStatus - 节点状态计算

**位置**: `SimpleGanttChartRefactored.vue` Line 2430-2560

**核心逻辑**（以卸柜节点为例）：


还箱节点同理，检查卸柜节点是否完成（`unloadOp?.unloadDate`）。

#### 2. getNodeDisplayType - 显示类型决策

**位置**: `SimpleGanttChartRefactored.vue` Line 1828-1894

**重构优化**：拆分为4个职责单一的函数

| 函数                        | 职责                       | 行数 |
| --------------------------- | -------------------------- | ---- |
| getInspectionNodeType       | 查验节点特殊处理           | 5    |

## 问题修复记录

### 问题描述

已提柜的3个货柜（HMMU6855127、GAOU6195045、KOCU5129260）未显示卸柜和还箱节点的计划圆点。

### 根因分析

**两层问题**：

1. **后端数据错误**：`ganttDerived` 将后续节点错误标记为 `completed`
2. **前端逻辑缺陷**：`getNodeDisplayType` 优先使用后端数据，导致本地状态被覆盖
3. **虚线圆点样式错误**：模板中 `isMainTask` 返回 false 时背景色为透明（白色）

| 节点 | 本地状态       | ganttDerived | 旧逻辑结果   | 新逻辑结果          |
| ---- | -------------- | ------------ | ------------ | ------------------- |
| 卸柜 | active/pending | completed    | null（错误） | main/dashed（正确） |
| 还箱 | active/pending | completed    | null（错误） | main/dashed（正确） |

**根本原因**：

```typescript
// 旧逻辑：优先使用后端 ganttDerived

if (ganttType !== null) {
  return ganttType // ← 后端返回 null（因为 gn.completed=true），直接返回

return getNodeTypeFromLocalStatus(node) // ← 永远不会执行
```
