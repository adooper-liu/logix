# 部署与本地运行指南

本文说明 LogiX **不同场景下应使用的 Compose 文件、脚本与命令**，避免混用导致「没有前端 / 后端起不来 / 旧镜像」等问题。

---

## 1. 核心文件一览

| 文件 | 作用 |
|------|------|
| `docker-compose.timescaledb.yml` | **开发用**整栈（容器内后端 dev 镜像等），与 `Makefile` 的 `tsdb-up` 对应。 |
| `docker-compose.timescaledb.prod.yml` | **生产形态**整栈定义：DB、Redis、ES、**容器内 backend**、Nginx、Prometheus、Grafana。**不含前端构建、不含前端容器**。 |
| `docker-compose.prod.local-demo.yml` | **仅叠加在 prod 上**：把 `frontend/dist` 挂进 Nginx、略降 ES 内存；**不能单独使用**。 |
| `docker-compose.admin-tools.yml` | 叠加 **Adminer / pgAdmin**；**不含前端**。 |
| `scripts/local-prod-demo.ps1` | 本机 **npm 构建前端** + 拉起 **prod + prod.local-demo** 的自动化脚本（须在**仓库根目录**执行）。 |
| `start-logix-dev.ps1` | **日常开发**：用 prod compose **只起部分基础设施** + **本机** `npm run dev` 跑前后端。 |

---

## 2. 三种正确用法（按你的目的选一种）

### 2.1 日常开发（推荐）

**目标**：改代码、热重载、调试 API。

| 层级 | 做法 |
|------|------|
| 数据库 / Redis / 监控 | `start-logix-dev.ps1` 会用 **`docker-compose.timescaledb.prod.yml` 的部分服务**（postgres、redis、prometheus、grafana）+ **`admin-tools`**（Adminer、pgAdmin）。 |
| 后端 | **本机** `backend` 目录：`npm run dev`（脚本会新开窗口）。 |
| 前端 | **本机** `frontend` 目录：`npm run dev` → 通常 **http://localhost:5173**，API 走开发代理到 **3001**。 |
| 物流路径微服务 | 若存在 `logistics-path-system`，脚本会尝试本机 `npm run dev`。 |

**停止**：`.\stop-logix-dev.ps1`（若项目中有）。

**注意**：此模式 **不会在 Docker 里构建前端**；前端始终是 Vite 开发服务器。

---

### 2.2 仅数据库 + 管理工具（不写业务代码时）

**目标**：只要 Postgres/Redis 和 Adminer/pgAdmin，不要整套生产栈。

示例（可按需增减服务名）：

```bash
docker compose -f docker-compose.timescaledb.prod.yml -f docker-compose.admin-tools.yml --env-file .env up -d postgres redis adminer pgadmin
```

或先起主栈网络再只起工具（见 `docker-compose.admin-tools.yml` 文件头注释）。

**说明**：这条路径 **没有 backend 容器、没有 Nginx、没有前端构建**。

---

### 2.3 本地「生产形态」联调 / 演示

**目标**：尽量接近线上：容器内 backend、Nginx、同源 `/api/`、静态前端。

**正确步骤**：

1. 在**仓库根目录**准备 `.env`（可复制 `.env.local-docker-prod.example`）。
2. 执行：

   ```powershell
   cd E:\logix
   .\scripts\local-prod-demo.ps1
   ```

   脚本会：`frontend` 下 **`npm run build`（Vite）** → 再 **`docker compose`** 使用：

   - `-f docker-compose.timescaledb.prod.yml`
   - `-f docker-compose.prod.local-demo.yml`
   - `--env-file .env`

3. 浏览器：**http://localhost/**（前端 + 经 Nginx 的 API）。

**停止**：

```bash
docker compose -f docker-compose.timescaledb.prod.yml -f docker-compose.prod.local-demo.yml --env-file .env down
```

**要点**：

- **不要在 `frontend` 子目录里执行** `.\scripts\local-prod-demo.ps1`（路径不对）。
- 修改后端代码后需 **重新构建 backend 镜像**（例如 `docker compose ... build backend --no-cache` 或脚本里带 `--build`）。
- 修改前端后需 **重新 `npm run build`** 再重启或重建 Nginx 挂载的 `dist`（脚本会 build；手动改代码后要再 build）。

---

## 3. `docker compose up` 与 `build` 的关系

- **`up -d`**：默认**不强制重建镜像**；若本地已有旧 `logix-backend` 等镜像，可能仍在跑旧代码。
- **`up -d --build`**：对 compose 里带 `build:` 的服务（如 **backend**）会构建/更新镜像。
- **前端**：在 **`docker-compose.timescaledb.prod.yml` + `admin-tools.yml` 的组合里没有任何前端服务**，因此 **Compose 永远不会替你 `npm run build` 前端**。本地演示前端只能：
  - 主机执行 `npm run build`，再通过 **`docker-compose.prod.local-demo.yml`** 挂载 `frontend/dist`，或
  - 自行增加前端 Dockerfile 服务（当前仓库默认未做）。

## 4. 前端构建与类型检查

| 命令（在 `frontend` 目录） | 含义 |
|---------------------------|------|
| `npm run build` | **生产包**：仅 **Vite 打包**（当前仓库配置），用于 Docker 挂载 `dist` 或静态托管。 |
| `npm run build:check` | **先 `vue-tsc` 再 Vite**；类型未修完时会失败。 |
| `npm run type-check` | 仅类型检查，不产出 `dist`。 |

CI / 合并前质量门禁建议逐步让 `build:check` 或 `type-check` 通过；日常演示可用 `npm run build` 先跑通镜像。

## 5. 正式环境（服务器）

1. 准备服务器 `.env`（密码与域名勿用示例文件默认值）。
2. 使用 **`docker-compose.timescaledb.prod.yml`**（视情况叠加监控、SSL 证书目录 `nginx/ssl` 等）。
3. **前端**通常二选一：
   - 构建产物由 Nginx 或 CDN 托管（需自行配置 `location /` 与 `/api/` 反代）；或
   - 在 compose 中增加前端静态服务（本仓库默认 prod 文件未包含）。

飞驼等变量须与 `backend/src/config` 一致（如 `FEITUO_API_BASE_URL`、`FEITUO_CLIENT_ID` 等）。

## 6. 常见问题（速查）

| 现象 | 可能原因 |
|------|----------|
| 访问 localhost 无前端或 403 | `prod` 默认 Nginx **未挂前端**；演示需 **`prod.local-demo.yml` + 已存在的 `frontend/dist`**。 |
| `backend-prod` 反复退出 | 旧镜像未重建（`build backend --no-cache`）；或 `.env` 中 DB/Redis 密码与容器/数据卷初始化时不一致。查看 `docker logs logix-backend-prod`。 |
| `local-prod-demo.ps1` 找不到 | 须在**仓库根目录**执行。 |
| orphan 容器提示 | 曾用 `admin-tools` 起过 Adminer/pgAdmin，换 compose 组合时可用 `docker compose ... up --remove-orphans` 清理。 |
