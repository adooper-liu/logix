# 智能体测试与排错规范使用说明

## 概述

这套规范为 LogiX 项目的智能体（AI 助手）提供标准化的测试与排错流程，确保开发质量和效率。

## 文档结构

```
第 2 层 - 测试指南/
├── 01-测试指南.md                          # 基础测试工具使用（Jest/Vitest/Playwright）
├── 02-智能体测试与排错 SOP.md              # 核心标准作业程序（必读）
├── 03-智能体测试排错快速参考卡.md          # 快速查询手册（打印置顶）
├── 04-智能体测试与排错检查清单.md          # 全流程检查清单（执行时对照）
└── 05-问题诊断记录模板.md                  # 问题记录标准化模板
```

## 使用场景

### 场景 1: 新功能开发

**流程**:

```
1. 阅读 SOP 第二章（测试流程规范）
   └─> 确定测试策略（TDD/后补测试）

2. 使用检查清单 1.1（现有测试覆盖检查）
   └─> 了解现有测试情况

3. 编写单元测试（参考 SOP 2.2 测试模板）
   └─> 后端：Jest 模板
   └─> 前端：Vitest + Vue Test Utils 模板

4. 运行测试验证
   └─> npm run test (后端)
   └─> npm run test:unit (前端)

5. 提交前使用检查清单 6.1（提交前检查）
   └─> 确保 100% 通过
```

### 场景 2: Bug 修复

**流程**:

```
1. 使用快速参考卡第二章（排错流程速查）
   └─> 5 分钟快速诊断

2. 按照 SOP 第三章（排错流程规范）执行
   ├─> 第一步：信息收集（5W1H）
   ├─> 第二步：问题分类
   └─> 第三步：根因分析（五问法/二分法）

3. 使用调试工具（SOP 3.4 调试工具集）
   ├─> 后端调试脚本
   ├─> 数据库查询
   └─> 日志分析

4. 修复后使用检查清单 5.2（测试验证）
   └─> 确保回归测试通过

5. 填写问题诊断记录模板（文档 05）
   └─> 知识沉淀
```

### 场景 3: 性能优化

**流程**:

```
1. 使用 SOP 第三章定位性能瓶颈
   └─> SQL 查询分析（EXPLAIN）
   └─> 代码 Profiling

2. 编写性能基准测试（SOP 2.1 测试策略）
   └─> 记录优化前指标

3. 实施优化
   └─> 索引优化
   └─> 缓存策略
   └─> 算法改进

4. 运行对比测试（检查清单 5.3）
   └─> 验证性能提升

5. 确保功能回归测试通过（检查清单 5.2）
```

### 场景 4: 复杂问题排查

**流程**:

```
1. 判断是否需要升级（SOP 第六章）
   └─> 自主排查 15 分钟未果 -> 请求人工协助

2. 准备会诊材料（检查清单 7.2）
   ├─> 问题现象详细描述
   ├─> 已尝试的排查步骤
   ├─> 相关日志片段
   └─> 数据库查询结果

3. 使用问题诊断记录模板（文档 05）
   └─> 详细记录诊断过程

4. 团队会诊后更新知识库（SOP 第七章）
```

## 每日开发流程

### 早上开始工作

```bash
# 1. 运行全部测试，确认基线正常
cd backend && npm run test
cd frontend && npm run test:unit

# 2. 查看是否有失败的测试
# 如有，优先修复

# 3. 检查今日开发任务
# - 新功能 -> 先写测试
# - Bug 修复 -> 先复现问题
```

### 开发过程中

```
每次修改代码前思考:
1. 这个修改需要什么类型的测试？
2. 现有测试是否已覆盖？
3. 是否需要补充边界场景测试？

修改完成后:
1. 运行相关单元测试
2. 运行集成测试（如适用）
3. 手动验证功能（如需要）
```

### 提交代码前

```bash
# 执行完整检查清单（检查清单 6.1）
npm run lint          # ESLint 检查
npm run type-check    # TypeScript 类型检查
npm run format        # 代码格式化
npm run test          # 全部单元测试
npm run skill:check   # SKILL 合规检查

# 确认所有检查 100% 通过
```

## 常见问题解答

### Q1: 测试失败怎么办？

**A**:

1. 首先阅读错误信息，定位失败原因
2. 检查是语法错误、逻辑错误还是环境问题
3. 使用快速参考卡第三章定位问题
4. 如是代码问题，立即修复；如是测试问题，更新测试

### Q2: 如何确定测试覆盖率是否足够？

**A**:

1. 运行带覆盖率的测试：`npm run test -- --coverage`
2. 查看核心业务逻辑覆盖率（目标≥80%）
3. 检查是否有未覆盖的边界场景
4. 使用检查清单 2.2 验证测试用例设计

### Q3: 遇到偶发性问题如何排查？

**A**:

1. 使用 SOP 3.2 的二分法定位
2. 增加详细日志记录
3. 添加监控告警
4. 收集更多样本数据
5. 必要时请求人工协助（SOP 第六章）

### Q4: 如何平衡测试时间和开发进度？

**A**:

1. 遵循测试优先原则，前期投入节省后期调试时间
2. 使用 TDD 方法，边写测试边开发
3. 优先覆盖核心业务逻辑和边界场景
4. 利用自动化检查工具提高效率

### Q5: 问题修复后如何防止复发？

**A**:

1. 必须添加对应的单元测试（检查清单 5.2）
2. 将问题场景加入回归测试套件
3. 考虑添加监控告警
4. 更新文档和知识库（SOP 第七章）
5. 团队分享，避免其他人踩坑

## 最佳实践建议

### 对新开发者的建议

1. **第一周**:
   - 精读 SOP 文档（文档 02）
   - 打印快速参考卡（文档 03）置顶
   - 熟悉常用调试脚本

2. **第一个月**:
   - 严格按照检查清单执行（文档 04）
   - 每次问题修复都填写记录模板（文档 05）
   - 积累常见错误排查经验

3. **三个月后**:
   - 能够独立排查复杂问题
   - 优化和改进测试流程
   - 指导新人，分享经验

### 对团队的建议

1. **晨会**: 分享昨日遇到的典型问题和解决方案
2. **周会**: 复盘本周重大故障，更新 SOP
3. **月度**: 组织技术分享，培训测试技能
4. **季度**: 审查测试覆盖率，制定提升计划
