# 测试指南

LogiX 项目测试规范与实践。

## 运行测试的前提条件

### 1. Node.js 版本要求

```bash
# 检查 Node.js 版本（必须 >= 18.0.0）
node --version

# 检查 npm 版本
npm --version
```

**推荐版本**:
- Node.js: 18.x 或更高
- npm: 9.x 或更高

### 2. 安装依赖

```bash
# 后端测试依赖
cd backend
npm install

# 前端测试依赖
cd frontend
npm install
```

### 3. 环境变量配置

**后端测试需要配置环境变量**：

```bash
# 创建测试环境文件
cd backend
cp .env.example .env

# 或使用测试专用配置
cp .env.test.example .env.test
```

**最小配置要求**（`backend/.env` 或 `backend/.env.test`）:
```env
NODE_ENV=test
DB_HOST=localhost
DB_PORT=5432
DB_USERNAME=postgres
DB_PASSWORD=postgres
DB_DATABASE=logix_test
```

**重要说明**: 
- 单元测试使用 Mock，不需要真实数据库连接
- 集成测试需要真实的测试数据库
- 测试数据库应该与开发数据库隔离

### 4. TypeScript 编译

```bash
# 确保 TypeScript 配置正确
cd backend
npm run type-check

# 如果有类型错误，先修复
```

### 5. 测试目录结构

```
backend/
├── src/
│   ├── services/
│   │   ├── ContainerService.ts
│   │   └── ContainerService.test.ts    # 同级放置
│   ├── tests/
│   │   ├── cost-optimizer-calculation.test.ts
│   │   └── integration/                # 集成测试
│   └── test/
│       └── setup.ts                    # 全局测试配置
```

## 测试入门路径（由浅入深）

### 第一阶段：理解测试基础（1-2 天）

#### Step 1: 认识测试文件结构

**最简单的测试示例**：`backend/src/config/scheduling.config.test.ts`

```typescript
import { SchedulingConfig } from './scheduling.config';

describe('SchedulingConfig', () => {
  it('should have correct default values', () => {
    // Arrange
    const config = new SchedulingConfig();
    
    // Act & Assert
    expect(config.maxAttempts).toBeDefined();
    expect(config.timeout).toBeGreaterThan(0);
  });
});
```

**学习要点**:
- `describe()`: 描述被测试的模块
- `it()`: 具体的测试用例
- `expect()`: 断言预期结果

#### Step 2: 运行第一个测试

```bash
# 进入后端目录
cd backend

# 运行单个测试文件
npm run test -- scheduling.config.test.ts

# 查看输出
# PASS  src/config/scheduling.config.test.ts
#   SchedulingConfig
#     ✓ should have correct default values (3 ms)
```

**如何识别错误**:

```bash
# ❌ 失败的测试输出示例
FAIL  src/config/scheduling.config.test.ts
  SchedulingConfig
    ✕ should have correct default values (5 ms)

  ● SchedulingConfig › should have correct default values

    expect(received).toBeGreaterThan(expected)

    Expected: > 0
    Received:   -1

      5 |     expect(config.maxAttempts).toBeDefined();
      6 | >   expect(config.timeout).toBeGreaterThan(0);
        |                            ^
      7 |   });
```

**错误解读**:
- **错误类型**: `expect().toBeGreaterThan()` 失败
- **期望值**: > 0
- **实际值**: -1
- **错误位置**: 第 6 行
- **修复方法**: 检查 `config.timeout` 为什么是 -1

### 第二阶段：编写服务层测试（3-5 天）

#### Step 3: 学习 Mock 基础

**示例**: `backend/src/services/ContainerFilterService.test.ts`

```typescript
// 1. Mock 外部依赖（数据库、API 等）
jest.mock('../database', () => ({
  AppDataSource: {
    getRepository: jest.fn()
  }
}));

describe('ContainerFilterService', () => {
  let service: ContainerFilterService;
  let mockQueryBuilder: any;

  beforeEach(() => {
    // 2. 创建 Mock 对象
    mockQueryBuilder = {
      leftJoinAndSelect: jest.fn().mockReturnThis(),
      where: jest.fn().mockReturnThis(),
      andWhere: jest.fn().mockReturnThis(),
      getMany: jest.fn().mockResolvedValue([])
    };

    // 3. 配置 Mock
    (AppDataSource.getRepository as jest.Mock).mockReturnValue({
      createQueryBuilder: jest.fn().mockReturnValue(mockQueryBuilder)
    });

    service = new ContainerFilterService();
  });

  it('should filter containers by port codes', async () => {
    // 4. Arrange - 准备测试数据
    const options = {
      portCodes: ['USLAX'],
      minFreeDays: 3
    };

    // 5. Act - 执行被测试的方法
    const result = await service.filter(options);

    // 6. Assert - 验证结果
    expect(result).toBeDefined();
    expect(Array.isArray(result)).toBe(true);
  });
});
```

**Mock 的作用**:
- 隔离外部依赖（数据库、HTTP 请求）
- 控制测试环境
- 加快测试执行速度
- 避免副作用

### 第三阶段：复杂业务逻辑测试（1-2 周）

#### Step 5: 学习复杂场景测试

**示例**: `backend/src/services/DemurrageDateCalculator.test.ts`

这个测试文件包含 286 行，覆盖了：
- 日期计算（addDays, daysBetween）
- 周末识别（isWeekend）
- 工作日计算（addWorkingDays, workingDaysBetween）
- 免费期计算（calculateLastFreeDate）
- 计费天数计算（calculateChargeDays）

**关键测试模式**:

```typescript
describe('addWorkingDays', () => {
  it('应该跳过周末添加工作日', () => {
    // 从周一开始添加 5 个工作日
    const monday = new Date('2026-03-09');
    const result = calculator.addWorkingDays(monday, 5);
    
    // Mon(1), Tue(2), Wed(3), Thu(4), Fri(5) -> 2026-03-13
    expect(result.toISOString()).toBe('2026-03-13T00:00:00.000Z');
  });

  it('应该正确处理跨越周末的情况', () => {
    // 从周五开始添加 3 个工作日
    const friday = new Date('2026-03-13');
    const result = calculator.addWorkingDays(friday, 3);
    
    // Fri(1), skip Sat/Sun, Mon(2), Tue(3) -> 2026-03-17
    expect(result.toISOString()).toBe('2026-03-17T00:00:00.000Z');
  });
});
```

**学习要点**:
- 覆盖边界条件（周末、节假日）
- 使用具体日期而非相对日期
- 注释清晰说明计算过程

### 第四阶段：集成测试（2-3 天）

#### Step 6: 集成测试示例

**示例**: `backend/tests/integration/scheduling/intelligent-scheduling.e2e.test.ts`

```typescript
import { AppDataSource } from '../../../src/database';
import { IntelligentSchedulingService } from '../../../src/services/intelligentScheduling.service';

describe('IntelligentScheduling E2E', () => {
  beforeAll(async () => {
    // 初始化真实数据库连接
    await AppDataSource.initialize();
  });

  afterAll(async () => {
    // 关闭数据库连接
    await AppDataSource.destroy();
  });

  it('should complete full scheduling process', async () => {
    // 使用真实数据进行端到端测试
    const service = new IntelligentSchedulingService();
    const result = await service.schedule('CONTAINER_001');
    
    expect(result.success).toBe(true);
    expect(result.truckingCompany).toBeDefined();
    expect(result.warehouse).toBeDefined();
  });
});
```

**集成测试 vs 单元测试**:

| 对比项 | 单元测试 | 集成测试 |
|--------|---------|---------|
| 依赖 Mock | 全部 Mock | 部分或无 Mock |
| 执行速度 | 快（毫秒级） | 慢（秒级） |
| 测试范围 | 单一功能 | 多个模块协作 |
| 数据库 | 不需要 | 需要 |
| 用途 | 开发驱动 | 验收测试 |

## 如何识别和调试测试错误

### 错误类型 1: 语法错误

**症状**:
```bash
FAIL  src/services/MyService.test.ts
  ● Test suite failed to run

    SyntaxError: Unexpected token (10:4)
```

**原因**:
- TypeScript 语法错误
- 缺少 import 语句
- 括号不匹配

**解决方法**:
```bash
# 1. 运行 TypeScript 检查
npm run type-check

# 2. 查看具体错误位置
# 编辑器会显示红色波浪线

# 3. 修复语法后重新运行测试
```

### 错误类型 2: Mock 配置错误

**症状**:
```bash
FAIL  src/services/ContainerService.test.ts
  ● ContainerService › should get container by id

    TypeError: Cannot read property 'findOne' of undefined

      15 |   const container = await this.containerRepository.findOne({
      16 |     where: { id }
    > 17 |   });
         |                    ^
```

**原因**:
- Mock 对象未正确配置
- 缺少必要的 Mock 方法

**解决方法**:
```typescript
// ❌ 错误示例
const mockRepo = {};

// ✅ 正确示例
const mockRepo = {
  findOne: jest.fn().mockResolvedValue({ id: 1, name: 'Test' }),
  find: jest.fn().mockResolvedValue([]),
  save: jest.fn().mockResolvedValue({}),
  createQueryBuilder: jest.fn().mockReturnValue({
    where: jest.fn().mockReturnThis(),
    getOne: jest.fn().mockResolvedValue(null)
  })
};
```

### 错误类型 3: 异步测试错误

**症状**:
```bash
FAIL  src/services/AsyncService.test.ts
  ● AsyncService › should fetch data

    thrown: "Exceeded timeout of 5000 ms for a test.
    Use jest.setTimeout(newTimeout) to increase the timeout value, if this is a long-running test."
```

**原因**:
- 异步方法未使用 `async/await`
- Promise 未正确返回
- 超时设置过短

**解决方法**:
```typescript
// ❌ 错误示例
it('should fetch data', () => {
  service.fetchData(); // 缺少 await
});

// ✅ 正确示例
it('should fetch data', async () => {
  const data = await service.fetchData();
  expect(data).toBeDefined();
});

// 增加超时时间
it('should fetch large dataset', async () => {
  jest.setTimeout(30000); // 30 秒
  const data = await service.fetchLargeDataset();
  expect(data.length).toBeGreaterThan(0);
});
```

### 错误类型 4: 断言失败

**症状**:
```bash
FAIL  src/services/Calculator.test.ts
  ● Calculator › should add numbers

    expect(received).toBe(expected) // Object.is equality

    Expected: 5
    Received: 4

      10 |     const result = calculator.add(2, 2);
    > 11 |     expect(result).toBe(5);
         |                    ^
      12 |   });
```

**原因**:
- 断言预期值错误
- 业务逻辑 bug
- 测试数据不正确

**解决方法**:
```typescript
// 1. 检查业务逻辑
const result = calculator.add(2, 2);
console.log('Actual result:', result); // 调试输出

// 2. 修正断言
expect(result).toBe(4); // 正确的预期值

// 3. 或者修复业务逻辑
if (result !== 4) {
  throw new Error('Calculator bug!');
}
```

### 错误类型 5: 测试污染

**症状**:
```bash
PASS  src/services/Service1.test.ts
FAIL  src/services/Service2.test.ts
  ● Service2 › should work

    expect(received).toHaveLength(expected)

    Expected length: 0
    Received length: 3
    Received array:  [{id: 1}, {id: 2}, {id: 3}]
```

**原因**:
- 前一个测试修改了全局状态
- Mock 未正确清理
- 数据库数据未清理

**解决方法**:
```typescript
// 在 setup.ts 中配置自动清理
beforeEach(() => {
  jest.clearAllMocks(); // 清理 Mock
});

afterEach(() => {
  jest.resetAllMocks(); // 重置 Mock
});

// 或在测试文件中手动清理
describe('MyService', () => {
  beforeEach(() => {
    // 清理特定状态
    cleanupTestData();
  });
});
```

### 错误类型 6: 环境变量缺失

**症状**:
```bash
FAIL  src/services/ConfigService.test.ts
  ● ConfigService › should load config

    Error: Missing required environment variable: API_KEY
```

**原因**:
- 测试需要环境变量但未配置
- `.env` 文件不存在

**解决方法**:
```typescript
// 在测试中 Mock 环境变量
process.env.API_KEY = 'test-key';
process.env.DB_HOST = 'localhost';

// 或在 setup.ts 中统一配置
// backend/src/test/setup.ts
process.env.NODE_ENV = 'test';
process.env.DB_HOST = 'localhost';
process.env.DB_PORT = '5432';
// ...
```

## 最佳实践

### 测试命名

```
describe: [模块名]
it: should [预期行为]
```

### 测试覆盖

```
核心业务逻辑: 必须覆盖
API 接口: 必须覆盖
复杂组件: 必须覆盖
工具函数: 必须覆盖
```

### Mock 原则

```
外部依赖: 必须 Mock
数据库: 使用内存数据库或 Mock
API 调用: 使用 Mock Service
时间: Mock Date
```
