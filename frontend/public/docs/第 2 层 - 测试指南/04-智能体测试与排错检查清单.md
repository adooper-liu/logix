# 智能体测试与排错检查清单

## 一、开发前测试检查清单

### 1.1 现有测试覆盖检查

- [ ] 运行全部单元测试，确认通过率 100%
- [ ] 查看目标模块测试覆盖率（核心逻辑≥80%）
- [ ] 确认有集成测试/E2E 测试覆盖关键路径
- [ ] 检查测试用例是否覆盖边界场景
- [ ] 确认 Mock 策略合理（外部依赖全部 Mock）

**命令**:
```bash
cd backend && npm run test
cd frontend && npm run test:unit -- --coverage
```

### 1.2 测试策略确定

| 修改类型 | 必须执行的测试 |
|---------|--------------|
| Bug 修复 | ✅ 复现问题的失败测试<br>✅ 修复后的回归测试 |
| 新功能 | ✅ TDD 测试用例<br>✅ 边界场景测试<br>✅ 异常处理测试 |
| 重构 | ✅ 现有功能回归测试<br>✅ 性能对比测试（如适用） |
| 性能优化 | ✅ 性能基准测试<br>✅ 负载测试<br>✅ 回归测试 |

## 二、单元测试编写检查清单

### 2.1 测试结构检查

- [ ] 测试文件命名正确（`*.test.ts`或同级放置）
- [ ] describe 块清晰描述被测试模块
- [ ] beforeEach 正确初始化 Mock 对象
- [ ] afterEach 清理 Mock 状态
- [ ] 测试用例独立，无相互依赖

### 2.2 测试用例设计检查

- [ ] 覆盖正常流程（Happy Path）
- [ ] 覆盖边界条件（空值、极值、特殊字符）
- [ ] 覆盖异常情况（错误输入、网络失败）
- [ ] 覆盖业务规则（状态转换、权限校验）
- [ ] 测试命名清晰（should + 预期行为）

### 2.3 Mock 检查

- [ ] 所有外部依赖已 Mock（数据库、API、文件系统）
- [ ] Mock 数据符合实际业务场景
- [ ] Mock 返回值覆盖各种情况（成功、失败、空值）
- [ ] 验证 Mock 调用参数正确性

### 2.4 断言检查

- [ ] 使用合适的断言方法（toBeDefined, toEqual, toHaveLength 等）
- [ ] 断言信息充分，能从错误信息定位问题
- [ ] 避免过度断言（一个测试只验证一个行为）
- [ ] 异步测试正确使用 async/await 或 done

## 三、问题诊断检查清单

### 3.1 信息收集（5W1H）

- [ ] **What**: 具体错误现象是什么
- [ ] **When**: 何时发生（首次出现时间、频率、周期性）
- [ ] **Where**: 在哪个环节/页面/接口出现
- [ ] **Who**: 哪些用户受影响（特定用户还是全部）
- [ ] **Why**: 可能的触发条件
- [ ] **How**: 如何复现（详细步骤）

### 3.2 日志检查

- [ ] 查看后端错误日志（backend/logs/error.log）
- [ ] 搜索相关关键词（容器号、订单号、错误码）
- [ ] 检查日志时间戳，确认问题发生时间点
- [ ] 查看前后关联日志，理解完整流程

**命令**:
```bash
tail -f backend/logs/error.log
grep "[container_number]" backend.log
```

### 3.3 数据库检查

- [ ] 查询相关表的数据是否存在
- [ ] 检查字段值是否正确
- [ ] 验证关联关系是否完整
- [ ] 查看数据时间戳，确认数据更新时间

**命令**:
```sql
-- 检查数据量
SELECT COUNT(*) FROM [table_name];

-- 检查详情
SELECT * FROM [table_name] WHERE [condition];

-- 检查关联
SELECT t1.*, t2.* 
FROM [table1] t1
LEFT JOIN [table2] t2 ON t1.id = t2.foreign_id
WHERE t1.id = 'XXX';
```

### 3.4 API 检查

- [ ] 使用 Postman/curl 直接调用 API
- [ ] 检查请求参数是否正确
- [ ] 检查响应状态码和返回数据
- [ ] 检查 Header 是否完整（Content-Type、Authorization）

**命令**:
```bash
curl -v http://localhost:3001/api/v1/[endpoint]
```

## 四、根因分析检查清单

### 4.1 五问法分析

- [ ] Why 1: 为什么会出现这个现象
- [ ] Why 2: 为什么会导致这个问题
- [ ] Why 3: 为什么会有这个缺陷
- [ ] Why 4: 为什么会遗漏这个场景
- [ ] Why 5: 为什么制度/流程未预防

### 4.2 二分法定位

- [ ] 确定问题范围（前端/后端/数据库）
- [ ] 在数据流中间点检查数据状态
- [ ] 根据检查结果缩小范围
- [ ] 重复检查直到定位根因

### 4.3 常见原因排查

**数据问题**:
- [ ] 数据库表结构是否正确
- [ ] 数据导入逻辑是否有 bug
- [ ] 映射关系是否配置正确
- [ ] 数据同步是否延迟

**逻辑问题**:
- [ ] 业务逻辑代码是否有 bug
- [ ] 算法实现是否正确
- [ ] 边界条件是否处理
- [ ] 状态机转换是否正确

**接口问题**:
- [ ] 路由定义是否正确
- [ ] 参数校验是否通过
- [ ] 响应处理是否正确
- [ ] 错误处理是否完善

**前端问题**:
- [ ] 组件逻辑是否正确
- [ ] 状态管理是否混乱
- [ ] 网络请求参数是否正确
- [ ] 数据渲染逻辑是否有误

**性能问题**:
- [ ] SQL 查询是否优化（EXPLAIN 分析）
- [ ] 是否有循环嵌套
- [ ] 缓存策略是否合理
- [ ] 是否有内存泄漏
