## 系统概述

### 业务价值

甘特图系统是 LogiX 物流系统的核心可视化组件，用于展示货柜从清关到还箱的完整物流生命周期。

**核心价值**：
- **可视化追踪**：直观展示货柜在各物流节点的计划与实际进度
- **智能预警**：自动识别超期、即将超期等风险货柜（红/橙/灰三色）
- **交互操作**：支持拖拽改期、右键菜单、详情查看
- **多维度分组**：按目的港 -> 节点 -> 供应商三级分组展示
- **数据容错**：通过反向链式依赖推断缺失的前置节点状态

### 核心架构

```
SimpleGanttChart (主组件)
├── useGanttLogic (核心逻辑 Composable)
│   ├── 数据加载与过滤
│   ├── 日期范围计算
│   ├── 分组与排序
│   └── Tooltip 管理
├── 三级分组结构
│   ├── Level 1: 目的港汇总行
│   ├── Level 2: 五节点泳道（清关/提柜/卸柜/还箱/查验）
│   └── Level 3: 供应商分组
├── 圆点渲染引擎
│   ├── calculateNodeStatus (状态计算)
│   ├── getNodeDisplayType (显示类型决策)
│   └── 正向+反向链式依赖检查

    ├── 拖拽改期

```

## 核心原理

### 甘特图数据结构

#### 三级分组结构

**Level 1: 目的港汇总行**
- 显示港口名称、总柜数、预警统计
- 支持折叠/展开所有子节点

**Level 2: 五节点泳道**
- 清关（Customs）
- 查验（Inspection，可选）
- 提柜（Pickup）
- 卸柜（Unload）
- 还箱（Return）

**Level 3: 供应商分组**
- 清关行（customsBrokerCode）
- 车队（truckingCompanyId）
- 仓库（warehouseId）
- 还箱码头（returnTerminalCode）

#### 节点状态机

每个节点有三种状态：

| 状态 | 含义 | 视觉表现 | 触发条件 |
|------|------|----------|----------|
| `pending` | 待激活 | 空心圆点（白色填充 + 虚线边框） | 有计划日期，但前置节点未完成 |
| `active` | 可执行 | 实心圆点（有背景色 + 实线边框） | 有计划日期，且前置节点已完成 |
| `completed` | 已完成 | 不显示圆点（销毁） | 有实际日期，或通过后向推导得出 |

### 圆点渲染流程

```
calculateNodeStatus() 
    ↓ 计算节点状态 (pending/active/completed)
    ↓ 考虑正向依赖（前置节点完成检查）
    ↓ 考虑反向依赖（后续节点完成反推）
getNodeDisplayType() 
    ↓ 决定显示类型 (main/dashed/null)
    ↓ 优先使用本地状态（卸柜/还箱）
    ↓ 回退到 ganttDerived（清关/提柜）
getNodeTypeFromLocalStatus()
    ↓ completed → null, active → main, pending → dashed
模板 v-if="displayType"
    ↓ null 时不渲染 DOM
圆点显示/销毁
```

## 圆点生命周期

### 完整生命周期

```
创建 → 待激活(pending) → 可执行(active) → 已完成(completed) → 销毁(null)
         ↓                    ↓                  ↓
      空心圆点            实心圆点           不显示圆点
      (虚线边框)          (实线边框)         (DOM移除)
```

### 各阶段详解

#### 1. 创建阶段

**触发条件**：
- 货柜有供应商信息（清关行/车队/仓库）
- 有计划日期或实际日期

**示例**：
```typescript
// 清关节点创建
if (customsSupplier) {
  nodes.清关.supplier = customsSupplier
  if (destPortOp.plannedCustomsDate) {
    nodes.清关.plannedDate = new Date(destPortOp.plannedCustomsDate)
  }
}
```

#### 2. 待激活阶段 (pending)

**特征**：
- 有空心圆点（白色填充 + 虚线边框）
- 表示"有计划，但前置条件未满足"

**判断逻辑**：
```typescript
// 提柜节点：清关未完成 + 有计划日期
if (nodes.提柜.plannedDate) {
  const customsCompleted = destPortOp?.actualCustomsDate
  if (!customsCompleted) {
    nodes.提柜.status = 'pending'  // 空心圆点
  }
}
```

**视觉效果**：
- 背景色：transparent（白色填充）
- 边框：2px dashed #c0c4cc（灰色虚线）
- 边框颜色根据预警等级变化（红/橙/灰）

#### 3. 可执行阶段 (active)

**特征**：
- 有实心圆点（有背景色 + 实线边框）
- 表示"前置条件已满足，可以执行"

**判断逻辑**：
```typescript
// 提柜节点：清关已完成 + 有计划日期
if (nodes.提柜.plannedDate) {
  const customsCompleted = destPortOp?.actualCustomsDate
  if (customsCompleted) {
    nodes.提柜.status = 'active'  // 实心圆点
  }
}
```

**视觉效果**：
- 背景色：根据物流状态（蓝/绿/橙/灰）
- 边框：2px solid（实线）
- 边框颜色根据预警等级变化（红/橙/灰）

#### 4. 已完成阶段 (completed)

**特征**：
- 圆点销毁，不显示
- Tooltip 中仍可查看实际日期（绿色标识）

**判断逻辑**：
```typescript
// 提柜节点：有实际日期
if (pickupTransport.deliveryDate || pickupTransport.pickupDate) {
  nodes.提柜.status = 'completed'  // 销毁
}
```

**销毁机制**：
```vue
<div v-if="displayType" class="container-dot">
  <!-- displayType 为 null 时，Vue 不渲染该 DOM -->
</div>
```

#### 5. 销毁阶段 (null)

**转换规则**：
```typescript
const getNodeTypeFromLocalStatus = (node: any): 'main' | 'dashed' | null => {
  if (node.status === 'completed') return null  // 销毁
  if (node.status === 'active') return 'main'   // 实心圆点
  if (node.status === 'pending') return 'dashed' // 空心圆点
  return null
}
```

### 生命周期示例

#### 示例1：正常流程

```
时间线: T1        T2        T3        T4        T5
事件:   计划清关  实际清关  计划提柜  实际提柜  计划卸柜

清关:   pending → active  → completed(销毁)
提柜:             pending → active    → completed(销毁)
卸柜:                       pending   → active
```

#### 示例2：数据缺失（反向依赖生效）

```
数据状态:
actualCustomsDate: null      // 清关日期缺失
deliveryDate: "2026-04-11"   // 但已有送仓日期

反向推导:
nodes.清关.status = 'completed'  // 反推清关已完成 -> 销毁
nodes.提柜.status = 'completed'  // 提柜本身完成 -> 销毁

最终显示: 卸柜=main, 还箱=dashed（2个圆点）
```

### 设计原则

1. **背景色独立于边框色**：背景色反映物流状态，边框色反映预警等级
2. **所有圆点都有边框**：不使用透明边框，统一使用红/橙/灰三色
3. **虚线圆点保持空心**：使用 `!important` 确保白色填充不被内联样式覆盖
4. **实际日期用绿色**：在 Tooltip 中使用绿色标识已完成节点

## 链式依赖逻辑

### 正向依赖（前置检查）

**完整依赖链条**：清关 -> 提柜 -> 卸柜 -> 还箱

| 节点 | 前置依赖 | 判断逻辑 |
|------|----------|----------|
| 清关 | 无 | 有计划日期 -> active；有实际日期 -> completed |
| 提柜 | **清关完成** | 清关未完成+有计划 -> pending<br>清关已完成+有计划 -> active<br>有实际日期 -> completed |
| 卸柜 | **提柜完成** | 提柜未完成+有计划 -> pending<br>提柜已完成+有计划 -> active<br>有实际日期 -> completed |
| 还箱 | **卸柜完成** | 卸柜未完成+有计划 -> pending<br>卸柜已完成+有计划 -> active<br>有实际日期 -> completed |

**前置节点完成判断**：
- 清关完成：`destPortOp?.actualCustomsDate` 存在
- 提柜完成：`pickupTransport?.deliveryDate || pickupTransport?.pickupDate` 存在
- 卸柜完成：`unloadOp?.unloadDate` 存在
- 还箱完成：`emptyReturn?.returnTime` 存在

**代码实现**（`SimpleGanttChartRefactored.vue` Line 2576-2587）：

```typescript
// 提柜状态判断：增加对清关的前置依赖检查
if (pickupTransport.deliveryDate || pickupTransport.pickupDate) {
  nodes.提柜.status = 'completed'  // 已完成
} else if (nodes.提柜.plannedDate) {
  // 检查前置节点（清关）是否完成
  const customsCompleted = destPortOp?.actualCustomsDate
  if (customsCompleted) {
    nodes.提柜.status = 'active'   // 前置已完成 + 有计划 = 实心圆点
  } else {
    nodes.提柜.status = 'pending'  // 前置未完成 + 有计划 = 空心圆点
  }
}
```

### 反向依赖（后续推导）

**业务规则**：后续节点完成意味着前置节点必然已完成，即使前置节点的实际日期字段缺失。

| 触发条件 | 反推结果 | 适用场景 |
|---------|---------|----------|
| **已提柜** (`deliveryDate` 或 `pickupDate` 存在) | 清关 -> completed | 数据录入不完整，但业务上清关必然已完成 |
| **已卸柜** (`unloadDate` 存在) | 清关 -> completed<br>提柜 -> completed | 只记录了卸柜日期，前置日期缺失 |
| **已还箱** (`returnTime` 存在) | 清关 -> completed<br>提柜 -> completed<br>卸柜 -> completed | 只记录了还箱日期，所有前置日期缺失 |

**设计原则**：
- 优先使用实际日期字段判断状态
- 如果实际日期缺失，但后续节点已完成，则反推前置节点为 completed
- 反推的前提是供应商信息存在（确保节点应该显示）
- 避免显示大量虚线圆点（待激活），提升用户体验

**代码实现**（`SimpleGanttChartRefactored.vue` Line 2652-2686）：

```typescript
// ========== 反向链式依赖检查 ==========
// 业务规则：后续节点完成意味着前置节点必然已完成
// 已提柜 -> 清关完成 | 已卸柜 -> 清关+提柜完成 | 已还箱 -> 清关+提柜+卸柜完成

// 1. 已还箱 -> 反推清关、提柜、卸柜全部完成
if (emptyReturn?.returnTime) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed'  // 反推清关已完成
  }
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate) && pickupSupplier) {
    nodes.提柜.status = 'completed'  // 反推提柜已完成
  }
  if (!unloadOp?.unloadDate && unloadSupplier) {
    nodes.卸柜.status = 'completed'  // 反推卸柜已完成
  }
}
// 2. 已卸柜 -> 反推清关、提柜完成
else if (unloadOp?.unloadDate) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed'  // 反推清关已完成
  }
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate) && pickupSupplier) {
    nodes.提柜.status = 'completed'  // 反推提柜已完成
  }
}
// 3. 已提柜 -> 反推清关完成
else if (pickupTransport?.deliveryDate || pickupTransport?.pickupDate) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed'  // 反推清关已完成
  }
}
```

### 双向依赖的优势

1. **容错性强**：即使前置节点日期缺失，也能正确推断状态
2. **用户体验好**：避免显示大量虚线圆点（待激活）
3. **业务准确性高**：符合物流实际流程（后续完成意味着前置必然完成）
4. **数据一致性**：减少因数据录入顺序不同导致的显示差异

## 反向链式依赖

### 业务场景分析

#### 场景1：仅提柜（清关日期缺失）

**数据状态**：
```typescript
actualCustomsDate: null      // 清关日期缺失
deliveryDate: "2026-04-11"   // 但已有送仓日期
plannedUnloadDate: "2026-04-14"
lastReturnDate: "2026-04-16"
```

**正向依赖结果**（无反向推导）：
- 清关：无 actualCustomsDate，但有 plannedCustomsDate -> `active`（实心圆点）❌ 错误
- 提柜：有 deliveryDate -> `completed`（销毁）✅
- 卸柜：提柜已完成 + 有计划 -> `active`（实心圆点）✅
- 还箱：卸柜未完成 + 有计划 -> `pending`（空心圆点）✅

**问题**：清关不应该显示实心圆点，因为实际上已经完成（只是日期缺失）

**反向依赖结果**：
```typescript
// 检测到 deliveryDate 存在，反推清关已完成
if (pickupTransport?.deliveryDate && !destPortOp?.actualCustomsDate) {
  nodes.清关.status = 'completed'  // 反推清关已完成 -> 销毁
}
```

**最终显示**：
- 清关：`completed` -> `null`（销毁）✅ 正确
- 提柜：`completed` -> `null`（销毁）✅
- 卸柜：`active` -> `main`（实心圆点）✅
- 还箱：`pending` -> `dashed`（空心圆点）✅

**圆点总数**：2个（1实1虚）

| **容错性** | 弱 | 强 |

### 核心函数详解

#### 1. calculateNodeStatus - 节点状态计算

**位置**: `SimpleGanttChartRefactored.vue` Line 2498-2697

**职责**：计算货柜在各节点的状态（pending/active/completed）

**输入**：container 对象  
**输出**：ContainerNodeStatus 对象

**核心逻辑**：

```typescript
const calculateNodeStatus = (container: any): ContainerNodeStatus => {
  const nodes: any = {
    清关: { status: 'pending', supplier: '未指定', plannedDate: undefined, actualDate: undefined },
    查验: { status: 'pending', supplier: '未指定', plannedDate: undefined, actualDate: undefined },
    提柜: { status: 'pending', supplier: '未指定', plannedDate: undefined, actualDate: undefined },
    卸柜: { status: 'pending', supplier: '未指定', plannedDate: undefined, actualDate: undefined },
    还箱: { status: 'pending', supplier: '未指定', plannedDate: undefined, actualDate: undefined },
  }

  // 1. 判断清关状态
  const destPortOp = container.portOperations?.find((op: any) => op.portType === 'destination')
  const customsSupplier = destPortOp?.customsBrokerCode || destPortOp?.customsBroker
  if (customsSupplier) {
    nodes.清关.supplier = customsSupplier
    // ... 设置 plannedDate 和 actualDate
    
    if (destPortOp.actualCustomsDate) {
      nodes.清关.status = 'completed'
    } else if (nodes.清关.plannedDate) {
      nodes.清关.status = 'active'
    }
  }

  // 2. 判断提柜状态（检查清关前置依赖）
  const pickupTransport = container.truckingTransports?.[0]
  const pickupSupplier = pickupTransport?.truckingCompanyId || pickupTransport?.carrierCompany
  if (pickupSupplier) {
    nodes.提柜.supplier = pickupSupplier
    // ... 设置 plannedDate 和 actualDate
    
    if (pickupTransport.deliveryDate || pickupTransport.pickupDate) {
      nodes.提柜.status = 'completed'  // 有实际日期 = 完成
    } else if (nodes.提柜.plannedDate) {
      // 检查前置节点（清关）是否完成
      const customsCompleted = destPortOp?.actualCustomsDate
      if (customsCompleted) {
        nodes.提柜.status = 'active'   // 前置已完成 + 有计划 = 活跃
      } else {
        nodes.提柜.status = 'pending'  // 前置未完成 + 有计划 = 待激活
      }
    }
  }

  // 3. 判断卸柜状态（检查提柜前置依赖）
  // 4. 判断还箱状态（检查卸柜前置依赖）
  // ...

  // 5. 反向链式依赖检查
  if (emptyReturn?.returnTime) {
    // 已还箱 -> 反推清关、提柜、卸柜全部完成
    if (!destPortOp?.actualCustomsDate && customsSupplier) {
      nodes.清关.status = 'completed'
    }
    // ...
  } else if (unloadOp?.unloadDate) {
    // 已卸柜 -> 反推清关、提柜完成
    // ...
  } else if (pickupTransport?.deliveryDate || pickupTransport?.pickupDate) {
    // 已提柜 -> 反推清关完成
    // ...
  }

  return {
    containerNumber: container.containerNumber,
    portCode,
    nodes,
  }
}
```

#### 2. getNodeDisplayType - 显示类型决策

**位置**: `SimpleGanttChartRefactored.vue` Line 1932-1957

**职责**：决定节点在甘特图中的显示类型（main/dashed/null）

**输入**：container 对象, nodeName 字符串  
**输出**：'main' | 'dashed' | null

**核心逻辑**：

```typescript
const getNodeDisplayType = (container: any, nodeName: string): 'main' | 'dashed' | null => {
  const nodeStatus = calculateNodeStatus(container)
  const node = nodeStatus.nodes[nodeName as keyof typeof nodeStatus.nodes]

  // 边界检查：无供应商或供应商未指定
  if (!node?.supplier || node.supplier === '未指定') return null

  // 1. 查验节点特殊处理（后端 gantt-v1 无此节点，仅信本地状态）
  if (nodeName === '查验') {
    return getInspectionNodeType(node)
  }

  // 2. 卸柜/还箱节点：优先使用本地状态（避免后端 ganttDerived 错误标记为 completed）
  if (nodeName === '卸柜' || nodeName === '还箱') {
    return getNodeTypeFromLocalStatus(node)
  }

  // 3. 其他节点使用后端 ganttDerived（如果存在且有效）
  const ganttType = getNodeTypeFromGanttDerived(container, nodeName)
  if (ganttType !== null) {
    return ganttType
  }

  // 4. 回退到本地状态判断
  return getNodeTypeFromLocalStatus(node)
}
```

**关键策略**：
- 卸柜/还箱节点跳过 `ganttDerived`，直接使用本地状态（避免后端错误标记）
- 清关/提柜节点优先使用 `ganttDerived`，失败后回退本地状态

#### 3. getNodeTypeFromLocalStatus - 状态转换

**位置**: `SimpleGanttChartRefactored.vue` Line 1919-1924

**职责**：将节点状态转换为显示类型

**输入**：node 对象  
**输出**：'main' | 'dashed' | null

**核心逻辑**：

```typescript
const getNodeTypeFromLocalStatus = (node: any): 'main' | 'dashed' | null => {
  if (node.status === 'completed') return null  // completed 转换为 null，不显示
  if (node.status === 'active') return 'main'   // active 转换为 main，实心圆点
  if (node.status === 'pending') return 'dashed' // pending 转换为 dashed，空心圆点
  return null
}
```

**核心转换规则**：
- `completed` → `null`：**销毁圆点，不渲染DOM**
- `active` → `'main'`：显示实心圆点
- `pending` → `'dashed'`：显示空心圆点

## 常见问题

### 问题1：已完成节点仍显示圆点

**现象**：有实际日期的节点仍然显示圆点

**可能原因**：
1. 实际日期字段为空（数据库未更新）
2. `calculateNodeStatus` 未正确识别实际日期
3. `getNodeTypeFromLocalStatus` 返回错误类型

**排查步骤**：
```typescript
// 1. 检查数据库字段
console.log('actualCustomsDate:', destPortOp.actualCustomsDate)
console.log('deliveryDate:', pickupTransport.deliveryDate)
console.log('unloadDate:', unloadOp.unloadDate)

console.log('提柜状态:', nodeStatus.nodes.提柜.status)

// 3. 检查显示类型
const displayType = getNodeDisplayType(container, '提柜')
console.log('提柜显示类型:', displayType)  // 应为 null
```

**解决方案**：
- 确认数据库中实际日期字段已正确更新
- 检查 `calculateNodeStatus` 中的判断逻辑
- 确认 `getNodeTypeFromLocalStatus` 正确转换状态

### 问题2：Tooltip 中不显示实际日期

**现象**：圆点已销毁，但 Tooltip 中看不到实际日期

**可能原因**：
1. `hasActualDates` 返回 `false`
2. 辅助函数返回 `null`
3. 模板条件判断失败

**排查步骤**：
```typescript

// 3. 检查 Tooltip 容器数据

```

**解决方案**：
- 确认实际日期字段存在
- 检查辅助函数的返回值
- 确认 Tooltip 模板的条件判断正确

### 问题3：链式依赖失效

**现象**：前置节点已完成，后续节点仍显示 `pending`（空心圆点）

**可能原因**：
1. 前置节点的实际日期字段为空
2. 链式依赖检查逻辑错误
3. 反向依赖未生效

**排查步骤**：

- 检查 `calculateNodeStatus` 中的链式依赖逻辑

### 问题4：后端 ganttDerived 数据不可信

**现象**：卸柜/还箱节点被错误标记为 `completed`

**根本原因**：后端 `ganttDerived` 可能错误地将后续节点标记为 `completed`
