## 🔴 Critical：必须修复

### 1. 数据库表名与字段名错误

**位置**: 方案 3.3 节「新增索引」

**问题**:
```sql
-- ❌ 方案中的错误
CREATE INDEX IF NOT EXISTS idx_demurrage_standard_port_company
ON ext_demurrage_standard(destination_port_code, shipping_company_code, is_chargeable);
-- 表名错误：应为 ext_demurrage_standards（复数）

CREATE INDEX IF NOT EXISTS idx_warehouse_occupancy_date
ON ext_warehouse_daily_occancy(occupancy_date, warehouse_id);
-- 表名拼写错误：occancy → occupancy
-- 字段名错误：实际为 date, warehouse_code（非 occupancy_date, warehouse_id）

CREATE INDEX IF NOT EXISTS idx_trucking_slot_occupancy
ON ext_trucking_slot_occupancy(occupancy_date, trucking_company_id, port_code);
-- 字段名错误：实际为 date（非 occupancy_date）
```

**修正**:
```sql
-- ✅ 正确写法
CREATE INDEX IF NOT EXISTS idx_demurrage_standard_port_company
ON ext_demurrage_standards(destination_port_code, shipping_company_code, is_chargeable);

CREATE INDEX IF NOT EXISTS idx_warehouse_occupancy_date
ON ext_warehouse_daily_occupancy(date, warehouse_code);

CREATE INDEX IF NOT EXISTS idx_trucking_slot_occupancy
ON ext_trucking_slot_occupancy(date, trucking_company_id, port_code);
```

**依据**: `backend/03_create_tables_supplement.sql`、`backend/src/entities/` 实体定义

### 2. 不存在的服务依赖

**位置**: 方案 3.2 节「接口 2: 评估总成本」

**问题**:

2. **滞箱费预测**: 
   - 在 `demurrage.service.ts` 中新增 `predictDetentionForReturnDate(containerNumber, proposedReturnDate)` 
   - 内部复用 `calculateSingleDemurrage`，传入 `plannedPickupDate` 与 `proposedReturnDate` 作为起止日

### 3. 与 intelligent-scheduling-mapping Skill 的约束

**位置**: 方案 3.2 节「generateAllFeasibleOptions」、3.2 节「evaluateTotalCost」

**Skill 要求**:
- 仓库必须来自 `port → trucking → warehouse` 映射链
- 车队必须来自 `dict_warehouse_trucking_mapping`，且须同时服务该港口
- 无映射时返回 `[]`，禁止回退到「该国全部仓库」

**方案符合点**: ✅ `getCandidateWarehouses(countryCode, portCode)` 已遵循上述规则

**需补充**:
- `generateDropOffOptions`、`generateExpeditedOptions` 中的车队选择必须同样遵循 `selectTruckingCompany` 逻辑，仅从 `warehouse_trucking_mapping` 选
- 在方案文档中明确：**所有方案生成均基于现有映射关系，不引入新的数据源**

## 🟡 Suggestion：建议改进

### 4. 车队与成本评估的先后顺序

**问题**: 运输费依赖车队，但 `evaluateTotalCost` 时 `UnloadOption` 可能尚未确定车队。

**当前方案**:
```typescript
interface UnloadOption {
  warehouse: Warehouse;
  unloadDate: Date;
  strategy: 'Direct' | 'Drop off' | 'Expedited';
  truckingCompany?: TruckingCompany;  // 可选
}
```

**建议**: 在 `generateAllFeasibleOptions` 阶段即为每个 `(warehouse, unloadDate, strategy)` 确定 `truckingCompany`，再传入 `evaluateTotalCost`。这样：
- 运输费可基于实际车队估算
- 与现有 `selectTruckingCompany(warehouseCode, portCode, pickupDate, country)` 逻辑一致

---

### 5. 滞箱费预测的业务前提

**问题**: `demurrage.service` 中滞箱费计算依赖 `pickupDateActual`（实际提柜日），排产时仅有计划日期。

**建议**: 
- `predictDetentionForReturnDate` 应明确使用 `plannedPickupDate` 作为起算日
- 在接口注释中说明：**预测模式**下用计划日期，与**实际结算**用实际日期的差异
- 可选：在 `DemurrageCalculationResult` 中增加 `calculationMode: 'actual' | 'forecast'` 以区分

### 6. 前端国际化

**位置**: 方案 3.4 节 Vue 模板

**问题**: 存在硬编码中文，违反开发准则「用户可见文案用 $t()/t()」

```vue
<!-- ❌ 硬编码 -->
<h4>成本明细</h4>
<cost-row label="滞港费" ... />
<span class="savings-best">最优方案</span>
```

**建议**:
```vue
<!-- ✅ 国际化 -->
<h4>{{ $t('scheduling.costBreakdown') }}</h4>
<cost-row :label="$t('scheduling.demurrageCost')" ... />
<span class="savings-best">{{ $t('scheduling.bestOption') }}</span>
```

### 8. 组件职责划分

**问题**: 方案同时提到：
- `schedulingCostOptimizer.service.ts` — 成本优化服务
- `unloadOptionGenerator.service.ts` — 方案生成器  
- `costEvaluation.service.ts` — 成本评估器

但 Phase 2 任务 2.1 仅创建 `SchedulingCostOptimizerService`，且其内部已包含 `generateAllFeasibleOptions`、`evaluateTotalCost`、`selectBestOption`。

**建议**: 
- **Phase 2**: 保持单一 `SchedulingCostOptimizerService`，职责内聚，便于测试
- **Phase 4 或后续**: 若文件超过 300 行，再拆分为 `UnloadOptionGenerator`、`CostEvaluator` 两个子服务
