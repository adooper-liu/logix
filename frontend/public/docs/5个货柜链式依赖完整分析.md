## 关键发现

### 1. 反向链式依赖的价值

**场景**：3个已提柜货柜（HMMU6855127、GAOU6195045、KOCU5129260）

**问题**：

- 清关日期缺失（`actualCustomsDate = null`）
- 正向链式误判清关为 `active`（实心圆点）
- 用户看到实心圆点，以为清关还需执行

**解决**：

- 反向检测到 `deliveryDate` 存在
- 反推清关必然已完成
- 将清关状态修正为 `completed` → 销毁

**效果**：

- 消除3个误导性圆点
- 圆点总数从17个降至14个
- 用户看到的都是真正需要关注的节点

### 2. 未提柜货柜不受影响

**场景**：2个未提柜货柜（HMMU6232153、HMMU6019657）

**原因**：

- 没有任何后续节点完成（提柜、卸柜、还箱都未完成）
- 反向推导条件不满足
- 保持正向链式结果不变

**结果**：

- 清关显示为 `active`（正确，因为有计划日期且无前置依赖）
- 提柜、卸柜、还箱显示为 `pending`（正确，因为前置未完成）
- 共4个圆点，符合预期

### 3. 数据完整性的重要性

| 数据状态                     | 正向链式准确性 | 反向链式必要性 |
| ---------------------------- | -------------- | -------------- |
| 完整（所有日期都有）         | 100% 准确      | 不需要         |
| 部分缺失（如清关日期缺失）   | 有误判风险     | **必需**       |
| 严重缺失（只有最后一个日期） | 大量误判       | **关键**       |

**结论**：

- 数据越不完整，反向链式依赖越重要
- 反向推导是数据容错的关键机制

// ========== 反向链式依赖检查 ==========
// 业务规则：后续节点完成意味着前置节点必然已完成
// 已提柜 -> 清关完成 | 已卸柜 -> 清关+提柜完成 | 已还箱 -> 清关+提柜+卸柜完成

// 1. 已还箱 -> 反推清关、提柜、卸柜全部完成
if (emptyReturn?.returnTime) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推清关已完成
  }
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate) && pickupSupplier) {
    nodes.提柜.status = 'completed' // 反推提柜已完成
  }
  if (!unloadOp?.unloadDate && unloadSupplier) {
    nodes.卸柜.status = 'completed' // 反推卸柜已完成
  }
}
// 2. 已卸柜 -> 反推清关、提柜完成
else if (unloadOp?.unloadDate) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推清关已完成
  }
  if (!(pickupTransport?.deliveryDate || pickupTransport?.pickupDate) && pickupSupplier) {
    nodes.提柜.status = 'completed' // 反推提柜已完成
  }
}
// 3. 已提柜 -> 反推清关完成
else if (pickupTransport?.deliveryDate || pickupTransport?.pickupDate) {
  if (!destPortOp?.actualCustomsDate && customsSupplier) {
    nodes.清关.status = 'completed' // 反推清关已完成
  }
}

## 总结

### 正向链式 vs 反向链式

| 维度            | 正向链式            | 反向链式            |
| --------------- | ------------------- | ------------------- |
| **检查方向**    | 前置 → 当前         | 后续 → 当前         |
| **触发条件**    | 自身有实际日期      | 后续节点有实际日期  |
| **适用场景**    | 数据完整            | 数据缺失            |
| **准确性**      | 依赖数据完整性      | 容错性强            |
| **5个货柜结果** | 17个圆点（3个错误） | 14个圆点（0个错误） |

### 核心价值

1. **数据容错**：即使清关日期缺失，也能通过提柜日期推断清关已完成
2. **用户体验**：减少误导性圆点，只显示真正需要关注的节点
3. **业务准确**：符合物流实际流程（能提柜说明清关必然完成）
4. **视觉清晰**：圆点总数从17个降至14个，降低视觉干扰

### 最佳实践

- **优先使用正向链式**：数据完整时，正向检查最准确
- **补充使用反向链式**：数据缺失时，反向推导提供容错
- **两者结合**：实现最佳的准确性和容错性平衡
