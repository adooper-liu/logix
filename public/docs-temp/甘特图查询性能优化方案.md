# 甘特图查询性能优化方案

## 问题描述

甘特图页面在加载数据时存在以下性能问题：

1. **阻塞页面渲染**：在 `onMounted` 中直接 `await` 查询500条货柜数据 + 统计数据
2. **用户体验差**：页面白屏时间长，缺少加载状态反馈
3. **无错误处理**：查询失败时没有友好提示

## 已实施的优化（方案1：异步加载 + 骨架屏）

### 1. 异步加载数据

```typescript
// 延迟数据加载到下一个事件循环
onMounted(() => {
  setTimeout(() => {
    loadData()
  }, 0)
})
```

**优势**：
- 页面先渲染骨架屏，用户立即看到加载状态
- 不阻塞初始渲染

### 2. 骨架屏加载状态

```vue
<el-card class="gantt-card" v-loading="loading">
  <!-- 初始加载时显示骨架屏 -->
  <div v-if="isInitialLoad && loading" class="gantt-skeleton">
    <div class="skeleton-header">
      <div class="skeleton-title"></div>
    </div>
    <div class="skeleton-lanes">
      <div v-for="i in 4" :key="i" class="skeleton-lane">
        <div class="skeleton-lane-title"></div>
        <div class="skeleton-lane-content"></div>
      </div>
    </div>
  </div>

  <!-- 加载完成或刷新时显示正常内容 -->
  <template v-else>
    <ContainerGanttChart ... />
  </template>
</el-card>
```

**优势**：
- 用户感知加载进度，减少焦虑
- 骨架屏与实际布局一致，切换无闪烁

### 3. 错误状态处理

```typescript
const error = ref<string | null>(null)

const loadData = async () => {
  loading.value = true
  error.value = null
  try {
    // ... 查询逻辑
  } catch (error: any) {
    error.value = error.message || '未知错误'
    // ElMessage 错误提示
  } finally {
    loading.value = false
    isInitialLoad.value = false
  }
}
```

**优势**：
- 明确的错误提示
- 区分初始加载和刷新

## 可选的进一步优化方案

### 方案2：分页/增量加载（适用于大数据量场景）

**适用场景**：当货柜数据超过500条时

**实现方式**：
1. 后端支持分页参数 `page` 和 `pageSize`
2. 前端使用虚拟滚动组件（如 `vue-virtual-scroller`）
3. 用户滚动到底部时自动加载下一页

**优势**：
- 减少单次查询数据量
- 降低内存占用

**劣势**：
- 需要较大改动
- 甘特图的时间范围计算需要调整

### 方案3：Web Worker 后台查询

**适用场景**：当数据处理逻辑复杂时

**实现方式**：
1. 将数据查询和计算逻辑移到 Web Worker
2. 主线程负责渲染，Worker负责数据处理
3. 通过 `postMessage` 通信

**优势**：
- 不阻塞主线程渲染
- 提升页面响应速度

**劣势**：
- 代码复杂度增加
- 调试难度较高

### 方案4：缓存策略

**适用场景**：用户频繁刷新或切换筛选条件时

**实现方式**：
1. 使用 `Map` 缓存已加载的数据
2. 缓存键：`filterCondition + startDate + endDate`
3. 设置缓存过期时间（如5分钟）

```typescript
const cache = new Map<string, { data: any; timestamp: number }>()

const loadData = async () => {
  const cacheKey = `${filterCondition}-${startDate}-${endDate}`
  const cached = cache.get(cacheKey)

  if (cached && Date.now() - cached.timestamp < 300000) {
    containers.value = cached.data
    return
  }

  // ... 查询逻辑
  cache.set(cacheKey, { data: containers.value, timestamp: Date.now() })
}
```

**优势**：
- 减少重复请求
- 提升响应速度

**劣势**：
- 需要处理缓存失效
- 占用额外内存

## 后端优化建议

### 1. 查询优化

**当前问题**：
```typescript
// container.controller.ts 第857-862行
const withRelations = await this.containerRepository.find({
  where: { containerNumber: In(containerNumbers) },
  relations: ['seaFreight']
})
```

**优化建议**：
- 使用 `select` 只查询必要字段
- 添加数据库索引（`container_number`、`ata_dest_port` 等）
- 使用 `QueryBuilder` 优化关联查询

### 2. 批量查询优化

```typescript
// 优化前：N+1查询
for (const container of containers) {
  const status = await this.containerStatusService.getStatus(container.containerNumber)
}

// 优化后：批量查询
const containerNumbers = containers.map(c => c.containerNumber)
const statuses = await this.containerStatusService.batchGetStatuses(containerNumbers)
```

### 3. API超时配置

**当前配置**：
```typescript
// frontend/src/services/container.ts 第24行
timeout: 120000  // 2分钟
```

**建议**：
- 根据实际查询时间调整
- 对大数据量查询返回部分数据，并提供分页

## 性能监控建议

### 1. 添加性能日志

```typescript
const loadData = async () => {
  const startTime = performance.now()
  loading.value = true
  try {
    // ... 查询逻辑
    const endTime = performance.now()
    logger.info(`[Performance] Data loaded in ${(endTime - startTime).toFixed(2)}ms`)
  } finally {
    loading.value = false
  }
}
```

### 2. 用户埋点

```typescript
// 记录加载次数、失败率、平均加载时间
analytics.track('gantt_chart_load', {
  duration: endTime - startTime,
  success: true,
  containerCount: containers.value.length
})
```
