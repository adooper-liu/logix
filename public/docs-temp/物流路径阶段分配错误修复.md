## 问题清单

### 问题 1: FETA 阶段分配错误

**现象**: FETA（交货地抵达）被分配到"提柜"阶段，但按时间顺序应在"到港"阶段

**根因**: `STANDARD_MODE_TEMPLATE` 中 `DELIVERY_ARRIVED` 被错误放入提柜阶段的 statuses 数组

**影响**: 
- FETA 节点显示在错误的阶段
- 阶段顺序混乱
- 不符合业务逻辑

**修复方案**:
```typescript
// 修改前（第 96-102 行）
{
  order: 10,
  status: StandardStatus.GATE_OUT,
  label: "提柜",
  statuses: [
    StandardStatus.GATE_OUT,
    StandardStatus.IN_TRANSIT_TO_DEST,
    StandardStatus.DELIVERY_ARRIVED,  // ❌ 错误
    StandardStatus.STRIPPED,
  ],
}

// 修改后
{
  order: 7,
  status: StandardStatus.ARRIVED,
  label: "抵港",
  statuses: [
    StandardStatus.ARRIVED,
    StandardStatus.BERTHED,      // POCA 靠泊
    StandardStatus.DELIVERY_ARRIVED,  // ✅ FETA 交货地抵达
  ],
}
```

### 问题 3: "可提 缺数据"判断逻辑

**现象**: 界面显示"可提 缺数据"，但实际有 STCS（提柜）数据

**根因**: 
- "可提"阶段对应 `StandardStatus.AVAILABLE`
- 系统只认 `AVAILABLE` 状态的事件
- 但飞驼数据中使用的是 `PCAB`（可提货）状态码
- `PCAB` 可能被映射到其他标准状态，不是 `AVAILABLE`

**当前映射逻辑**（第 259-267 行）:
```typescript
function mapStatusCodeToStandard(code: string | null): StandardStatus {
  if (!code) return StandardStatus.UNKNOWN;
  const upper = String(code).trim().toUpperCase();
  const fromWarning = FEITUO_WARNING_MAP[upper];
  if (fromWarning) return fromWarning;
  const fromMain = FEITUO_STATUS_MAP[upper];
  if (fromMain) return fromMain;
  return StandardStatus.UNKNOWN;
}
```

**待排查**: 
1. `FEITUO_STATUS_MAP['PCAB']` 映射到什么？
2. 是否应该映射到 `StandardStatus.AVAILABLE`？

**数据库证据**:
```
1570 | HMMU6232153 | STCS | 提柜 (货) | 2026-03-30 11:48:00 | is_estimated=1
```

注意：STCS 是"提柜"动作，不是"可提货"状态。"可提货"应该是 PCAB。

## 业务逻辑验证

### FETA 的正确阶段

根据业务时间顺序：
```
1. 抵港（BDAR）: 2026-03-17 15:41:00
2. 交货地抵达（FETA）: 2026-03-17 15:41:00  ← 与抵港同时
3. 靠泊（POCA）: 2026-03-21 15:13:00
4. 卸船（DSCH）: 2026-03-22 07:48:00
5. 提柜（STCS）: 2026-03-30 11:48:00
```

**结论**: FETA 与 BDAR 同时发生，应该在"到港"阶段，不是在"提柜"阶段。

### "可提"状态的正确理解

**业务流程**:
```
卸船完成 → 可提货（PCAB/AVAILABLE） → 提柜（STCS/GTOT） → 还箱（RCVE）
```

**当前问题**:
- 有 STCS（提柜）数据
- 但没有 PCAB（可提货）数据
- 系统认为"可提"阶段缺数据

**待确认**:
1. PCAB 是否应该映射到 `StandardStatus.AVAILABLE`？
2. 如果 PCAB 缺失但有 STCS，是否应该跳过"可提"阶段？
