# 中转港与目的港混淆问题 - 根本解决方案

## 问题根源

### 1. 状态机逻辑错误（containerStatus.service.ts:189-201）

```typescript
private isAtPort(dates: DateInfo): boolean {
  // ❌ 错误：已到中转港被判定为 at_port
  if (dates.hasTransit && !dates.hasDestinationAta) {
    return true; // 已到中转港，等待到目的港
  }

  // ✓ 正确：已到目的港
  if (dates.hasDestinationAta) {
    return true;
  }

  return false;
}
```

**问题：**
- "有中转港但无目的港ATA"被判定为 `at_port`
- 但这实际上是 `in_transit`（运输途中）

### 2. 数据导入逻辑问题（import.controller.ts）

导入时直接更新状态，未使用状态机判断：

```typescript
// ❌ 错误：直接设置状态
const logisticsStatus = this.validateLogisticsStatus(
  containerData.logisticsStatus || ''
);

if (existingContainer) {
  Object.assign(existingContainer, {
    ...containerData,
    logisticsStatus  // 直接使用导入的状态
  });
}
```

### 3. 统计查询问题（containerStatistics.service.ts）

所有统计查询都需要排除中转港：

```sql
-- ❌ 当前方式：每个查询都要手动过滤
WHERE port_type = 'destination'
AND NOT EXISTS (
  SELECT 1 FROM process_port_operations transit
  WHERE transit.container_number = po.container_number
  AND transit.port_type = 'transit'
)
```

## 解决方案

### 方案1：修正状态机逻辑（立即执行）

**修改 `isAtPort` 方法：**

```typescript
/**
 * 判断是否已到目的港（不包括中转港）
 * @param dates 日期信息
 * @returns 是否已到目的港
 */
private isAtPort(dates: DateInfo): boolean {
  // ✓ 只判断目的港ATA，不考虑中转港
  // 中转港货柜无目的港ATA时，应为 in_transit
  return dates.hasDestinationAta === true;
}
```

**影响范围：**
- 状态判断逻辑
- 自动状态更新
- 批量状态更新任务

---

### 方案2：数据导入时禁用手动状态设置（中期）

**修改 `import.controller.ts`：**

```typescript
// 导入时不设置 logistics_status，由状态机自动判断
if (existingContainer) {
```

### 方案3：创建港口操作视图（长期）

**创建视图统一排除中转港：**

```sql
-- 创建目的港视图（自动排除中转港）
CREATE OR REPLACE VIEW v_destination_ports AS
SELECT
    po.*
FROM process_port_operations po
WHERE po.port_type = 'destination'
AND NOT EXISTS (
    SELECT 1 FROM process_port_operations transit
    WHERE transit.container_number = po.container_number
    AND transit.port_type = 'transit'
)
WITH CHECK OPTION;
```

**使用视图简化查询：**

```sql
-- ✓ 简化后的统计查询
SELECT COUNT(DISTINCT c.container_number)
FROM biz_containers c
INNER JOIN v_destination_ports po ON c.container_number = po.container_number
WHERE po.ata_dest_port IS NOT NULL;
```

### 方案4：添加数据库约束（预防性）

**添加CHECK约束防止状态设置错误：**

```sql
-- 添加约束：at_port 状态必须有目的港ATA
ALTER TABLE biz_containers
ADD CONSTRAINT chk_at_port_requires_dest_ata
CHECK (
    logistics_status != 'at_port'
    OR EXISTS (
        SELECT 1 FROM process_port_operations po
        WHERE po.container_number = container_number
        AND po.port_type = 'destination'
        AND po.ata_dest_port IS NOT NULL
    )
);
```

### 方案5：统一状态管理接口（架构级）

**创建统一的状态更新接口：**

```typescript
// containerStatus.service.ts

/**
 * 统一的状态更新入口
 * 所有状态更新必须通过此方法
 */
async updateStatus(containerNumber: string): Promise<boolean> {
  try {
    const expectedStatus = await this.determineStatus(containerNumber);

    await this.containerRepository.update(
      { containerNumber },
      { logisticsStatus: expectedStatus }
    );

    return true;
  } catch (error) {
    logger.error(`更新状态失败: ${containerNumber}`, error);
    return false;
  }
}

/**
 * 批量更新状态
 */
async batchUpdateStatuses(limit: number = 1000): Promise<number> {
  const containers = await this.containerRepository.find({ take: limit });

  let updatedCount = 0;
  for (const container of containers) {
    const success = await this.updateStatus(container.containerNumber);
    if (success) updatedCount++;
  }

  return updatedCount;
}
```

## 关键要点

1. **中转港定义**：只是途经港口，不影响最终状态
2. **目的港定义**：真正的目的地，决定 `at_port` 状态
3. **状态规则**：
   - 有中转港 + 无目的港ATA = `in_transit`
   - 有目的港ATA = `at_port`
4. **统一原则**：所有状态判断都通过状态机，禁止手动设置
