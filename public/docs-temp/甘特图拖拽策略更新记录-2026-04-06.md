## 紧急修复 (2026-04-06 第二次更新)

### 问题1: 还箱日没有重新计算或同步移动

**原因分析**:

- `performBackwardSyncDecision` 函数中只处理了提柜日、送柜日、卸柜日的同步
- **没有处理还箱日的自动重算逻辑**
- `mergeReturnDateIntoUpdateData` 只在 `updateData.plannedUnloadDate` 存在时才生效
- 当拖拽提柜日时，如果卸柜日也被同步更新，还箱日应该重新计算，但当前逻辑**没有触发这个重算**

**修复方案**:

```typescript
// ✅ 如果卸柜日被同步更新，也需要重算还箱日
if (syncDecision.autoSyncFields.includes("plannedUnloadDate") && !updateData.plannedUnloadDate) {
  // 手动设置卸柜日以便触发还箱日重算
  updateData.plannedUnloadDate = newDate;
  mergeReturnDateIntoUpdateData(container, updateData, unloadMode);
}
```

### 问题2: 还箱节点不可以拖拽

**原因分析**:

- `NODE_TO_FIELD_MAP` 中已经定义了还箱节点的映射
- 但 `performBackwardSyncDecision` 中**没有处理还箱日拖拽的逻辑**
- 还箱日的前置约束已定义（卸柜日），但后向同步决策中缺少对还箱日拖拽的处理

**修复方案**:

```typescript
// 4. ✅ 还箱日拖动：允许拖拽，但不同步其他节点
if (currentField === "plannedReturnDate") {
  // 还箱日是最终节点，不需要同步后续节点
  // 但需要确保还箱日 >= 卸柜日（前向检查已处理）
  return {
    autoSyncFields: [],
    needsUserConfirm: false,
  };
}
```

### 修复后的行为

| 场景                 | 修复前            | 修复后                    |
| -------------------- | ----------------- | ------------------------- |
| 拖拽提柜日(LIVE)     | 还箱日不变 ❌     | 还箱日自动重算 ✅         |
| 拖拽提柜日(DROP往后) | 还箱日不变 ❌     | 还箱日自动重算 ✅         |
| 拖拽卸柜日           | 还箱日自动重算 ✅ | 还箱日自动重算 ✅         |
| 拖拽还箱日           | 不允许拖拽 ❌     | 允许拖拽，仅更新还箱日 ✅ |
| 还箱日前向检查       | 无 ❌             | 确保还箱日>=卸柜日 ✅     |

## 更新内容

### 1. 核心改进

#### 1.1 新增前向检查机制

- **函数**: `performForwardCheck`
- **作用**: 确保目标日期 >= 前节点日期
- **约束关系**:
  - 提柜日的前置: 清关日
  - 送柜日的前置: 提柜日
  - 卸柜日的前置: 送柜日
  - 还箱日的前置: 卸柜日

#### 1.2 新增后向同步决策机制

- **函数**: `performBackwardSyncDecision`
- **作用**: 根据拖拽方向和卸柜模式决定后续节点是否同步
- **决策逻辑**:
  - LIVE模式: 提=送=卸强制同步
  - DROP模式往后移: 自动同步(如需要)
  - DROP模式往前移: 询问用户是否同步

### 2. 代码变更

#### 2.1 useGanttLogic.ts

**新增函数**:

- `performForwardCheck`: 前向检查函数(约80行)
- `performBackwardSyncDecision`: 后向同步决策函数(约120行)

**修改函数**:

- `handleDrop`: 重构为调用新的检查和决策函数
  - Step 1: 获取基本信息
  - Step 2: 前向检查(失败则阻止拖拽)
  - Step 3: 后向同步决策
  - Step 4: 应用同步决策
  - Step 5: 设置待确认数据

- `handleDragEnd`: 优化确认对话框文案
  - 统一提示: "检测到往前移动,后续节点是否同步前移?"
  - 按钮文案: "同步调整" / "仅调整当前节点"

**删除逻辑**:

- 原有的硬编码策略判断(约50行)
- 分散在各处的if-else分支

### 3. 业务规则对比

#### 3.1 LIVE模式

| 项目       | 旧逻辑        | 新逻辑                 |
| ---------- | ------------- | ---------------------- |
| 提柜日拖动 | 强制同步送/卸 | 强制同步送/卸(不变)    |
| 卸柜日拖动 | 强制同步提/送 | 强制同步提/送(不变)    |
| 前向检查   | 无            | **新增**: 目标>=前节点 |

#### 3.2 DROP模式

| 项目                   | 旧逻辑                         | 新逻辑                       |
| ---------------------- | ------------------------------ | ---------------------------- |
| 提柜日往后移           | 条件同步(送柜日晚于新提柜日时) | **优化**: 自动同步(更清晰)   |
| 提柜日往前移           | 询问用户                       | **优化**: 询问用户(文案统一) |
| 卸柜日往前移且早于提柜 | 标记但未使用                   | **修复**: 正确询问用户       |
| 前向检查               | 无                             | **新增**: 目标>=前节点       |
