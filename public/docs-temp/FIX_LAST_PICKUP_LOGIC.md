# 最晚提柜统计逻辑修复

## 🐛 问题描述

**现象**: "最晚提柜"卡片所有分类都显示为 0，但"按提柜统计"显示有 67 个货柜

**原因**: 查询条件错误，将"已安排拖卡但未实际提柜"的货柜排除在外

---

## ✅ 业务逻辑澄清

### **错误的理解**（之前的实现）
```
最晚提柜 = 只统计完全没有拖卡运输记录的货柜
查询条件：tt.containerNumber IS NULL
```

### **正确的理解**（修复后的实现）
```
最晚提柜 = 统计所有未实际提柜的货柜（不管有没有计划）
查询条件：tt.pickupDate IS NULL
```

---

## 📊 业务场景对比

| 场景 | 有计划日期 | 有实际提柜日期 | 之前是否统计 | **现在是否统计** |
|-----|----------|-------------|------------|----------------|
| 完全未安排拖卡 | ❌ | ❌ | ✅ | ✅ |
| 已安排未执行 | ✅ | ❌ | ❌ | **✅ (修复!)** |
| 已执行 | ✅ | ✅ | ❌ | ❌ |

---

## 🔧 代码修改

### 修改文件
`backend/src/services/containerStatistics.service.ts`

### 修改内容
修改所有最晚提柜分布查询的条件：

```typescript
// 修改前（错误）
.andWhere('tt.containerNumber IS NULL')

// 修改后（正确）
.andWhere('(tt.containerNumber IS NULL OR tt.pickupDate IS NULL)')
```

### 影响的方法
- `getExpiredCount()` - 已超时
- `getUrgentCount()` - 即将超时 (1-3 天)
- `getWarningCount()` - 预警 (4-7 天)
- `getNormalCount()` - 时间充裕 (7 天以上)
- `getNoLastFreeDateCount()` - 缺最后免费日

## 💡 业务价值

### 修复前的问题
- 无法监控"已安排但未执行"的货柜
- 这些货柜可能因为各种原因迟迟未提柜
- 导致错过最佳提柜时机，产生滞港费

### 修复后的价值
1. ✅ **全面监控**: 所有未实际提柜的货柜都在监控范围内
2. ✅ **风险预警**: 及时发现逾期或即将逾期的货柜
3. ✅ **避免损失**: 提前干预，避免产生滞港费
4. ✅ **业务闭环**: 与"按提柜统计"形成互补关系

---

## 🔄 与其他统计的关系

### 按提柜统计 vs 最晚提柜

| 维度 | 按提柜统计 | 最晚提柜 |
|-----|----------|---------|
| **目标集** | 所有 at_port 货柜 | 所有**未实际提柜**的货柜 |
| **包含已安排未执行** | ✅ | ✅ |
| **包含完全未安排** | ✅ | ✅ |
| **包含已执行** | ❌ | ❌ |
| **业务用途** | 整体进度监控 | 风险预警和倒计时 |

### 数据一致性
- "按提柜统计"总数 = "最晚提柜"总数（理论上应该相等）
- 差异："按提柜"包含今日实际提柜的货柜，"最晚提柜"不包含
