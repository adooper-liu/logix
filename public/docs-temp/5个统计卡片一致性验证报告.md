# 5个统计卡片一致性验证报告

## 问题摘要

5个统计卡片的统计逻辑、查询逻辑和前端映射存在以下不一致问题：

---

## 1. 【卡片1: 按物流状态】✅ 正确

### 状态机定义（containerStatus.service.ts）
```typescript
not_shipped: 无任何流程记录
shipped: 有海运记录但无港口操作记录
in_transit: 有港口操作记录但无目的港ATA
at_port: 有目的港ATA
picked_up: 有拖车运输记录且已提柜
unloaded: 有仓库操作记录
returned_empty: 有还空箱记录
```

### 数据库实际状态
```
not_shipped: 0
shipped: 0
in_transit: 167
arrived_at_transit: 68  ← 这是一个统计字段，不是状态机状态
at_port: 10
picked_up: 39
unloaded: 15
returned_empty: 119
```

### ✅ 结论
- 状态机逻辑正确
- 前端显示与数据库状态一致
- 注意：`arrived_at_transit`不是真实状态，只是统计子集

---

## 2. 【卡片2: 按到港】⚠️ 存在逻辑矛盾

### 后端统计（arrivalDistribution）
```json
{
  "arrivedAtDestination": 219,  // 已到目的港
  "arrivedAtTransit": 68,      // 已到中转港
  "expectedArrival": 82,      // 预计到港
  "today": 0,
  "arrivedBeforeTodayNotPickedUp": 46,
  "arrivedBeforeTodayPickedUp": 173
}
```

### 数据库实际验证
```
已到目的港: 219
已到中转港: 68
预计到港: 63  ← 与后端返回82不一致！
```

### ❌ 问题分析

**问题1：预计到港数量不一致**
- 后端统计：82
- 数据库验证：63
- 差异：19个货柜

**问题原因：**
状态机逻辑与统计逻辑不一致：
1. 状态机：`at_port`状态要求**必须有目的港ATA**
2. 统计逻辑：`expectedArrival`的统计条件是 `shipped OR in_transit`
3. 实际情况：某些货柜状态是`in_transit`但实际已有目的港ATA，被误统计为"预计到港"

**问题2：已到目的港数量过大**
- 后端统计：219个"已到目的港"
- 实际`at_port`状态：只有10个
- 原因：统计逻辑将所有有ATA的货柜都算作"已到目的港"，包括`picked_up`、`unloaded`、`returned_empty`状态

### ✅ 中转港排除逻辑正确
- `expectedArrival`统计正确排除了有中转港记录的货柜
- `arrivedAtTransit`与`expectedArrival`不重叠

- `unloaded`状态的货柜

## 根本原因总结

### 1. 状态机与统计逻辑不一致
**状态机定义**（containerStatus.service.ts:189-198）：
```typescript
private isAtPort(dates: DateInfo): boolean {
  return dates.hasDestinationAta === true;  // 必须有目的港ATA
}
```

**按到港统计**（arrivalDistribution）：
```typescript
arrivedAtDestination = 有ATA的货柜（包括picked_up/unloaded/returned_empty）
expectedArrival = shipped OR in_transit（排除中转港）
```

**矛盾点**：
- 状态机：`at_port`必须有目的港ATA
- 统计：`expectedArrival`包括`in_transit`状态（可能有ATA）
- 结果：某些货柜被重复统计或遗漏

### 2. 目标集定义不统一

| 维度 | 后端目标集 | 数据库验证 | 是否一致 |
|------|----------|----------|---------|
| 按到港 | 所有出运状态 | 所有出运状态 | ✅ |
| 按计划提柜 | 46个（错误） | 10个（at_port） | ❌ |
| 按最晚提柜 | 10个（at_port） | 10个（at_port） | ✅ |
| 按最晚还箱 | 54个（picked_up+unloaded） | 54个 | ✅ |

### 3. SQL查询条件复杂度

**ETA统计**（EtaStatistics.service.ts）：
```typescript
// 排除中转港
query.andWhere(`NOT EXISTS (
  SELECT 1 FROM process_port_operations transit_po
  WHERE transit_po.container_number = container.container_number
  AND transit_po.port_type = 'transit'
)`);
```

**按到港统计**（ArrivalStatistics.service.ts）：
```typescript
// 已到中转港：有transit记录 + dest无ATA + 状态在shipped/in_transit/at_port
query.andWhere('container.logisticsStatus IN (:...statuses)', {
  statuses: ['shipped', 'in_transit', 'at_port']
});
```

矛盾点：
- ETA统计排除中转港
- 按到港统计包含中转港
- 但两个维度都称为"按到港"，容易混淆

## 解决方案

### 方案1：统一目标集定义（推荐）

**核心原则**：
1. 所有统计维度必须明确目标集（基础条件）
2. 统计和查询必须使用相同的目标集SQL
3. 前端filterCondition直接对应后端方法，无中间映射

**具体修改**：

#### 修改1：按计划提柜目标集
```typescript
// PlannedPickupStatistics.service.ts
async getDistribution(startDate?: string, endDate?: string): Promise<Record<string, number>> {
  // 目标集：at_port状态（与按最晚提柜一致）
  const targetSetQuery = `
    SELECT c.container_number, tt.planned_pickup_date
    FROM biz_containers c
    INNER JOIN process_port_operations po ON c.container_number = po.container_number AND po.port_type = 'destination'
    LEFT JOIN process_trucking_transport tt ON c.container_number = tt.container_number
    WHERE c.logistics_status = 'at_port'
  `;
  // ... 统计逻辑
}
```

#### 修改2：按最晚还箱逻辑检查
```typescript
// LastReturnStatistics.service.ts
async getReturnExpiredCount(today: Date, startDate?: string, endDate?: string): Promise<number> {
  // 检查是否lastReturnDate字段命名错误
  // 应该是last_return_date而不是lastReturnDate
  query.andWhere('er.last_return_date < :today', { today });
}
```

### 方案2：重新设计按到港统计

**新分组逻辑**：
```
按到港：
  ├─ 已到目的港（at_port状态）
  │   ├─ 今日到港
  │   ├─ 之前未提柜
  │   └─ 之前已提柜
  ├─ 已到中转港（有中转港记录，无目的港ATA）
  │   └─ 按ETA细分
  └─ 运输途中（shipped/in_transit，无中转港，无目的港ATA）
      └─ 按ETA细分
```

**优势**：
- 与状态机一致
- 三个分组不重叠
- 逻辑清晰易懂
