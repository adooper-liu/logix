### 2.2 简化版状态机+时间机实现

#### 核心数据流

```
飞驼API / Excel导入
    ↓
ext_container_status_events + process_port_operations 核心字段
    ↓
状态机（calculateLogisticsStatus）→ 更新 biz_containers.logistics_status
    ↓
预警引擎（定时 SQL 检查）→ 写入 ext_container_alerts
    ↓

    ↓

```

#### 事件流与状态重算（已实现）

飞驼同步 / Excel 导入 → 写入 `ext_container_status_events` → 更新 `process_port_operations` 核心字段 → 触发 `calculateLogisticsStatus` 重算 → 更新 `biz_containers.logistics_status`。

**注意**：`calculateLogisticsStatus` 需传入完整参数（container, portOperations, seaFreight, truckingTransport, warehouseOperation, emptyReturn）才能正确判断 7 层状态。见 `container.service.ts`、`containerStatus.service.ts`。

**调度**：使用 `setInterval`（与 `containerStatusScheduler` 一致），不引入 `node-schedule`。

**SQL 注意**：清关超时检查需加 `po.status_occurred_at IS NOT NULL`，避免 NULL 比较。

**规则**：

- `checkCustomsOverdue`：清关超 48 小时

- `checkETADeviation`：需单独实现或暂缓

**createAlert**：同类型未解决预警不重复创建（去重）。

## 三、ComfyUI 集成方案（可选）

### 为什么不建议全量集成？

1. **复杂性高**：ComfyUI 的节点化架构需要大量自定义开发
2. **性能问题**：实时物流数据不适合工作流模式（延迟敏感）
3. **维护成本高**：每个节点都需要单独维护和测试

### 推荐方案：ComfyUI 作为分析工具

```typescript
// backend/src/services/comfyuiIntegration.service.ts

/**
 * ComfyUI集成服务：仅用于复杂分析和报告生成
 */
export class ComfyUIIntegrationService {
  /**
   * 生成货柜分析报告（非实时）
   */
  static async generateContainerAnalysis(containerNumber: string): Promise<{
    reportUrl: string;
    summary: string;
  }> {
    // 1. 获取货柜完整数据
    const container = await containerService.getContainerWithDetails(containerNumber);

    // 2. 构建ComfyUI工作流输入
    const workflowInput = {
      container_number: containerNumber,
      timeline: this.buildTimelineData(container),
      alerts: container.alerts,
      time_deviations: this.calculateDeviations(container),
      risk_factors: this.identifyRiskFactors(container),
    };

    // 3. 调用ComfyUI API（异步，不阻塞主流程）
    const comfyuiResponse = await this.callComfyUIWorkflow("container_analysis", {
      input: workflowInput,
      // ComfyUI工作流配置
      nodes: {
        text_analysis: { model: "gpt-4" },
        risk_scoring: { algorithm: "random_forest" },
        visualization: { type: "timeline" },
      },
    });

    // 4. 返回报告链接
    return {
      reportUrl: comfyuiResponse.output.images[0],
      summary: comfyuiResponse.output.text,
    };
  }

  /**
   * 调用ComfyUI工作流（异步）
   */
  private static async callComfyUIWorkflow(workflowName: string, params: any): Promise<any> {
    // 注意：这里是异步调用，不等待结果
    // 实际项目中可使用消息队列（如RabbitMQ）解耦

    return fetch(`${COMFYUI_URL}/api/workflows/${workflowName}/run`, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(params),
    }).then((r) => r.json());
  }
}
```

**使用场景**：

- 点击货柜详情页的"生成 AI 分析报告"按钮
- 非实时，后台处理，完成后发送通知
- 用于复杂问题诊断和趋势分析

## 六、关键设计决策

### 1. 为什么不用复杂的工作流引擎？

**答**：物流事件是**线性且实时**的，不需要分支和回退。简单的状态机+定时任务足够。

### 2. 为什么 ComfyUI 只用于分析，不用于实时处理？

**答**：ComfyUI 的节点化架构有**执行延迟**（秒级），不适合实时预警（需要毫秒级响应）。

### 3. 如何确保性能？

**答**：

- 事件处理：异步，不阻塞主流程
- 预警检查：定时批量，每小时一次
- 前端显示：WebSocket 推送，按需更新
- 数据库：索引优化，分区存储历史数据

### 4. 如何扩展新规则？

**答**：在`AlertService`中添加新方法，无需修改核心逻辑。规则即代码，简单直接。

2.4 定时任务：不应引入 node-schedule
问题：方案使用 node-schedule，而 LogiX 现有调度器使用 setInterval。

建议：与 containerStatusScheduler 保持一致，使用 setInterval：

// backend/src/schedulers/alertScheduler.ts
import { logger } from '../utils/logger.js';
export class AlertScheduler {
private alertService = new AlertService();
private intervalId: NodeJS.Timeout | null = null;
start(intervalMinutes: number = 60, delaySeconds: number = 10): void {
if (this.intervalId) return;
const intervalMs = intervalMinutes _ 60 _ 1000;
this.intervalId = setInterval(() => this.executeTask(), intervalMs);
setTimeout(() => this.executeTask(), delaySeconds \* 1000);
logger.info('[AlertScheduler] Started');
}
private async executeTask(): Promise<void> {
try {
await this.alertService.checkAllAlerts();
} catch (error) {
logger.error('[AlertScheduler] Error:', error);
}
}
}
export const alertScheduler = new AlertScheduler();
