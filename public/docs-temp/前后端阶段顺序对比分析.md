# 前后端阶段顺序对比分析

## 问题发现

**基准**：后端阶段模板定义了三种运输模式的 order，但前端 STAGE_MAP 只有一套固定的 order。

## 纯海运模式对比

| 状态 | 后端 order | 前端 order | 一致性 |
|------|-----------|-----------|--------|
| NOT_SHIPPED | 1 | 1 | ✅ |
| EMPTY_PICKED_UP | 2 | 1 | ⚠️ 前端合并为起运阶段 |
| GATE_IN | 3 | 1 | ⚠️ 前端合并为起运阶段 |
| LOADED | 4 | 1 | ⚠️ 前端合并为起运阶段 |
| DEPARTED | 5 | 1 | ⚠️ 前端合并为起运阶段 |
| SAILING | 6 | 5 | ✅ |
| ARRIVED/BERTHED/DISCHARGED/AVAILABLE | 7-9 | 6 | ⚠️ 后端分 3 个阶段，前端合并为到港 |
| GATE_OUT/STRIPPED | 10-11 | 10 | ✅ |
| RETURNED_EMPTY/COMPLETED | 11 | 11 | ✅ |

**结论**：前端将起运阶段（1-5）合并为"origin"，将抵港/卸船/可提（7-9）合并为"arrival"。

## 海铁联运模式对比

| 状态 | 后端 order | 前端 order | 一致性 |
|------|-----------|-----------|--------|
| RAIL_LOADED | 9 | 9 | ✅ |
| RAIL_DEPARTED | 10 | 9 | ⚠️ 前端合并为铁路阶段 |
| RAIL_ARRIVED | 11 | 9 | ⚠️ 前端合并为铁路阶段 |
| RAIL_DISCHARGED | 12 | 9 | ⚠️ 前端合并为铁路阶段 |
| GATE_OUT | 13 | 10 | ⚠️ 后端 order=13，前端 order=10 |
| IN_TRANSIT_TO_DEST | 13 | 10 | ⚠️ 后端 order=13，前端 order=10 |

**问题**：前端铁路阶段 order=9，但后端铁路段 order=9-12，导致铁路阶段在"海运"（order=5）之前显示。

## 驳船联运模式对比

| 状态 | 后端 order | 前端 order | 一致性 |
|------|-----------|-----------|--------|
| FEEDER_LOADED | 4 | 4 | ✅ |
| FEEDER_DEPARTED | 5 | 4 | ⚠️ 前端合并为驳船阶段 |
| FEEDER_ARRIVED | 6 | 4 | ⚠️ 前端合并为驳船阶段 |
| FEEDER_DISCHARGED | 7 | 4 | ⚠️ 前端合并为驳船阶段 |
| LOADED | 8 | 1 | ⚠️ 后端 order=8，前端 order=1（起运） |
| DEPARTED | 9 | 1 | ⚠️ 后端 order=9，前端 order=1（起运） |
| SAILING | 10 | 5 | ⚠️ 后端 order=10，前端 order=5 |

**问题**：驳船模式下，海运装船后的阶段 order 全部错位。

## 根本原因

**设计差异**：
- **后端**：每种运输模式有独立的 order，精确到每个阶段
- **前端**：只有一套固定的 STAGE_MAP，通过 stage 分组 + order 排序

**前端策略**：
```typescript
// 前端按 stage 分组，同组内不区分 order
NOT_SHIPPED: { stage: 'origin', label: '起运', order: 1 },
EMPTY_PICKED_UP: { stage: 'origin', label: '起运', order: 1 },  // 同组
GATE_IN: { stage: 'origin', label: '起运', order: 1 },          // 同组
```

**后端策略**：
```typescript
// 后端每个阶段有独立 order
{ order: 1, status: NOT_SHIPPED, ... }
{ order: 2, status: EMPTY_PICKED_UP, ... }
{ order: 3, status: GATE_IN, ... }
```

## 解决方案

### 方案 A：保持现状（推荐）

**理由**：
1. 前端按 stage 分组展示，同组内的节点按时间顺序排列
2. 前端的 order 只用于组间排序，组内顺序由节点时间决定
3. 后端的 order 用于路径生成和验证

**验证**：
- 检查前端实际渲染逻辑：是否真的只依赖 stage 分组？
- 检查节点排序：是否按时间顺序而非 order？

### 方案 B：统一 order

**修改前端 STAGE_MAP**，使其与后端完全一致：
- 需要支持多种运输模式的不同 order
- 实现复杂度高，收益低

## 实际影响评估

### 测试场景 1：纯海运

**后端顺序**：
1. 未出运 (1) -> 2. 提空箱 (2) -> 3. 进港 (3) -> 4. 装船 (4) -> 5. 离港 (5) -> 6. 海运 (6) -> 7. 抵港 (7) -> 8. 卸船 (8) -> 9. 可提 (9) -> 10. 提柜 (10) -> 11. 还箱 (11)

**前端显示**：
- 起运组 (order=1): [未出运，提空箱，进港，装船，离港] - 按时间排序
- 海运组 (order=5): [海运]
- 到港组 (order=6): [抵港，靠泊，卸船，可提] - 按时间排序
- 提柜组 (order=10): [已提柜，拆箱] - 按时间排序
- 还箱组 (order=11): [已还空箱，完成] - 按时间排序

**结论**：前端显示顺序正确 ✅

### 测试场景 2：海铁联运

**后端顺序**：
... -> 6. 海运 (6) -> 7. 抵港 (7) -> 8. 卸船 (8) -> 9. 铁路装车 (9) -> 10. 铁路运输 (10) -> 11. 铁路到达 (11) -> 12. 铁路卸箱 (12) -> 13. 提柜 (13) -> 14. 还箱 (14)

**前端显示**：
- 起运组 (order=1)
- 海运组 (order=5)
- 到港组 (order=6)
- 铁路组 (order=9): [铁路装车，铁路运输，铁路到达，铁路卸箱] - 按时间排序
- 提柜组 (order=10): [已提柜，提柜 (货)] - 按时间排序
- 还箱组 (order=11)

**问题**：铁路组 (order=9) 在到港组 (order=6) 之后显示 ✅ 正确

### 测试场景 3：驳船联运

**后端顺序**：
1. 未出运 (1) -> 2. 提空箱 (2) -> 3. 进港 (3) -> 4. 驳船装船 (4) -> 5. 驳船离港 (5) -> 6. 驳船抵达 (6) -> 7. 驳船卸船 (7) -> 8. 海运装船 (8) -> 9. 海运离港 (9) -> 10. 海运 (10) -> 11. 抵港 (11) -> ...

**前端显示**：
- 起运组 (order=1): [未出运，提空箱，进港]
- 驳船组 (order=4): [驳船装船，驳船离港，驳船抵达，驳船卸船] - 按时间排序
- 海运组 (order=5): [海运装船，海运离港，海运] - 按时间排序
- 到港组 (order=6): [抵港，靠泊，卸船，可提] - 按时间排序
- 提柜组 (order=10)
- 还箱组 (order=11)

**问题**：驳船组 (order=4) 在起运组 (order=1) 之后、海运组 (order=5) 之前显示 ✅ 正确

## 最终结论

**当前实现是正确的！**

前端的设计策略：
1. **stage 分组**：决定节点属于哪个视觉分组
2. **order 排序**：决定分组之间的先后顺序
3. **组内排序**：由节点的实际时间决定，不是 order

因此：
- 前端 STAGE_MAP 的 order 只需要保证**组间顺序正确**
- 同组内的多个状态共享同一个 order
- 运输模式过滤通过 stage 名称控制（rail/feeder）

**无需修改代码**，当前实现符合业务需求。
