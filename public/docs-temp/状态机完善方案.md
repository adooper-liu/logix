# LogiX 状态机完善方案

## 📋 当前问题分析

### 1.1 状态机定义

```
未出运 (not_shipped)
  ↓ 有出运记录 (shipment_date)
在途 (in_transit)
  ↓ 有到港记录 (ata_dest_port)
已到港 (at_port)
  ↓ 有提柜记录 (pickup_date)
已提柜 (picked_up)
  ↓ 有卸柜记录 (unload_date)
已卸柜 (unloaded)
  ↓ 有还箱记录 (return_time)
已还箱 (returned_empty)
```

### 1.2 当前数据库状态分布

| 状态 | 数量 |
|------|------|
| at_port | 92 |
| in_transit | 85 |
| picked_up | 54 |
| returned_empty | 119 |
| 未出运 (not_shipped) | 0 |
| 已卸柜 (unloaded) | 0 |

### 1.3 初步检查结果

✅ **数据一致性良好的情况**：
- `picked_up`状态都有拖卡记录
- `returned_empty`状态都有还箱时间记录
- `picked_up`状态没有卸柜记录
- `at_port`状态没有拖卡记录
- `in_transit`状态没有到港记录

⚠️ **潜在问题**：
- `unloaded`状态数量为0（可能是业务场景较少或状态未更新）
- 数据库中缺少`not_shipped`状态记录
- 需要进一步检查`at_port`状态是否有到港记录

## 🎯 状态机完善方案

### 2.1 状态更新机制

#### 方案A：自动状态更新（推荐）

**触发时机**：
1. 创建/更新海运记录时 → 检查是否需要更新为`in_transit`
2. 创建/更新港口操作记录时 → 检查是否需要更新为`at_port`
3. 创建/更新拖卡运输记录时 → 检查是否需要更新为`picked_up`
4. 创建/更新仓库操作记录时 → 检查是否需要更新为`unloaded`
5. 创建/更新还空箱记录时 → 检查是否需要更新为`returned_empty`

**更新规则**：
```typescript
// 伪代码
function updateContainerStatus(containerNumber: string) {
  const status = determineContainerStatus(containerNumber);
  await containerRepository.update(containerNumber, { logistics_status: status });
}

function determineContainerStatus(containerNumber: string): string {
  // 按优先级从后往前检查
  if (hasReturnTime(containerNumber)) return 'returned_empty';
  if (hasUnloadDate(containerNumber)) return 'unloaded';
  if (hasPickupDate(containerNumber)) return 'picked_up';
  if (hasAtaDate(containerNumber)) return 'at_port';
  if (hasShipmentDate(containerNumber)) return 'in_transit';
  return 'not_shipped';
}
```

**实现位置**：
- 在各服务（SeaFreightService、PortOperationService等）的create/update方法中调用状态更新
- 或创建一个独立的`ContainerStatusService`统一管理状态更新

#### 方案B：定时任务批量更新

**执行频率**：每小时或每天一次

**更新逻辑**：
- 扫描所有货柜
- 根据操作记录计算应该的状态
- 更新状态不匹配的货柜

**优点**：
- 实现简单
- 不影响业务逻辑

**缺点**：
- 状态可能延迟更新
- 不能实时反映状态变化

### 2.2 数据验证机制

#### 验证规则

1. **状态一致性验证**：
   ```sql
   -- 示例：检查at_port状态是否有到港记录
   SELECT container_number
   FROM biz_containers
   WHERE logistics_status = 'at_port'
     AND NOT EXISTS (
       SELECT 1 FROM process_port_operations
       WHERE container_number = biz_containers.container_number
         AND ata_dest_port IS NOT NULL
     )
   ```

2. **日期完整性验证**：
   - 如果状态是`picked_up`，必须有`shipment_date`和`ata_dest_port`
   - 如果状态是`unloaded`，必须有`shipment_date`、`ata_dest_port`、`pickup_date`
   - 如果状态是`returned_empty`，必须有`shipment_date`、`ata_dest_port`、`pickup_date`、`unload_date`、`return_time`

3. **日期顺序验证**：
   ```sql
   -- 检查日期是否按顺序递增
   shipment_date <= ata_dest_port <= pickup_date <= unload_date <= return_time
   ```

#### 验证实现

**方案1：数据库触发器**
- 在各操作表上创建AFTER INSERT/UPDATE触发器
- 触发器中验证数据完整性
- 发现问题时记录到日志表或抛出异常

**方案2：应用层验证**
- 在各服务的create/update方法中添加验证逻辑
- 发现问题时抛出业务异常
- 前端显示错误信息

**方案3：定期数据质量检查**
- 创建定时任务
- 每天执行一次完整的数据一致性检查
- 生成报告并通知相关人员

### 2.3 数据修复机制

#### 修复策略

**策略1：基于最新记录推断状态**
```sql
-- 示例：根据最新操作记录更新状态
UPDATE biz_containers c
SET logistics_status =
  CASE
    WHEN EXISTS (SELECT 1 FROM process_empty_returns WHERE container_number = c.container_number AND return_time IS NOT NULL)
    THEN 'returned_empty'
    WHEN EXISTS (SELECT 1 FROM process_warehouse_operations WHERE container_number = c.container_number AND unload_date IS NOT NULL)
    THEN 'unloaded'
    WHEN EXISTS (SELECT 1 FROM process_trucking_transport WHERE container_number = c.container_number AND pickup_date IS NOT NULL)
    THEN 'picked_up'
    WHEN EXISTS (SELECT 1 FROM process_port_operations WHERE container_number = c.container_number AND port_type = 'destination' AND ata_dest_port IS NOT NULL)
    THEN 'at_port'
    WHEN EXISTS (SELECT 1 FROM process_sea_freight WHERE container_number = c.container_number AND shipment_date IS NOT NULL)
    THEN 'in_transit'
    ELSE 'not_shipped'
  END
WHERE logistics_status != (
  CASE
    WHEN EXISTS (SELECT 1 FROM process_empty_returns WHERE container_number = c.container_number AND return_time IS NOT NULL)
    THEN 'returned_empty'
    WHEN EXISTS (SELECT 1 FROM process_warehouse_operations WHERE container_number = c.container_number AND unload_date IS NOT NULL)
    THEN 'unloaded'
    WHEN EXISTS (SELECT 1 FROM process_trucking_transport WHERE container_number = c.container_number AND pickup_date IS NOT NULL)
    THEN 'picked_up'
    WHEN EXISTS (SELECT 1 FROM process_port_operations WHERE container_number = c.container_number AND port_type = 'destination' AND ata_dest_port IS NOT NULL)
    THEN 'at_port'
    WHEN EXISTS (SELECT 1 FROM process_sea_freight WHERE container_number = c.container_number AND shipment_date IS NOT NULL)
    THEN 'in_transit'
    ELSE 'not_shipped'
  END
);
```

**策略2：人工审核修复**
- 对于复杂的数据不一致问题
- 通过管理界面标记问题记录
- 人工确认后修复

**策略3：数据补全**
- 对于缺失的关键日期数据
- 通过历史日志或外部系统找回
- 无法找回的标记为"数据缺失"

### 2.4 前端展示方案

#### 统计卡片展示

**原则**：
- 统计直接基于`logistics_status`字段
- 不需要在查询中添加额外的约束条件
- 状态维护逻辑确保数据的准确性

**示例**：
```typescript
// 统计各状态货柜数量
SELECT logistics_status, COUNT(*) as count
FROM biz_containers
GROUP BY logistics_status;
```

#### 状态可视化

**桑基图**：
- 按状态流转关系展示
- 每个节点的数量基于`logistics_status`
- 流转关系基于操作记录的日期顺序

**甘特图**：
- 展示每个货柜的时间线
- 关键节点：出运、到港、提柜、卸柜、还箱
- 自动根据操作记录计算当前状态

#### 异常状态标记

**标记规则**：
1. **状态不一致**：状态与实际操作记录不匹配
   - 前端显示警告图标
   - 提示"状态可能不正确，请联系管理员"

2. **数据缺失**：缺少关键日期记录
   - 前端显示信息图标
   - 提示"部分数据缺失，状态可能不准确"

3. **日期异常**：日期顺序不正确
   - 前端显示错误图标
   - 提示"日期顺序异常，请检查数据"

**实现方式**：
```typescript
// 前端接口返回格式
interface ContainerStatus {
  status: string;
  isConsistent: boolean;
  hasMissingData: boolean;
  hasDateAnomaly: boolean;
  warnings: string[];
}

// 后端计算
function calculateContainerStatus(container: Container): ContainerStatus {
  const status = container.logistics_status;
  const warnings: string[] = [];

  // 检查状态一致性
  const expectedStatus = determineExpectedStatus(container);
  const isConsistent = status === expectedStatus;
  if (!isConsistent) {
    warnings.push(`状态不一致：当前状态${status}，预期状态${expectedStatus}`);
  }

  // 检查数据缺失
  const hasMissingData = checkMissingData(container, status);
  if (hasMissingData) {
    warnings.push('缺少关键日期记录');
  }

  // 检查日期异常
  const hasDateAnomaly = checkDateAnomaly(container);
  if (hasDateAnomaly) {
    warnings.push('日期顺序异常');
  }

  return {
    status,
    isConsistent,
    hasMissingData,
    hasDateAnomaly,
    warnings
  };
}
```

### 2.5 状态机监控与告警

#### 监控指标

1. **状态不一致率**：
   - 计算：状态不一致的货柜数 / 总货柜数
   - 告警阈值：> 5%

2. **数据缺失率**：
   - 计算：有关键数据缺失的货柜数 / 总货柜数
   - 告警阈值：> 10%

3. **状态更新延迟**：
   - 计算：操作记录创建时间与状态更新时间的差值
   - 告警阈值：> 1小时

#### 告警机制

**实时告警**：
- 状态更新失败时立即发送告警
- 通过邮件、Slack、企业微信等渠道通知

**定期报告**：
- 每天生成数据质量报告
- 包含状态不一致、数据缺失等问题的详细信息
