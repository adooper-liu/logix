## 二、问题发现与修复

### 2.1 初始问题

**现象**: 前3个货柜的清关节点 `taskRole=main`，应该为 `none`

**根本原因**: 后端代码只检查了 `deliveryDate`，没有检查 `pickupDate`

### 2.2 修复内容

#### 修复1: `computeGanttPhase` 函数（Line 155）

**修改前**:

```typescript
if (trucking?.deliveryDate) return 3;
```

**修改后**:

```typescript
// 已提柜：deliveryDate 或 pickupDate 任一存在即可
if (trucking?.deliveryDate || trucking?.pickupDate) return 3;
```

**影响**:

- 有 `pickupDate` 的货柜现在会被正确识别为阶段3（提柜阶段）
- 而不是阶段1（清关阶段）

#### 修复2: `buildNodes` 函数（Line 191-221）

**修改前**:

```typescript
const hasDeliveryDate = !!trucking?.deliveryDate; // 已提柜
// ...
else if (hasDeliveryDate && key === 'customs') {
  taskRole = 'none';
}
```

**修改后**:

```typescript
const hasDeliveryDate = !!trucking?.deliveryDate; // 已提柜（送达）
const hasPickupDate = !!trucking?.pickupDate; // 已提柜（提货）
// ...
// 已提柜 -> 清关销毁（deliveryDate 或 pickupDate 任一存在即可）
else if ((hasDeliveryDate || hasPickupDate) && key === 'customs') {
  taskRole = 'none';
}
```

**影响**:

- 即使只有 `pickupDate`，清关节点也会被正确设置为 `taskRole='none'`
- 符合业务规则：已提柜意味着清关必然已完成

## 四、前后端对比分析

### 4.1 架构设计

| 维度         | 后端                           | 前端                        |
| ------------ | ------------------------------ | --------------------------- |
| **数据源**   | 流程表实际日期                 | 流程表 + 飞驼API + 本地推断 |
| **计算时机** | 流程表更新时                   | 每次渲染时                  |
| **存储位置** | `biz_containers.gantt_derived` | 内存中（不持久化）          |
| **优势**     | 单一真相源、可缓存             | 实时响应、兼容旧数据        |
| **劣势**     | 需要触发更新                   | 重复计算、性能开销          |

### 4.2 双重保障机制

✅ **后端 gantt-v3**:

- 基于流程表实际日期
- 在数据更新时计算并存储
- 提供稳定的数据源

✅ **前端 calculateNodeStatus**:

- 基于多数据源综合判断
- 在渲染时实时计算
- 即使后端数据不完整，也能正确显示

### 4.3 为什么需要双重机制？

**场景**: 这5个测试货柜

- 物流状态: `picked_up`（来自飞驼API）
- 流程表: 只有 `pickupDate`，没有 `deliveryDate`

如果只依赖后端：

- 修复前：清关节点错误地显示为 `main`
- 修复后：清关节点正确地为 `none`

如果只依赖前端：

- 前端会从飞驼API获取更完整的信息
- 即使流程表数据不完整，也能正确显示

**结论**: 双重机制提供了**容错能力**和**平滑过渡路径**。
