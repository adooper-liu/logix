---

## 🎯 核心原则

**每次只拆分一个域，立即验证，确保零偏差**

这是重构安全的**最后一道防线**，避免"改完发现错了，但不知道错在哪"的灾难。

---

## 📋 完整操作流程

### Step 1: 备份原文件（必须）

**在开始每个 Phase 前，先备份原文件**

```bash
# 方法 1: 文件备份（简单直接）
cp src/components/common/gantt/useGanttLogic.ts \
   src/components/common/gantt/useGanttLogic.ts.backup.phase1

# 方法 2: Git tag 标记（推荐，可追溯）
git tag refactor-before-phase1
git commit -m "backup: before phase 1 refactoring - useGanttLogic.ts"

# 方法 3: Git stash（临时备份）
git stash push -m "backup before phase 1" -- src/components/common/gantt/useGanttLogic.ts
```

**备份命名规范**:
```
{原文件名}.backup.{phase编号}
例如:
- useGanttLogic.ts.backup.phase1
- useGanttLogic.ts.backup.phase2
```

---

### Step 2: 提取单个域（原子性操作）

**✅ 正确做法**: 一次只提取一个域

```typescript
// Phase 1: 只提取纯函数 dateRangeUtils
// 1. 创建新文件
// src/components/common/gantt/dateRangeUtils.ts
export function calculateDateRange(...) { ... }
export function generateDateRange(...) { ... }

// 2. 修改原文件，删除已提取的函数
// src/components/common/gantt/useGanttLogic.ts
import { calculateDateRange, generateDateRange } from './dateRangeUtils'
// 删除原来的函数定义

// 3. 验证导入是否正确
npm run type-check
```

**❌ 错误做法**: 同时提取多个域

```typescript
// ❌ 不要这样做！
// 同时提取 dateRangeUtils + alerts + costOptimization
// 一旦出错，无法定位是哪个域的问题
```

---

### Step 3: 无损核对（Diff 对比）

**目标**: 确保重构前后功能完全等价

#### 方法 1: Git Diff（推荐）

```bash
# 查看变更内容
git diff HEAD -- src/components/common/gantt/useGanttLogic.ts

# 重点关注:
# - 删除的函数是否都已导入
# - 调用点是否正确更新
# - 没有意外修改其他代码
```

#### 方法 2: VS Code 对比工具

```bash
# 打开对比视图
code --diff \
  src/components/common/gantt/useGanttLogic.ts.backup.phase1 \
  src/components/common/gantt/useGanttLogic.ts
```

**检查要点**:
- ✅ 只有预期的变更（删除函数 + 添加 import）
- ✅ 没有意外的空白行、注释变化
- ✅ 函数调用点全部正确更新

#### 方法 3: Checksum 验证（功能等价性）

```bash
# 运行相同的测试用例，对比输出
npm run test -- useGanttLogic

# 如果有集成测试，运行端到端测试
npm run test:e2e -- gantt
```

**验证标准**:
- 输入相同 → 输出必须相同
- 行为一致 → UI 表现、API 响应无变化
- 性能相当 → 无明显性能退化

---

### Step 4: 验证清单（必须全部通过）

在提交前，必须完成以下检查：

```markdown
## Phase X 验证清单

### 自动化检查
- [ ] **类型检查通过**: `npm run type-check` 无新增错误
- [ ] **单元测试通过**: 相关测试全部 green
- [ ] **Lint 通过**: `npm run lint` 无警告
- [ ] **构建成功**: `npm run build` 无错误

### 手动检查
- [ ] **功能等价**: 手动测试核心功能，行为一致
  - [ ] 甘特图渲染正常
  - [ ] 拖拽功能正常
  - [ ] 成本优化计算正确
  - [ ] 预警显示正常
- [ ] **无回归**: 其他模块不受影响
  - [ ] 相关页面加载正常
  - [ ] API 调用正常
- [ ] **性能检查**: 无明显性能退化
  - [ ] 首屏加载时间 < X 秒
  - [ ] 交互响应时间 < X ms

### 代码审查
- [ ] **至少一人 Review 通过**
- [ ] **Diff 合理**: 变更符合预期
- [ ] **注释更新**: 关键逻辑有注释
- [ ] **文档同步**: 架构文档已更新
```

---

### Step 5: 提交或回滚

#### ✅ 验证通过：提交并清理

```bash
# 1. 提交变更
git add .
git commit -m "refactor: extract dateRangeUtils (phase 1)

- 提取日期计算纯函数到 dateRangeUtils.ts
- 减少 useGanttLogic.ts 约 150 行
- TS 错误从 35 降到 28
- 所有测试通过"

# 2. 删除备份文件
rm src/components/common/gantt/useGanttLogic.ts.backup.phase1

# 3. 推送分支
git push origin refactor/useGanttLogic-phase1
```

#### ❌ 验证失败：立即回滚

```bash
# 1. 恢复原文件
git checkout HEAD -- src/components/common/gantt/useGanttLogic.ts

# 或者从备份恢复
cp src/components/common/gantt/useGanttLogic.ts.backup.phase1 \
   src/components/common/gantt/useGanttLogic.ts

# 2. 分析问题
# - 查看 Diff，找出问题所在
# - 检查测试失败原因
# - 修复后重试

# 3. 记录问题
echo "Phase 1 失败原因: [具体问题]" >> REFACTOR_LOG.md
```

**重要**: 不要试图"勉强通过"，有问题就回滚！

## 🛠️ 自动化工具

### 工具 1: 备份脚本 (`scripts/refactor-backup.sh`)

```bash
#!/bin/bash
# 用法: ./scripts/refactor-backup.sh useGanttLogic phase1

set -e

FILE=$1
PHASE=$2
BACKUP="${FILE}.ts.backup.${PHASE}"

if [ -z "$FILE" ] || [ -z "$PHASE" ]; then
  echo "❌ Usage: ./refactor-backup.sh <file> <phase>"
  exit 1
fi

# 检查文件是否存在
if [ ! -f "${FILE}.ts" ]; then
  echo "❌ File not found: ${FILE}.ts"
  exit 1
fi

# 备份
cp "${FILE}.ts" "$BACKUP"
echo "✅ Backup created: $BACKUP"

# 创建 Git tag
git tag "refactor-before-${PHASE}"
echo "✅ Git tag created: refactor-before-${PHASE}"

# 提示下一步
echo ""
echo "📝 Next steps:"
echo "   1. Extract the domain to new file(s)"
echo "   2. Update imports in original file"
echo "   3. Run validation: npm run type-check && npm run test"
echo "   4. If passed, run: ./scripts/refactor-verify.sh $FILE $PHASE"
echo "   5. If failed, run: git checkout HEAD -- ${FILE}.ts"
```

### 工具 2: 验证脚本 (`scripts/refactor-verify.sh`)

```bash
#!/bin/bash
# 用法: ./scripts/refactor-verify.sh useGanttLogic phase1

set -e

FILE=$1
PHASE=$2
BACKUP="${FILE}.ts.backup.${PHASE}"

echo "🔍 Starting verification for $PHASE..."
echo ""

# 1. 类型检查
echo "Step 1/4: Type checking..."
if npm run type-check > /dev/null 2>&1; then
  echo "✅ Type check passed"
else
  echo "❌ Type check failed"
  echo "   Run 'npm run type-check' for details"
  exit 1
fi

# 2. 单元测试
echo "Step 2/4: Running tests..."
if npm run test -- $FILE > /dev/null 2>&1; then
  echo "✅ Tests passed"
else
  echo "❌ Tests failed"
  echo "   Run 'npm run test -- $FILE' for details"
  exit 1
fi

# 3. Lint 检查
echo "Step 3/4: Linting..."
if npm run lint > /dev/null 2>&1; then
  echo "✅ Lint passed"
else
  echo "❌ Lint failed"
  echo "   Run 'npm run lint' for details"
  exit 1
fi

# 4. Diff 对比
echo "Step 4/4: Comparing with backup..."
if [ -f "$BACKUP" ]; then
  DIFF_LINES=$(git diff --no-index "$BACKUP" "${FILE}.ts" | wc -l)
  echo "   Diff lines: $DIFF_LINES"
  if [ $DIFF_LINES -gt 500 ]; then
    echo "⚠️  Warning: Large diff detected ($DIFF_LINES lines)"
    echo "   Please review carefully"
  fi
else
  echo "⚠️  Warning: Backup file not found"
fi

echo ""
echo "✅ All checks passed!"
echo ""
echo "📝 Next steps:"
echo "   1. Review the changes: git diff"
echo "   2. Manual testing: [列出需要手动测试的功能]"
echo "   3. If all good, commit and delete backup:"
echo "      git add ."
echo "      git commit -m \"refactor: $PHASE\""
echo "      rm $BACKUP"
```

## ⚠️ 常见陷阱与对策

### 陷阱 1: 忘记备份

**症状**: 改完后发现错了，但没有备份可回滚

**对策**: 
- 将备份脚本加入 `.husky/pre-commit` hook
- 或使用 Git alias: `git config alias.backup '!cp $1 $1.backup'`

### 陷阱 2: 一次性提取太多

**症状**: 同时提取 3-4 个域，出错了不知道是哪个域的问题

**对策**: 
- 严格遵守"一次一个域"原则
- 使用 TODO 列表跟踪进度

### 陷阱 3: 验证不充分

**症状**: 类型检查通过，但运行时出错

**对策**: 
- 必须包含手动测试
- 建立回归测试套件

### 陷阱 4: 备份文件堆积

**症状**: 目录下全是 `.backup.*` 文件

**对策**: 
- 验证通过后立即删除备份
- 使用 `.gitignore` 忽略备份文件（但不推荐，因为备份很重要）
