## 🎯 核心价值

### 1. 唯一事实标准

**之前的问题**:
- 规范分散在多个文档
- 部分内容重复
- 缺少统一入口

**现在的解决方案**:
- 建立质量门禁 SKILL 作为唯一入口
- 自动触发规则确保一致性
- 所有检查项基于权威源验证

---

### 2. 零学习成本

**之前的问题**:
- 需要记住多个 Skill 名称
- 不知道何时调用哪个 Skill
- 容易遗漏关键检查

**现在的解决方案**:
- 自然语言触发，AI 自动识别
- 12 条自动触发规则覆盖所有场景
- 违规处罚机制确保执行

---

### 3. 防止技术债

**之前的问题**:
- 临时补丁修数据
- 统计与列表不一致
- 缺少自动化检查

**现在的解决方案**:
- 合并前强制质量门禁
- 自动化检查脚本
- 技术债登记与追踪

## 🛠️ 使用指南

### 开发者日常流程

```bash
# 1. 开发前
git pull origin develop

# 2. 开发中（自然语言提问）
用户: "这个字段名对吗？"
AI: [自动验证] → 输出验证报告

# 3. 提交前（根目录执行）
npm run quality              # 完整质量检查
# 或快速检查
npm run validate             # 类型检查 + Lint

# 4. 提交代码
git add .
git commit -m "feat: 添加新功能"

# 5. 推送并创建 PR
git push origin feature/xxx
```

### Code Review 流程

```markdown
1. 作者提交 PR
   ↓
2. CI 自动运行质量门禁
   ↓
3. Reviewer 检查:
   - [ ] 代码是否符合规范？
   - [ ] 测试是否充分？
   - [ ] 是否有技术债？
   ↓
4. 通过后合并到 develop
   ↓
5. 定期合并到 main
```

## ⚠️ 注意事项

### 1. 避免过度检查

**原则**: 只检查能防止代码开发出错的项目

**示例**:
- ✅ 检查：数据库字段是否存在
- ❌ 不检查：变量命名风格偏好

---

### 2. 保持文档同步

**原则**: 代码变更后，相关文档必须同步更新

**检查清单**:
- [ ] SKILL 文档是否反映最新代码？
- [ ] 自动触发规则是否需要调整？
- [ ] 示例代码是否可运行？

---

### 3. 定期验证权威性

**原则**: 所有检查项必须基于权威源

**权威源列表**:
- 数据库表结构：`backend/sql/schema/03_create_tables.sql`
- TypeORM 实体：`backend/src/entities/*.ts`
- API 路由：`backend/src/routes/index.ts`
- 前端服务：`frontend/src/services/*.ts`
