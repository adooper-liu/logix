# 状态机更新数据库状态问题分析

## 问题描述

数据库中产生了两个表:
- `process_empty_return` (无数据)
- `process_empty_returns` (有数据)

这导致状态机更新逻辑可能存在问题。

## 状态机完整更新逻辑

### 1. 状态判断优先级

```typescript
determineStatus(containerNumber: string): Promise<string> {
  const dates = await this.getDateInfo(containerNumber);

  // 1. 已还箱 (最高优先级)
  if (dates.returnTime) return 'returned_empty';

  // 2. 已卸柜 (WMS已确认)
  if (this.isWmsConfirmed(dates)) return 'unloaded';

  // 3. 已提柜
  if (dates.pickupDate) return 'picked_up';

  // 4. 已到港 (包括中转港和目的港)
  if (dates.hasTransit || dates.hasDestinationAta) return 'at_port';

  // 5. 在途
  if (dates.shipmentDate) return 'in_transit';

  // 6. 未出运
  return 'not_shipped';
}
```

### 2. 状态判断条件

| 状态 | 表名 | 关键字段 | 判断条件 |
|------|------|----------|----------|
| not_shipped | - | - | 初始状态,无其他操作记录 |
| in_transit | process_sea_freight | shipment_date | 有出运记录 |
| at_port | process_port_operations | ata_dest_port | 已到中转港或目的港 |
| picked_up | process_trucking_transport | pickup_date | 有提柜记录 |
| unloaded | process_warehouse_operations | wms_status, ebs_status, wms_confirm_date | WMS已确认 |
| returned_empty | process_empty_returns | return_time | 有还箱记录 |

### 3. 关键SQL查询 (必须修正)

**错误的SQL** (backend/src/services/containerStatus.service.ts):
```typescript
LEFT JOIN process_empty_return er ON c.container_number = er.container_number
```

**正确的SQL**:
```typescript
LEFT JOIN process_empty_returns er ON c.container_number = er.container_number
```

## 数据一致性问题

### 问题1: JOIN表名错误导致查询失败

当 `process_empty_return` 表不存在或无数据时:
- `getDateInfo` 方法无法获取还箱时间
- 即使 `process_empty_returns` 表有 `return_time`,状态机也无法判断为 `returned_empty`
- 导致状态停留在 `unloaded` 或 `picked_up`

### 问题2: 状态不一致示例

假设货柜 `FANU3376528`:
- `process_empty_returns.return_time` = '2026-03-01'
- 由于JOIN到错误的表,`dates.returnTime` = `undefined`
- 状态机判断为 `unloaded` 而不是 `returned_empty`

## 总结

### 核心问题

表名不统一导致状态机无法正确判断状态:
- 正确表名: `process_empty_returns` (带s)
- 错误表名: `process_empty_return` (不带s)

### 修复要点

1. **所有SQL使用 `process_empty_returns`**
2. **TypeORM Entity使用正确的表名**
3. **删除或重命名错误的表**
4. **验证状态机逻辑正确性**

### 预防措施

1. **代码审查**: 确保所有表名与Entity定义一致
2. **自动化测试**: 添加表名一致性检查
3. **命名规范**: 统一使用复数形式的表名
