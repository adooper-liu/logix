```typescript
if (field === "plannedUnloadDate") {
  const oldUnload = warehouse?.plannedUnloadDate ? dayjs(warehouse.plannedUnloadDate).format("YYYY-MM-DD") : null;
  const pickupStr = trucking?.plannedPickupDate ? dayjs(trucking.plannedPickupDate).format("YYYY-MM-DD") : null;

  if (!isDropOffMode) {
    // Live load: 三日期强制同步
    updateData["plannedPickupDate"] = newDate;
    updateData["plannedDeliveryDate"] = newDate;
    updateData["plannedUnloadDate"] = newDate;
  } else {
    // Drop off: 仅同步送柜日和卸柜日
    updateData["plannedDeliveryDate"] = newDate;
    updateData["plannedUnloadDate"] = newDate;

    // 如果前移且早于提柜日，标记需要询问
    const forward = oldUnload && dayjs(newDate).isBefore(dayjs(oldUnload), "day");
    if (forward && pickupStr && dayjs(newDate).isBefore(dayjs(pickupStr), "day")) {
      unloadForwardNeedsPickupConfirm = true;
    }
  }
}
```

#### 3. 还箱日自动重算

**实现位置**：

- `mergeReturnDateIntoUpdateData` L29-44
- `mergeReturnDateWhenPickupOnlyForward` L50-78
- `computePlannedReturnFromUnload` L10-26

**规则**：

- **Live load**: 还箱日 = 卸柜日
- **Drop off**:
  - 如有现有还箱日：还箱日 = max(卸柜日, 现有还箱日)
  - 如无现有还箱日：还箱日 = 卸柜日 + 1天

**调用时机**：

1. `handleDrop` 中（L1244-1245）：准备更新数据时
2. `executeUpdateWithData` 中（L1049）：发送请求前再次确保
3. `handleDragEnd` 二次确认后（L999）：用户选择同步时

## 二、数据流完整性检查

### ✅ 完整的数据流

```
用户拖拽节点
    ↓
handleDragStart (记录 draggingContainer, draggingNodeName)
    ↓
handleDragOver (更新 dragOverDate, dropIndicatorPosition)
    ↓
handleDrop (核心逻辑)
    ├─ 获取卸柜模式 (unloadMode)
    ├─ 判断拖拽方向 (isForwardDrag)
    ├─ 根据策略矩阵构建 updateData
    ├─ 调用 mergeReturnDateIntoUpdateData 计算还箱日
    ├─ 生成 confirmMsg
    └─ 设置 pendingDropConfirm
    ↓
handleDragEnd
    ├─ 清理拖拽状态
    ├─ 读取 pendingDropConfirm
    ├─ 判断是否需要二次确认
    │   ├─ Drop off + 向前拖拽提柜日 → 弹窗询问
    │   └─ 其他情况 → 直接显示确认对话框
    ├─ 用户确认后
    │   ├─ 如需要同步 → 添加 plannedDeliveryDate/plannedUnloadDate
    │   └─ 调用 executeUpdateWithData
    └─ 执行更新
    ↓
executeUpdateWithData
    ├─ 再次调用 mergeReturnDateIntoUpdateData (双重保险)
    ├─ 调用 containerService.updateSchedule
    ├─ 成功后刷新数据
    └─ 失败后显示错误信息
```

### ✅ 关键字段传递

| 字段            | handleDrop | pendingDropConfirm | handleDragEnd | executeUpdateWithData       | 后端 API             |
| --------------- | ---------- | ------------------ | ------------- | --------------------------- | -------------------- |
| container       | ✅         | ✅                 | ✅            | ✅                          | ✅ (containerNumber) |
| newDate         | ✅         | ✅                 | ✅            | ❌ (使用 updateData)        | ✅                   |
| updateField     | ✅         | ✅                 | ✅            | ❌ (使用 updateData)        | ✅                   |
| extraUpdateData | ✅         | ✅                 | ✅            | ✅ (合并到 finalUpdateData) | ✅                   |
| unloadMode      | ✅         | ✅                 | ✅            | ✅ (重新获取)               | ✅ (unloadModePlan)  |
| isDropOffMode   | ✅         | ✅                 | ✅            | ❌ (重新计算)               | ❌                   |

## 四、与后端交互检查

### ✅ 正确的部分

1. **字段名称**：前端使用 camelCase（plannedPickupDate），后端 controller 也期望 camelCase
2. **日期格式**：前端发送 'YYYY-MM-DD' 字符串，后端接收后转换为 Date 对象
3. **多字段更新**：`updateSchedule` 接口支持同时更新多个字段

### ⚠️ 需要注意的部分

1. **后端校验**：
   - 位置：`backend/src/controllers/container.controller.ts` L1199-1233
   - 校验规则：
     - 计划清关日 ≤ 计划提柜日
     - 计划提柜日 ≤ 计划送仓日
     - 计划送仓日 ≤ 计划卸柜日
     - 计划卸柜日 ≤ 计划还箱日

   **风险**：如果前端发送的数据违反这些规则，后端会返回 400 错误。

2. **映射关系校验**：
   - 位置：`backend/src/controllers/container.controller.ts` L1169-1197
   - 校验规则：
     - 车队必须支持该港口
     - 仓库必须支持该车队

   **风险**：如果只更新日期而不更新车队/仓库，可能触发校验失败。
