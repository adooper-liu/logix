# 统计和表格过滤不一致问题全面修复

## 问题描述

用户反馈：统计卡片显示"紧急：倒计时 3 天内"有 **3 个货柜**，但点击后表格过滤出 **5 条记录**，其中 **2 条费利克斯托（英国）** 的货柜不应该显示。

## 排查范围

全面检查了所有统计服务的 6 个子服务：

1. ✅ LastReturnStatisticsService - 最晚还箱统计
2. ✅ LastPickupStatisticsService - 最晚提柜统计  
3. ✅ PlannedPickupStatisticsService - 提柜计划统计
4. ⚠️ EtaStatisticsService - 预计到港统计
5. ⚠️ ArrivalStatisticsService - 实际到港统计
6. ✅ StatusDistributionService - 状态分布统计

## 问题根因

统计方法和表格过滤方法使用了**不同的日期范围过滤逻辑**，导致国家过滤条件未正确应用到表格过滤中。

### 两种实现模式对比

#### 模式一：子查询模式（正确）✅

```typescript
applyDateFilterToQuery(query, this.containerRepository, startDate, endDate, extraParams);
```

- 使用 `IN (子查询)` 实现
- 子查询自动包含国家过滤
- 统计和表格过滤逻辑一致

#### 模式二：直接字段比较（错误）❌

```typescript
ContainerQueryBuilder.addDateFilters(query, startDate, endDate);
ContainerQueryBuilder.addCountryFilters(query);
```

- 直接对 `order.actualShipDate` 等字段做比较
- 虽然调用了 `addCountryFilters`，但与统计方法的子查询模式不同
- 可能导致过滤逻辑不一致

### 代码对比

**统计方法** (`getReturnUrgentCount`):
```typescript
async getReturnUrgentCount(today: Date, threeDaysLater: Date, startDate?: string, endDate?: string): Promise<number> {
  const query = ContainerQueryBuilder.createBaseQuery(this.containerRepository);
  ContainerQueryBuilder.addCountryFilters(query);  // ✅ 国家过滤

  query.andWhere('er.last_return_date IS NOT NULL');
  query.andWhere('er.last_return_date >= :today', { today });
  query.andWhere('er.last_return_date <= :threeDays', { threeDays: threeDaysLater });
  
  // ✅ 使用 applyDateFilterToQuery -> IN (子查询) -> 自动包含国家过滤
  applyDateFilterToQuery(query, this.containerRepository, startDate, endDate, {
    today,
    threeDays: threeDaysLater
  });
  
  const result = await query.select('COUNT(DISTINCT container.containerNumber)', 'count').getRawOne();
  return parseInt(result?.count || '0');
}
```

**表格过滤方法** (`getContainersByReturnUrgent` - 修复前):

### 关键差异

| 方法 | 日期过滤函数 | 实现方式 | 国家过滤 |
|------|------------|---------|---------|
| 统计方法 | `applyDateFilterToQuery` | `IN (子查询)` | ✅ 子查询中包含 |
| 表格过滤（修复前） | `addDateFilters` | 直接字段比较 | ❌ 缺失 |

**为什么会有这个差异？**

- `applyDateFilterToQuery` 内部会调用 `createDateRangeSubQuery`，该子查询会关联 `biz_customers` 表并根据国家代码过滤
- `addDateFilters` 只是简单地对 `order.actualShipDate` 等字段做日期比较，**不包含国家过滤逻辑**

## 技术要点

### applyDateFilterToQuery 的工作原理

```typescript
// DateRangeSubquery.ts
export function applyDateFilterToQuery(
  query: SelectQueryBuilder<any>,
  containerRepository: Repository<Container>,
  startDate?: string,
  endDate?: string,
  extraParams?: Record<string, unknown>,
  countryCode?: string
): void {
  if (startDate && endDate) {
    // ✅ 创建子查询，自动包含国家过滤
    const subQuery = createDateRangeSubQuery(containerRepository, startDate, endDate, countryCode);
    query.andWhere(`container.containerNumber IN (${subQuery.getQuery()})`)
         .setParameters({
           ...subQuery.getParameters(),
           ...(extraParams || {})
         });
  } else {
    // 无日期范围时使用基础过滤
    ContainerQueryBuilder.addDateFilters(query, startDate, endDate);
    DateFilterBuilder.addCountryFilters(query, countryCode);
  }
}
```

### createDateRangeSubQuery 的国家过滤逻辑

```typescript
// DateRangeSubquery.ts
export function createDateRangeSubQuery(
  containerRepository: Repository<Container>,
  startDate: string,
  endDate: string,
  countryCode?: string
): SelectQueryBuilder<Container> {
  const qb = containerRepository
    .createQueryBuilder('c')
    .select('DISTINCT c.containerNumber')
    .leftJoin('c.replenishmentOrders', 'o')
    .leftJoin('c.seaFreight', 'sf')
    .where('COALESCE(o.actualShipDate, o.expectedShipDate, sf.shipmentDate) >= :startDate', { startDate })
    .andWhere('COALESCE(o.actualShipDate, o.expectedShipDate, sf.shipmentDate) <= :endDate', { endDate });
  
  // ✅ 关键：如果有国家代码，关联 biz_customers 并过滤
  const code = normalizeCountryCode(countryCode || getScopedCountryCode());
  if (code) {
    qb.leftJoin('biz_customers', 'cust', TYPEORM_O_CUSTOMER_JOIN_ON)
      .andWhere('cust.country = :countryCode', { countryCode: code });
  }
  
  return qb;
}
```

## 经验教训

### 问题本质

这是典型的**代码重复导致的逻辑不一致**问题：

1. **统计模块**和**过滤模块**都实现了日期范围过滤功能
2. 部分服务（LastPickup/PlannedPickup）使用 SQL 模板模式，统计和查询共用同一套 SQL
3. 部分服务（LastReturn/Eta/Arrival）使用了不同的实现：
   - 统计方法使用 `applyDateFilterToQuery`（包含国家过滤的子查询）
   - 表格过滤方法使用 `addDateFilters`（只有日期比较）
4. 开发者误以为两者等价，实际上后者缺少完整的国家过滤逻辑

### 最佳实践

1. **统一工具函数**：日期范围过滤应该只有一个实现，避免多处维护
2. **组合优于复制**：如果多个地方需要相同逻辑，应该复用而非复制
3. **测试覆盖**：应该有集成测试验证统计数量和表格过滤结果的一致性
4. **代码审查重点**：涉及数据过滤的逻辑，必须审查是否包含所有必要的过滤条件（国家、日期、状态等）

### 后续改进建议

1. **重构 `addDateFilters`**：考虑删除此函数，统一使用 `applyDateFilterToQuery`
2. **添加集成测试**：验证每个统计卡片点击后，表格过滤结果与统计数量一致
3. **日志增强**：在关键过滤点添加日志，便于排查问题
4. **代码注释**：在 `applyDateFilterToQuery` 处添加 JSDoc 说明其重要性
