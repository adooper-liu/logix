## 一、统计维度概览

### 按到港维度统计结构

```
按到港总计 (全部货柜)
├── ① 实际已到目的港 (有ATA)
│   ├── 今日到港
│   └── 今日之前到港
│       ├── 今日之前到港未提柜
│       └── 今日之前到港已提柜
└── ② 未到目的港 (无ATA)
    ├── 已逾期未到港
    ├── 3日内预计到港
    ├── 7日内预计到港
    ├── 7日后预计到港
    └── 其他记录
```

### 数据来源

- **主表**: `containers` (货柜表)
- **港口操作表**: `process_port_operations` (海运港口操作记录)
- **关联表**: `replenishment_orders` (备货单), `sea_freight` (海运信息)

### 核心规则

1. **只统计目的港**: `port_type = 'destination'`，排除中转港 `port_type = 'transit'`
2. **只统计主要目的港**: 对于多港经停的货柜，只统计 `port_sequence` 最大的 destination 记录
3. **数据一致性**: 所有子查询都使用 `port_sequence` 过滤，不使用 `MAX(ata_dest_port) GROUP BY`

## 二、统计项详细说明

### 1. 按到港总计 (全部货柜)

| 属性 | 值 |
|-----|---|
| **统计项名称** | 按到港总计 / 合计 |
| **对应方法** | `getArrivalDistribution()` - totalTarget |
| **取数范围** | 所有已出运的货柜 |
| **包含状态** | shipped, in_transit, at_port, picked_up, unloaded, returned_empty |
| **时间条件** | 无时间限制，包含所有货柜 |
| **港口条件** | 无港口条件限制 |
| **数据验证** | `按到港总计 = 各子项之和` |

**SQL 逻辑**:
```sql
SELECT COUNT(DISTINCT container_number)
FROM containers
WHERE logistics_status IN ('shipped', 'in_transit', 'at_port', 'picked_up', 'unloaded', 'returned_empty')
```

---

### 2. 今日到港

| 属性 | 值 |
|-----|---|
| **统计项名称** | 今日到港 |
| **对应方法** | `getArrivedToday()` |
| **取数范围** | 今天实际到港的货柜 |
| **包含状态** | shipped, in_transit, at_port, picked_up, unloaded, returned_empty |
| **时间条件** | `ATA = 今天` |
| **港口条件** | `port_type = 'destination'` 且为主要目的港 (`port_sequence` 最大) |

**SQL 逻辑**:
```sql
SELECT COUNT(DISTINCT c.container_number)
FROM containers c
INNER JOIN (
  SELECT po1.container_number, po1.ata_dest_port
  FROM process_port_operations po1
  WHERE po1.port_type = 'destination'
    AND po1.ata_dest_port IS NOT NULL
    AND po1.port_sequence = (
      SELECT MAX(po2.port_sequence)
      FROM process_port_operations po2
      WHERE po2.container_number = po1.container_number
        AND po2.port_type = 'destination'
    )
) latest_po ON latest_po.container_number = c.container_number
WHERE DATE(latest_po.ata_dest_port) = :today
  AND c.logistics_status IN ('shipped', 'in_transit', 'at_port', 'picked_up', 'unloaded', 'returned_empty')
```

**业务含义**: 今天刚到目的港的货柜数量

### 5. 已逾期未到港

| 属性 | 值 |
|-----|---|
| **统计项名称** | 已逾期未到港 |
| **对应方法** | `getOverdueNotArrived()` |
| **取数范围** | 预计到港时间已过期但还未实际到港的货柜 |
| **包含状态** | shipped, in_transit, at_port (仅这3个状态) |
| **时间条件** | `ETA < 今天` 或 `eta_correction < 今天` |
| **港口条件** | `port_type = 'destination'` 且为主要目的港，`ATA IS NULL` |
| **状态限制** | **重要**: 只包含 shipped/in_transit/at_port 状态 |

**SQL 逻辑**:
```sql
SELECT COUNT(DISTINCT c.container_number)
FROM containers c
INNER JOIN (
  SELECT po1.container_number
  FROM process_port_operations po1
  WHERE po1.port_type = 'destination'
    AND po1.ata_dest_port IS NULL
    AND (po1.eta_dest_port IS NOT NULL OR po1.eta_correction IS NOT NULL)
    AND (po1.eta_dest_port < :today OR po1.eta_correction < :today)
    AND po1.port_sequence = (
      SELECT MAX(po2.port_sequence)
      FROM process_port_operations po2
      WHERE po2.container_number = po1.container_number
        AND po2.port_type = 'destination'
    )
) dest_po ON dest_po.container_number = c.container_number
WHERE c.logistics_status IN ('shipped', 'in_transit', 'at_port')
```

**业务含义**: 应该已经到港但实际未到港，可能存在延误风险

|-----|---|
| **统计项名称** | 3日内预计到港 |

| **取数范围** | 预计3天内到港的货柜（含今天） |
| **包含状态** | shipped, in_transit, at_port (仅这3个状态) |

| **港口条件** | `port_type = 'destination'` 且为主要目的港，`ATA IS NULL` |
| **状态限制** | **重要**: 只包含 shipped/in_transit/at_port 状态 |

**SQL 逻辑**:
```sql
SELECT COUNT(DISTINCT c.container_number)
FROM containers c
INNER JOIN (
  SELECT po1.container_number
  FROM process_port_operations po1
  WHERE po1.port_type = 'destination'
    AND po1.ata_dest_port IS NULL

    AND po1.eta_dest_port >= :today
    AND po1.eta_dest_port <= :threeDaysLater

      SELECT MAX(po2.port_sequence)

      WHERE po2.container_number = po1.container_number

WHERE c.logistics_status IN ('shipped', 'in_transit', 'at_port')
```

**业务含义**: 近期即将到港的货柜，需要提前做好接柜准备

## 三、统计规则与过滤条件

### 1. 核心规则

| 规则名称 | 规则说明 | 适用范围 |
|---------|---------|---------|
| **目的港过滤** | 只统计 `port_type = 'destination'` 的记录，排除中转港 `port_type = 'transit'` | 所有统计项 |
| **主要目的港过滤** | 只统计 `port_sequence` 最大的 destination 记录 | 所有统计项 |
| **状态一致性** | ATA分组包含所有状态，ETA分组只包含 shipped/in_transit/at_port | 所有统计项 |

### 2. 多港经停处理

当一个货柜有多条 destination 记录时：
- **使用 `port_sequence` 过滤**: 选择 `port_sequence` 最大的记录作为主要目的港
- **不使用 `MAX(ata_dest_port)` GROUP BY**: 避免获取到非主要目的港的数据

**正确做法**:
```sql
AND po1.port_sequence = (
  SELECT MAX(po2.port_sequence)
  FROM process_port_operations po2
  WHERE po2.container_number = po1.container_number
    AND po2.port_type = 'destination'
)
```

### 3. 状态过滤说明

| 维度 | 包含的状态 | 说明 |
|-----|-----------|------|
| **实际已到港（ATA）** | shipped, in_transit, at_port, picked_up, unloaded, returned_empty | 所有已出运状态 |
| **未到港（ETA）** | shipped, in_transit, at_port | 仅包含未到港之后状态 |
| **未提柜** | shipped, in_transit, at_port | 排除 picked_up, unloaded, returned_empty |
| **已提柜** | picked_up, unloaded, returned_empty | 仅包含已提柜状态 |

### 4. 时间条件说明

| 时间字段 | 用途 | 说明 |
|---------|-----|------|
| **ATA** (ata_dest_port) | 实际到港时间 | 用于"实际已到港"维度 |
| **ETA** (eta_dest_port) | 预计到港时间 | 用于"未到港"维度 |
| **ETA修正** (eta_correction) | 预计到港修正时间 | 用于逾期判断，优先级高于ETA |

## 四、数据验证规则

### 1. 汇总关系验证

```
按到港总计 = 今日到港 + 今日之前到港未提柜 + 今日之前到港已提柜 + 已逾期未到港 + 3日内预计到港 + 7日内预计到港 + 7日后预计到港 + 其他记录
```

**验证方式**:
```typescript
const sum = arrivedTodayResult + arrivedBeforeTodayNotPickedUpResult + arrivedBeforeTodayPickedUpResult + overdueNotArrivedResult + within3DaysResult + within7DaysResult + over7DaysResult + otherRecordsResult;
console.log('[getArrivalDistribution]', {
  targetTotal: parseInt(totalTarget.count),
  calculatedSum: sum,
  diff: parseInt(totalTarget.count) - sum
});
```

### 2. 互斥性验证

- **实际已到港** vs **未到港**: 基于 `ATA IS NOT NULL` vs `ATA IS NULL` 互斥
- **今日之前到港未提柜** vs **今日之前到港已提柜**: 基于 `logistics_status` 互斥
- **未到港各子项**: 基于 `ETA` 时间范围互斥

### 3. 覆盖性验证

所有 `logistics_status IN ('shipped', 'in_transit', 'at_port', 'picked_up', 'unloaded', 'returned_empty')` 的货柜都必须属于某个分类

## 七、常见问题

### Q1: 为什么 ETA 分组只包含 shipped/in_transit/at_port 状态？

**A**: 因为 picked_up/unloaded/returned_empty 状态的货柜已经提柜，不需要用 ETA 预测到港时间。这些货柜应该归类到"今日之前到港已提柜"分类中。

### Q2: 如何处理多港经停的货柜？

**A**: 只统计主要目的港（port_sequence 最大的 destination 记录），不统计中转港和非主要目的港。

### Q3: 为什么使用 port_sequence 而不是 MAX(ata_dest_port)？

**A**:
- `port_sequence` 明确标识了港口访问顺序，语义更清晰
- `MAX(ata_dest_port)` 可能获取到非主要目的港的 ATA
- 符合业务逻辑：只统计最终目的港的状态

### Q4: 统计结果与实际不符怎么办？

**A**:
1. 检查是否有多港经停的货柜
2. 检查 `port_sequence` 字段是否正确设置
3. 检查 `port_type` 是否正确标识目的港/中转港
4. 检查 `logistics_status` 是否正确
5. 检查 `ATA` 和 `ETA` 日期字段是否完整

### Q5: 如何验证统计结果的正确性？

**A**:
1. 检查汇总关系：各子项之和应等于总计
2. 检查互斥性：每个货柜只能属于一个分类
3. 检查覆盖性：所有目标状态的货柜都应被统计
4. 对比前端显示的数据和数据库查询结果
