# TimescaleDB 超表兼容性分析

## 核心结论

**LogiX 项目中所有原本计划使用 TimescaleDB 超表的表,都不适合设为超表。**

原因:物流业务中时间字段的可空性是正常业务状态,不能使用默认值填充。

---

## 冲突表详细分析

### 1. container_status_events (集装箱状态事件表)

#### 原计划
- **分区键**: `occurred_at` (状态发生时间)
- **用途**: 存储货柜在各个节点的状态事件(进港、卸船、提柜等)

#### 问题分析

**数据库定义**:
```sql
occurred_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
```

**业务场景**:
1. **Excel 导入历史数据**: 很多历史事件的准确时间未知或记录不完整
2. **飞驼 API 同步**: 某些状态码可能没有具体时间戳
3. **手动录入**: 用户可能只知道事件发生了,但不知道确切时间

**为什么不能用默认值**:
- 需要区分"确切知道的时间"(如:2024-01-15 10:30:00)和"未知时间(用当前时间占位)"
- 如果用 `DEFAULT CURRENT_TIMESTAMP`,会丢失语义信息
- 统计查询时无法区分真实时间和占位时间

**示例场景**:
```
货柜 HMMU6232153:
- 2024-01-10: 离开起运港 (已知时间)
- ????: 到达中转港 (时间未知,Excel中为空)
- ????: 离开中转港 (时间未知,Excel中为空)
- 2024-01-25: 到达目的港 (已知时间)

如果设置 DEFAULT CURRENT_TIMESTAMP:
- 导入时(2024-02-01),两个未知时间会被填充为 2024-02-01
- 这会导致时间顺序混乱,统计数据错误
```

#### 结论
❌ **不适合作为超表**

**替代方案**:
- 保持普通表
- 使用复合索引优化查询性能(已实现)
- 应用层处理时间字段的NULL值逻辑

---

### 2. process_port_operations (港口操作表)

#### 原计划
- **分区键**: `gate_in_time` (进港时间)
- **用途**: 记录货柜在各个港口的操作信息(ETA/ATA/ETD/ATD等)

#### 问题分析

这是一个**多阶段物流过程表**,包含大量时间字段,所有字段都可能为空:

| 字段类别 | 字段名 | 可空 | 业务说明 | 为何不能设默认值 |
|---------|--------|------|----------|----------------|
| **预计时间** | eta | ✅ | 预计到达时间 | 船期未确定时为空 |
| | etd | ✅ | 预计离开时间(中转) | 同上 |
| | revised_eta | ✅ | 修正ETA | 无修正时为空 |
| **实际时间** | ata | ✅ | 实际到达时间 | 未到港时为空(正常状态) |
| | atd | ✅ | 实际离开时间 | 未离开时为空(正常状态) |
| | discharged_time | ✅ | 卸船时间 | 未卸船时为空 |
| **港口操作** | gate_in_time | ✅ | 进港时间 | 未进港时为空(正常状态) |
| | gate_out_time | ✅ | 出港时间 | 未出港时为空 |
| | available_time | ✅ | 可提箱时间 | 未就绪时为空 |
| **清关相关** | planned_customs_date | ✅ | 计划清关日期 | 未计划时为空 |
| | actual_customs_date | ✅ | 实际清关日期 | 未清关时为空 |
| | isf_declaration_date | ✅ | ISF申报日期 | 未申报时为空 |
| **滞留/放行** | customs_hold_date | ✅ | 海关滞留日期 | 未滞留时为空 |
| | customs_release_date | ✅ | 海关放行日期 | 未放行时为空 |
| | carrier_hold_date | ✅ | 船公司滞留日期 | 未滞留时为空 |
| **特殊运输** | train_arrival_date | ✅ | 火车到站日期 | 非海铁联运时为空 |
| | train_departure_time | ✅ | 火车出发时间 | 同上 |
| **免费期** | last_free_date | ✅ | 最后免费日 | 未计算时为空 |
| | rail_last_free_date | ✅ | 铁路最后免费日 | 非铁路运输时为空 |

**关键问题**:
1. **时间字段逐步填充**: 货柜从起运到还箱,时间字段是随着物流进程逐步填充的
2. **空值是正常状态**: 很多时间字段在某个阶段为空是完全正常的
3. **无法找到合适的分区键**:
   - `gate_in_time`: 未进港时为空
   - `eta`: 预计时间可能多次变更
   - `ata`: 未到港时为空
   - 任何时间字段都可能为空!

**示例场景**:
```
货柜 ABCU1234567 的物流过程:

第1天: 创建记录
  - eta: 2024-02-15 (预计)
  - 其他时间字段: 全部为空 (正常!)

第5天: 进港
  - gate_in_time: 2024-02-10 08:00
  - ata: 仍为空 (还未到港)

第15天: 到达目的港
  - ata: 2024-02-15 14:30
  - gate_out_time: 仍为空 (还未提柜)

第20天: 提柜
  - gate_out_time: 2024-02-20 10:00

如果使用超表:
- 第1天创建记录时,gate_in_time 为空 → 违反超表约束!
- 即使设置 DEFAULT,也会造成时间顺序混乱
```

#### 结论
❌ **完全不适合作为超表**

**替代方案**:
- 保持普通表
- 使用常规索引优化常用查询
- 应用层根据业务阶段动态更新各个时间字段

## TimescaleDB 高级功能依赖关系

### 功能清单及依赖

| 功能 | 是否需要超表 | 当前状态 | 影响 |
|------|------------|---------|------|
| **超表(Hypertable)** | ✅ 必需 | ❌ 已禁用 | 自动分区、时间范围查询优化 |
| **压缩(Compression)** | ✅ 必需 | ❌ 已禁用 | 节省存储空间(通常90%+) |
| **保留策略(Retention)** | ✅ 必需 | ❌ 已禁用 | 自动删除过期数据 |
| **连续聚合(Continuous Aggregate)** | ✅ 必需 | ❌ 已禁用 | 预聚合统计,提升查询性能 |

### 当前禁用的功能

#### 1. 数据压缩
```sql
-- 原本计划:压缩30天前的状态事件,90天前的港口操作
-- 效果:节省约90%存储空间
ALTER TABLE container_status_events SET (timescaledb.compress);
SELECT add_compression_policy('container_status_events', INTERVAL '30 days');
```

**影响**:
- 数据存储成本增加
- 但对于中小规模系统(几十万条记录),影响不大

#### 2. 数据保留策略
```sql
-- 原本计划:自动删除2年前的状态事件,3年前的港口操作
SELECT add_retention_policy('container_status_events', INTERVAL '2 years');
```

**影响**:
- 需要手动清理历史数据
- 建议:定期执行归档脚本,将老数据移到归档表

#### 3. 连续聚合视图
```sql
-- 原本计划:预聚合每日统计,每小时/每天自动刷新
CREATE MATERIALIZED VIEW container_status_daily_stats
WITH (timescaledb.continuous) AS ...
```

**影响**:
- 统计查询性能下降
- 替代方案:使用物化视图(Materialized View)+定时刷新,或应用层缓存

## 最终建议

### 短期方案(当前实施)

1. **不使用超表**: 两个表都保持为普通PostgreSQL表
2. **使用常规索引**: 通过复合索引优化查询性能(已实现)
3. **允许NULL值**: 时间字段保持 `nullable: true`,不设默认值
4. **应用层处理**: 在Service层处理NULL值的业务逻辑

### 中期优化方案

如果未来数据量增长到百万级,考虑以下优化:

1. **表分区(Partitioning)**:
   - 使用 PostgreSQL 原生表分区(按月份或年份)
   - 不依赖 TimescaleDB,更灵活

2. **物化视图(Materialized Views)**:
   - 替代连续聚合视图
   - 使用 pg_cron 定时刷新

3. **数据归档**:
   - 定期将老数据移到归档表
   - 主表只保留最近1-2年的活跃数据

### 长期架构方案

如果确实需要 TimescaleDB 的高级功能:

1. **重新设计表结构**:
   - 创建专门的时序数据表,只存储"已发生且有明确时间"的事件
   - 原表作为业务表,保持灵活性

2. **双写策略**:
   - 业务写入原表(允许NULL)
   - 同时写入时序表(时间字段必填)
   - 使用时序表做分析和统计

3. **异步同步**:
   - 使用 CDC(Change Data Capture) 将数据同步到时序表
   - 过滤掉时间字段为NULL的记录

---

## 总结

**LogiX 项目的业务特点决定了不适合使用 TimescaleDB 超表:**

1. ✅ 时间字段的可空性是正常业务状态
2. ✅ 物流过程是渐进式的,时间字段逐步填充
3. ✅ 需要区分"未知时间"和"当前时间"
4. ❌ 超表要求分区键不能为NULL,与业务需求冲突

**当前方案(普通表+索引)是最合适的选择**,既保证了业务灵活性,又通过索引提供了足够的查询性能。
