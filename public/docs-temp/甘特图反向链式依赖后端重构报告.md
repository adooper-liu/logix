# 甘特图反向链式依赖后端重构报告

## 概述

本次重构将甘特图的**反向链式依赖逻辑**从前端迁移到后端，实现单一真相源（Single Source of Truth），确保前后端数据一致性。

**版本**: gantt-v3  
**日期**: 2026-04-04  
**作者**: 刘志高

---

## 问题分析

### 1. 原有架构问题

**前端逻辑**（已废弃）：

- 在 `calculateNodeStatus` 中实现反向推导
- 已提柜 → 清关 completed
- 已卸柜 → 清关 + 提柜 completed
- 已还箱 → 清关 + 提柜 + 卸柜 completed

**后端逻辑**（gantt-v2）：

- 只根据实际日期判断阶段
- 不考虑反向推导
- 已提柜货柜的清关节点返回 `taskRole='dashed'`（应显示虚线圆点）

**矛盾**：

- 前端认为清关节点应该销毁（completed）
- 后端认为清关节点应该显示（dashed）
- 导致前端显示错误

### 2. 根本原因

后端的 `buildNodes` 函数没有实现反向链式依赖逻辑，导致：

```typescript
// gantt-v2 的逻辑
if (nodeIndex === stage - 1) {
  taskRole = "main";
} else {
  taskRole = "dashed"; // ❌ 所有前置节点都显示为 dashed
}
```

## 解决方案

### 1. 后端重构（已完成 ✅）

**文件**: `backend/src/utils/ganttDerivedBuilder.ts`

**修改内容**：

```typescript
function buildNodes(...): GanttDerivedNode[] {
  // 反向链式依赖：检测后续节点是否已完成
  const hasDeliveryDate = !!trucking?.deliveryDate; // 已提柜
  const hasUnloadDate = !!warehouse?.unloadDate; // 已卸柜
  const hasReturnTime = !!emptyReturn?.returnTime; // 已还箱

  return NODE_ORDER.map((key, nodeIndex) => {
    let taskRole: GanttDerivedNode['taskRole'];

    if (stage === 5) {
      taskRole = 'none';
    } else if (nodeIndex >= stage) {
      taskRole = 'none';
    } else if (nodeIndex === stage - 1) {
      taskRole = 'main';
    } else {
      // ✅ 新增：应用反向链式依赖
      if (hasReturnTime) {
        taskRole = 'none'; // 已还箱 -> 所有前置节点销毁
      } else if (hasUnloadDate && (key === 'customs' || key === 'pickup')) {
        taskRole = 'none'; // 已卸柜 -> 清关、提柜销毁
      } else if (hasDeliveryDate && key === 'customs') {
        taskRole = 'none'; // 已提柜 -> 清关销毁
      } else {
        taskRole = 'dashed'; // 其他情况：显示为虚线
      }
    }

    return { key, taskRole, completed, plannedDate, actualDate };
  });
}
```

**规则版本更新**：

```typescript
export const GANTT_RULE_VERSION = "gantt-v3"; // v3: 添加反向链式依赖逻辑
```

### 2. 前端调整（已完成 ✅）

**文件**: `frontend/src/components/common/SimpleGanttChartRefactored.vue`

**修改内容**：

- 保持 `useLocalState = true`，优先使用本地状态
- 移除调试日志
- 注释说明：后端 gantt-v3 已实现反向链式依赖，但前端仍优先使用本地状态以确保兼容性

**原因**：

- 即使后端已更新，旧数据的 `gantt_derived` 可能仍是 v2 版本
- 前端本地状态可以立即生效，无需等待数据库更新
- 未来可以逐步过渡到完全使用后端数据

### 3. 数据迁移脚本（已创建 ✅）

## 业务规则

### 反向链式依赖规则

| 后续节点完成            | 影响的前置节点   | taskRole 变化     |
| ----------------------- | ---------------- | ----------------- |
| 已提柜 (`deliveryDate`) | 清关             | `dashed` → `none` |
| 已卸柜 (`unloadDate`)   | 清关、提柜       | `dashed` → `none` |
| 已还箱 (`returnTime`)   | 清关、提柜、卸柜 | `dashed` → `none` |

### 预期效果

**已提柜货柜**（HMMU6855127、GAOU6195045、KOCU5129260）：

- 清关节点：`taskRole='none'` → **不显示** ✅
- 提柜节点：`taskRole='main'` → 显示实线圆点
- 卸柜节点：`taskRole='dashed'` → 显示虚线圆点
- 还箱节点：`taskRole='dashed'` → 显示虚线圆点

**未提柜货柜**（HMMU6232153、HMMU6019657）：

- 清关节点：`taskRole='dashed'` 或 `'main'` → **正常显示** ✅
- 提柜节点：`taskRole='dashed'` → 显示虚线圆点
- 卸柜节点：`taskRole='dashed'` → 显示虚线圆点

## 技术优势

### 1. 单一真相源

- 后端是业务逻辑的唯一真相源
- 前端只负责展示和用户交互
- 符合 RESTful API 设计原则

### 2. 数据一致性

- 前后端使用相同的计算逻辑
- 避免数据不一致导致的显示错误
- 便于调试和维护

### 3. 性能优化

- 后端计算一次，多次使用
- 支持缓存和索引
- 减少前端重复计算

### 4. 可扩展性

- 规则版本化管理（gantt-v1/v2/v3）
- 便于未来添加新的业务规则
- 支持灰度发布和回滚
