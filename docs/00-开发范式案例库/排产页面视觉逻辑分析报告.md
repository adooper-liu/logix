## 🎯 视觉逻辑分析

### ✅ 优秀设计 (符合认知流)

#### 1. **F 型浏览模式** ✓
```
用户视线轨迹:
左上 (过滤条件) → 中上 (统计信息) → 右上 (操作按钮)
         ↓
     中部 (流程图 + 日志)
         ↓
     下方 (结果表格)
```

**优点**:
- ✅ 符合 Nielsen Norman Group 的 F 型阅读模式
- ✅ 重要信息优先展示 (统计 > 详情)
- ✅ 操作按钮位于视觉热点区域

#### 2. **注意力路线分组** ✓
```
顶部操作栏三组设计:
┌───────────┬───────────┬──────────┐
│ 过滤条件  │ 高级设置  │ 操作按钮 │
│  (核心)   │ (辅助)    │ (行动)   │
└───────────┴───────────┴──────────┘
```

**优点**:
- ✅ 左 → 右：信息输入 → 参数配置 → 执行动作
- ✅ 符合"观察 → 思考 → 行动"的决策流程
- ✅ 减少认知负担，每组职责清晰

#### 3. **渐进式信息展示** ✓
```
信息密度递增:
统计数字 (4 个指标) 
    ↓
流程图 (5 个步骤)
    ↓
执行日志 (详细过程)
    ↓
结果表格 (完整数据)
```

**优点**:
- ✅ 从抽象到具体
- ✅ 从概览到详情
- ✅ 避免信息过载

## ⚠️ 问题识别与优化建议

### 🔴 问题 1: 成本优化面板位置不当

**当前布局**:
```
执行日志卡 (Line 160-197)
    ↓
成本优化面板 (Line 200-207) ← 问题位置
    ↓
排产结果卡 (Line 217-...)
```

**问题分析**:

1. **视觉焦点分散**
   - 成本优化面板在日志和结果之间
   - 打断"预览 → 结果"的视觉连续性
   - 用户需要上下滚动查看完整流程

2. **执行流程割裂**
   ```
   用户操作流:
   点击"预览排产" → 查看日志 → 查看结果
                          ↑
                    被成本优化面板打断
   ```

3. **注意力竞争**
   - 日志卡 (执行过程) vs 成本优化 (额外功能)
   - 两个卡片宽度相同，视觉权重相等
   - 用户不知道应该先看哪个

**优化方案**:

```vue
<!-- 方案 A: 移到最底部 (推荐) -->
<el-row :gutter="12">
  <el-col :span="24">
    <!-- ① 执行日志 -->
    <LogCard v-if="logs.length > 0" />
  </el-col>
</el-row>

<el-row :gutter="12">
  <el-col :span="24">
    <!-- ② 排产结果 (主内容) -->
    <ResultCard />
  </el-col>
</el-row>

<!-- ✅ 成本优化面板移到最底部 -->
<el-row :gutter="12" style="margin-top: 16px">
  <el-col :span="24">
    <CostOptimizationPanel
      :selected-containers="selectedPreviewContainers"
      @applied="handleOptimizationApplied"
    />
  </el-col>
</el-row>
```

**优点**:
- ✅ 保持"预览 → 日志 → 结果"的主流程完整性
- ✅ 成本优化作为"后续操作",位置更自然
- ✅ 减少视觉干扰，焦点集中在核心任务

### 🟡 问题 2: 执行日志卡片高度过大

**当前设计**:
```vue
<el-card class="log-card" v-if="logs.length > 0">
  <div class="log-container" ref="logContainer">
    <!-- 日志内容可能很长 -->
  </div>
</el-card>
```

**问题分析**:

1. **日志可能占据大量空间**
   - 批量排产 50 个货柜 → 50+ 条日志
   - 日志卡片可能占据 2-3 屏高度
   - 用户需要大量滚动才能看到结果

2. **信息优先级错位**
   ```
   用户关心度:
   结果 > 统计 > 过程日志
   但实际布局: 日志在结果之前
   ```

**优化方案**:

```vue
<!-- 方案：可折叠的日志卡片 -->
<el-card class="log-card" v-if="logs.length > 0">
  <template #header>
    <div class="card-header">
      <span><el-icon><Document /></el-icon> 执行日志</span>
      <div class="header-actions">
        <el-tag size="small" :type="logs.length > 0 ? 'success' : 'info'">
          {{ logs.length }} 条记录
        </el-tag>
        <!-- ✅ 新增：折叠按钮 -->
        <el-button text size="small" @click="isLogCollapsed = !isLogCollapsed">
          <el-icon><ArrowUp v-if="isLogCollapsed" /><ArrowDown v-else /></el-icon>
          {{ isLogCollapsed ? '展开' : '收起' }}
        </el-button>
        <el-button text size="small" @click="logs = []">
          <el-icon><Delete /></el-icon> 清空
        </el-button>
      </div>
    </div>
  </template>

  <!-- ✅ 折叠时隐藏内容 -->
  <div v-show="!isLogCollapsed" class="log-container" ref="logContainer">
    <!-- 日志内容 -->
  </div>
</el-card>
```

**优点**:
- ✅ 默认展开 (需要时查看过程)
- ✅ 可收起 (专注结果查看)
- ✅ 用户自主控制信息密度

### 🟢 问题 3: 统计栏与流程图信息重复

**当前设计**:
```
统计栏 (Line 108-129):
┌──────┬──────┬───────┬──────┐
│待排产│initial│issued │仓库  │
└──────┴──────┴───────┴──────

流程图 (Line 132-157):
查询 → 排序 → 计划日 → 资源 → 写回
```

**问题分析**:

1. **信息重叠**
   - 统计栏显示"待排产"数量
   - 流程图显示"查询"步骤
   - 两者都表达"开始状态"

2. **认知负荷**
   - 用户需要理解两个组件的关系
   - 统计栏是"状态快照"
   - 流程图是"过程描述"
   - 同时存在造成信息冗余

**优化方案**:

```vue
<!-- 方案 A: 合并统计与流程 (推荐) -->
<div class="flow-bar-with-stats">
  <div class="flow-step" :class="{ active: currentStep >= 1 }">
    <span class="step-num">1</span>
    <span class="step-text">查询</span>
    <!-- ✅ 在步骤中显示统计 -->
    <span class="step-stat">{{ overview.pendingCount }} 待排产</span>
  </div>
  <span class="flow-arrow">→</span>
  <div class="flow-step" :class="{ active: currentStep >= 2 }">
    <span class="step-num">2</span>
    <span class="step-text">排序</span>
    <span class="step-stat">{{ overview.warehouses?.length }} 仓库</span>
  </div>
  <!-- ... 其他步骤 -->
</div>
```

**优点**:
- ✅ 减少垂直空间占用
- ✅ 状态与过程紧密结合
- ✅ 视觉更简洁

## 🔍 执行流程分析

### 当前执行流程

```
1. 用户操作：点击"预览排产"
   ↓
2. 前端：调用 batchSchedule API
   ↓
3. 后端：执行批量排产逻辑
   ↓
4. 前端：接收结果
   ↓
5. 显示执行日志 (自动滚动)
   ↓
6. 显示排产结果表格
   ↓
7. 用户查看日志
   ↓
8. 用户查看结果
   ↓
9. 用户选择要保存的记录
   ↓
10. 点击"确认保存"
```

**问题**:
- ⚠️ Step 5-6: 日志自动滚动可能跳过结果区域
- ⚠️ Step 7-8: 用户需要在日志和结果之间切换注意力
- ⚠️ 成本优化面板在 Step 6 后显示,打断流程

### 优化后的执行流程

```
1. 用户操作：点击"预览排产"
   ↓
2. 前端：调用 batchSchedule API
   ↓
3. 后端：执行批量排产逻辑
   ↓
4. 前端：接收结果
   ↓
5. ✅ 滚动到结果区域 (自动聚焦核心内容)
   ↓
6. ✅ 显示排产结果表格 + 统计徽章
   ↓
7. ✅ 日志卡片默认收起 (显示"5 条日志"标签)
   ↓
8. 用户查看结果表格
   ↓
9. 用户选择要保存的记录
   ↓
10. 点击"确认保存"
   ↓

   ↓
12. [可选] 点击"批量优化"进行成本优化
```

**优化点**:
- ✅ 自动聚焦核心内容 (结果表格)
- ✅ 辅助信息 (日志) 默认收起
- ✅ 成本优化作为可选后续步骤
- ✅ 减少不必要的视觉干扰

## 🎯 总结

### 当前优势 ✅

1. **F 型浏览模式**: 符合用户视觉习惯
2. **注意力分组**: 操作栏三组设计合理
3. **渐进式展示**: 从概览到详情层次清晰

### 待优化点 ⚠️

1. **成本优化面板位置**: 打断主流程视觉连续性
2. **执行日志过大**: 占据过多视觉空间
3. **统计与流程分离**: 信息重复且占用空间

### 优化方向 📈

1. **移**: 成本优化面板移到底部
2. **折**: 执行日志添加折叠功能
3. **并**: 统计栏与流程图合并
4. **聚**: 自动滚动聚焦结果区域

### 核心原则 🎓

**视觉逻辑优化三原则**:
1. **主次分明**: 核心内容占据视觉焦点
2. **流程顺畅**: 操作顺序符合用户预期
3. **信息适度**: 避免过载，按需展示
