## ✅ 成功案例

### 案例 1: 排产确认保存功能完善 (2026-03-27)

**标签**: `架构分析` `SKILL 原则` `最小改动`

#### 背景

排产系统预览功能已完成，但确认保存时只更新状态，没有实际保存计划日期到数据库和扣减资源档期。

#### 问题

- `savePreviewResults` 方法只有日志记录，没有实际保存逻辑
- 缺少档期扣减机制
- 需要支持快速路径（使用预览数据）和慢速路径（重新计算）

#### 解决方案

**应用开发范式步骤**:

1. **架构分析**
   - 业务架构：预览 → 确认 → 保存流程
   - 数据模型：WarehouseOperation, TruckingTransport, 档期表
   - 服务架构：IntelligentSchedulingService, SchedulingController
   - 数据流：dryRun 参数控制是否保存

2. **问题诊断**
   - 现象：保存后数据库无记录
   - 根因：方法未实现完整逻辑
   - 影响：档期数据不准确，可能超卖

3. **策略选择**
   - ✅ 方案 A：完善现有的 `savePreviewResults` 方法（推荐）
   - 方案 B：创建新的 CapacityCheckService（不推荐）
   - 方案 C：重写整个保存逻辑（反对）

4. **实施**
   - 新增 `savePlannedDates` 方法
   - 新增 `occupyCapacity` 方法
   - 前端添加 `refreshCapacityData` 调用

5. **结果**
   - 工作量：2 小时
   - 风险：低（基于现有方法完善）
   - 效果：完美，符合 SKILL 原则

#### 经验教训

✅ **成功经验**:
- 先深度分析架构，再动手修改
- 复用现有代码，避免重复造轮子
- 小步迭代，渐进式完善
- 档期字段验证：plannedCount vs occupied

📝 **改进建议**:
- 应该在第一次实现时就完整
- 需要补充集成测试

#### 代码片段

```typescript
// ✅ 正确：完善现有方法
private async savePreviewResults(previewResults: any[]) {
  // ... 现有代码 ...
  
  // 新增：保存计划日期
  await this.savePlannedDates(preview.plannedData, manager);
  
  // 新增：扣减资源档期
  await this.occupyCapacity(preview.plannedData, manager);
}
```

### 案例 2: 资源状态与日期状态分离显示 (2026-03-26)

**标签**: `UI 优化` `用户体验` `渐进式重构`

#### 背景

资源状态列同时显示产能负荷和日期紧急度，导致用户困惑。

#### 问题

- 0% 占用率显示"紧张"（因为日期临近）
- 用户无法区分是产能紧张还是日期紧急
- 逻辑耦合，难以维护

#### 解决方案

应用五维分析法：

1. **业务架构**: 用户需要清晰了解两个维度
2. **数据模型**: 档期数据 + 日期数据
3. **前端架构**: 分离显示逻辑
4. **用户体验**: 不同列显示不同信息

**实施**:
- 资源状态列：只显示产能负荷（正常/紧张/超负荷）
- 计划日期列：增加日期状态标签（剩余 X 天/超期 X 天）

#### 结果

- 用户满意度提升
- 代码逻辑更清晰
- 可维护性提高

## ❌ 失败案例

### 案例 1: 档期字段名错误 (2026-03-27)

**标签**: `字段验证` `数据库优先` `反面教材`

#### 问题

在 `occupyCapacity` 方法中使用了错误的字段名：

```typescript
// ❌ 错误：臆想字段名
warehouseOccupancy.occupied += 1;  // 不存在这个字段！
truckingOccupancy.occupied += 1;   // 不存在这个字段！
```

#### 根因

- 没有先读取实体定义
- 凭经验假设字段名是 `occupied`
- 违反"数据库优先开发原则"

#### 正确做法

应该先验证实体字段：

```typescript
// ✅ 正确：先读取实体定义
const warehouseOccupancy = await manager.findOne(ExtWarehouseDailyOccupancy, {
  where: { warehouseCode, date }
});

if (warehouseOccupancy) {
  // 验证字段名
  warehouseOccupancy.plannedCount += 1;  // 正确的字段名
}
```

#### 教训

📌 **必须遵守的原则**:
1. 修改数据库相关代码前，必须先读取实体定义
2. 不要臆想字段名
3. 使用 TypeORM 时，注意字段名映射（camelCase ↔ snake_case）

### 案例 2: Vue 组件类型冲突 (2026-03-25)

**标签**: `TypeScript` `Vue` `类型安全`

#### 问题

在 Vue 组件中使用严格类型导致编译错误：

```typescript
// ❌ 错误：类型过于严格
interface PlanDateItem {
  plannedPickupDate: string;  // 必须是 string
}

// 但实际数据可能是 undefined
row.plannedData?.plannedPickupDate  // 可能是 undefined
```

#### 解决方案

```typescript
// ✅ 正确：使用 any 或可选属性
interface PlanDateItem {
  plannedPickupDate?: string;  // 可选
}

// 或使用 any（Vue 组件推荐）
const row: any = props.row;
```

#### 教训

- Vue 组件中过度追求类型安全会导致复杂性
- 适当使用 `any` 是可以接受的
- 优先保证功能，再考虑类型完美

## ⚠️ 最佳实践

### BP-1: 先理解后修改

**场景**: 任何代码修改前

**步骤**:
1. 读取相关文件（至少 2-3 个依赖文件）
2. 理解业务逻辑
3. 找到权威源（数据库实体、接口定义）
4. 制定修改方案
5. 方案评审（复杂修改）

**检查清单**:
- [ ] 是否读取了实体定义？
- [ ] 是否理解了业务流程？
- [ ] 是否找到了现有代码？
- [ ] 是否有更简单的方案？

---

### BP-2: 复用优先于创新

**场景**: 需要新功能时

**优先级**:
1. ✅ 复用现有服务/组件
2. ✅ 扩展现有功能
3. ⚠️ 适度重构
4. ❌ 从头开始（最后选择）

**检查清单**:
- [ ] 是否搜索了现有代码？
- [ ] 是否有类似实现？
- [ ] 是否可以参数化？
- [ ] 是否可以继承扩展？

---

### BP-3: 小步迭代

**场景**: 大型重构/新功能

**步骤**:
1. 第一阶段：核心功能（MVP）
2. 第二阶段：增强功能
3. 第三阶段：优化完善

**每个阶段**:
- 可独立验证
- 可回滚
- 不影响现有功能

---

### BP-4: 文档即代码

**要求**:
- 代码修改必须同步更新文档
- 注释解释"为什么"而不是"是什么"
- 复杂逻辑必须有示例

**模板**:
```typescript
/**
 * 功能说明
 * 
 * @param xxx - 参数说明
 * @returns - 返回值说明
 * 
 * @example
 * ```typescript
 * const result = await func(xxx);
 * ```
 */
```

## 🔧 反模式

### AP-1: 重复造轮子

**表现**:
- 已有类似功能，却重新实现
- 复制粘贴代码，稍作修改
- "这个很简单，我几分钟就能写好"

**危害**:
- 代码冗余
- 维护成本翻倍
- 行为不一致

**预防**:
- 开发前先搜索
- Code Review 重点检查
- 建立通用组件库

---

### AP-2: 过度设计

**表现**:
- 为未来可能的需求编程
- 过度抽象，难以理解
- 设计模式滥用

**危害**:
- 代码复杂
- 性能下降
- 维护困难

**预防**:
- YAGNI 原则（You Aren't Gonna Need It）
- KISS 原则（Keep It Simple, Stupid）
- 简单优于复杂

---

### AP-3: 盲目修改

**表现**:
- 不理解代码就修改
- 试错法编程
- "先试试，不行再改"

**危害**:
- 引入新 Bug
- 破坏现有逻辑
- 技术债务累积

**预防**:
- 先读后写
- 方案评审
- 充分测试
