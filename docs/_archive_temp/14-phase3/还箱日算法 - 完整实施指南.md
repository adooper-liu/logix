## 🎯 问题背景

### 原有问题

1. **Drop off 模式还箱能力检查顺序错误**
   - ❌ 直接检查"卸柜日 +1"，跳过当天检查
   - ❌ 可能导致不必要的堆场费用

2. **提柜日兜底逻辑定位不清晰**
   - ❌ 担心破坏"先卸柜，后倒退"的逻辑
   - ❌ 需要明确三种调整模式的定位

### 业务规则

```
Drop off 模式:
  Step 1: 优先检查卸柜日当天能力 → 当天还箱（最优）
  Step 2: 其次检查卸柜日 +1 能力 → 次日还箱（标准）
  Step 3: 最后顺延查找有能力日期 → 减少堆场费用

Live load 模式:
  还 = 卸（同日）
  若能力不足 → 调整卸柜日到最早有能力日期

最晚还箱日:
  红线约束，不可超过
```

## ✅ 解决方案

### 核心代码修改

**文件**: `backend/src/services/intelligentScheduling.service.ts`  
**位置**: L992-L1047

```typescript
} else {
  // ✅ Drop off 模式：优先当天还箱，其次卸 +1，再往后顺延

  // Step 1: 先检查卸柜日当天的还箱能力
  const availableOnUnloadDate = await this.findEarliestAvailableReturnDate(
    truckingCompanyId,
    returnDateOnly,
    lastReturnDate
  );

  if (availableOnUnloadDate) {
    // 如果卸柜日当天有能力，当天还箱（最优解，减少堆场费用）
    return {
      returnDate: availableOnUnloadDate,
      adjustedUnloadDate: undefined
    };
  }

  // Step 2: 如果卸柜日当天没能力，再检查卸柜日 +1
  const nextDay = new Date(returnDateOnly);
  nextDay.setUTCDate(nextDay.getUTCDate() + 1);

  const availableOnNextDay = await this.findEarliestAvailableReturnDate(
    truckingCompanyId,
    nextDay,
    lastReturnDate
  );

  if (availableOnNextDay) {
    // 卸柜日 +1 有能力，次日还箱（标准 Drop off 模式）
    return {
      returnDate: availableOnNextDay,
      adjustedUnloadDate: undefined
    };
  }

  // Step 3: 如果都没能力，继续顺延查找
  const availableDate = await this.findEarliestAvailableReturnDate(
    truckingCompanyId,
    nextDay,
    lastReturnDate
  );

  if (!availableDate) {
    return {
      returnDate: nextDay,
      adjustedUnloadDate: undefined
    };
  }

  return {
    returnDate: availableDate,
    adjustedUnloadDate: undefined
  };
}
```

### 架构设计

```
正向推导 + 反向修正 + 兜底保护

正向推导（默认）:
  清关日 → 提柜日 → 卸柜日 → 送仓日 → 还箱日

兜底保护（提柜日 ≤ 今天）:
  提柜日 → 明天
  后续步骤继续正向推导

反向修正（还箱能力不足）:
  还箱日 → 调整卸柜日 → 调整送仓日
```
