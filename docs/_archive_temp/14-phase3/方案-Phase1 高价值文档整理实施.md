## 一、总体策略

### 1.1 整理方法论

参考 `计算逻辑场景模拟.md` 风格，采用：

```markdown
✅ 场景模拟法：通过具体参数和计算过程展示逻辑
✅ 对比表格：清晰展示不同方案/状态/结果的差异
✅ 权威源验证：所有数据基于实际代码和数据库表结构
✅ 渐进式整理：每个专题独立成篇，逐步完善
```

### 1.2 SKILL 原则落地

| SKILL 原则 | 在文档整理中的体现 |
|------------|-------------------|
| **数据库优先** | 所有费用计算基于真实表结构，引用实体文件路径 |
| **权威源验证** | 每个案例都标注来源文件、方法、行号 |
| **代码简洁** | 文档精简，只保留核心场景和关键差异 |
| **业务导向** | 围绕真实业务场景（滞港费、排柜、档期等） |

   - API 路径重复 404 场景
   - Excel导入列名不匹配场景
   - FullCalendar 初始化错误场景

#### 执行步骤

**Step 1: 分类 Bug 报告** (1.5h)
```bash
# 使用 PowerShell 分类
Get-ChildItem "*修复报告*.md" | Where-Object { $_.Name -match "UI|样式 | 图表" } | Move-Item "UI-Bugs/"
Get-ChildItem "*修复报告*.md" | Where-Object { $_.Name -match "数据 | 字段 | 计算" } | Move-Item "Data-Bugs/"
Get-ChildItem "*修复报告*.md" | Where-Object { $_.Name -match "API|路径 | 导入" } | Move-Item "API-Bugs/"
```

**Step 2: 提取共性场景** (2h)
```markdown
从每个 Bug 中提取:
- 现象描述（What/When/Where）
- 根因分析（直接/间接/系统性）
- 修复方案（临时/永久/预防）
- 验证方法（如何确认修复成功）
```

**Step 3: 编写场景模拟** (2h)
```markdown
每个 Bug 类型编写 3-5 个场景:
- 场景 1: 典型错误
- 场景 2: 边界情况
- 场景 3: 并发/特殊情况
```

**Step 4: 创建检查清单** (0.5h)
```markdown
针对每类 Bug，创建 Code Review 检查项:
- [ ] 字段名是否与数据库一致
- [ ] CSS 注释是否使用/* */
- [ ] API 路径是否有重复
```


### 6.1 识别的风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| R1: 源文档质量差 | 中 | 中 | 提前识别，补充验证 |

| R4: 理解有偏差 | 中 | 中 | 多人审查，交叉验证 |

### 6.2 风险触发时的行动

```markdown
如果 R2 触发（代码已变更）:
  Step 1: 暂停文档编写
  Step 2: 重新读取最新代码
  Step 3: 更新场景模拟
  Step 4: 标注版本号和更新日期
```
