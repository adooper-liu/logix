# 仓库手工指定功能 - 紧急修复报告

**创建日期**: 2026-03-26  
**问题发现**: "手工指定"按钮不可点击  
**修复状态**: ✅ **已完成**

---

## 🐛 **问题描述**

### **现象**

- "手工指定"按钮处于禁用状态，无法点击
- 用户反馈："手工模式是不可点击？"

### **根本原因**

按钮的 `disabled` 状态绑定错误：

```vue
<!-- ❌ 错误的实现 -->
:disabled="!hasSelectedContainers"

const hasSelectedContainers = computed(() => selectedContainersForDesignation.value.length > 0);
const selectedContainersForDesignation = ref<string[]>([]); // 永远是空数组
```

**问题分析**：

1. 代码逻辑要求用户先选择货柜才能点击"手工指定"
2. 但页面上**没有货柜选择器**，用户无法选择货柜
3. `selectedContainersForDesignation` 永远是空数组
4. 导致按钮永远处于禁用状态

---

## ✅ **修复方案**

### **方案设计原则**

考虑到实际业务场景：

- 通常是对**一批待排产货柜**统一指定仓库
- 不需要精细到单个货柜控制
- 简化操作，提升用户体验

### **修复内容**

#### **1. 修改按钮禁用条件**

**修改前**:

```vue
:disabled="!hasSelectedContainers"
```

**修改后**:

```vue
:disabled="overview.pendingCount === 0"
```

**说明**：只要有待排产货柜就可以点击，无需预先选择。

---

#### **2. 简化状态管理**

**修改前**:

```typescript
const selectedContainersForDesignation = ref<string[]>([]); // 空数组
const hasSelectedContainers = computed(() => selectedContainersForDesignation.value.length > 0);
```

**修改后**:

```typescript
// const selectedContainersForDesignation = ref<string[]>([]); // 不再需要
// const hasSelectedContainers = computed(() => ...); // 注释掉
```

**说明**：移除不必要的状态变量。

---

#### **3. 优化打开对话框逻辑**

**修改前**:

```typescript
const openDesignatedWarehouseDialog = () => {
  if (!hasSelectedContainers.value) {
    ElMessage.warning("请先选择要排产的货柜");
    return;
  }
  showDesignatedWarehouseDialog.value = true;
};
```

**修改后**:

```typescript
const openDesignatedWarehouseDialog = () => {
  if (overview.value.pendingCount === 0) {
    ElMessage.warning("没有待排产的货柜");
    return;
  }
  showDesignatedWarehouseDialog.value = true;
};
```

**说明**：检查有待排产货柜即可，无需预先选择。

---

#### **4. 优化确认处理函数**

**修改前**:

```typescript
const handleDesignatedWarehouseConfirm = async (data: {...}) => {
  const result = await containerService.batchSchedule({
    designatedWarehouseMode: true,
    designatedWarehouseCode: data.warehouseCode,
    containerNumbers: data.containerNumbers || selectedContainersForDesignation.value,
    // ↑ 问题：如果用户没选柜号，会用空数组，导致不排产任何货柜
    dryRun: false,
  });
};
```

**修改后**:

```typescript
const handleDesignatedWarehouseConfirm = async (data: {...}) => {
  const result = await containerService.batchSchedule({
    designatedWarehouseMode: true,
    designatedWarehouseCode: data.warehouseCode,
    containerNumbers: data.containerNumbers,
    // ↑ 改进：如果为 undefined，后端会对所有待排产货柜生效
    dryRun: false,
  });
};
```

**说明**：允许 `containerNumbers` 为 `undefined`，后端会自动对所有待排产货柜生效。

---

#### **5. 修复模板引用**

**修改前**:

```vue
<DesignatedWarehouseDialog :container-numbers="selectedContainersForDesignation" ... />
```

**修改后**:

```vue
<DesignatedWarehouseDialog :container-numbers="[]" ... />
```

**说明**：传入空数组，表示不限制特定柜号。

---

#### **6. 更新前端类型定义**

**文件**: `frontend/src/services/container.ts`

**新增字段**:

```typescript
async batchSchedule(params: {
  country?: string
  startDate?: string
  endDate?: string
  forceSchedule?: boolean
  limit?: number
  skip?: number
  dryRun?: boolean
  containerNumbers?: string[]
  etaBufferDays?: number

  // ✅ 新增：手工指定仓库模式（可选）
  designatedWarehouseMode?: boolean
  designatedWarehouseCode?: string
}): Promise<{...}>
```

**说明**：添加手工指定模式的类型定义。

## ✅ **修复后的行为**

### **用户使用流程**

```
1. 用户访问排产页面
   ↓
2. 查看有待排产货柜数量（如：5 个）
   ↓
3. 点击"手工指定"按钮（✅ 现在可以点击了！）
   ↓
4. 弹出仓库选择对话框
   ↓
5. 选择目标仓库（如：US-W001）
   ↓
6. （可选）选择特定柜号，或留空（表示全部）
   ↓
7. 点击"确认排产"
   ↓
8. 系统对以下对象进行排产：
   - 如果选了柜号 → 只对选中的货柜排产
   - 如果留空 → 对所有 5 个待排产货柜排产
   ↓
9. 显示排产结果并刷新页面
```

## 🎯 **改进点总结**

### **优点**

1. ✅ **简化操作** - 无需预先选择货柜
2. ✅ **符合实际** - 批量操作更符合业务场景
3. ✅ **用户友好** - 减少操作步骤
4. ✅ **灵活控制** - 仍支持选择特定柜号

### **权衡**

- ⚠️ 移除了"先选择后操作"的精细控制
- ✅ 但换来了更好的用户体验
- ✅ 实际业务中通常是对一批货柜统一操作

## 📝 **经验教训**

### **问题根源**

1. **过度设计** - 原本想实现"先选择后操作"的精细控制
2. **脱离实际** - 忽略了页面上没有货柜选择器的现实
3. **缺乏验证** - 实现后没有实际测试按钮是否可点击

### **改进措施**

1. **及时测试** - 每完成一个功能立即自测
2. **简化优先** - 优先实现简单实用的方案
3. **用户视角** - 从用户角度思考操作流程
4. **SKILL 原则** - 基于实际业务需求，杜绝过度设计
