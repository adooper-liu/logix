# Phase 2 中价值文档筛选实施方案

**制定日期**: 2026-03-29  
**执行原则**: 价值导向 · 场景优先 · 渐进整理 · 避免过度  
**目标范围**: 从约 200+ 篇文档中筛选出 50-80 篇中价值文档

---

## 一、总体策略

### 1.1 筛选方法论

采用 **三层漏斗筛选法**:

```
第一层：快速扫描 (按标题和关键词)
    ↓ 剔除明显低价值文档 (重复/过时/无实质内容)
第二层：内容评估 (按章节结构和场景)
    ↓ 识别中价值文档 (有场景/有对比/有验证)
第三层：价值排序 (按使用频率和影响范围)
    ↓ 选出 Top 50-80 篇中价值文档
```

### 1.2 文档价值定义

| 价值等级 | 特征 | 占比 | 处理方式 |
|----------|------|------|----------|
| **高价值** | ✅ 核心计算逻辑<br>✅ 关键业务流程<br>✅ 重大 Bug 修复<br>✅ 架构级决策 | 10%<br>(约 20 篇) | Phase 1 已完成<br>创建专题文档 |
| **中价值** | ✅ 常见场景处理<br>✅ 一般功能优化<br>✅ 典型错误案例<br>✅ 配置和使用指南 | 30%<br>(约 50-80 篇) | Phase 2 重点<br>分类整理成册 |
| **低价值** | ⚠️ 日常维护记录<br>⚠️ 简单问题修复<br>⚠️ 重复性报告<br>⚠️ 临时性方案 | 60%<br>(约 120 篇) | 归档备查<br>不主动整理 |

## 二、筛选标准

### 2.1 中价值文档识别标准

#### 技术类中价值

| 类型 | 识别标准 | 示例 |
|------|----------|------|
| **功能实现** | - 涉及 3 个以上文件修改<br>- 有新的设计模式<br>- 解决了复杂问题 | - 智能排产车队选择算法<br>- 成本优化多目标评分<br>- 状态机实现 |
| **Bug 修复** | - 根因分析清晰<br>- 有预防措施<br>- 影响范围≥2 个模块 | - 日期时区问题修复<br>- 数据一致性问题<br>- 并发控制问题 |
| **性能优化** | - 有前后对比数据<br>- 优化幅度>20%<br>- 可复用性强 | - 查询优化 (100ms→20ms)<br>- 缓存策略优化<br>- 批量处理优化 |
| **重构改进** | - 提升代码质量<br>- 降低耦合度<br>- 提高可维护性 | - 组件抽取和复用<br>- 服务层重构<br>- 数据结构优化 |

#### 业务类中价值

| 类型 | 识别标准 | 示例 |
|------|----------|------|
| **业务流程** | - 涉及多环节协作<br>- 有状态流转<br>- 影响用户体验 | - 提柜→卸柜→还箱流程<br>- 费用计算流程<br>- 审批流程 |
| **数据流转** | - 跨系统/模块<br>- 有关键转换<br>- 易出错环节 | - Excel 导入数据清洗<br>- API 数据映射<br>- 数据同步机制 |
| **配置规则** | - 需要理解业务背景<br>- 有多种配置组合<br>- 容易配置错误 | - 滞港费标准配置<br>- 仓库 - 车队映射配置<br>- 档期配置 |

### 2.2 排除标准 (低价值文档)

```markdown
❌ 纯事务性记录
  - 日常会议纪要
  - 简单任务分配
  - 进度汇报 (无技术含量)

❌ 临时性方案
  - 一次性脚本说明
  - 临时数据修复
  - 测试性代码

❌ 重复性内容
  - 同一问题的多个类似报告
  - 已有更好文档的早期版本
  - 简单的复制粘贴

❌ 过于简单的问题
  - 拼写错误修复
  - 简单样式调整
  - 单行代码修改
```

### 3.2 第二步：快速扫描 (Triage)

**时间**: 2-3 小时  
**方法**: 基于标题和关键词快速分类

#### 分类关键词

| 类别 | 关键词 | 预期数量 |
|------|--------|----------|
| **高价值候选** | 智能排产、成本优化、深度分析、方案设计、重构 | 20-30 篇 |
| **中价值候选** | 修复报告、优化报告、实施报告、功能说明、配置指南 | 80-100 篇 |
| **低价值候选** | 会议记录、简单修复、临时方案、测试记录、日常汇报 | 100-120 篇 |

#### 执行脚本

```powershell
# 定义关键词
$highValueKeywords = @("智能排产", "成本优化", "深度分析", "方案设计", "重构")
$midValueKeywords = @("修复报告", "优化报告", "实施报告", "功能说明")

# 遍历文档并标记
Get-ChildItem "docs\_archive_temp" -Recurse -Filter *.md | ForEach-Object {
    $content = Get-Content $_.FullName -Raw
    $tags = @()
    
    # 检查高价值关键词
    foreach ($keyword in $highValueKeywords) {
        if ($content -match $keyword) {
            $tags += "HighValue"
            break
        }
    }
    
    # 检查中价值关键词
    foreach ($keyword in $midValueKeywords) {
        if ($content -match $keyword) {
            $tags += "MidValue"
            break
        }
    }
    
    # 无关键词则为 LowValue
    if ($tags.Count -eq 0) {
        $tags += "LowValue"
    }
    
    [PSCustomObject]@{
        FileName = $_.Name
        Path = $_.DirectoryName
        Size = $_.Length
        Tags = $tags -join ","
        LastModified = $_.LastWriteTime
    }
} | Export-Csv "docs\00-开发范式案例库\phase2-triage-result.csv" -NoTypeInformation
```

### 3.3 第三步：内容评估

**时间**: 4-6 小时  
**方法**: 阅读摘要和关键章节，打分排序

#### 评分维度

| 维度 | 权重 | 评分标准 (1-5 分) |
|------|------|------------------|
| **技术深度** | 30% | 5=有创新设计<br>3=有技术亮点<br>1=常规实现 |
| **业务价值** | 30% | 5=核心业务逻辑<br>3=重要功能<br>1=边缘功能 |
| **可复用性** | 20% | 5=可多处复用<br>3=部分复用<br>1=特定场景 |
| **文档质量** | 20% | 5=结构清晰 + 有示例<br>3=结构完整<br>1=简单描述 |

#### 评分卡模板

```markdown
## 文档评分卡

**文档名称**: [文件名]
**路径**: [相对路径]
**大小**: [KB]

### 评分

| 维度 | 得分 | 理由 |
|------|------|------|
| 技术深度 (30%) | /5 | |
| 业务价值 (30%) | /5 | |
| 可复用性 (20%) | /5 | |
| 文档质量 (20%) | /5 | |

### 加权总分

总分 = 技术深度×0.3 + 业务价值×0.3 + 可复用性×0.2 + 文档质量×0.2
     = ___ × 0.3 + ___ × 0.3 + ___ × 0.2 + ___ × 0.2
     = ___ / 5.0

### 评级

□ S 级 (4.5-5.0): 必须整理，放入高价值集合
□ A 级 (4.0-4.4): 重点整理，放入中价值集合
□ B 级 (3.0-3.9): 选择性整理
□ C 级 (<3.0): 归档备查

### 整理建议

推荐整理方式:
- [ ] 创建独立专题文档
- [ ] 合并到现有专题
- [ ] 作为附录/案例
- [ ] 仅归档，不整理
```

## 六、风险控制

### 6.1 风险识别

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| R1: 文档太多，时间不足 | 高 | 中 | 优先处理高价值和高中价值 |
| R2: 评分标准不一致 | 中 | 中 | 制定详细评分细则，双人校准 |
| R3: 分类困难，边界模糊 | 高 | 低 | 允许交叉引用，不强求唯一 |
| R4: 文档质量差，难以整理 | 中 | 中 | 提取核心价值，重新组织 |

### 6.2 风险应对策略

```markdown
如果 R1 触发 (时间不足):
  Step 1: 优先处理评分≥4.0 的文档
  Step 2: 简化评分流程，只评高/中价值
  Step 3: 延长 Phase 2 时间到 5 天

如果 R3 触发 (分类困难):
  Step 1: 允许多标签分类
  Step 2: 建立"其他"类别作为缓冲
  Step 3: 在 README 中添加交叉引用
```
