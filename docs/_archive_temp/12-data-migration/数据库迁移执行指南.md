## 四、日期类型统一（按情况）🔵

**背景**: 项目早期版本使用 DATE 类型，后期统一改为 TIMESTAMP。

### 4.1 统一脚本选择

**二选一原则**: 两个脚本功能重叠，只执行一个即可。

| 脚本                            | 特点                   | 推荐场景                   |
| ------------------------------- | ---------------------- | -------------------------- |
| `unify-datetime-types.sql`      | 直接转换，无检查       | 新库、确认需要转换         |
| `convert_date_to_timestamp.sql` | 带条件判断，可重复执行 | 生产环境、可能需要多次执行 |

**推荐**: 使用 `convert_date_to_timestamp.sql`（更安全）

### 4.2 转换的表与字段

**受影响的表**（基于代码验证）:

```sql
✅ process_port_operations
   - port_arrival_date (DATE → TIMESTAMP)
   - last_pickup_date (DATE → TIMESTAMP)
   - last_free_date (DATE → TIMESTAMP)

✅ process_sea_freight
   - etd_origin_port (DATE → TIMESTAMP)
   - eta_dest_port (DATE → TIMESTAMP)
   - actual_departure_date (DATE → TIMESTAMP)
   - actual_arrival_date (DATE → TIMESTAMP)

✅ biz_containers
   - last_free_date (DATE → TIMESTAMP) [如有]
```

### 4.3 执行检查

```bash
# 检查当前类型
psql $DATABASE_URL -c "\d process_port_operations" | grep "Date\|Timestamp"

# 如果需要回滚（不推荐）
psql $DATABASE_URL -f migrations/rollback_timestamp_to_date.sql
```

**注意**: `rollback_timestamp_to_date.sql` 与上述转换相反，**不要同时使用**！

## 九、常见问题 FAQ 💡

### Q1: 如何确认迁移脚本都执行成功了？

**A**: 使用以下验证查询:

```sql
-- 1. 检查表数量
SELECT COUNT(*) AS total_tables
FROM information_schema.tables
WHERE table_schema = 'public';
-- 应该返回 30+ 张表

-- 2. 检查关键表是否存在
SELECT tablename FROM pg_tables
WHERE schemaname = 'public'
AND tablename IN (
    'biz_containers', 'biz_replenishment_orders',
    'dict_scheduling_config', 'ext_demurrage_records'
);

-- 3. 检查配置项
SELECT config_key, config_value
FROM dict_scheduling_config
ORDER BY config_key;
```

### Q2: 执行顺序能否调整？

**A**: 基本原则:

- ✅ 主初始化脚本必须按顺序（01→03→supplement→02→04→05）
- ✅ CREATE 表必须在 ALTER 表之前
- ✅ 有依赖关系的必须按依赖顺序（如 A1 在 B1 之前）
- ⚠️ 同一组内的可以适当调整，但建议按文档顺序

### Q3: 某个脚本执行失败怎么办？

**A**: 排查步骤:

1. 查看错误信息，确定是语法错误还是依赖问题
2. 检查前置依赖表是否存在
3. 如果是 ALTER 表，检查目标表结构
4. 使用 `\d+ table_name` 查看表详情
5. 参考 `backend/docs/DATABASE_SCRIPTS_INDEX.md`

### Q4: 是否需要每次部署都执行所有脚本？

**A**: 不需要！分场景:

- **新建数据库**: 执行全部必须脚本
- **增量更新**: 只执行新增的迁移脚本
- **生产环境**: 使用 Flyway/Liquibase等迁移工具管理

### Q5: 如何处理脚本重叠问题？

**A**: 常见重叠:

- `create_resource_occupancy_tables.sql` ↔ `03_create_tables_supplement.sql`
  → 二选一，推荐用 supplement
- `unify-datetime-types.sql` ↔ `convert_date_to_timestamp.sql`
  → 二选一，推荐用 convert（更安全）
