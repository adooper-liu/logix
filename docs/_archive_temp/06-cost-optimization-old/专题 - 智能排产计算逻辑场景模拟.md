## 一、概述

### 1.1 适用范围

本文档详细说明 LogiX 系统**智能排产模块**的计算逻辑，通过**场景模拟**和**对比表格**展示不同参数下的计算结果。

适用于:
- ✅ 开发者理解排产算法
- ✅ 测试人员设计测试用例
- ✅ 产品经理审查业务逻辑
- ✅ 运维人员排查问题

### 1.2 核心概念

智能排产以 **卸柜日 (plannedUnloadDate)** 为核心锚点，向前向后推导其他 4 个计划日期。

#### 五个计划日期

| 序号 | 字段名 | 中文名 | 说明 |
|------|--------|--------|------|
| 1 | `plannedCustomsDate` | 计划清关日 | ETA/ATA - 1 天 |
| 2 | `plannedPickupDate` | 计划提柜日 | 清关日 + 1天，且 ≤ lastFreeDate |
| 3 | `plannedDeliveryDate` | 计划送仓日 | 提柜日 + 运输天数 (Live load=提，Drop off=卸) |
| 4 | `plannedUnloadDate` | 计划卸柜日 | **核心锚点**：基于仓库可用性计算 |
| 5 | `plannedReturnDate` | 计划还箱日 | 卸柜日 + 1天 (受还箱能力约束) |

#### 三种排产模式

| 模式 | 触发方式 | dryRun 参数 | 数据保存 | 用途 |
|------|----------|------------|----------|------|
| **① 预览排产** | 前端点击"预览排产" | `true` | ❌ 不保存 | 预览方案，审查后批量保存 |
| **② 单柜优化** | 前端点击"优化"按钮 | `false` | ❌ 可选保存 | 针对单个货柜的成本优化 |
| **③ 批量排产** | 前端点击"执行排产" | `false` | ✅ 保存到 DB | 正式排产，批量处理 |

## 二、核心计算逻辑

### 2.1 日期推导关系

```
         ←─────────── 向前推导 ───────────→        ←── 向后推导 ──→

┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   计划清关日    计划提柜日    计划送仓日    计划卸柜日    计划还箱日        │
│  ───────────    ──────────    ──────────    ──────────    ─────────        │
│                                                                             │
│   planned      planned        planned        planned        planned         │
│  CustomsDate  PickupDate    DeliveryDate    UnloadDate    ReturnDate        │
│                                                                             │
│       ↑            ↑             ↑             ↑              ↑            │
│       │            │             │             │              │            │
│    ETA-1 天     customs+1 天    提 + 运输天      锚点          卸 + 在仓天       │
│                 (≤lastFree)                                           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 计算公式总览

| 日期 | 计算公式 | 约束条件 |
|------|----------|----------|
| plannedCustomsDate | ATA/ETA - 1天 - ETA 缓冲天数 | - |
| plannedPickupDate | plannedCustomsDate + 1天 | ≤ lastFreeDate, ≥ today |
| plannedDeliveryDate | Live load: = pickup<br>Drop off: = unload | 模式决定 |
| plannedUnloadDate | 基于仓库可用性 | 日产能限制 |
| plannedReturnDate | Live load: = unload<br>Drop off: = unload + 1 | ≤ lastReturnDate |

### 2.3 关键参数

| 参数 | 来源 | 说明 |
|------|------|------|
| ATA | `port_operations.ata_dest_port` | 实际到港日期 |
| ETA | `port_operations.eta_dest_port` | 预计到港日期 |
| lastFreeDate | `process_port_operations.last_free_date` | 最晚提柜日 (免费期截止) |
| lastReturnDate | `process_empty_return.last_return_date` | 最晚还箱日 |
| dailyUnloadCapacity | `ext_warehouse.warehouse_daily_unload_capacity` | 仓库日产能 |
| dailyReturnCapacity | `trucking_company.daily_return_capacity` | 车队日还箱能力 |

### 3.4 车队选择优化

#### 场景 10: 多车队综合评分

**参数**:
- 仓库：UK-S005
- 港口：UKFXT
- 候选车队:
  - A 车队：成本£100, 能力充足，合作关系 3 年
  - B 车队：成本£120, 能力充足，合作关系 5 年
  - C 车队：成本£90, 能力紧张，合作关系 1 年

**评分逻辑**:
```typescript
// 代码位置：intelligentScheduling.service.ts:L1772-L1990

Step 1: 成本评分 (最低成本=100 分)
minCost = £90, maxCost = £120
A 车队：(120-100)/(120-90) * 100 = 66.7 分
B 车队：(120-120)/(120-90) * 100 = 0 分
C 车队：(120-90)/(120-90) * 100 = 100 分

Step 2: 能力评分
A 车队：有产能 → 100 分
B 车队：有产能 → 100 分
C 车队：能力紧张 → 50 分

Step 3: 关系评分
A 车队：3 年 → 60 分
B 车队：5 年 → 100 分
C 车队：1 年 → 20 分

Step 4: 综合评分 (假设权重：成本 40%, 能力 40%, 关系 20%)
A 车队：66.7*0.4 + 100*0.4 + 60*0.2 = 78.7 分
B 车队：0*0.4 + 100*0.4 + 100*0.2 = 60.0 分
C 车队：100*0.4 + 50*0.4 + 20*0.2 = 64.0 分

结果：选择 A 车队 (综合得分最高)
```

**结果**:
| 车队 | 成本分 | 能力分 | 关系分 | 总分 | 排名 |
|------|--------|--------|--------|------|------|
| A 车队 | 66.7 | 100 | 60 | **78.7** | 🥇 |
| C 车队 | 100 | 50 | 20 | 64.0 | 🥈 |
| B 车队 | 0 | 100 | 100 | 60.0 | 🥉 |

## 四、对比汇总表

### 4.1 卸柜模式对比

| 维度 | Live load | Drop off |
|------|-----------|----------|
| **提→送关系** | 提=送 (同日) | 提 < 送 |
| **送→卸关系** | 送=卸 | 送=卸 |
| **卸→还关系** | 还=卸 (同日) | 还=卸 +1 |
| **日期链** | 提=送=卸=还 | 提 < 送=卸 < 还 |
| **适用场景** | 直接送仓 | 堆场中转 |
| **车队要求** | 无需堆场 | 需要有堆场 |

---

### 4.2 排产模式对比

| 维度 | 预览排产 | 单柜优化 | 批量排产 |
|------|----------|----------|----------|
| **dryRun** | true | false | false |
| **数据保存** | ❌ 不保存 | ❌ 可选保存 | ✅ 保存 |
| **处理数量** | 批量 | 单个 | 批量 |
| **用途** | 方案审查 | 成本优化 | 正式执行 |
| **扣减产能** | ❌ 否 | ❌ 否 | ✅ 是 |

---

### 4.3 日期约束对比

| 约束类型 | 约束条件 | 违反处理 |
|----------|----------|----------|
| **提柜日≥today** | 不能是过去日期 | 顺延到明天 |
| **提柜日≤lastFreeDate** | 不能超过免费期 | 截断到 lastFreeDate |
| **卸柜日≥提柜日** | 卸柜不能在提柜前 | 从提柜日起查产能 |
| **还箱日≤lastReturnDate** | 不能超过最晚还箱日 | 停止查找，返回 null |

## 六、经验教训

### ✅ 成功经验

1. **以卸柜日为锚点的设计**
   - ✅ 所有日期围绕卸柜日推导，逻辑清晰
   - ✅ 便于调整和优化（只改锚点，其他自动重算）

2. **产能约束检查**
   - ✅ 仓库日产能和车队还箱能力独立管理
   - ✅ 自动顺延到最早可用日期，避免人工干预

3. **三种排产模式分离**
   - ✅ 预览、优化、批量各司其职
   - ✅ dryRun 参数控制是否保存，灵活安全

4. **车队综合评分机制**
   - ✅ 不仅考虑成本，还考虑能力和关系
   - ✅ 多维度平衡，避免单一指标偏差

---

### 📝 改进建议

1. **日期链验证增强**
   - 📝 当前只检查基本顺序，建议增加业务规则验证
   - 📝 例如：提柜日与卸柜日间隔不能超过 N 天

2. **产能预警机制**
   - 📝 当仓库连续 N 天产能满时，提前预警
   - 📝 建议增加产能趋势分析

3. **车队评分权重可配置**
   - 📝 当前权重硬编码在代码中 (40%, 40%, 20%)
   - 📝 建议改为配置文件或数据库配置

4. **lastFreeDate 和 lastReturnDate 自动生成**
   - 📝 当前依赖滞港费服务预先计算
   - 📝 建议在排产时实时计算，避免数据不一致

## 七、附录：关键流程图

### 7.1 单柜排产流程

```
开始
  ↓
获取目的港操作记录
  ↓
计算清关日和提柜日
  ↓
检查提柜日是否过期？
  ├─ 是 → 顺延到明天
  └─ 否 → 继续
  ↓
检查提柜日 ≤ lastFreeDate?
  ├─ 否 → 截断到 lastFreeDate
  └─ 是 → 继续
  ↓
查找候选仓库
  ↓
从提柜日起查仓库产能
  ├─ 产能满 → 顺延下一天
  └─ 有产能 → 确定为卸柜日
  ↓
选择车队 (综合评分)
  ↓
计算送仓日和还箱日
  ↓
验证日期链
  ├─ 失败 → 返回错误
  └─ 成功 → 继续
  ↓
dryRun?
  ├─ true → 返回结果，不保存
  └─ false → 保存 DB，扣减产能
  ↓
结束
```
