# 飞驼数据接入方式解读

> 基于飞驼官方文档与产品说明，梳理飞驼数据的接入方式、订阅模式与推送机制。

---

## 一、图片解读摘要

### 1.1 图一：飞驼-船公司信息表（2.1.3.2.1）

**数据范围**：码头信息、北美铁路信息与船公司信息放在同一逻辑表内。

| 接入方式 | 订阅参数 | 机制 |
|:---------|:---------|:-----|
| **船公司订阅** | MBL No. + 船公司 | 订阅与查询接口一致；订阅后收到飞驼数据变更消息时**重新查询**船司信息 |
| **海铁联运订阅** | MBL Number + 集装箱号 + 船公司 | 同上；收到变更消息时**重新查询**海铁联运信息 |

**要点**：
- 订阅与查询**接口一致**，同一接口既用于订阅也用于查询
- 采用「订阅 → 变更通知 → 再查询」模式，非直接推送完整数据
- 支持 MBL Number 变更场景

**测试接口**：
- 中国码头+船公司：https://doc.prd.freightower.com/web/#/2/2
- 北美铁路：https://doc.prd.freightower.com/web/#/2/35

---

### 1.2 图二：飞驼异常信息推送 LMS（2.1.3.2.2）

**接入方式**：LMS 提供回调接口，飞驼主动推送。

| 步骤 | 说明 |
|:-----|:-----|
| 1 | LMS 提供回调接口（Webhook URL） |
| 2 | 飞驼调用回调接口，推送 **类型 2：时间变化的数据** 给 LMS |

**要点**：
- 飞驼**主动推送**异常与时间变更数据
- 需 LMS 暴露可被飞驼调用的 HTTP 接口
- 推送内容含异常类型、状态码等

**文档**：https://doc.prd.freightower.com/web/#/2/5

---

### 1.3 图三：飞驼-码头信息表（2.1.3.2.3）

**订阅方式**：集装箱号 + 港口（码头代码）

| 项目 | 说明 |
|:-----|:-----|
| **订阅与查询** | **接口不一致**（与船公司表不同） |
| **更新机制** | 定时任务自动更新已订阅码头信息 |
| **前置条件** | 货物已卸在码头 且 货物未提离码头；不满足则无码头信息 |
| **开始订阅时间** | MIN(目的港到达日期, 预计到港日 - 3 天) |
| **订阅后** | 开启查询 |

**当前订阅逻辑**（物流状态 ≠ 已还箱，且满足以下之一）：
- 预计到港日期为空 或 目的港到达日期为空
- 或 预计到港日期 ≤ 当前日期 + 3 天 或 目的港到达日期 ≤ 当前日期 + 3 天

**测试接口**：美国+加拿大+英国码头 https://doc.prd.freightower.com/web/#/2/43

## 二、飞驼官方 API 接入方式（doc.freightower.com）

### 2.1 集装箱综合跟踪（订阅+查询）

**接口**：`POST https://openapi.freightower.com/application/v1/query`

**认证**：`Authorization: Bearer <Token>`（需先调用获取 Token 接口）

**请求参数**（示例）：

| 参数 | 必填 | 说明 |
|:-----|:-----|:-----|
| billNo | 否 | 提单号、订舱号、港区分单号 |
| containerNo | 否 | 箱号；单号+箱号同传时以单号为准 |
| carrierCode | 条件 | 船公司代码（与 portCode 二选一） |
| portCode | 条件 | 港区代码（与 carrierCode 二选一，与 isExport 绑定） |
| isExport | 条件 | E 出口 / I 进口 |
| billCategory | 否 | BL 提单号 / BK 订舱号 |

**机制**：
- 首次调用即完成**订阅**，后续可重复调用**查询**
- 订阅与查询为**同一接口**
- 支持近百船公司及中国港区、海关全程跟踪

---

### 2.2 集装箱综合跟踪（增量+预警推送）

**机制**：飞驼向 LMS 提供的 URL **主动推送**变更与预警。

**配置流程**：
1. 使用「订阅+查询」接口完成订阅
2. 将回调 URL 提供给飞驼客服配置
3. 配置成功后接收已订阅运单的实时推送

**推送内容**：
- 订阅成功时：推送当前最新状态
- 跟踪过程中：有变化时推送最新事件（ETD/ETA、箱动态、异常预警）
- 运单完成后：停止推送

**签名验证**（可选）：
- 使用 HmacSHA1，headers：`x-ft-timestamp`、`x-ft-nonce`、`x-ft-client`、`x-ft-signature`

## 三、飞驼数据接入方式汇总

| 接入方式 | 触发方 | 数据流向 | 适用场景 |
|:---------|:-------|:---------|:---------|
| **1. 订阅+查询（拉取）** | LogiX 主动 | LogiX → 飞驼 API → LogiX | 按需查询、定时轮询、初始化数据 |
| **2. 异常/增量推送（推送）** | 飞驼主动 | 飞驼 → LogiX 回调 URL | 实时变更、异常预警 |
| **3. 码头信息（订阅+定时更新）** | 飞驼定时 | 飞驼定时更新已订阅数据 | 美国/加拿大/英国码头信息 |

## 四、LogiX 推荐接入策略

### 4.1 主链路：订阅+查询（拉取）

```
LogiX 后端
    │
    ├─ 1. 获取 Token (CLIENTID + SECRET)
    ├─ 2. POST /application/v1/query
    │      Body: { billNo, containerNo, carrierCode, ... }
    │      Header: Authorization: Bearer <Token>
    │
    └─ 3. 解析返回 → 写入 ext_container_status_events
              → 更新 process_port_operations 等核心字段
              → 触发状态重算
```

**使用场景**：
- 货柜详情页「同步飞驼」按钮
- 定时任务批量同步（已出运未还箱）
- Excel 导入后补充/校验数据

### 4.2 增强：异常推送（Webhook）

```
飞驼
    │ 数据变更 / 异常预警
    ▼
POST {LogiX 回调 URL}
    │
    ▼
LogiX Webhook 接口
    ├─ 验证签名（可选）
    ├─ 解析 payload
    ├─ 写入 ext_container_status_events
    ├─ 更新核心字段
    └─ 触发状态重算
```

**前置**：向飞驼提供回调 URL（如 `https://your-domain.com/api/v1/webhooks/feituo`）

### 4.3 码头信息（可选）

- 订阅参数：集装箱号 + 港口
- 订阅与查询**接口不一致**，需单独对接
- 适用：美国/加拿大/英国码头场景

| 图片/文档 | 飞驼产品模块 | LogiX 对接优先级 |
|:----------|:-------------|:-----------------|

## 六、实施顺序建议

1. **P0**：对接「集装箱综合跟踪（订阅+查询）」— 主数据源
2. **P1**：配置「异常/增量推送」回调 — 实时更新
3. **P2**：按需对接船公司信息表、码头信息表 — 扩展能力
