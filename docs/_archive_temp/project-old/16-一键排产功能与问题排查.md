# 一键排产功能与问题排查

> Shipments 页「一键排产」功能说明、数据流与常见问题排查指南

---

## 一、功能入口与流程

### 1.1 入口位置

| 页面 | 路径 | 按钮 |
|------|------|------|
| Shipments 货柜列表 | `/#/shipments` | 顶部「一键排产」按钮 |

### 1.2 用户流程

1. 用户选择顶部日期范围（出运日期口径）
2. 点击「一键排产」
3. 确认弹窗（说明智能排产 6 步流程）
4. 调用 `POST /api/v1/scheduling/batch-schedule`
5. 成功后刷新列表

### 1.3 请求参数

```typescript
// Shipments.vue handleBatchSchedule
{
  country: undefined,           // 当前传 undefined，未使用国家筛选
  startDate: shipmentDateRange[0],  // 顶部日期范围
  endDate: shipmentDateRange[1]
}
```

---

## 二、后端数据流

```
POST /api/v1/scheduling/batch-schedule
  → scheduling.controller.batchSchedule
  → intelligentSchedulingService.batchSchedule
  → 1. getContainersToSchedule：查询待排产货柜
  → 2. 按需写回：await 滞港费计算，合并 lastPickupDateComputed 到容器，确保排产受最晚提柜日约束
  → 3. sortByClearanceDate：按清关可放行日排序（先到先得）
  → 4. 逐柜 scheduleSingleContainer
  → 5. updateContainerSchedule、decrementWarehouseOccupancy、decrementTruckingOccupancy
```

---

## 三、滞港费按需写回（必须受最晚提柜日约束）

排产前对本次待排产货柜 **await** `calculateForContainer`，将 `lastPickupDateComputed` 合并到容器对象，确保 `calculatePlannedPickupDate` 使用最新 last_free_date，计划提柜日 ≤ 最晚提柜日。并发 5。

---

## 四、待排产货柜条件

| 条件 | 说明 |
|------|------|
| `schedule_status` | `initial` 或 `issued`（不含 `dispatched`、`adjusted`） |
| 目的港 ATA/ETA | 必须有 `ata_dest_port` 或 `eta_dest_port` |
| 日期范围 | 若传 startDate/endDate，按目的港 ATA/ETA 过滤 |

**国家筛选**：`getContainersToSchedule` 已支持 `request.country`，按备货单关联的 `customer.country` 过滤。

---

## 五、单柜排产失败常见原因

| 失败原因 | 返回 message | 排查方向 |
|----------|--------------|----------|
| 无目的港记录 | 无目的港操作记录 | 检查 `process_port_operations` 是否有 `port_type='destination'` |
| 无到港日期 | 无到港日期（ATA/ETA），无法排产 | 检查目的港 `ata_dest_port`、`eta_dest_port`、`revised_eta_dest_port`、`eta_correction` |
| 无可用仓库 | 无可用仓库 | 检查 `dict_warehouses`、`dict_warehouse_trucking_mapping`、`dict_trucking_port_mapping` |
| 仓库产能不足 | 仓库产能不足，无法排产 | 检查 `ext_warehouse_daily_occupancy` 或仓库 `daily_unload_capacity` |
| 无可用车队 | 无可用车队 | 检查 `dict_warehouse_trucking_mapping`（仓库→车队）、`dict_trucking_port_mapping`（港口→车队）；若无数据需在字典管理中配置 |

## 七、排查步骤

### 7.1 ERR_CONNECTION_RESET（连接被重置）

`POST .../batch-schedule net::ERR_CONNECTION_RESET` 表示后端连接被重置，常见原因：

1. **后端进程崩溃**：查看后端终端/日志，是否有未捕获异常或 OOM
2. **数据库迁移未执行**：执行 `migrations/add_schedule_status.sql`、`migrations/add_last_free_date_mode.sql`
3. **请求超时**：排产柜数多时可能超 120 秒，可适当增加前端 timeout 或分批排产
4. **后端未启动**：确认 `http://localhost:3001` 可访问

### 7.2 一键排产无反应或报错

1. **检查 API 是否可达**
   - 确认后端 `http://localhost:3001` 已启动
   - 确认前端 `VITE_API_BASE_URL` 指向正确

2. **检查数据库迁移**
   ```sql
   -- biz_containers 需有 schedule_status
   SELECT column_name FROM information_schema.columns 
   WHERE table_name = 'biz_containers' AND column_name = 'schedule_status';
   -- 若无，执行 migrations/add_schedule_status.sql
   ```

3. **relation "ext_warehouse_daily_occupancy" does not exist**
   - 智能排产依赖仓库/车队日产能占用表，需执行资源占用表迁移：
   ```bash
   # 在项目根目录，按实际 DB 连接修改参数
   psql -h localhost -p 5432 -U logix_user -d logix_db -f migrations/create_resource_occupancy_tables.sql
   ```
   - 或使用 pgAdmin/DBeaver 等工具执行 `migrations/create_resource_occupancy_tables.sql`

4. **检查 process_port_operations.last_free_date_mode**
   ```sql
   -- 若 demurrage 写回失败，可能是缺少此列
   SELECT column_name FROM information_schema.columns 
   WHERE table_name = 'process_port_operations' AND column_name = 'last_free_date_mode';
   -- 若无，执行 migrations/add_last_free_date_mode.sql
   ```

### 7.3 排产成功数为 0

1. **确认有待排产货柜**
   ```sql
   SELECT c.container_number, c.schedule_status, po.ata_dest_port, po.eta_dest_port
   FROM biz_containers c
   LEFT JOIN process_port_operations po ON c.container_number = po.container_number AND po.port_type = 'destination'
   WHERE c.schedule_status IN ('initial', 'issued')
   AND (po.ata_dest_port IS NOT NULL OR po.eta_dest_port IS NOT NULL);
   ```

2. **检查日期范围**
   - 顶部日期范围会过滤 ATA/ETA，若范围过窄可能无结果

3. **检查映射数据**
   - `dict_warehouse_trucking_mapping`：仓库→车队
   - `dict_trucking_port_mapping`：港口→车队
   - 需有 `is_active=true` 且 `country` 与备货单销往国一致

### 7.4 部分柜排产失败

查看接口返回的 `results` 数组，每个元素的 `message` 即失败原因。
