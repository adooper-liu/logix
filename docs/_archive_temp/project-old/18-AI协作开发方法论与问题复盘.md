# AI 协作开发方法论与问题复盘

> 总结 LogiX 项目开发中的技能调用、需求理解、错误排查与改进方法，形成可复用的规范与流程。

---

## 一、技能体系与调用规范

### 1.1 项目技能（LogiX 专用）

| 技能 | 路径 | 触发场景 | 调用时机 |
|------|------|----------|----------|
| **logix-development** | `.cursor/skills/logix-development/SKILL.md` | 新功能、修 Bug、改代码 | **首选**，任何开发任务先读此技能 |
| **database-query** | `.cursor/skills/database-query/SKILL.md` | 写 SQL、查表结构、分析数据 | 涉及数据库时 |
| **excel-import-requirements** | `.cursor/skills/excel-import-requirements/SKILL.md` | Excel 导入、映射、类型转换 | 导入/导出相关 |
| **document-processing** | `.cursor/skills/document-processing/SKILL.md` | Excel/PDF 处理、表单提取 | 文档类任务 |
| **code-review** | `.cursor/skills/code-review/SKILL.md` | PR 审查、代码质量检查 | 提交前或用户要求审查时 |
| **commit-message** | `.cursor/skills/commit-message/SKILL.md` | 生成提交信息 | 用户要求写 commit 时 |

### 1.2 通用技能（Cursor 内置）

| 技能 | 触发场景 |
|------|----------|
| **create-rule** | 创建 .cursor/rules、AGENTS.md、项目约定 |
| **create-skill** | 新建 Agent Skill、编写 SKILL.md |
| **update-cursor-settings** | 修改 settings.json、主题、格式化 |

### 1.3 规则（Rules，始终生效）

| 规则 | 路径 | 作用 |
|------|------|------|
| **logix-project-map** | `.cursor/rules/logix-project-map.mdc` | 表结构、实体、API、业务逻辑速查 |
| **logix-development-standards** | `.cursor/rules/logix-development-standards.mdc` | 数据库优先、命名、日期口径、单一职责 |

### 1.4 技能调用流程（标准操作）

```
用户需求
    ↓
1. 判断任务类型（功能开发 / Bug 修复 / 数据迁移 / 文档 / 配置）
    ↓
2. 读取对应技能（开发→logix-development；数据库→database-query；导入→excel-import-requirements）
    ↓
3. 查阅 Rules（logix-project-map 查表/API；logix-development-standards 查规范）
    ↓
4. 定位代码/文档（Grep、SemanticSearch、Read）
    ↓
5. 执行修改（遵循开发顺序：SQL → 实体 → API → 前端）
    ↓
6. 验证（ReadLints、npm run lint / type-check）
```

## 二、需求理解：步骤与方法

### 2.1 需求解析四步法

| 步骤 | 动作 | 产出 |
|------|------|------|
| **1. 明确意图** | 用户说的是「要做什么」还是「解释/排查」？ | 避免过度编辑（仅回答 vs 动手改） |
| **2. 拆解范围** | 涉及哪些层：DB / 实体 / API / 前端？ | 影响分析清单 |
| **3. 查现有实现** | 搜索相关代码、文档、映射表 | 不重复造轮子、不破坏已有逻辑 |
| **4. 对齐规范** | 对照 logix-development-standards、命名规范 | 符合项目约定 |

### 2.2 典型需求类型与处理方式

| 需求类型 | 理解要点 | 典型动作 |
|----------|----------|----------|
| **新增字段/表** | 数据库优先，先 SQL 再实体再 API | 查 03_create_tables.sql、实体、Excel 映射 |
| **修改业务逻辑** | 找 Service 层，不直接改 Controller | SemanticSearch 找核心逻辑 |
| **修复报错** | 先看错误信息，再查依赖（表/迁移/映射） | relation does not exist → 执行迁移 |
| **优化体验** | 分步、确认、日志、错误提示 | 如分步排产 + 确认弹窗 |
| **术语澄清** | 仅回答，不编辑 | 如「提柜日被提升」= 日期下限为今天 |

### 2.3 需求理解检查清单

- [ ] 用户是要「改代码」还是「问问题」？
- [ ] 涉及哪些模块（智能排产 / 导入 / 统计 / 甘特图 / …）？
- [ ] 是否有文档可参考（11-project/*.md、05-state-machine/）？
- [ ] 命名是否与数据库一致（snake_case）？
- [ ] 日期口径是否统一（actual_ship_date / shipment_date）？

## 三、错误排查：场景与思路

### 3.1 错误分类与排查路径

| 错误类型 | 典型表现 | 排查思路 | 常见根因 |
|----------|----------|----------|----------|
| **数据库不存在** | `relation "xxx" does not exist` | 1. 查 migrations 是否有建表脚本<br>2. 执行对应迁移 | 迁移未执行 |
| **API 无响应/超时** | ERR_CONNECTION_RESET、长时间无返回 | 1. 后端日志是否有异常<br>2. 是否全量处理导致超时<br>3. 改为分批/分步 | 全量写回、未分页 |
| **业务逻辑错误** | 提<送、日期在过去、卸柜方式不对 | 1. 查文档（04-五节点、10-智能排柜）<br>2. 对照业务规则实现 | 写死逻辑、未按规则 |
| **映射/字段错误** | 无可用车队、无可用仓库 | 1. 查 dict_*_mapping 表是否有数据<br>2. 查 country/portCode 是否匹配<br>3. 加回退逻辑 | 映射表空、过滤过严 |
| **命名不一致** | dailyCapacity vs dailyUnloadCapacity | 1. 查实体定义<br>2. 查项目地图 | 字段名写错 |

### 3.2 智能排产专项排查（按错误 message）

| message | 排查顺序 | 解决动作 |
|---------|----------|----------|
| 无目的港操作记录 | process_port_operations 是否有 port_type='destination' | 补数据或修查询 |
| 无到港日期 | ata_dest_port、eta_dest_port、revised_eta、eta_correction | 补数据 |
| 无可用仓库 | dict_warehouses、dict_warehouse_trucking_mapping、dict_trucking_port_mapping | 配置映射 |
| 仓库产能不足 | ext_warehouse_daily_occupancy、daily_unload_capacity | 执行迁移、检查产能 |
| 无可用车队 | dict_warehouse_trucking_mapping、dict_trucking_port_mapping | 配置映射；加 trucking_port 回退 |
| relation "ext_xxx" does not exist | migrations/create_resource_occupancy_tables.sql | 执行迁移 |

### 3.3 排查工具与命令

```bash
# 验证表/列存在
psql -c "SELECT column_name FROM information_schema.columns WHERE table_name='biz_containers' AND column_name='schedule_status';"

# 执行迁移
psql -h localhost -U logix_user -d logix_db -f migrations/create_resource_occupancy_tables.sql

# 单柜模拟排产
cd backend && npx tsx scripts/simulate-schedule-one.ts TEMU1234567

# Lint / 类型检查
npm run lint && npm run type-check
```

## 四、项目开发问题复盘

### 4.1 重复发生的问题（浪费时间的根因）

| 问题 | 发生次数 | 根因 | 预防措施 |
|------|----------|------|----------|
| **迁移未执行** | 多次 | 新表/新列未在目标库执行 | 迁移清单 + 部署检查；文档 16 写清「需执行哪些迁移」 |
| **全量处理导致超时** | 1 次 | 滞港费写回对全部货柜执行 | 仅对本批（limit/skip）写回 |
| **映射表无数据** | 1 次 | 依赖 warehouse_trucking_mapping，表空 | 加 trucking_port_mapping 回退 |
| **字段名写错** | 1 次 | dailyCapacity vs dailyUnloadCapacity | 查实体再写；Rules 中强调 |
| **业务规则未查文档** | 多次 | 提/送/卸顺序、卸柜方式写死 | 开发前先读 04、10 文档 |

### 4.2 未一次性做对的问题

| 问题 | 第一次做法 | 正确做法 | 教训 |
|------|------------|----------|------|
| 送仓日计算 | 从卸柜日反推，导致送<提 | 从提柜日正推，保证提≤送≤卸 | 先理解业务时序，再写代码 |
| 卸柜方式 | 写死 Drop off | 按仓库当日产能：有→Live，无→Drop off | 查文档 2.3.1 再实现 |
| 清关日 | ATA 优先 | ETA 优先（用户明确要求） | 需求澄清优先于默认实现 |
| 计划日期下限 | 无 | 至少今天，历史 ATA 不排到过去 | 边界条件要显式处理 |
| 分步排产 | 无 | 每批 3 个 + 确认弹窗 | 长耗时操作要分步 + 可中断 |

### 4.3 做得更好的方法

1. **开发前必读**：04-五节点调度、10-智能排柜、12-国家概念、13-销往国家到排柜资源
2. **新增表/字段**：先写 migrations/*.sql，在文档中注明「需执行 xxx.sql」
3. **长耗时 API**：支持 limit/skip 或流式；前端分步 + 确认
4. **映射类逻辑**：主路径 + 回退路径（如 warehouse_trucking → trucking_port）
5. **边界条件**：日期下限、空值、表无数据，都要显式处理
6. **需求澄清**：术语（如「提升」）、优先级（ETA vs ATA）先确认再实现

## 五、标准操作流程（SOP）汇总

### 5.1 新增功能 SOP

```
1. 读 logix-development 技能
2. 查 logix-project-map 确认表/API
3. 数据库：03_create_tables.sql + migrations/xxx.sql
4. 实体：entities/ + index.ts + database 配置
5. 后端：Service → Controller → Routes
6. 前端：services → views → components
7. 文档：16/17 或对应模块文档补充排查说明
8. npm run lint / type-check
```

### 5.2 修复 Bug SOP

```
1. 看错误 message / 堆栈
2. 分类：DB / API / 业务逻辑 / 映射
3. 按 3.1 错误分类表定位
4. 修改 + 验证
5. 若为常见问题，更新 16 文档排查步骤
```

### 5.3 需求澄清 SOP

```
1. 用户是否要改代码？（避免过度编辑）
2. 术语是否明确？（如「提升」「直接送」）
3. 优先级是否明确？（ETA vs ATA）
4. 有文档可依则先读再答
```
