## 3. 重大架构决策

### 3.1 Express vs NestJS

**决策**: 采用 Express + TypeORM 为主，部分模块引入 NestJS 概念

**对比**:

| 维度       | Express    | NestJS       |
| ---------- | ---------- | ------------ |
| 学习曲线   | 低         | 中高         |
| 灵活性     | 高         | 中（约束多） |
| 团队熟悉度 | 熟悉       | 一般         |
| 快速开发   | 极快       | 快           |
| 大型项目   | 需自行组织 | 结构化       |
| 依赖注入   | 需手动实现 | 原生支持     |

**实施方案**:

- 主服务保持 Express 快速迭代
- 新模块可尝试 NestJS 风格（如 services 目录）
- 不强制全面迁移，避免推翻重来

---

### 3.2 TypeORM 命名策略

**问题**: TypeScript 使用 camelCase，数据库使用 snake_case

**解决方案**: 每个字段显式指定列名

示例：

```typescript
@Entity({ name: "biz_containers" })
export class Container {
  @PrimaryColumn({ name: "container_number", type: "varchar", length: 50 })
  containerNumber: string;

  @Column({ name: "container_type_code", type: "varchar", length: 20 })
  containerTypeCode: string;

  @Column({ name: "logistics_status", type: "varchar", length: 30 })
  logisticsStatus: string;
}
```

**为什么不使用自动映射**:

- 自动映射不完全可靠
- 显式指定更清晰、更易维护
- 避免查询时字段名错误

### 3.3 PostgreSQL + TimescaleDB

**使用场景**: 状态事件表 `ext_container_status_events`

**转换命令**:

```sql
SELECT create_hypertable('ext_container_status_events', 'occurred_at');
```

**已转换为 hypertable 的表**:

1. ext_container_status_events - 分区列：occurred_at
2. process_port_operations - 分区列：ata
3. process_sea_freight - 分区列：actual_loading_date
4. sys_data_change_log - 分区列：created_at

**好处**:

- 自动按时间分区
- 压缩旧数据（节省 90% 存储空间）
- 高效时间范围查询（性能提升 10-100x）

## 4. 技术债务与踩坑记录

### 4.1 TypeORM 实体定义

**问题**: 查询时字段名不匹配

**错误示例**:

```typescript
// ❌ 错误：未指定列名
@Entity()
export class Container {
  @PrimaryColumn()
  containerNumber: string; // 查询时变成 container_number
}
```

**正确示例**:

```typescript
// ✅ 正确：显式指定列名
@Entity({ name: "biz_containers" })
export class Container {
  @PrimaryColumn({ name: "container_number", type: "varchar", length: 50 })
  containerNumber: string;
}
```

---

### 4.2 日期处理

**问题**: 日期格式不统一

**解决方案**: 统一使用 Day.js，所有日期显示 ISO 8601 格式

示例：

```typescript
import dayjs from "dayjs";

const date = dayjs("2026-03-24");
const isoString = date.toISOString(); // 2026-03-24T00:00:00.000Z
```

---

### 4.3 跨域问题

**问题**: 前端开发环境跨域访问后端 API

**解决方案**: Express 配置 CORS

示例：

```typescript
import cors from "cors";
app.use(
  cors({
    origin: ["http://localhost:5173", "http://localhost:3000"],
    credentials: true,
  }),
);
```

---

### 4.4 错误处理

**问题**: 未捕获的异常导致服务崩溃

**解决方案**: 全局错误处理中间件

示例：

```typescript
// backend/src/middleware/error.middleware.ts
export function errorMiddleware(err: any, req: Request, res: Response, next: NextFunction) {
  console.error("Error:", err);
  res.status(err.status || 500).json({
    message: err.message || "Internal server error",
  });
}
```

## 总结

### 核心技术选型

1. **后端**: Express + TypeORM + PostgreSQL + TimescaleDB
2. **前端**: Vue 3 + Vite + Element Plus + Pinia
3. **微服务**: GraphQL + Socket.IO
4. **缓存**: Redis 7

### 关键架构决策

1. 保持 Express 快速迭代，逐步引入 NestJS 概念
2. TypeORM 显式指定列名，避免自动映射
3. 使用 TimescaleDB 优化时间序列数据
4. 微服务化，核心服务 + 专业微服务
