## 1. 整体项目结构

## 2. 多服务架构

### 2.1 核心服务

| 服务 | 端口 | 技术栈 | 职责 |
|------|------|--------|------|
| **后端 API** | 3001 | Express + TypeORM | 业务逻辑、数据库操作 |
| **前端应用** | 5173 | Vue 3 + Vite | 用户界面、交互逻辑 |
| **物流路径微服务** | 4000 | GraphQL | 路径规划、运费计算 |
| **PostgreSQL** | 5432 | TimescaleDB | 数据存储 |
| **Redis** | 6379 | Redis 7 | 缓存、会话管理 |
| **Prometheus** | 9090 | Prometheus | 性能监控 |
| **Grafana** | 3000 | Grafana | 监控仪表盘 |

### 2.2 服务间通信

```
前端 (5173)
  ↓ HTTP/REST
后端 API (3001)
  ↓ GraphQL
物流路径微服务 (4000)
  ↓ SQL
PostgreSQL (5432)
  ↓
Redis (6379) [缓存]
```

## 3. 数据库结构布局

### 3.1 核心表分类

#### 业务表 (biz_*)
- `biz_containers` - 货柜信息表
- `biz_replenishment_orders` - 备货单表
- `biz_customer_code_mappings` - 客户代码映射表

#### 过程表 (process_*)
- `process_sea_freight` - 海运信息表
- `process_port_operations` - 港口操作表
- `process_trucking_transports` - 拖车运输表
- `process_warehouse_operations` - 仓库操作表
- `process_empty_returns` - 还空箱表

#### 扩展表 (ext_*)
- `ext_feituo_import_tables` - 飞驼导入原始数据
- `ext_container_status_events` - 集装箱状态事件（TimescaleDB hypertable）
- `ext_feituo_places` - 飞驼地点信息
- `ext_feituo_vessels` - 飞驼船舶信息

#### 系统表 (sys_*)
- `sys_users` - 用户表
- `sys_data_change_log` - 数据变更日志（TimescaleDB hypertable）
- `sys_flow_instances` - 流程实例表
- `sys_flow_definitions` - 流程定义表

#### 字典表 (dict_*)
- `dict_countries` - 国家字典
- `dict_ports` - 港口字典
- `dict_trucking_companies` - 车队公司字典
- `dict_warehouses` - 仓库字典

### 3.2 关键表关系

```
biz_containers (货柜)
├── process_sea_freight (海运信息)
├── process_port_operations (港口操作)
├── process_trucking_transports (拖车运输)
├── process_warehouse_operations (仓库操作)
└── process_empty_returns (还空箱)

biz_replenishment_orders (备货单)
└── biz_containers (关联货柜)

ext_feituo_import_tables (飞驼原始数据)
└── biz_containers (通过 MBL Number 关联)
```

### 3.3 TimescaleDB hypertable

已转换为 hypertable 的表：
1. `ext_container_status_events` - 分区列：occurred_at
2. `process_port_operations` - 分区列：ata
3. `process_sea_freight` - 分区列：actual_loading_date
4. `sys_data_change_log` - 分区列：created_at

## 4. 文件命名规范

### 4.1 后端命名规范

**实体文件**:
```
{EntityName}.ts
示例：Container.ts, ReplenishmentOrder.ts
```

**控制器文件**:
```
{功能}.controller.ts
示例：container.controller.ts, scheduling.controller.ts
```

**服务文件**:
```
{功能}.service.ts
示例：container.service.ts, demurrage.service.ts
```

**路由文件**:
```
{功能}.routes.ts
示例：container.routes.ts, scheduling.routes.ts
```

**中间件文件**:
```
{功能}.middleware.ts
示例：error.middleware.ts, auth.middleware.ts
```

### 4.2 前端命名规范

**Vue 组件**:
```
PascalCase.vue
示例：ShipmentsList.vue, ContainerDetail.vue
```

**Composable 函数**:
```
use+PascalCase.ts
示例：useGanttLogic.ts, useContainerData.ts
```

**服务文件**:
```
{功能}.service.ts
示例：container.service.ts, scheduling.service.ts
```

**样式文件**:
```
kebab-case.scss
示例：shipments-list.scss, gantt-chart.scss
```

### 4.3 数据库脚本命名

```
{序号}_{描述}.sql
示例：001_add_scheduling_optimization_config.sql
add_cost_optimization_config.sql
```

## 5. 代码组织原则

### 5.1 分层架构

```
表现层 (Controllers/Routes)
    ↓
业务逻辑层 (Services)
    ↓
数据访问层 (Entities/Repositories)
    ↓
数据库 (PostgreSQL/TimescaleDB)
```

### 5.2 依赖注入

所有服务通过依赖注入方式提供，便于测试和维护。

示例：
```typescript
// backend/src/main.ts
const app = express();

// 注册服务
const containerService = new ContainerService();
const schedulingService = new SchedulingService();

// 注册控制器
app.use('/api/containers', new ContainerController(containerService));
app.use('/api/scheduling', new SchedulingController(schedulingService));
```

### 5.3 单一职责原则

每个文件只负责一个职责：
- Controller: 处理 HTTP 请求
- Service: 实现业务逻辑
- Entity: 定义数据结构
- Middleware: 处理横切关注点

### 5.4 约定优于配置

通过命名规范和目录结构减少配置文件：
- 文件名表明用途（`.controller.ts`, `.service.ts`）
- 目录名表明层级（`controllers/`, `services/`）
- 类名与文件名一致（`Container.ts` → `export class Container`）
