## 1. 数据库优先原则 🔥

### 1.1 核心原则

```
✅ 唯一基准：数据库表结构是唯一基准
✅ 开发顺序：SQL → 实体 → API → 前端 → 联调
✅ 禁止反向：代码对齐数据库，不反向改库补数据
```

### 1.2 开发流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                    数据库优先开发流程                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. 设计数据库                                                      │
│     └── 创建/修改 SQL 脚本                                          │
│          └── 03_create_tables.sql                                   │
│                                                                     │
│  2. 生成实体                                                        │
│     └── 使用 TypeORM 定义实体                                        │
│          └── entities/*.ts                                         │
│                                                                     │
│  3. 开发 API                                                        │
│     └── 控制器 → 服务 → 路由                                         │
│          └── controllers/, services/, routes/                       │
│                                                                     │
│  4. 前端对接                                                        │
│     └── API 调用 → 类型定义 → 组件实现                               │
│          └── services/, types/, views/                             │
│                                                                     │
│  5. 联调测试                                                        │
│     └── 前后端联调 → Bug 修复 → 完成                                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.3 禁止事项

```typescript
// ❌ 错误：先写代码，后改数据库
// 某开发者：直接在代码中新增字段，没有创建 migration

// ✅ 正确：先改数据库，再改代码
// 1. 创建 migration: ALTER TABLE xxx ADD COLUMN yyy;
// 2. 修改实体: @Column({ name: 'yyy' }) yyy: string
// 3. 修改 API: 添加字段映射
```

## 2. 命名规范 🔥

### 2.1 完整命名对照表

| 层级 | 格式 | 示例 |
|------|------|------|
| 数据库表名 | snake_case + 前缀 | `biz_containers` |
| 数据库字段 | snake_case | `container_number` |
| 实体属性 | camelCase | `containerNumber` |
| API 请求/响应 | snake_case | `{ container_number: '...' }` |
| 前端组件 | PascalCase | `ContainerDetails.vue` |
| 组合式函数 | use + PascalCase | `useContainerData()` |
| CSS 类名 | kebab-case | `.container-card` |

### 2.2 表名前缀含义

```typescript
dict_     // 字典表（基础数据）
           // dict_ports, dict_countries, dict_container_types

biz_      // 业务表（核心业务数据）
           // biz_containers, biz_replenishment_orders

process_  // 过程表（业务流程数据）
           // process_sea_freight, process_port_operations

ext_      // 扩展表（外部系统数据）
           // ext_feituo_status_events, ext_container_charges

flow_     // 流程定义表
           // flow_definitions, flow_instances
```

### 2.3 TypeORM 命名策略 🔥

**问题**: TypeORM 默认 camelCase，数据库使用 snake_case

**解决方案**:

```typescript
// 方案1：每个 @Column 显式指定 name（推荐）
@Entity({ name: 'biz_containers' })
export class Container {
  @PrimaryColumn({ name: 'container_number', type: 'varchar', length: 50 })
  containerNumber: string

  @Column({ name: 'container_type_code', type: 'varchar', length: 20 })
  containerTypeCode: string

  @Column({ name: 'logistics_status', type: 'varchar', length: 30 })
  logisticsStatus: string
}
```

## 3. 代码风格规范

### 3.1 后端风格

```
✅ 缩进：2 空格
✅ 引号：单引号
✅ 分号：必须使用
✅ 行宽：~120 字符
✅ 命名：camelCase（变量）、PascalCase（类）
```

### 3.2 前端风格

```
✅ 缩进：2 空格
✅ 引号：单引号
✅ 分号：不使用
✅ 行宽：~100 字符
✅ 命名：camelCase（变量）、PascalCase（组件）
```

### 3.3 禁止事项 🔥

```typescript
// ❌ console.log() 生产环境
console.log('debug')  // 禁止

// ✅ 使用日志
import { logger } from '../utils/logger'
logger.info('container loaded', { containerNumber })

// ❌ 硬编码中文
const statusText = '已出运'  // 禁止

// ✅ 使用 i18n
import { useI18n } from 'vue-i18n'
const { t } = useI18n()
const statusText = t('status.shipped')

// ❌ 硬编码色值
background-color: #409EFF  // 禁止

// ✅ 使用 SCSS 变量
background-color: $primary-color

// ❌ 魔法数字
if (days > 3)  // 禁止

// ✅ 使用常量
const ALERT_THRESHOLD_DAYS = 3
if (days > ALERT_THRESHOLD_DAYS)
```

## 4. 组件拆分规范 🔥

### 4.1 单一职责原则

```
✅ 一个文件只做一类事
✅ 组件职责清晰明确
✅ 避免巨型组件
```

### 4.2 组件大小指南

```
推荐：< 150 行/组件
适中：150-300 行/组件
警告：> 300 行/组件（需要重构）
```

### 4.3 组件层次设计

```
主组件（整合、协调）
  ↓
子组件（专用功能）
  ↓
微组件（原子功能）
```

## 5. 日期口径统一 🔥

### 5.1 核心原则

```
✅ 全项目统一：所有数据展示使用顶部日期范围筛选
✅ 后端口径：actual_ship_date（备货单）→ shipment_date（海运）
✅ 页面规范：必须有日期选择器，卡片/表格/图表共用同一套日期
```

### 5.2 日期字段映射

| 表 | 字段 | 用途 |
|---|------|------|
| biz_replenishment_orders | actual_ship_date | 实际出运日期 |
| process_sea_freight | shipment_date | 出运日期 |
| process_sea_freight | eta | 预计到港 |
| process_sea_freight | ata | 实际到港 |
| process_port_operations | ata_dest_port | 目的港 ATA |
| process_port_operations | eta_dest_port | 目的港 ETA |

### 5.3 时区处理 🔥

**问题**: JavaScript `new Date('2026-01-01')` 转换为带时区的 timestamp

**解决方案**:

```typescript
// 方案1：SQL层用 ::date 转换
const sql = `
  SELECT * FROM biz_containers 
  WHERE COALESCE(ata_dest_port, eta_dest_port)::date >= $1::date
    AND COALESCE(ata_dest_port, eta_dest_port)::date <= $2::date
`

// 方案2：前端传纯字符串
const dateStr = '2026-01-01'  // 不使用 new Date()
```

## 6. API设计规范

### 6.1 RESTful 规范

```typescript
GET    /api/v1/containers              // 列表
GET    /api/v1/containers/:id          // 详情
POST   /api/v1/containers              // 创建
PATCH  /api/v1/containers/:id          // 更新
DELETE /api/v1/containers/:id          // 删除
```

### 6.2 响应格式

```typescript
// 成功响应
{
  success: true,
  data: {
    // 业务数据
  },
  message?: string
}

// 错误响应
{
  success: false,
  error: {
    code: string,
    message: string,
    details?: any
  }
}
```

## 7. 数据导入规范 🔥

### 7.1 Excel 导入流程

```
上传文件 → 解析 Excel → 字段映射 → 数据验证 → 
按表分组 → 批量发送 → 后端事务处理 → 返回结果
```

### 7.2 字段映射配置

```typescript
interface FieldMapping {
  excelField: string      // Excel 列名
  table: string           // 数据库表名
  field: string          // 数据库字段名
  required: boolean       // 是否必填
  transform?: Function   // 转换函数
  aliases?: string[]     // 列名别名
}
```

### 7.3 多别名容错机制 🔥

```typescript
// 兼容多种列名格式
aliases: [
  '箱号 (集装箱号)',  // 半角括号无空格
  '箱号 (集装箱号)',  // 半角括号有空格
  '箱号（集装箱号）',  // 全角括号
  '箱号',             // 简化版本
]
```

### 7.4 子表自动关联

```typescript
// container_number 自动关联
// 1. 先导入主表（biz_containers）
// 2. 获取 container_number 映射
// 3. 用 container_number 关联子表
// 4. 批量导入子表
```
