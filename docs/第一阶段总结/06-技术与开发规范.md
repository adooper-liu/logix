## 1. 数据库优先原则

### 1.1 核心原则

1. **唯一基准**: 数据库表结构是唯一基准
2. **开发顺序**: SQL → 实体 → API → 前端 → 联调
3. **禁止反向**: 代码对齐数据库，不反向改库补数据

### 1.2 开发流程

步骤 1: 设计数据库
- 创建/修改 SQL 脚本
- 文件：migrations/*.sql

步骤 2: 生成实体
- 使用 TypeORM 定义实体
- 文件：backend/src/entities/*.ts

步骤 3: 开发 API
- 控制器 → 服务 → 路由
- 文件：controllers/, services/, routes/

步骤 4: 前端对接
- API 调用 → 类型定义 → 组件实现
- 文件：services/, types/, views/

步骤 5: 联调测试
- 前后端联调 → Bug 修复 → 完成

### 1.3 禁止事项

错误做法：先写代码，后改数据库
- 直接在代码中新增字段，没有创建 migration

正确做法：先改数据库，再改代码
1. 创建 migration: ALTER TABLE xxx ADD COLUMN yyy;
2. 修改实体：@Column({ name: 'yyy' }) yyy: string
3. 修改 API: 添加字段映射

## 2. 命名规范

### 2.1 完整命名对照表

| 层级 | 格式 | 示例 |
|------|------|------|
| 数据库表名 | snake_case + 前缀 | biz_containers |
| 数据库字段 | snake_case | container_number |
| 实体属性 | camelCase | containerNumber |
| API 请求/响应 | snake_case | { container_number: '...' } |
| 前端组件 | PascalCase | ContainerDetails.vue |
| 组合式函数 | use + PascalCase | useContainerData() |
| CSS 类名 | kebab-case | .container-card |

### 2.2 表名前缀含义

**dict_**: 字典表（基础数据）
- dict_ports, dict_countries, dict_container_types

**biz_**: 业务表（核心业务数据）
- biz_containers, biz_replenishment_orders

**process_**: 过程表（业务流程数据）
- process_sea_freight, process_port_operations

**ext_**: 扩展表（外部系统数据）
- ext_feituo_status_events, ext_container_charges

**flow_**: 流程定义表
- flow_definitions, flow_instances

### 2.3 TypeORM 命名策略

问题：TypeORM 默认 camelCase，数据库使用 snake_case

解决方案：每个 @Column 显式指定列名

示例：
```typescript
@Entity({ name: 'biz_containers' })
export class Container {
  @PrimaryColumn({ name: 'container_number', type: 'varchar', length: 50 })
  containerNumber: string

  @Column({ name: 'container_type_code', type: 'varchar', length: 20 })
  containerTypeCode: string

  @Column({ name: 'logistics_status', type: 'varchar', length: 30 })
  logisticsStatus: string
}
```

## 3. 代码风格规范

### 3.1 后端风格

- 缩进：2 空格
- 引号：单引号
- 分号：必须使用
- 行宽：~120 字符
- 命名：camelCase（变量）、PascalCase（类）

### 3.2 前端风格

- 缩进：2 空格
- 引号：单引号
- 分号：不使用
- 行宽：~100 字符
- 命名：camelCase（变量）、PascalCase（组件）

## 4. 组件拆分规范

### 4.1 组件大小控制

- 单一职责：一个组件只做一件事
- 行数限制：单个组件不超过 300 行
- 复杂度控制：嵌套层级不超过 3 层

### 4.2 组件分类

**通用组件**: 
- 位置：src/components/common/
- 特点：可复用、无业务逻辑
- 示例：UniversalImport, GanttChart

**业务组件**:
- 位置：src/components/business/ 或 views/
- 特点：特定业务场景
- 示例：ContainerDetails, SchedulingPanel

## 5. 日期口径统一

### 5.1 时区处理

所有日期统一使用 UTC，显示时转换为本地时间

示例：
```typescript
// 存储：UTC
const createdAt = new Date().toISOString(); // 2026-03-24T12:00:00.000Z

// 显示：本地时间
const localTime = dayjs(createdAt).local().format('YYYY-MM-DD HH:mm:ss');
```

### 5.2 日期格式

ISO 8601 格式：YYYY-MM-DDTHH:mm:ss.sssZ

显示格式：YYYY-MM-DD HH:mm:ss

## 6. API 设计规范

### 6.1 RESTful 风格

资源命名：
- GET /api/v1/containers - 获取货柜列表
- GET /api/v1/containers/:id - 获取单个货柜
- POST /api/v1/containers - 创建货柜
- PUT /api/v1/containers/:id - 更新货柜
- DELETE /api/v1/containers/:id - 删除货柜

### 6.2 响应格式

成功响应：
```json
{
  "data": { ... },
  "message": "Success"
}
```

错误响应：
```json
{
  "error": {
    "code": "CONTAINER_NOT_FOUND",
    "message": "货柜不存在"
  }
}
```

### 6.3 分页参数

```
GET /api/v1/containers?page=1&limit=20&sort=created_at&order=desc
```

响应：
```json
{
  "data": [...],
  "pagination": {
    "total": 100,
    "page": 1,
    "limit": 20,
    "totalPages": 5
  }
}
```

## 7. 数据导入规范

### 7.1 Excel 导入流程

1. 文件上传
2. 数据解析和验证
3. 主键判断（创建 or 更新）
4. 批量写入数据库
5. 返回导入结果

### 7.2 字段映射规则

支持列名变体：
- container_number, Container Number, CONTAINER NO., 箱号
- order_number, Order Number, ORDER NO., 订单号

全角半角兼容：
- （英文括号）和 () (中文括号) 视为相同

### 7.3 错误处理

数据验证失败时：
- 记录错误行号
- 说明错误原因
- 提供修正建议
- 允许部分导入成功

## 总结

### 核心规范

1. **数据库优先**: SQL → 实体 → API → 前端
2. **命名规范**: 严格遵循前后缀和大小写规则
3. **代码风格**: 后端用分号，前端不用分号
4. **组件拆分**: 单一职责，不超过 300 行
5. **日期口径**: UTC 存储，本地显示
6. **API 设计**: RESTful 风格，统一响应格式
7. **数据导入**: 支持列名变体，容错处理
