## 一、方案概述

### 1.1 核心洞察

**问题**: 
- 当前单柜优化无差别搜索前后 7 天（15 天），计算量大（45 次/柜）
- 不区分免费期内 vs 已超期，优化策略不合理
- 批量优化尚未实现

**解决方案**:
1. ✅ **分类处理**: 根据免费期状态分类（免费期内 vs 已超期）
2. ✅ **智能搜索**: 不同类别使用不同搜索方向
3. ✅ **统一架构**: 单柜和批量复用相同的底层逻辑
4. ✅ **跳过机制**: 无能力时保持原日期，避免强制优化

## 五、业务逻辑评审

### 5.1 分类策略 ✅

**分类标准**:
```typescript
remainingDays >= 0  → 'within_free_period' (免费期内)
remainingDays < 0   → 'overdue' (已超期)
```

**业务意义**:
- ✅ **免费期内**: 成本为 0，优化目标是灵活性（靠近免费期）
- ✅ **已超期**: 已经产生费用，优化目标是减少损失（尽早处理）

**评审意见**: ✅ **分类标准符合业务实际**

---

### 5.2 搜索策略 ✅

**已超期策略** (`OVERDUE`):
```typescript
{
  searchDirection: 'forward',     // ✅ 从今天开始往后找
  searchStartOffset: 0,           // ✅ 从今天开始
  searchEndOffset: 7,             // ✅ 往后 7 天
  prioritizeZeroCost: false,      // ✅ 已经不可能的，尽量减少损失
  allowSkipIfNoCapacity: false    // ✅ 超期的必须处理
}
```

**业务解释**: 已经超期，每天都在产生费用，应该尽早处理，不能跳过。

---

**免费期内策略** (`WITHIN_FREE_PERIOD`):
```typescript
{
  searchDirection: 'backward',    // ✅ 从免费期截止日往前找
  searchStartOffset: 0,           // ✅ 从免费期开始
  searchEndOffset: -7,            // ✅ 往前 7 天
  prioritizeZeroCost: true,       // ✅ 必须成本为 0
  allowSkipIfNoCapacity: true     // ✅ 无能力时保持原日期
}
```

**业务解释**: 还在免费期内，成本都是 0，应该尽量靠近免费期（灵活性最高），如果仓库紧张，可以让出能力给超期的。

**评审意见**: ✅ **搜索策略符合业务逻辑**

---

### 5.3 优先级策略 ✅

**批量优化优先级**:
```typescript
assignPriorities(withinFreePeriod, overdue) {
  // ✅ 已超期的优先级最高（1 ~ N）
  overdue.forEach((container, index) => {
    priorityQueue.push({ ...container, priority: index + 1 });
  });
  
  // ✅ 免费期内的优先级较低（N+1 ~ M）
  withinFreePeriod.forEach((container, index) => {
    priorityQueue.push({ ...container, priority: overdue.length + index + 1 });
  });
}
```

**业务意义**: 紧急优先，超期的优先处理，免费期内的可以等待。

**评审意见**: ✅ **优先级分配合理**
