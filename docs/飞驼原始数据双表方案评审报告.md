# 飞驼原始数据双表方案评审报告

**方案**：places + status 双原始表（完整审计）  
**版本**：1.0  
**日期**：2026-03-19  
**状态**：已评审通过

---

## 一、方案概述

### 1.1 设计目标

在飞驼 API 数据接入流程中，新增**原始数据存储层**，实现：

1. **完整审计**：API 返回的 places[] 和 status[] 在解析前完整落库
2. **数据追溯**：解析失败或业务表被覆盖时，可从原始表恢复
3. **问题排查**：支持按原始结构查询，便于定位映射/解析问题
4. **数据隔离**：原始数据与业务数据分离，职责清晰

### 1.2 方案选型

| 方案 | 说明 | 结论 |
|------|------|------|
| 方案 A | 仅新增 ext_feituo_places，status 依赖 ext_container_status_events.raw_data | 未采用 |
| **方案 B** | **places + status 双原始表** | **已采用** |

采用方案 B 的理由：

- 原始数据与业务数据完全分离，审计边界清晰
- 解析失败时原始数据不丢失
- 支持按 event_code、event_time、port_code 等原始字段查询
- 与现有 ext_feituo_import_*（Excel 原始）架构一致

## 二、完整表结构设计

### 2.1 ext_feituo_places（飞驼地点原始数据表）

**定位**：存储飞驼 API 返回的 `result.places[]` 原始数据，一条记录对应一个地点。

```sql
-- ============================================================
-- 飞驼地点原始数据表 (ext_feituo_places)
-- 定位：存储飞驼API places[] 原始数据，支持完整审计
-- ============================================================
CREATE TABLE IF NOT EXISTS ext_feituo_places (
    id SERIAL PRIMARY KEY,
    container_number VARCHAR(50) NOT NULL,
    bill_of_lading_number VARCHAR(50),
    
    -- 地点基本信息
    place_index INT NOT NULL,
    place_type INT NOT NULL,
    port_code VARCHAR(50),
    port_name VARCHAR(100),
    port_name_en VARCHAR(100),
    port_name_cn VARCHAR(100),
    name_origin VARCHAR(200),
    
    -- 原始时间字段（飞驼 API 返回）
    sta TIMESTAMPTZ,
    eta TIMESTAMPTZ,
    ata TIMESTAMPTZ,
    ata_ais TIMESTAMPTZ,
    atb_ais TIMESTAMPTZ,
    disc TIMESTAMPTZ,
    std TIMESTAMPTZ,
    etd TIMESTAMPTZ,
    atd TIMESTAMPTZ,
    atd_ais TIMESTAMPTZ,
    atbd_ais TIMESTAMPTZ,
    load TIMESTAMPTZ,
    
    -- 运输信息
    vessel_name VARCHAR(100),
    voyage_number VARCHAR(50),
    transport_mode_in VARCHAR(20),
    transport_mode_out VARCHAR(20),
    terminal_name VARCHAR(100),
    container_count_in VARCHAR(20),
    container_count_out VARCHAR(20),
    
    -- 坐标与时区
    latitude DECIMAL(10,6),
    longitude DECIMAL(10,6),
    port_timezone VARCHAR(10),
    firms_code VARCHAR(50),
    
    -- 元数据
    sync_request_id VARCHAR(100),
    data_source VARCHAR(50) DEFAULT 'API',
    raw_json JSONB NOT NULL,
    created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    
    FOREIGN KEY (container_number) REFERENCES biz_containers(container_number) ON DELETE CASCADE
);

CREATE INDEX idx_feituo_places_container ON ext_feituo_places(container_number);
CREATE INDEX idx_feituo_places_bol ON ext_feituo_places(bill_of_lading_number);
CREATE INDEX idx_feituo_places_type ON ext_feituo_places(place_type);
CREATE INDEX idx_feituo_places_created ON ext_feituo_places(created_at DESC);

COMMENT ON TABLE ext_feituo_places IS '飞驼API places[] 原始数据，支持完整审计与追溯';
COMMENT ON COLUMN ext_feituo_places.place_type IS '飞驼type: 1=起运港接货, 2=起运港码头, 4=中转港, 5=目的港';
```

**place_type 映射**（飞驼 API 真实返回值）：

| place_type | 含义 | port_type |
|------------|------|-----------|
| 1 | 起运港（接货/堆场） | origin |
| 2 | 起运港（码头装船） | origin |
| 4 | 中转港 | transit |
| 5 | 目的港 | destination |

### 2.2 ext_feituo_status_events（飞驼状态原始数据表）

**定位**：存储飞驼 API 返回的 `result.containers[].status[]` 原始数据，一条记录对应一个状态节点。

```sql
-- ============================================================
-- 飞驼状态节点原始数据表 (ext_feituo_status_events)
-- 定位：存储飞驼API status[] 原始数据，支持完整审计
-- ============================================================
CREATE TABLE IF NOT EXISTS ext_feituo_status_events (
    id SERIAL PRIMARY KEY,
    container_number VARCHAR(50) NOT NULL,
    bill_of_lading_number VARCHAR(50),
    
    -- 状态基本信息
    status_index INT NOT NULL,
    event_code VARCHAR(20) NOT NULL,
    description_cn VARCHAR(200),
    description_en VARCHAR(200),
    event_description_origin VARCHAR(500),
    
    -- 时间信息
    event_time TIMESTAMPTZ NOT NULL,
    is_estimated BOOLEAN DEFAULT false,
    port_timezone VARCHAR(10),
    
    -- 地点信息
    event_place VARCHAR(100),
    event_place_origin VARCHAR(200),
    port_code VARCHAR(50),
    terminal_name VARCHAR(100),
    
    -- 运输信息
    transport_mode VARCHAR(20),
    vessel_name VARCHAR(100),
    voyage_number VARCHAR(50),
    
    -- 关联
    related_place_index INT,
    source INT,
    firms_code VARCHAR(50),
    
    -- 扩展字段（部分状态码特有）
    bill_no VARCHAR(50),
    declaration_no VARCHAR(100),
    
    -- 元数据
    sync_request_id VARCHAR(100),
    data_source VARCHAR(50) DEFAULT 'API',
    raw_json JSONB NOT NULL,
    created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    
    FOREIGN KEY (container_number) REFERENCES biz_containers(container_number) ON DELETE CASCADE
);

CREATE INDEX idx_feituo_status_container ON ext_feituo_status_events(container_number);
CREATE INDEX idx_feituo_status_bol ON ext_feituo_status_events(bill_of_lading_number);
CREATE INDEX idx_feituo_status_code ON ext_feituo_status_events(event_code);
CREATE INDEX idx_feituo_status_time ON ext_feituo_status_events(event_time DESC);
CREATE INDEX idx_feituo_status_created ON ext_feituo_status_events(created_at DESC);

COMMENT ON TABLE ext_feituo_status_events IS '飞驼API status[] 原始数据，支持完整审计与追溯';
```

### 3.2 写入顺序

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 写入 ext_feituo_places | 按 place_index 顺序，先原始后解析 |
| 2 | 写入 ext_feituo_status_events | 按 status_index 顺序 |
| 3 | 解析 places → process_port_operations | 根据 place_type 映射 port_type |
| 4 | 解析 status → ext_container_status_events | 标准化 + 更新 process_port_operations 核心字段 |
| 5 | 汇总 receipt/delivery → process_sea_freight | 提单级汇总 |

### 3.3 与现有表的关系

| 数据来源 | 原始表 | 业务表 |
|----------|--------|--------|
| Excel 表一 | ext_feituo_import_table1 | process_sea_freight 等 |
| Excel 表二 | ext_feituo_import_table2 | process_port_operations, ext_container_status_events |
| **API places[]** | **ext_feituo_places** | process_port_operations |
| **API status[]** | **ext_feituo_status_events** | ext_container_status_events, process_port_operations |

## 四、设计决策说明

### 4.1 为何不建 bill_of_lading_number 的 FK

- API 同步时可能先有 container，process_sea_freight 尚未创建
- 原始表以「先落库、后关联」为原则，避免插入失败
- bill_of_lading_number 保留为普通字段，用于查询与关联，不做 FK

### 4.2 为何保留 raw_json

- 飞驼 API 可能新增字段，列式存储无法覆盖
- raw_json 保证原始响应完整可追溯
- 列式字段便于常用查询，raw_json 作为兜底

### 4.3 sync_request_id 用途

- 标识单次 API 请求/同步批次
- 支持按请求维度审计、重跑、回滚
- 可选实现，初期可为 NULL

### 4.4 与 ext_container_status_events 的职责

| 表 | 定位 | 写入时机 |
|----|------|----------|
| ext_feituo_status_events | API 原始，解析前 | API 返回后立即写入 |
| ext_container_status_events | 标准化状态事件 | 解析后写入，供物流路径等使用 |

两者为上下游关系，非替代关系。

## 五、实施要点

### 5.1 写入流程伪代码

```typescript
async function processFeituoAPIResponse(data: FeituoAPIResponse) {
  const { containerNo, billNo, places, containers } = data.result;
  const syncRequestId = generateSyncRequestId();
  
  // 1. 写入 places 原始数据
  for (let i = 0; i < places.length; i++) {
    await extFeituoPlacesRepo.save({
      container_number: containerNo,
      bill_of_lading_number: billNo,
      place_index: i,
      place_type: places[i].type,
      port_code: places[i].code,
      port_name: places[i].name,
      // ... 其他列式字段
      raw_json: places[i],
      sync_request_id: syncRequestId,
      data_source: 'API',
    });
  }
  
  // 2. 写入 status 原始数据
  const statusList = containers?.[0]?.status ?? [];
  for (let i = 0; i < statusList.length; i++) {
    const s = statusList[i];
    await extFeituoStatusEventsRepo.save({
      container_number: containerNo,
      bill_of_lading_number: billNo,
      status_index: i,
      event_code: s.eventCode,
      event_time: s.eventTime,
      is_estimated: s.isEsti === 'Y',
      // ... 其他列式字段
      raw_json: s,
      sync_request_id: syncRequestId,
      data_source: 'API',
    });
  }
  
  // 3. 解析并写入业务表
  await syncPlacesToPortOperations(containerNo, places);
  await syncStatusToContainerEvents(containerNo, statusList);
  await syncReceiptDeliveryToSeaFreight(containerNo, data.result);
}
```

### 5.2 数据保留策略建议

| 表 | 建议保留期 | 说明 |
|----|------------|------|
| ext_feituo_places | 1 年 | 原始数据，可归档 |
| ext_feituo_status_events | 1 年 | 同上 |
| ext_container_status_events | 长期 | 业务使用 |
| process_port_operations | 长期 | 业务使用 |

### 5.3 依赖与前置条件

- biz_containers 中需存在对应 container_number
- 若 process_sea_freight 尚未创建，需在解析阶段创建或更新
- 建议在 API 适配层统一实现「先写原始表，再解析」的流程

## 七、风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| 存储增长 | 制定保留策略，定期归档或清理 |
