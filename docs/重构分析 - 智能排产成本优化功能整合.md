## 🔍 二、根本原因

### 2.1 缺乏统一的参数提取逻辑

**核心问题**：没有遵循 **SKILL 原则** 中的 **Single Source of Truth（单一事实来源）**

- ❌ 两个组件各自实现参数提取逻辑
- ❌ 没有统一的工具函数或 Hook
- ❌ 字段映射关系不明确（warehouseId vs warehouseCode）

### 2.2 对排产数据的理解不足

**数据流追踪**：

1. **后端排产** (`backend/src/services/intelligentScheduling.service.ts` 第 674 行):

```typescript
  truckingCompanyId: truckingCompany.companyCode,
```

2. **前端显示** (`SchedulingPreviewModal.vue` 第 96 行):

```vue
<el-table-column prop="warehouseName" label="仓库" />
<!-- 显示的是 warehouseName，不是 warehouseCode -->
```

3. **优化功能提取** (修复前):


### 2.3 缺乏复用意识

用户指出的关键问题：

> "如果这个仓库不存在，那么预览排柜的数据从哪日在哪里获取的，说明优化又是自己创造了一套轮子，没有复用或者采用首轮排柜获取数据的逻辑"

**核心洞察**：

- ✅ 排产功能已经正确获取了仓库编码（`warehouse.warehouseCode`）
- ❌ 优化功能没有复用排产的数据获取逻辑
- ❌ 优化功能尝试从显示数据（warehouseName）中提取
- ❌ 导致数据不一致和优化失败

## ✅ 三、重构方案

### 3.1 设计原则

遵循 **SKILL 原则**：

1. **Single Source of Truth（单一事实来源）**: 统一从 `plannedData` 获取参数
2. **Keep It Simple（保持简单）**: 提取公共逻辑到 Composable
3. **Leverage Existing（利用现有）**: 复用排产数据，不重新创造轮子

### 3.2 架构设计

```
┌─────────────────────────────────────┐
│   useCostOptimization Hook          │  ← 新增：统一优化逻辑
├─────────────────────────────────────┤
│ • extractOptimizeParams()           │  ← 参数提取（单一事实源）
│ • executeOptimization()             │  ← 执行优化（统一 API 调用）
│ • optimizing (state)                │  ← 加载状态
│ • optimizationResult (state)        │  ← 优化结果
└─────────────────────────────────────┘
           ↑               ↑
           │               │
┌──────────┴──────┐ ┌─────┴──────────────┐
│ SchedulingVisual │ │ SchedulingPreview  │
│ (单柜优化)       │ │ Modal (批量优化)   │
└─────────────────┘ └────────────────────┘
```

### 3.3 核心实现

#### 文件 1: `frontend/src/composables/useCostOptimization.ts`

```typescript
/**
 * 智能排产成本优化组合式函数
 * 遵循 SKILL 原则：Single Source of Truth
 */

import { ref } from "vue";
import { costOptimizerService, type Alternative } from "@/services/costOptimizer.service";
import type { ScheduleResult } from "@/types/scheduling";

interface OptimizeParams {
  containerNumbers: string[];
  warehouseCode: string;
  truckingCompanyId: string;
  basePickupDate: string;
}

interface OptimizeResult {
  optimizedCount: number;
  totalSavings: number;
  alternatives: Alternative[];
}

/**
 * 从排产结果中提取优化参数（单一事实来源）
 */
export function extractOptimizeParams(scheduleResults: ScheduleResult[] | ScheduleResult): OptimizeParams | null {
  const results = Array.isArray(scheduleResults) ? scheduleResults : [scheduleResults];

  if (!results || results.length === 0) {
    return null;
  }

  const successfulResults = results.filter((r) => r.success && r.plannedData);
  if (successfulResults.length === 0) {
    return null;
  }

  const containerNumbers = successfulResults.map((r) => r.containerNumber);

  // ✅ 关键修复：plannedData.warehouseId 存储的就是 warehouseCode
  const firstPlannedData = successfulResults[0].plannedData!;
  const warehouseCode =
    firstPlannedData.warehouseId || // ✅ 首选
    firstPlannedData.warehouseCode || // 备选
    firstPlannedData.warehouseName?.split(" ")[0] || // 降级
    "";

  const truckingCompanyId = firstPlannedData.truckingCompanyId || firstPlannedData.truckingCompany || "";

  const basePickupDate = firstPlannedData.plannedPickupDate || "";

  // 验证必要参数
  if (!warehouseCode || !truckingCompanyId || !basePickupDate) {
    return null;
  }

  return {
    containerNumbers,
    warehouseCode,
    truckingCompanyId,
    basePickupDate,
  };
}

/**
 * 智能成本优化 Hook
 */
export function useCostOptimization(autoExtractParams = true) {
  const optimizing = ref(false);
  const optimizationResult = ref<OptimizeResult | null>(null);

  async function executeOptimization(paramsOrResults: OptimizeParams | ScheduleResult[] | ScheduleResult): Promise<OptimizeResult> {
    optimizing.value = true;
    optimizationResult.value = null;

    try {
      // 自动提取参数或直接使用传入的参数
      let params: OptimizeParams;
      if (autoExtractParams) {
        const extracted = extractOptimizeParams(paramsOrResults as any);
        if (!extracted) {
          throw new Error("无法提取优化参数");
        }
        params = extracted;
      } else {
        params = paramsOrResults as OptimizeParams;
      }

      // 调用后端优化 API
      const result = await costOptimizerService.suggestOptimalUnloadDate({
        containers: params.containerNumbers,
        warehouseCode: params.warehouseCode,
        truckingCompanyId: params.truckingCompanyId,
        basePickupDate: params.basePickupDate,
      });

      optimizationResult.value = {
        optimizedCount: result.alternatives.length,
        totalSavings: result.savings,
        alternatives: result.alternatives,
      };

      return optimizationResult.value;
    } catch (error: any) {
      console.error("[useCostOptimization] Optimization failed:", error);
      throw error;
    } finally {
      optimizing.value = false;
    }
  }

  function reset() {
    optimizing.value = false;
    optimizationResult.value = null;
  }

  return {
    optimizing,
    optimizationResult,
    executeOptimization,
    reset,
    extractOptimizeParams,
  };
}
```

## 📚 八、学习要点

### 8.1 识别重复代码的信号

1. **相似的参数提取**: 两个地方都在从 A 提取 B
2. **相同的 API 调用**: 调用同一个后端接口
3. **重复的状态管理**: 定义相同的 ref/reactive
4. **类似的结果处理**: 处理相同的返回数据结构

### 8.2 何时提取 Composable

当满足以下条件时，应该提取 Composable：

1. **跨组件复用**: 多个组件需要相同逻辑
2. **状态 + 逻辑**: 不仅有逻辑，还有状态管理
3. **生命周期相关**: 需要 onMounted/onUnmounted 等
4. **可测试性**: 希望独立测试业务逻辑

### 8.3 如何设计好的 Composable

1. **清晰的职责**: 一个 Hook 只做一件事
2. **明确的输入输出**: 参数和返回值类型清晰
3. **状态隔离**: 每个实例独立状态
