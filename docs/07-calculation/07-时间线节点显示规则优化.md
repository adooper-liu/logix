# 时间线节点显示规则优化

## 📋 需求背景

时间线包含 9 个节点：出运 → ETA → ATA → 卸船 → 最晚提柜 → 实际提柜 → 最晚还箱 → 实际还箱 → 当前

原有的显示逻辑将所有节点混为一谈，导致：
- 计划节点（如 ETA、最晚提柜、最晚还箱）与实际节点（如 ATA、实际提柜）显示规则不清晰
- 用户无法直观区分"计划"与"实际"的差异
- 超期提示不够准确

---

## ✅ 优化方案

### 节点分类（按业务性质）

#### **① 实际业务节点**（只显示历时）
- 出运
- ATA（实际到港）
- 卸船
- 实际提柜
- 实际还箱

**特点**：
- 业务已真实发生
- 日期是确定的历史事实
- 只显示"历时"（蓝色），表示从该节点到现在的天数

#### **② 计划与预警节点**（动态显示）
- ETA（预计到港）
- 修正 ETA
- 最晚提柜（LFD）
- 最晚还箱（LRD）

**特点**：
- 计划日期，用于参照对比
- 需要预警功能（倒计时/超期）
- 显示规则取决于后续实际节点是否已发生

---

## 🎯 显示规则

### 规则 1：实际业务节点

```
实际业务节点 → 总是显示「历时」（蓝色）
```

**示例**：
- 实际提柜（2026/03/28）→ 历时 2 天
- ATA（2026/03/25）→ 历时 5 天

### 规则 2：计划与预警节点

```
如果后续实际节点已发生：
  → 计算差值：实际日期 - 计划日期
  → 差值 > 0（实际晚于计划）：显示「实际超期：X 天」（橙色）
  → 差值 < 0（实际早于计划）：显示「实际提前：X 天」（绿色）
  → 差值 = 0（当天完成）：显示「按时完成」（蓝色）
否则：
  → 日期在未来：显示「倒计时」（绿色）
  → 日期在过去：显示「超期」（红色）
```

**示例场景**：

#### 场景 1：最晚提柜
- **情况 A**（实际提柜已发生）：
  - 最晚提柜：2026/03/26
  - 实际提柜：2026/03/28（已发生）
  - 显示：**历时 2 天**（实际提柜日 - 最晚提柜日）

- **情况 B**（实际提柜未发生）：
  - 最晚提柜：2026/03/26
  - 实际提柜：无
  - 今天：2026/03/30
  - 显示：**已超期 4 天**（最晚提柜日已过，但实际提柜未发生）

#### 场景 2：ETA
- **情况 A**（后续实际节点已发生）：
  - ETA: 2026/03/20
  - ATA: 2026/03/22（已发生）
  - 显示：**历时 2 天**（ATA - ETA）

- **情况 B**（所有实际节点都未发生）：
  - ETA: 2026/04/05
  - 今天：2026/03/30
  - 显示：**倒计时 6 天**

## 🔧 技术实现

### 修改文件 1: `KeyDatesTimeline.vue`

**函数**：`getEffectiveHasNextNode`

**逻辑**：
```typescript
const getEffectiveHasNextNode = (
  event: TimelineEvent,
  index: number,
  allEvents: TimelineEvent[]
): boolean => {
  // ✅ 实际业务节点：总是显示历时
  const actualEventLabels = ['出运', 'ATA', '卸船', '实际提柜', '实际还箱']
  if (actualEventLabels.includes(event.label)) {
    return true
  }
  
  // ✅ 计划与预警节点：检查后续实际节点是否已发生
  
  // 1. 最晚提柜：检查实际提柜是否已发生
  if (event.label === '最晚提柜') {
    const pickupDate = dates.value?.pickupDateActual
    if (!pickupDate) return false
    const pickupDateObj = typeof pickupDate === 'string' ? new Date(pickupDate) : pickupDate
    return pickupDateObj < new Date()
  }
  
  // 2. 最晚还箱：检查实际还箱是否已发生
  if (event.label === '最晚还箱') {
    const returnTime = dates.value?.returnTime
    if (!returnTime) return false
    const returnTimeObj = typeof returnTime === 'string' ? new Date(returnTime) : returnTime
    return returnTimeObj < new Date()
  }
  
  // 3. ETA/修正 ETA：检查所有后续实际业务节点是否有已发生的
  if (event.label === 'ETA' || event.label === '修正 ETA') {
    for (let i = index + 1; i < allEvents.length; i++) {
      const nextEvent = allEvents[i]
      // 只检查实际业务节点
      if (actualEventLabels.includes(nextEvent.label)) {
        // 如果该节点日期已发生（早于当前时间）
        if (nextEvent.date < new Date()) {
          return true
        }
      }
    }
    return false
  }
  
  return false
}
```

### 修改文件 2: `DurationDisplay.vue`

**函数**：`displayType`

**逻辑**：
```typescript
const displayType = computed(() => {
  // 显式指定模式
  if (props.mode !== 'auto') {
    return props.mode as 'countdown' | 'elapsed' | 'overdue'
  }

  // ✅ 有后一节点：显示历时（实际 - 计划）
  if (props.hasNextNode) {
    return 'elapsed'
  }

  // ✅ 无后一节点：根据时间判断
  const time = overdueTime.value
  if (!time) return 'elapsed'
  if (time < 0) return 'countdown'
  
  // 日期在过去：显示超期
  return 'overdue'
})
```

## 🎨 颜色规范

| 显示类型 | 颜色 | 含义 |
|---------|------|------|
| 历时 | 蓝色（info） | 实际业务节点或计划节点后已有实际发生 |
| 倒计时 | 绿色（success） | 计划节点在未来，时间充裕 |
| 超期 | 红色（danger） | 计划节点在过去，但后续实际未发生 |

## 📝 核心原则

1. **实际业务节点**：总是显示历时，因为业务已真实发生
2. **计划预警节点**：根据后续实际节点是否发生来决定显示方式
3. **计划服务于实际**：计划节点仅作为参照，不参与真实业务路径
4. **清晰的视觉区分**：蓝色（历时）、绿色（倒计时）、红色（超期）
