## 问题 3: 费用计算相同 - 根本原因分析

### 问题现象

**日志输出**（2026-03-27 18:17 成本优化调用）:

``log
[Demurrage] Temporarily overriding planned dates for ECMU5397691:
  originalPickup: 2026-03-27
  newPickup: 2026-03-31  ✅ 覆盖成功

[Demurrage] Cost calculation for ECMU5397691:
  chargeName: "Storage Charge"
  rangeEnd: "2026-03-27T00:00:00.000Z"  ❌ 应该是 2026-03-31
  rangeStart: "2026-02-11T00:00:00.000Z"
  freeDays: 7
  chargeDays: 38  ❌ 所有日期的 chargeDays 都是 38
  amount: 2200    ❌ 所有日期的费用都是 $2,200
```

### 你的疑问

> 而是在定时任务执行时，还没有传入优化的日期！这个是什么意思？

**两种不同的计算场景**：

#### 场景 1: 定时任务（实际费用计算）
- **触发时间**: 每 5 分钟自动执行
- **目的**: 计算当前**实际**产生的费用
- **参数来源**: 数据库中原始的 `plannedPickupDate`
- **计算逻辑**: 
  ```typescript
  rangeEnd = maxDate(plannedPickupDate, today)
  ```
- **这是正确的**！因为要计算到当前实际日期的费用

#### 场景 2: 成本优化功能（预测费用计算）
- **触发时间**: 用户点击"成本优化"按钮
- **目的**: 预测**不同候选日期**的费用
- **参数来源**: 用户传入的候选日期（如 2026-03-27, 2026-03-30, 2026-03-31）
- **问题逻辑**: 
  ```typescript
  // calculateTotalCostWithPlannedDates 中覆盖
  params.calculationDates.plannedPickupDate = new Date('2026-03-31');
  
  // 但 calculateForContainer 中又重新从数据库获取
  const params = await this.getContainerMatchParams(containerNumber);
  // ❌ 这里重新获取了数据库中的原始值，覆盖失效！
  
  const plannedPickupDate = params.calculationDates.plannedPickupDate;
  // ❌ 还是数据库中的 2026-03-27，不是覆盖后的 2026-03-31
  
  rangeEnd = maxDate(plannedPickupDate, today);
  // = maxDate(2026-03-27, 2026-03-27) = 2026-03-27
  ```

### 根本原因

**`calculateForContainer` 方法在内部重新从数据库获取了参数**，导致 `calculateTotalCostWithPlannedDates` 的日期覆盖**完全失效**！

**调用链**：
```
calculateTotalCostWithPlannedDates
  ↓ 覆盖 params.calculationDates.plannedPickupDate
  ↓ 调用 calculateForContainer
    ↓ 重新从数据库获取 params（覆盖的值丢失！）
    ↓ 使用数据库中的原始 plannedPickupDate
    ↓ 计算费用（所有日期都一样）
```

---

## 修复方案

### 核心思路

让 `calculateForContainer` 方法接受一个可选的 `paramsOverride` 参数，如果传入了覆盖的参数，就使用覆盖的值，否则从数据库获取。

### 修改 1: `calculateForContainer` 方法签名

**文件**: `backend/src/services/demurrage.service.ts`

**修改位置**: 第 1302-1324 行

```
// ✅ 修复前
async calculateForContainer(
  containerNumber: string,
  options?: { freeDateWriteMode?: 'batch' | 'none' }
): Promise<...> {
  const params = await this.getContainerMatchParams(containerNumber);
  // ...
}

// ✅ 修复后
async calculateForContainer(
  containerNumber: string,
  options?: { 
    freeDateWriteMode?: 'batch' | 'none';
    paramsOverride?: any; // ContainerMatchParams 类型
  }
): Promise<...> {
  // ✅ 关键修复：如果传入了 paramsOverride，使用覆盖的参数，否则从数据库获取
  const params = options?.paramsOverride || await this.getContainerMatchParams(containerNumber);
  // ...
}
```

### 修改 2: `calculateTotalCostWithPlannedDates` 方法调用

**文件**: `backend/src/services/demurrage.service.ts`

**修改位置**: 第 3564-3569 行

```
// ✅ 修复前
const demurrageResult = await this.calculateForContainer(containerNumber, {
  freeDateWriteMode: 'none'
});

// ✅ 修复后
// 3. ✅ 关键修复：调用 calculateForContainer 时传入覆盖后的 params
const demurrageResult = await this.calculateForContainer(containerNumber, {
  freeDateWriteMode: 'none',
  paramsOverride: params  // 传入覆盖后的参数
});
```

## 问题 4: forward/backward 搜索定义

### forward search (向前搜索)
**定义**: 从今天开始，向**未来**方向搜索

**示例**:
```
today = 2026-03-27
basePickupDate = 2026-03-28
lastFreeDate = 2026-03-27 (已超期)

forward search 生成的日期:
  2026-03-27 (今天)
  2026-03-28 (明天)
  2026-03-29
  2026-03-30
  ...
```

**应用场景**: 原计划已超期（`lastFreeDate < basePickupDate`），需要尽快提柜，从今天开始往后找最早的可行日期

### backward search (向后搜索)
**定义**: 从免费期截止日开始，向**过去**方向搜索

**示例**:
```
today = 2026-03-27
basePickupDate = 2026-03-28
lastFreeDate = 2026-03-30 (免费期内)

backward search 生成的日期:
  2026-03-30 (免费期截止日)
  2026-03-29
  2026-03-28 (原计划日期)
  // 2026-03-27 (今天) - 被过滤，因为 < basePickupDate
```

**应用场景**: 原计划在免费期内，可以灵活安排，从免费期截止日往前找最优日期
