## 问题与解决方案

### 问题 ①：计划提柜日是过去日期

**现象**：

- 当前日期：2026-03-26
- 排产显示：计划提柜日 = 2026-03-25（昨天）
- 问题：排产计划从一开始就过期了

**解决方案**：

- ✅ 新增 `eta_buffer_days` 配置（默认 2 天）
- ✅ 在 ETA 基础上顺延 N 天作为计划清关日
- ✅ 确保提柜日是未来日期

**代码修改**：

```typescript
// 新增：ETA 顺延逻辑
const etaBufferDays = await this.getEtaBufferDays();
if (etaBufferDays > 0) {
  plannedCustomsDate.setDate(plannedCustomsDate.getDate() + etaBufferDays);
}
```

### 问题 ②：需要 ETA 顺延功能

**业务需求**：

> "我希望是指定在 ETA 上再顺延 N 天，作为起始计划提柜日的开始日期"

**解决方案**：

- ✅ 配置化设计：通过 `dict_scheduling_config` 表管理
- ✅ 灵活可调：不同国家/港口可设置不同 buffer 天数
- ✅ 默认值：2 天（可根据实际情况调整）

**配置管理**：

```sql
-- 查看当前配置
SELECT config_key, config_value, description
FROM dict_scheduling_config
WHERE config_key = 'eta_buffer_days';

-- 修改顺延天数（如改为 3 天）
UPDATE dict_scheduling_config
SET config_value = '3', updated_at = NOW()
WHERE config_key = 'eta_buffer_days';
```

## 经验总结

### 设计原则

1. **主动规划 vs 被动调整**
   - 原有逻辑：过期后被动调整
   - 新逻辑：提前 buffer 主动规划
   - 改进：从"救火"到"防火"

2. **配置化 vs 硬编码**
   - 原有逻辑：buffer 天数硬编码为 0
   - 新逻辑：字典表配置，灵活可调
   - 改进：不同场景不同配置

3. **日志记录**
   - 记录 buffer 应用情况
   - 便于问题排查
   - 便于业务审计

### 最佳实践

1. **日期计算留有余地**
   - 清关、运输都需要时间
   - 不要安排太紧凑
   - 预留 1-3 天 buffer

2. **配置化管理**
   - 业务参数通过字典表管理
   - 避免硬编码
   - 支持动态调整

3. **文档化**
   - 详细的配置说明
   - 清晰的计算逻辑
   - 完整的验证方法
