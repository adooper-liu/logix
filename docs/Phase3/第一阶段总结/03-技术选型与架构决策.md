### 1.3 为什么选择 Express 而不是 NestJS？

**现状**: 项目采用 **Express + TypeORM** 为主，部分模块逐步引入 NestJS 概念

| 维度 | Express | NestJS |
|------|---------|--------|
| 学习曲线 | 低 | 中高 |
| 灵活性 | 高 | 中（约束多） |
| 团队熟悉度 | 熟悉 | 一般 |
| 快速开发 | ✅ 极快 | ✅ 快 |
| 大型项目 | ⚠️ 需自行组织 | ✅ 结构化 |
| 依赖注入 | 需手动实现 | ✅ 原生支持 |

**决策**: 
- 主服务保持 Express 快速迭代
- 新模块可尝试 NestJS 风格（如 services 目录）
- 不强制全面迁移，避免推翻重来

**踩坑记录 - SnakeNamingStrategy 问题**:


### 2.3 Composition API vs Options API

**决策**: 全面采用 **Composition API**

**实际代码示例**:

```typescript
// ✅ 推荐：Composition API
import { ref, computed, onMounted } from 'vue'

export function useContainerList() {
  const containers = ref<Container[]>([])
  const loading = ref(false)
  
  const filteredContainers = computed(() => {
    return containers.value.filter(c => c.logisticsStatus === 'at_port')
  })
  
  async function fetchContainers() {
    loading.value = true
    try {
      const res = await containerService.getList()
      containers.value = res.data
    } finally {
      loading.value = false
    }
  }
  
  onMounted(fetchContainers)
  
  return { containers, loading, filteredContainers, fetchContainers }
}

// ❌ 不推荐：Options API
export default {
  data() {
    return { containers: [] }
  },
  computed: {
    filteredContainers() {
      return this.containers.filter(c => c.logisticsStatus === 'at_port')
    }
  },
  mounted() { /* ... */ }
}
```

**优势**:
1. 逻辑复用 - 提取 composable 函数
2. 代码组织 - 相关逻辑放一起
3. 类型推断 - TypeScript 友好
4. Tree-shaking - 体积更小

### 2.5 状态管理 - Pinia 实践

```typescript
// frontend/src/store/container.ts
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'

export const useContainerStore = defineStore('container', () => {
  const containers = ref<Container[]>([])
  const scopedCountryCode = ref<string | null>(null)
  
  const filteredByCountry = computed(() => {
    if (!scopedCountryCode.value) return containers.value
    return containers.value.filter(c => c.destinationCountry === scopedCountryCode.value)
  })
  
  function setCountry(code: string) {
    scopedCountryCode.value = code
    localStorage.setItem('scopedCountryCode', code)
  }
  
  return { containers, scopedCountryCode, filteredByCountry, setCountry }
})
```

```typescript
// backend/src/middleware/countryFilter.middleware.ts
export function scopedCountryMiddleware(req: Request, res: Response, next: Next) {
  const countryCode = req.headers['x-country-code'] as string
  
  if (countryCode) {
    req.scopedCountryCode = countryCode
    // 存储到上下文，供后续服务使用
  }
  
  next()
}
```

**实际代码 - 过滤构建器**:

```typescript
// backend/src/services/statistics/common/DateFilterBuilder.ts
export class DateFilterBuilder {
  addCountryFilters(qb: SelectQueryBuilder<any>, countryCode: string | null) {
    if (!countryCode) return qb
    
    return qb
      .andWhere(' EXISTS (
        SELECT 1 FROM biz_containers c
        INNER JOIN biz_replenishment_orders o ON c.order_number = o.order_number
        INNER JOIN biz_customers cust ON o.customer_id = cust.id
        WHERE c.container_number = container.container_number
        AND cust.country = :countryCode
      )', { countryCode })
  }
}
```

## 4. 技术债务与踩坑记录

### 4.1 TypeORM SnakeNamingStrategy 问题 🔥

**问题描述**:
TypeORM 默认使用 camelCase 转换，但数据库使用 snake_case，导致字段映射失败

**错误现象**:
```
QueryFailedError: column container.containerNumber does not exist
```

**根本原因**:
```typescript
// 错误配置
new DataSource({
  namingStrategy: new SnakeNamingStrategy(),  // ❌ 不完全可靠
})
```

**解决方案**:
```typescript
// 方案1：每个 @Column 显式指定 name（推荐）
@Column({ name: 'container_number', type: 'varchar', length: 50 })
containerNumber: string

// 方案2：实体不使用任何装饰器参数，完全依赖全局策略（不推荐）
```

**问题描述**:
FullCalendar 在 Vue 3 中初始化失败

### 4.3 日期比较时区问题 🔥

**问题描述**:
JavaScript `new Date('2026-01-01')` 转换为带时区的 timestamp，与数据库 date 类型比较产生 ±1 天偏移

**错误现象**:
```sql
-- 查询 2026-01-01 的数据，结果为空（实际有数据）
SELECT * FROM biz_containers 
WHERE DATE(ata_dest_port) = '2026-01-01'
```

**解决方案**:
```typescript
// 方案1：SQL层用 ::date 转换
const sql = `
  SELECT * FROM biz_containers 
  WHERE COALESCE(ata_dest_port, eta_dest_port)::date = $1::date
`

// 方案2：前端传纯字符串，后端不转换
// 前端
const dateStr = '2026-01-01'  // 不使用 new Date()

// 后端接收
const today = params[0]  // 直接用字符串
```

### 4.4 筛选条件多层映射问题 🔥

**问题描述**:
前端 days 值 → filterCondition → 后端 service method，映射层过多导致不一致

**错误现象**:
```
前端点击"按计划提柜-逾期"
→ 传入 days='overdue'
→ 中间层映射为 'overduePlanned'
→ 后端路由到 getContainersByOverduePlanned()
→ 返回 0 条数据（实际有数据）
```

**解决方案**:
```typescript
// 前端直接使用后端 filterCondition（零映射）
// useContainerCountdown.ts
const days = 'overduePlanned'  // 直接使用后端值

// 删除中间映射层
// ❌ 之前
const filterCondition = FILTER_CONDITION_MAP[type][days]

// ✅ 现在  
const filterCondition = days  // 直接透传
```
