# 货币符号显示修复总结

## 问题描述

在排产预览中，英国仓库的货柜费用显示美元符号 `$` 而不是英镑符号 `£`。

### 问题现象

- **英国仓库**（GB/UK）：显示 `$2,410.00` ❌ 应该是 `£2,410.00`
- **美国仓库**（US）：显示 `$2,410.00` ✅ 正确
- **欧盟仓库**（DE/FR/IT/ES）：显示 `$2,410.00` ❌ 应该是 `€2,410.00`

## 根本原因

**前端 `formatCurrency` 函数需要国家代码参数，但后端返回的 `plannedData` 对象中没有提供仓库国家信息。**

### 代码分析

1. **前端格式化函数**（正确）

   ```typescript
   const formatCurrency = (amount: number, countryCode?: string): string => {
     const currencyMap: Record<string, { symbol: string; code: string }> = {
       US: { symbol: "$", code: "USD" },
       GB: { symbol: "£", code: "GBP" },
       EU: { symbol: "€", code: "EUR" },
       // ...
     };
     const currency = currencyMap[countryCode] || { symbol: "$", code: "USD" };
     return `${currency.symbol}${formattedAmount}`;
   };
   ```

2. **前端调用**（正确）

   ```vue
   {{ formatCurrency(row.estimatedCosts.totalCost, row.plannedData?.warehouseCountry || "US") }}
   ```

3. **后端返回**（问题所在）
   ```typescript
   const plannedData = {
     plannedCustomsDate: "...",
     warehouseId: "CA-S003",
     warehouseName: "Oshawa",
     // ❌ 缺少 warehouseCountry 字段！
   };
   ```

## 修复方案

### 后端修改

**文件**: `backend/src/services/intelligentScheduling.service.ts`

**修改内容**:

1. **添加字段到接口定义** (L61-73)

   ```typescript
   plannedData?: {
     // ... 其他字段
     warehouseCountry?: string; // ✅ 新增：仓库国家代码，用于前端货币格式化
     // ...

   ```

2. **填充国家信息** (L438-453)
   ```typescript

     // ... 其他字段

     // ...
   };
   ```

**修复要点**:

- ✅ 从仓库实体的 `country` 字段获取国家代码
- ✅ 回退机制：如果仓库没有国家信息，使用排产时的 `countryCode`
- ✅ 同步更新 TypeScript 接口定义

### 前端修改

无需修改！前端代码已经正确处理了国家参数。

## 经验总结

### 问题教训

1. **国家信息传递链断裂**
   - 后端有仓库国家信息，但没有传递给前端
   - 前端有货币格式化逻辑，但缺少国家参数
   - 导致默认使用 `$` 符号

2. **数据类型意识**
   - TypeORM 的 `decimal` 类型返回字符串
   - 需要显式转换为 `number` 避免字符串拼接

### 最佳实践

1. **完整的上下文信息**
   - 返回数据时应包含所有必要的上下文信息
   - 特别是国际化相关的国家、货币、语言等信息

2. **防御性编程**
   - 提供默认值回退机制
   - 使用 `||` 或 `??` 处理可能的空值

3. **类型安全**
   - TypeScript 接口定义应及时更新
   - 编译时检查有助于发现此类问题
