## 🔧 技术细节

### 后端 API 实现

**成本评估接口**:
```typescript
POST /api/v1/scheduling/evaluate-cost

Request:
{
  containerNumber: string,
  option: UnloadOption
}

Response:
{
  success: true,
  data: {
    containerNumber: string,
    option: UnloadOption,
    costBreakdown: CostBreakdown
  }
}
```

**方案对比接口**:
```typescript
POST /api/v1/scheduling/compare-options

核心逻辑:
1. 并行评估所有方案（Promise.all）
2. 按总成本升序排序
3. 计算排名
4. 计算节省金额
5. 返回完整对比结果
```

### 前端组件通信

**父子组件通信**:
```typescript
// 父组件 → 子组件（Props）
<UnloadOptionSelector 
  :options="availableOptions"
  v-model="selectedOption"
/>

// 子组件 → 父组件（Events）
emit('change', selectedOption)
emit('optionSelected', option)
```

**服务层调用**:
```typescript
const result = await costOptimizationService.evaluateCost(
  props.containerNumber,
  selectedOption.value
)
costBreakdown.value = result.data.costBreakdown
```

## ⚠️ 注意事项

### 1. 数据库依赖

**配置项要求**:
```sql
-- 必须确保以下配置项存在于 dict_scheduling_config 表
'transport_base_rate_per_mile'     -- 运输基础费率（USD/英里）
'transport_direct_multiplier'       -- Direct 模式倍数
'transport_dropoff_multiplier'      -- Drop off 模式倍数
'transport_expedited_multiplier'    -- Expedited 模式倍数
'external_storage_daily_rate'       -- 外部堆存日费率
'expedited_handling_fee'            -- 加急处理费
```

**实体元数据**:
- TypeORM 实体必须正确配置
- 表名、字段名必须与数据库一致
- 关联关系必须正确设置

### 2. 测试环境

**预期行为**:
- 如果数据库中没有实际数据，生成的方案数量可能为 0
- 这是正常现象，不影响核心功能验证
- 实际部署时需确保数据完整性

### 3. 性能考虑

**批量评估**:
- 当前实现支持并行评估（Promise.all）
- 建议单次对比不超过 10 个方案
- 大量方案对比时考虑分页或虚拟滚动

**图表渲染**:
- ECharts 实例在组件销毁时需要清理（已实现）
- 窗口 resize 时自动调整（已实现）
- 数据更新时重新初始化（已实现）
