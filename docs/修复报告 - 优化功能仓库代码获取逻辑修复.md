# 修复报告 - 优化功能仓库代码获取逻辑修复

## 📋 问题描述

用户反馈：点击"优化"按钮时提示"缺少仓库信息，无法进行优化"，后端返回"仓库不存在"错误。

### 错误日志
```
[handleOptimizeContainer] 检查参数：{
  containerNumber: 'ECMU5397691',
  warehouseCode: 'Bedford',  // ❌ 这是仓库名称，不是编码
  truckingCompanyId: 'CEVA_FREIGHT__UK__LTD',
  basePickupDate: '2026-03-28'
}

POST /api/v1/scheduling/optimize-container/ECMU5397691
Response: {
  success: false,
  message: '仓库不存在'  // WHERE warehouseCode = 'Bedford' 查找失败
}
```

## 🔍 根本原因分析

### 数据流追踪

#### 1. 排产数据存储（后端）
**文件**: `backend/src/services/intelligentScheduling.service.ts` (第 674 行)
```typescript
const plannedData = {
  // ...
  warehouseId: warehouse.warehouseCode,      // ✅ 存储的是仓库编码：'UK-S005'
  warehouseName: warehouse.warehouseName,    // 存储的是仓库名称：'Bedford'
  warehouseCountry: warehouse.country,
  // ...
}
```

**关键点**：`plannedData.warehouseId` 字段存储的实际上是 **warehouseCode**（仓库编码）

#### 2. 前端显示数据（预览表格）
**文件**: `frontend/src/views/scheduling/components/SchedulingPreviewModal.vue` (第 96 行)
```vue
<el-table-column prop="warehouseName" label="仓库" min-width="150" />
```
显示的是：`warehouseName = 'Bedford'`（仓库名称）

#### 3. 优化功能获取逻辑（修复前）
**文件**: `frontend/src/views/scheduling/SchedulingVisual.vue` (第 1629-1635 行)
```typescript
const warehouseCode =
  row.plannedData?.warehouseCode ||    // ❌ 这个字段不存在
  row.warehouseCode ||                 // ❌ 这个字段也不存在
  row.destinationWarehouse ||          // ❌ 不存在
  row.warehouseId ||                   // ⚠️ 根对象的 warehouseId，可能不存在
  row.warehouseName?.split(' ')[0] ||  // ❌ 得到 'Bedford'（名称，不是编码）
  row.plannedData?.warehouseName?.split(' '[0])  // ❌ 得到 'Bedford'（名称，不是编码）
```

**问题**：没有优先使用 `plannedData.warehouseId`，导致从 warehouseName 提取到的是仓库名称而非编码。

#### 4. 后端验证失败
**文件**: `backend/src/controllers/scheduling.controller.ts` (第 1928-1930 行)
```typescript
const warehouse = await warehouseRepo.findOne({
  where: { warehouseCode }  // ❌ 查找 'Bedford'，但数据库中是 'UK-S005'
});

if (!warehouse) {
  res.status(404).json({
    success: false,
    message: '仓库不存在'  // ← 返回的错误
  });
  return;
}
```

### 问题核心

用户指出的关键问题：
> "如果这个仓库不存在，那么预览排柜的数据从哪日在哪里获取的，说明优化又是自己创造了一套轮子，没有复用或者采用首轮排柜获取数据的逻辑"

**根本原因**：
- ✅ 排产功能正确使用了 `warehouse.warehouseCode`（如 `UK-S005`）
- ❌ 优化功能没有复用排产的数据获取逻辑，而是从显示数据中提取
- ❌ 显示数据使用的是 `warehouseName`（如 `Bedford`），不是编码
- ❌ 优化功能尝试从 warehouseName 提取，得到的是名称而非编码

## ✅ 修复方案

### 修复 1: SchedulingVisual.vue - 单柜优化

**文件**: `frontend/src/views/scheduling/SchedulingVisual.vue` (第 1625-1637 行)

**修复前**:
```typescript
// ✅ 优化：从多个可能的字段获取仓库代码
const warehouseCode =
  row.plannedData?.warehouseCode ||    // 优先从 plannedData 获取
  row.warehouseCode ||                 // 从根对象获取
  row.destinationWarehouse ||          // 备用字段 1
  row.warehouseId ||                   // 备用字段 2
  row.warehouseName?.split(' ')[0] ||  // ✅ 新增：从仓库名称提取
  row.plannedData?.warehouseName?.split(' '[0])  // ✅ 新增
```

**修复后**:
```typescript
// ✅ 关键修复：plannedData.warehouseId 存储的就是 warehouseCode（见后端代码第 674 行）
// 优先级：plannedData.warehouseId > warehouseCode > 其他备用字段
const warehouseCode =
  row.plannedData?.warehouseId ||      // ✅ 首选：排产数据中的 warehouseId（实际是 warehouseCode）
  row.plannedData?.warehouseCode ||    // 备选：plannedData 中的 warehouseCode 字段
  row.warehouseCode ||                 // 从根对象获取
  row.warehouseId ||                   // 备用字段（根对象的 warehouseId）
  row.destinationWarehouse ||          // 备用字段 1
  row.warehouseName?.split(' ')[0] ||  // 最后尝试：从仓库名称提取
  row.plannedData?.warehouseName?.split(' '[0])  // 从 plannedData 的仓库名称提取
```

**关键改进**：
1. ✅ 优先使用 `plannedData.warehouseId`，因为它就是 warehouseCode
2. ✅ 添加详细注释说明数据来源
3. ✅ 调整字段优先级，确保正确的数据源优先

## 📊 数据映射关系

| 字段名 | 存储位置 | 实际值示例 | 用途 |
|--------|----------|-----------|------|
| `warehouseCode` | Warehouse 表主键 | `UK-S005` | 仓库唯一标识（编码） |
| `warehouseName` | Warehouse 表字段 | `Bedford` | 仓库显示名称 |
| `plannedData.warehouseId` | TruckingTransport 表 | `UK-S005` | **实际存储的是 warehouseCode** |
| `plannedData.warehouseName` | TruckingTransport 表 | `Bedford` | 仓库显示名称 |

**重要说明**：
- `plannedData.warehouseId` 字段名具有误导性，它实际存储的是 `warehouseCode`
- 这是因为数据库设计时，`TruckingTransport` 表的 `warehouseId` 字段外键关联到 `Warehouse.warehouseCode`
