# 排柜

LogiX 智能排柜功能设计与实现。

## 一、核心概念

### 1.1 什么是排柜

智能排柜（Intelligent Scheduling）根据货柜的 ETA、清关日期、仓库可用性等因素，自动计算最优的：

- 计划清关日
- 计划提柜日
- 计划送仓日
- 计划卸柜日
- 计划还箱日
- 仓库选择
- 拖车公司选择
- 卸柜方案（Live load / Drop off）

### 1.2 排产状态

| 状态 | 说明 |
|------|------|
| `initial` | 待排产 |
| `issued` | 已排产 |
| `dispatched` | 已派工 |
| `adjusted` | 手工调整 |

### 1.3 排产模式

| 模式 | 说明 |
|------|------|
| **自动排产** | 系统自动选择最优仓库和日期 |
| **手工指定** | 用户指定仓库，手动排产 |
| **预览模式** | dryRun=true，只计算不保存 |

## 二、核心算法

### 2.1 锚点逻辑

以 **计划卸柜日（plannedUnloadDate）** 为核心锚点，向前向后推导其他日期：

```
锚点确定 → 向前推导 → 向后推导
```

**锚点确定**：
1. 从 ETA 后的首个可用仓库日期开始
2. 检查仓库日产能（dailyUnloadCapacity）
3. 向前查找首个有产能的日期

### 2.2 五个日期推导

```
plannedCustomsDate ──→ plannedPickupDate ──→ plannedDeliveryDate
      │                       │                      │
      │ 提柜日-1天            │ 送仓日=卸柜日        │
      ▼                       ▼                      ▼
   (向前推导)              (向前推导)              (计算得出)
                                                  
plannedReturnDate ◄──── plannedUnloadDate
      │                       │
      │ 卸柜日+在仓天数       │ (锚点)
      ▼                       ▼
   (向后推导)              (基于仓库可用性)
```

| 日期 | 推导方向 | 说明 |
|------|----------|------|
| plannedCustomsDate | 向前 | 提柜日 - 1天 |
| plannedPickupDate | 向前 | 清关日 + 1天，若已过期则至少为今天 |
| plannedDeliveryDate | 计算 | Live load：提=送=卸；Drop off：送=卸 |
| plannedUnloadDate | 锚点 | 基于仓库可用性计算 |
| plannedReturnDate | 向后 | 卸柜日 + 在仓天数，优先用 lastReturnDate |

### 2.3 卸柜方案

| 方案 | 说明 | 适用场景 |
|------|------|----------|
| **Live load** | 提=送=卸（同日） | 急货 |
| **Drop off** | 先放堆场，送=卸 | 普通货，省堆存费 |
| **Expedited** | 加急模式 | 紧急货物 |

### 2.4 约束校验

- **仓库产能**：检查 `ext_warehouse_daily_occupancy.planned_count < capacity`
- **拖车槽位**：检查 `ext_trucking_slot_occupancy`
- **还箱槽位**：检查 `ext_trucking_return_slot_occupancy`
- **滞港费**：计算预估费用，选择最优方案

## 三、排产流程

### 3.1 批量排产流程

```
batchSchedule(request)
       │
       ▼
查询待排产货柜（schedule_status=initial）
       │
       ▼
按清关可放行日排序（先到先得）
       │
       ▼
分批处理（limit/skip）
       │
       ▼
循环每个货柜：
  ① 查找目的港对应的仓库列表
  ② 找首个可用仓库和卸柜日
  ③ 计算五个计划日期
  ④ 计算预估费用
  ⑤ 保存到数据库
       │
       ▼
触发状态机更新
```

### 3.2 手工指定模式

```
designatedWarehouseMode=true
       │
       ▼
指定仓库 + 柜号列表
       │
       ▼
仅对指定柜号排产
       │
       ▼
跳过成本优化建议
```

|------|------|

```
建议优化条件：savings > 0（节省金额 > 0）
```

## 七、实施坑点

### 7.1 常见问题 Top 5

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 排产失败 - "无映射关系中的仓库" | 映射链不完整 | 补充 dict_trucking_port_mapping / dict_warehouse_trucking_mapping |
| 产能不足 | 当日产能已满 | 增加 ETA 顺延天数或临时增加产能 |
| 日期计算异常 | 周末配置未生效 | 检查 dict_scheduling_config.skip_weekends |
| 状态同步失败 | updateStatus 执行失败 | 手动触发 containerStatusService.updateStatus() |
| 成本计算为0 | 费率配置缺失 | 检查 dict_scheduling_config 费率配置 |

### 7.2 诊断 SQL

```sql
-- 检查车队-港口映射
SELECT * FROM dict_trucking_port_mapping
WHERE port_code = 'GBFXT' AND country = 'GB';

-- 检查仓库-车队映射
SELECT * FROM dict_warehouse_trucking_mapping
WHERE trucking_company_id = 'TC001';

-- 查看仓库占用情况
SELECT * FROM ext_warehouse_daily_occupancy
WHERE warehouse_code = 'WH001'
  AND date BETWEEN '2026-03-24' AND '2026-03-30'
ORDER BY date;

-- 检查排产配置
SELECT * FROM dict_scheduling_config
WHERE config_key IN ('skip_weekends', 'weekend_days', '%rate%');
```

### 7.3 问题演进

#### 问题1：三套成本计算逻辑不一致

**问题描述**：系统中存在三个成本计算场景，长期各自实现、口径不一。

##### 1.1 三个场景的定义与用途

| 场景 | 用户视角 | 入口方法 | 代码位置 | 用途 |
|------|----------|----------|----------|------|
| **A. 货柜详情页** | 财务/物流专员查看实际/预测滞港费 | `DemurrageService.calculateForContainer()` | demurrage.service.ts | 用户查看单个货柜的完整费用明细 |
| **B. 排产预览** | 物流专员预览排产方案费用 | `IntelligentScheduling.calculateEstimatedCosts()` | intelligentScheduling.service.ts | 排产前（dryRun 模式）预览预估总成本 |
| **C. 成本优化** | 系统自动决策最优方案 | `SchedulingCostOptimizer.evaluateTotalCost()` | schedulingCostOptimizer.service.ts | 自动排柜时选择最优卸柜策略 |

##### 1.2 三个场景的核心差异

| 维度 | A. 货柜详情 | B. 排产预览 | C. 成本优化 |
|------|-------------|-------------|-------------|
| **触发时机** | 用户主动查看详情 | 排产 dryRun 模式 | 自动排柜算法决策 |
| **输入参数** | containerNumber | containerNumber + 三个计划日期 + 卸柜方式 | containerNumber + 卸柜方案 |
| **费用范围** | D&D 费用（不含运输） | D&D + 堆存 + 运输 + 加急 | D&D + 堆存 + 运输 |
| **目标用户** | 人工决策参考 | 人工决策参考 | 算法自动决策 |
| **返回值粒度** | 按费用类型分类明细 | 按费用类型 + 额外费用 | 成本对比明细 |

##### 1.3 费用项覆盖对比（当前状态）

| 费用项 | A. 货柜详情 | B. 排产预览 | C. 成本优化 | 统一入口 |
|--------|-------------|-------------|-------------|----------|
| **滞港费** | ✅ calculateForContainer() | ✅ calculateTotalCost() | ✅ calculateTotalCost() | DemurrageService |
| **滞箱费** | ✅ calculateForContainer() | ✅ calculateTotalCost() | ✅ calculateTotalCost() | DemurrageService |
| **港口存储费** | ✅ 标准匹配 | ✅ calculateTotalCost() | ✅ calculateTotalCost() | DemurrageService |
| **D&D合并** | ✅ calculateForContainer() | ✅ calculateTotalCost() | ✅ calculateTotalCost() | DemurrageService |
| **外部堆存费** | ❌ 不涉及 | ✅ 独立计算（Drop off） | ✅ 独立计算（Drop off） | TruckingPortMapping |
| **运输费** | ❌ 不涉及 | ✅ calculateTotalCost() | ✅ calculateTotalCost() | WarehouseTruckingMapping |
| **加急费** | ❌ 不涉及 | ✅ 独立计算（Expedited） | ❌ 不涉及 | schedulingConfig |

##### 1.4 统一入口：calculateForContainer()

费用计算的**底层核心**是 `DemurrageService.calculateForContainer()`，三个场景的 D&D 费用计算最终都复用此方法：

```typescript
// demurrage.service.ts:calculateForContainer() - 核心底层方法
async calculateForContainer(
  containerNumber: string,
  options?: { freeDateWriteMode?: 'none' | 'actual' | 'forecast' }
): Promise<DemurrageResult>

// demurrage.service.ts:calculateTotalCost() - 统一封装层
async calculateTotalCost(
  containerNumber: string,
  options?: {
    mode?: 'actual' | 'forecast';
    plannedDates?: { plannedPickupDate, plannedUnloadDate, plannedReturnDate };
    includeTransport?: boolean;
    warehouse?: Warehouse;
    truckingCompany?: TruckuckingCompany;
    unloadMode?: string;
  }
): Promise<{
  demurrageCost: number;
  detentionCost: number;
  storageCost: number;
  ddCombinedCost: number;
  transportationCost: number;
  totalCost: number;
  currency: string;
  calculationMode: 'actual' | 'forecast';
}>
```

##### 1.5 调用链路

```
                    ┌──────────────────────────────────────┐
                    │   calculateForContainer()            │
                    │   (核心底层 - D&D费用计算)            │
                    └──────────────────┬───────────────────┘
                                         ↑
     ┌──────────────────────────────────┼────────────────────┐
     │                                  │                    │
     │ A.货柜详情直接调用                │ B.排产预览         │ C.成本优化
     │ calculateForContainer()           │ calculateTotalCost()→调用
     │                                  │                    │
     ▼                                  ▼                    ▼
┌─────────────┐              ┌──────────────────┐   ┌──────────────────┐
│ demurrage   │              │ calculateTotal   │   │ calculateTotal   │
│ .service    │              │ Cost()            │   │ Cost()           │
│ :calcForCon │              │ ├─ D&D → forCon   │   │ ├─ D&D → forCon   │
│ tainer()    │              │ ├─ transport      │   │ ├─ transport      │
│             │              │ └─ 返回完整结构    │   │ └─ 返回完整结构   │
└─────────────┘              └──────────────────┘   └──────────────────┘

独立费用（各场景自行处理）：
  • yardStorageCost（堆场堆存）→ B、C 独立计算
  • handlingCost（加急费）→ B 独立计算
```

**代码层面**：
```typescript
// A. 货柜详情页（直接调用核心方法）
demurrage.service.ts:calculateForContainer()
  └── calculateDemurrage()        // 滞港费
  └── calculateDetention()        // 滞箱费
  └── calculateStorage()          // 存储费

// B. 排产预览（B → calculateTotalCost → calculateForContainer）
intelligentScheduling.service.ts:calculateEstimatedCosts()
  └── demurrageService.calculateTotalCost()
       └── calculateForContainer()  // ✅ 复用底层
  └── 独立计算：yardStorageCost（堆场堆存）
  └── 独立计算：handlingCost（加急费）

// C. 成本优化（C → calculateTotalCost → calculateForContainer）
schedulingCostOptimizer.service.ts:evaluateTotalCost()
  └── demurrageService.calculateTotalCost()
       └── calculateForContainer()  // ✅ 复用底层
  └── 独立计算：yardStorageCost（堆场堆存）
```

##### 1.6 口径一致性保障

| 一致性维度 | 实现方式 |
|------------|----------|
| **费用计算基准** | 统一复用 `calculateForContainer()` 底层方法 |
| **状态机判定** | 统一使用 `isArrivedAtDestination()` 判断是否实际到港 |
| **起算日口径** | 统一使用 `calculationBasis`（按到港/按卸船） |
| **多行标准累加** | 统一遍历所有匹配标准并累加 |
| **预测模式** | 统一使用 `plannedDates` 参数传入计划日期 |

| 版本 | 状态 | 说明 |
|------|------|------|
| v1.0 | ❌ 各自实现 | 三套逻辑互不调用，结果不一致 |
