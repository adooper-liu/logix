## 🎯 **核心原则**

### 1. 简洁即美 ✅

**要求**:

- 去除所有 emoji 装饰（🎯📊✅❌等）

**示例**:

# TimescaleDB 测试操作指南

2. 后端服务已启动

   SELECT COUNT(\*) FROM biz_containers;

### 3. 遵循 SKILL ✅

**要求**:

- 遵循数据库优先开发原则
- 代码修改必须基于权威源验证
- 保持代码和文档的简洁性
- 围绕真实业务场景展开
- 禁止反向工程（代码对齐数据库）

**核心 SKILL**:

1. **数据库优先开发原则** 🔥
   - 数据库表结构是唯一基准
   - 开发顺序：SQL → 实体 → API → 前端 → 联调
   - 禁止反向：代码对齐数据库，不反向改库补数据

2. **基于权威源验证** 🔥
   - 所有修改必须基于真实代码
   - 所有命令必须可执行
   - 所有输出必须真实
   - 禁止凭空想象

3. **代码简洁性** 🔥
   - 优先修正源数据而非代码兼容
   - Excel 列名映射使用别名配置
   - 保持代码整洁，定期重构
   - 去除冗余逻辑

4. **业务导向** 🔥
   - 围绕真实业务场景（集装箱查询、滞港费计算等）
   - 解决实际问题
   - 提供实用工具和方法
   - 避免脱离业务的理论

---

## 📊 **文档精简指标**

### 量化目标

| 指标             | 目标值      | 计算方法                         |
| ---------------- | ----------- | -------------------------------- |
| **AI 虚幻内容**  | 0%          | 虚构内容行数 / 总行数            |
| **emoji 密度**   | < 1%        | emoji 数量 / 总字符数            |
| **真实可用内容** | > 90%       | 可执行内容行数 / 总行数          |
| **文档总数**     | 精简 20-30% | (原文档数 - 现文档数) / 原文档数 |
| **总行数**       | 精简 30-40% | (原总行数 - 现总行数) / 原总行数 |

### 质量评估表

| 维度         | 权重 | 评分标准                     | 得分 |
| ------------ | ---- | ---------------------------- | ---- |
| **真实性**   | 30%  | 所有命令可执行，所有输出真实 | /30  |
| **简洁性**   | 25%  | 无装饰，直截了当             | /25  |
| **实用性**   | 25%  | 解决实际问题，有明确用途     | /25  |
| **可维护性** | 20%  | 易于更新，结构清晰           | /20  |
| **总分**     | 100% |                              | /100 |

**合格标准**: ≥ 80 分  
**优秀标准**: ≥ 90 分
